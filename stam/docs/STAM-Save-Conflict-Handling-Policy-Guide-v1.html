01 Hero / Purpose 목적
STAM POLICY Candidate Guide 기준 / 검토 필요 Not Realtime Co-Edit
STAM 저장 요청 · 동시 편집 · 다중 탭 저장 · 중복 요청 · 최종값 · 변경이력 연계 정책 경계

본 가이드는 STAM에서 발생하는 저장 요청, 동시 편집, 동일 화면 다중 탭 저장, 버튼 더블클릭, 저장 실패, 저장 후 최종값 기준, 변경이력 연결 기준을 정의한다. 실시간 협업(Realtime Co-Editing) 또는 자동 병합(Auto-Merge) 기능을 약속하는 문서가 아니며, 구현 상세는 후속 기능 요청에서 확정한다.

02 Policy Summary Summary
Policy Key 정의 주요 책임 상태
Save Request 화면에서 발생하는 1건의 저장 요청 단위 — 식별·접수·처리·결과 기록 대상 요청 단위 추적성 확보 Candidate
Multi-tab Editing 동일 사용자가 동일 화면을 2개 이상 탭/창에서 동시에 여는 상황 무조건 차단하지 않고 마지막 저장 기준의 영향 안내 Candidate
Last Save Wins 일반 업무 저장에서 최종값을 늦은 저장 기준으로 정하는 정책 일반 업무 저장 한정 · Strict 영역 미적용 Candidate
Duplicate Request Block 버튼 더블클릭 등 동일 행위의 짧은 시간 중복 요청 차단 Idempotency Key / 클라이언트 비활성화 기준 Hard Limit
Strict Handling Areas 권한·보안·약관·삭제·복구·결제·감사 대상의 별도 엄격 처리 영역 Last Save Wins 미적용 · 충돌 시 별도 절차 Candidate
Change History Link 모든 저장 결과는 Change History / Recovery Policy Guide의 변경이력 후보로 연결 빠른 저장·늦은 저장 모두 이력 후보 Candidate
03 Scope — 포함 / 미포함 범위
포함 (In Scope)
  • 저장 요청(Save Request) 단위 정책 기준
  • 동시 편집(Concurrent Editing) 정책 기준
  • 동일 화면 다중 탭(Multi-tab) 저장 정책 기준
  • 버튼 더블클릭 등 중복 요청 차단 정책 기준
  • 일반 업무 저장에서의 Last Save Wins 정책
  • 저장 실패 안내 / 재시도(Retry) 정책 기준
  • 저장 결과의 Change History 연결 기준
  • Strict Handling Areas(권한·약관·삭제·복구·결제·감사) 분리 원칙
미포함 (Out of Scope)
  • 실시간 협업(Realtime Co-Editing) 기능 약속
  • 자동 병합(Auto-Merge) / OT / CRDT 알고리즘 약속
  • 자동 복구(Auto-Recovery) 약속
  • Undo / Revision 기능 약속
  • DB 테이블 / 컬럼 / ERD / 인덱스 확정
  • 특정 Firestore collection / lock 메커니즘 확정
  • 제품 화면 / WBS / Project Overview / nav-data 수정
  • API 코드 / Firebase Functions / Firestore rules 변경
04 Terminology 용어
용어정의주의
Save Request 1건의 저장 행위에 대응하는 요청 단위 엔티티 + 화면 + 주체 + 시점 + 페이로드
Idempotency Key 동일 행위의 중복 요청을 식별하기 위한 단일 키 후보 짧은 시간창에서 동일 키 = 동일 요청
Multi-tab Editing 동일 사용자가 동일 화면을 2개 이상 탭/창에서 동시에 여는 상황 무조건 차단하지 않음
Concurrent Editing 동일 대상에 대해 2명 이상이 동시에 편집하는 상황 실시간 협업 약속 아님
Last Save Wins 늦은 저장의 값이 최종값이 되는 정책 일반 업무 저장에 한정
Conflict Candidate 저장 시점에 같은 대상의 베이스 버전이 변경된 가능성이 감지된 상태 충돌 후보 표현 · 자동 병합 약속 아님
Strict Handling Areas 권한·보안·약관·삭제·복구·결제·감사 대상 영역 Last Save Wins 미적용
Duplicate Request 짧은 시간 내 동일 키 / 동일 행위의 중복 요청 차단 대상
Retry 저장 실패 후 같은 요청을 다시 시도하는 행위 Idempotency Key 동일 유지 후보
05 What This Guide Does Not Promise Anti-Promise
본 가이드가 약속하지 않는 것
  • 실시간 협업(Realtime Co-Editing) 기능 제공
  • 자동 병합(Auto-Merge / OT / CRDT) 동작
  • 자동 복구(Auto-Recovery) 동작
  • Undo / 단일 클릭 되돌리기 기능
  • 모든 시점에 대한 완전한 충돌 감지 보장
  • 특정 Firestore collection / lock 메커니즘 확정
  • DB 테이블 / 컬럼 / ERD / 인덱스 확정
  • 법무·보존기간·ISMS-P 통제 항목의 최종 확정 문구
Note: 본 가이드는 정책 경계만 정의한다. 실제 기능 범위·시한·우선순위는 별도 Product / Engineering 결정으로 정의한다.
06 Save Request Policy Policy
요청 단위
엔티티 + 화면 + 주체 + 시점 + 페이로드를 1건의 Save Request 단위 후보로 정의한다.
Idempotency Key
클라이언트는 1건의 행위에 대해 단일 Idempotency Key를 생성한다. 짧은 시간 내 동일 키 재요청은 중복으로 본다.
접수 → 처리 → 결과
접수(accepted), 처리 중(processing), 성공(succeeded), 실패(failed)의 후보 상태를 기록한다.
버튼 상태
처리 중 동안 저장 버튼은 비활성화 후보이며, 동일 요청 재발사를 막는 1차 가드로 둔다.
결과 안내
성공 / 실패 / 부분 성공의 결과 메시지를 후보로 분리한다. 실패 시 사유와 재시도 안내를 분리한다.
변경이력 연계
성공 결과는 Change History의 Change Record 후보로 연결한다.
07 Multi-tab Editing Policy Policy
  • 동일 화면 2개 탭 저장은 무조건 차단하지 않는다. 사용자는 동일 화면을 2개 이상 탭에서 열 수 있다.
  • 각 탭의 저장은 독립된 Save Request로 접수된다.
  • 일반 업무 저장 영역에서는 늦은 저장 기준의 최종값이 적용될 수 있음을 안내한다.
  • 두 저장 모두 변경이력 후보로 기록된다 (빠른 저장·늦은 저장 모두).
  • 탭 간 동일 키 충돌이 감지되면 Conflict Candidate로 표시하고 사용자에게 영향 안내를 후보로 한다.
  • Strict Handling Areas는 본 정책을 적용하지 않으며 별도 절차를 둔다.
08 Conflict Candidate Policy Policy
감지 기준 후보
베이스 버전 / 마지막 수정 시각 / 마지막 수정자 변화 등을 후보 감지 시그널로 평가한다.
표현 원칙
충돌은 후보(Candidate) 표현으로 시작한다. 자동 병합을 약속하지 않는다.
사용자 안내
충돌 후보 발생 시 다른 사용자/탭의 변경이 있었음을 안내하고, 진행/취소/검토 선택지를 후보로 제공한다.
Strict 영역 예외
권한·보안·약관·삭제·복구·결제·감사 대상은 충돌 후보 발생 시 일반 정책을 적용하지 않고 별도 절차를 따른다.
09 Last Save Wins Policy Policy
적용 범위 제한: Last Save Wins는 일반 업무 저장 기준에 한정한다. 권한·보안·약관·삭제·복구·결제·감사 영역(Strict Handling Areas)에는 적용하지 않는다.
  • 일반 업무 저장에서 늦은 저장의 값이 최종값이 될 수 있다.
  • 빠른 저장과 늦은 저장은 모두 변경이력 후보로 기록된다.
  • 늦은 저장은 직전 저장을 사일런트로 덮어쓰지 않는다 — 사용자 안내 메시지를 후보로 한다.
  • 충돌 후보가 감지된 경우, 사용자가 진행 여부를 선택하도록 한다.
  • Strict 영역에서는 본 정책을 적용하지 않는다 (S11 참조).
10 Duplicate Request Policy Policy
  • 버튼 더블클릭 등 짧은 시간 내 동일 행위는 Duplicate Request로 보고 차단 대상이다.
  • 클라이언트는 처리 중 동안 저장 버튼을 비활성화한다 (1차 가드).
  • 서버는 Idempotency Key 기준으로 동일 키 중복 수신을 무시 처리한다 (2차 가드).
  • 차단된 중복 시도는 변경이력에 별도 후보 행으로 남기지 않거나 단일 이력으로 통합 후보로 다룬다.
  • 네트워크 재시도와 사용자 더블클릭은 구분 가능한 한 분리 후보로 본다.

12 Save Failure / Retry Policy Policy
실패 유형 후보
네트워크 / 권한 / 검증 / 충돌 후보 / 서버 오류로 분리 후보.
사용자 안내
사유와 다음 행동(재시도 · 검토 · 취소)을 명확히 안내하는 후보 카피.
재시도 (Retry)
Idempotency Key는 동일하게 유지하는 것을 후보 원칙으로 한다.
실패 이력
실패 시도와 사유는 errorLogs / retryLogs 후보로 기록한다.
자동 재시도
사용자 의도 없는 자동 재시도는 권장하지 않는다. 네트워크 일시 오류에 한정 후보.
Strict 영역
권한·결제·감사 등 Strict 영역의 실패는 별도 검토 절차를 둔다.
13 Change History Integration Link
  • 모든 성공한 저장은 Change History / Recovery Policy Guide v1의 Change Record 후보로 연결된다.
  • 빠른 저장과 늦은 저장은 모두 변경이력 후보로 기록된다.
  • 최종값은 일반 업무 저장 기준으로 늦은 저장 값이 될 수 있다.
  • 충돌 후보가 발생한 저장도 별도 마커와 함께 변경이력 후보로 남긴다.
  • Duplicate Request 차단 결과는 변경이력에 별도 행을 만들지 않는 것을 후보 원칙으로 한다.
  • Strict 영역의 변경이력은 일반 변경이력과 별도 엄격 처리를 따른다.
