01 Overview — 문서 목적 목적

STAM은 PC 기준으로 Topbar / Left Navigation / App Shell이 정의되고 있다. 그러나 운영 단계에서 모바일은 반드시 별도 구조로 동작하므로, 지금 PC 기준을 닫기 전에 모바일에서의 동등 항목을 함께 확정해야 PC ↔ 모바일 사이의 충돌을 막을 수 있다. 본 가이드는 모바일 화면에서의 구조 / 액션 / 검색 / Drawer / 테이블 대응 기준을 단일 문서로 통합한다.

02 Breakpoint Policy — 모바일 / 태블릿 / 데스크탑 기준

STAM은 PC 기준과 충돌이 없도록 3-tier breakpoint를 채택한다. 모바일은 0–767px, 태블릿은 768–1023px, 데스크탑은 1024px 이상이다. 1280px 이하 구간은 Drawer를 fullscreen 으로 전환할 수 있는 후보 구간이다.

Tier Range Layout Left Nav Drawer Topbar Search
Mobile 0 – 767px Single column · Bottom action / nav 후보 Overlay drawer (좌) Fullscreen sheet icon-only → overlay 확장
Tablet 768 – 1023px Single column 우선 · 일부 2-pane 허용 Overlay drawer 또는 collapsed (64px) Fullscreen sheet 또는 right drawer icon + 짧은 input 가능
Desktop ≥ 1024px Topbar 48 / Left 256 / Content flex Persistent 256px Right drawer (PC 기준) CENTER 항상 노출 (input)
Drawer Limit < 1280px PC Right Drawer 폭 → fullscreen 으로 fallback 후보 Drawer 본문이 좁아져 PC 가독성 저하 우려 fullscreen sheet 검토 그대로
PC 기준과의 충돌 방지
  • PC가 사용하는 ≥ 1024px 구간은 그대로 둔다 — 기존 가이드 보존
  • 모바일은 PC 코드의 축소가 아니라 별도 분기로 처리
  • Left Nav 256px 폭은 모바일에서 절대 그대로 유지되지 않는다
  • Right Drawer 폭은 PC에서 고정, 모바일에서는 fullscreen 으로 전환
  • Tablet은 모바일 규칙을 따르는 것을 기본으로 하고, 일부만 PC 룰 차용
검토가 필요한 경계 케이스
  • 1280px ↓ 화면에서 PC Right Drawer 가 콘텐츠를 가리는 케이스
  • 768–1023px 태블릿에서 Bottom Nav 노출 여부
  • 가로모드(landscape)에서 mobile 정의의 적용 기준
  • UI baseline 변경 시 breakpoint 영향 회귀 점검
03 Mobile App Shell — 기본 구조 구조

모바일 App Shell은 Topbar / Content / Bottom (action 또는 nav) / Overlay drawer의 4 영역으로 구성된다. 페이지 레벨 가로 스크롤은 절대 발생하지 않으며, 긴 콘텐츠는 Content 영역 내부 스크롤로만 처리된다. safe-area-inset 은 Bottom action / Bottom nav 양쪽에 모두 반영된다.

  • WBS 보드는 등록을 Bottom sticky CTA 로 두는 것을 권장
05 Mobile Navigation — 3안 비교 3안

STAM은 산출물 SaaS 특성상 메뉴 수가 많고 deep navigation 이 필수이므로, 모바일 Navigation 은 A안(햄버거 + Drawer) 또는 C안(Hybrid · Bottom nav 4개 + Drawer 전체 메뉴) 중 하나가 적합하다. Bottom navigation 단독(B안) 은 메뉴 폭을 감당하지 못해 권장하지 않는다.

A 햄버거 + Left Drawer Navigation 권장
topbar LEFT의 햄버거에서 좌측 80% drawer를 슬라이드 인. 모든 메뉴를 단일 트리로 노출. STAM처럼 메뉴가 많은 산출물 SaaS에 가장 적합하다.
Pros
  • 모든 메뉴를 1depth로 평등하게 노출
  • PC Left Nav 와 정보 구조가 1:1로 대응 — 학습비용 낮음
  • 화면 콘텐츠가 가장 넓어진다
Cons
  • 주요 화면 진입에 2 tap 필요 (햄버거 → 항목)
  • 홈/탐색 빈도가 낮으면 단조롭게 느껴질 수 있음
B Bottom Navigation 단독 권장 아님
4–5 개의 핵심 영역만 Bottom nav에 노출. 햄버거 없음. 메뉴가 많은 STAM에는 부족.
Pros
  • 주요 화면 진입이 1 tap
  • 모바일 사용자에게 친숙한 패턴
Cons
  • STAM 메뉴 전부를 담을 수 없음
  • 설정 / 산출물 / 권한 / 결재 등이 누락되거나 강제 압축됨
  • PC Left Nav 구조와 매핑이 깨진다
C Hybrid — Bottom 4개 + Drawer 전체 대체 권장
자주 쓰는 4 영역(홈 / WBS / 산출물 / 메뉴)을 Bottom 으로, 전체 메뉴는 “메뉴” 항목 또는 햄버거로 Drawer 진입. FAB를 통해 등록 등 page action을 노출 가능.
Pros
  • 주요 4 화면은 1 tap
  • 전체 메뉴는 Drawer로 보장 (STAM 메뉴 폭 수용)
  • 중앙 FAB로 등록 / 작성 액션 강조 가능
Cons
  • 구현/유지 비용 가장 큼 (Bottom + Drawer 동시 운영)
  • Bottom 4 + 메뉴 라벨링이 일관되어야 함
추천 결정
  • 1차 권장: A안 (햄버거 + Drawer) — STAM 메뉴 폭, PC Left Nav 와의 1:1 매핑
  • 대체: C안 (Hybrid) — 주요 4개 노출이 사업적으로 중요해지는 시점에 전환
  • B안은 단독으로 채택하지 않는다 (STAM 메뉴 폭에 부적합)
  • 최종 결정 전까지 본 가이드는 A안 mock 을 기준 시안으로 둔다
결정이 보류된 항목
  • FAB 채택 여부 — C안 적용 시 활성, A안에서는 미사용 권장
  • Bottom Nav 위 sticky bottom action 동시 노출 정책
  • Drawer 내부 검색 input 노출 여부 (현재는 노출 안함)
06 Mobile Search — 검색 접근 패턴 패턴

모바일 topbar 의 CENTER는 검색을 직접 노출하지 않는다. 대신 RIGHT의 검색 아이콘을 누르면 topbar 아래로 inline search bar 가 확장되거나 (단순한 화면 검색), full search overlay 가 위에 올라온다 (전역 검색). WBS 처럼 테이블형 화면은 검색 접근성이 핵심이므로 검색 액션을 RIGHT 상단에 항상 고정한다.

금지
  • PC 컬럼 (WBS 코드 / 유형 / 승인 / 진척 / …) 을 카드에 모두 노출하는 것
  • PC 테이블을 그대로 모바일에 축소해 렌더링하는 것
  • 페이지 자체가 가로 스크롤되는 모든 형태
  • 카드 안 텍스트가 ellipsis 없이 줄바꿈 폭주하는 형태

09 Mobile Button Guideline — Touch Target 버튼

모바일은 touch target 최소 40px, 권장 44px를 기준으로 한다. PC의 sm 28px 버튼은 모바일 주요 액션에 사용 금지이며, icon-only 버튼 또한 최소 40px 컨테이너 안에서 운용된다. footer CTA는 44px 를 권장하고, topbar icon button 은 36 – 40px 사이를 채택한다.

TTouch Target Spec 28 (금지) · 32 / 36 (보조) · 40 (최소) · 44 (권장)
28 PC sm — 모바일 주요 액션 금지
32 topbar 내 보조 icon (선택)
36 topbar icon — 최소 후보
40 touch 최소 / icon-only 컨테이너
44 footer CTA · 권장 기본
위치 Variant Height 참고
Topbar icon ghost icon-only 36 – 40px 컨테이너 icon glyph 18px · padding으로 hit-area 확보
Bottom CTA (primary) primary 44px 권장 (40 허용) 화면 폭 전체에서 좌우 12px gutter
Bottom CTA (secondary) secondary / ghost 44px primary 옆 fixed-width (88 – 110px)
Fullscreen footer primary / secondary / ghost 40 – 44px PC drawer footer 32px → 모바일 40 + 권장
Bottom nav item nav icon + label 54px bar 내부 / 각 item ≥ 64w FAB 사용 시 중앙 item 만 -2px 띄움
Card 내 inline action chip / pill 실제 hit-area ≥ 40px 시각 padding 6px 라도 row 전체가 tap target
금지
  • PC sm 28px 버튼을 모바일 주요 CTA로 사용
  • icon-only 버튼을 32px 미만의 hit-area로 두는 것
  • 두 개의 primary 버튼을 같은 footer 에 동시에 두는 것
  • Bottom action 안에 icon 만 있는 ghost 버튼만 두는 것 (의도 불명)
PC와의 정합성
  • PC 의 button variant (primary / secondary / ghost) 명은 그대로 유지
  • size 토큰은 별도 — 모바일은 md / lg 만 사용
  • PC button CSS는 모바일에서 덮어쓰지 않고 media query로 size 보강
  • 실제 적용은 stam/css/stam.components.css 의 모바일 분기에서
10 Mobile Action Priority — 어떤 액션을 어디에 두는가 우선순위

모바일 topbar 의 RIGHT 에 둘 수 있는 액션은 검색 / 알림 / 더보기로 한정한다. 등록 / 내보내기 / 필터 / 보기 옵션은 Bottom action 또는 Bottom sheet 로 이동한다. sticky bottom action 은 content 위 fixed로 떠 있으나, scroll 시에도 content를 가리지 않도록 content 하단에 같은 높이의 spacer 를 자동으로 두는 것을 원칙으로 한다.

액션 Topbar Bottom action Bottom sheet / 더보기 비고
검색RIGHT 아이콘——전역 / 인라인 둘 다 검색은 topbar에서 시작
알림RIGHT 아이콘 (badge)——count는 알림 패널에서만 표시
사용자 메뉴——RIGHT 더보기 안topbar 직접 아바타 금지 (혼잡)
등록 / 작성—primary CTAFAB (C안)화면당 1개로 제한
필터—secondary / ghostbottom sheet조건 다수 시 bottom sheet 권장
보기 옵션 (카드/표)RIGHT 더보기 안—bottom sheet표 모드 진입은 더보기 → 새 화면
내보내기 (Excel / PDF)——RIGHT 더보기 또는 상세 footertopbar 직접 노출 금지
정렬——bottom sheetdrop-down 대신 시트로
상세 / 수정 / 삭제——상세 footer 또는 Quick Edit삭제는 ghost로만 노출
우선순위 원칙
  • topbar 액션 ≤ 3 — 검색 / 알림 / 더보기
  • page CTA 는 화면당 1개 — Bottom action 또는 FAB
  • 같은 화면에서 Bottom nav 와 Bottom action 동시 노출은 신중하게 검토
  • “하나의 화면에 두 가지 primary action” 금지
자주 발생하는 실수
  • topbar 에 등록 + 알림 + 사용자 + 메뉴 모두 노출 → 36px 미만 압축
  • Bottom CTA 가 content 마지막 카드를 가림
  • “더보기” 안에 너무 많은 항목을 평면적으로 나열 — 그루핑 필요
  • filter 와 search 의 책임이 섞여서 단일 UI로 합쳐지는 경우
