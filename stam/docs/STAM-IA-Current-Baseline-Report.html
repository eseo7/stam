변경 상세 내역 (Before → After)
총 5건 변경 — 이동 1건 + 신규 추가 4건
순서 변경
이동
WBS 작업 — 기존 9번 → B3 으로 이동
산출물 관리(B그룹) 내 WBS 작업 메뉴의 순서를 B9에서 B3으로 상향 조정합니다.
변경 사유: 분석 단계 초기에 WBS가 먼저 작성되는 실무 흐름 반영. 요구사항(B1) → 화면목록(B2) → WBS(B3)가 논리적 선행 순서에 부합합니다.
신규 추가
신규
DB 정보 — B9 신규 추가
DB 환경 정보(DBMS, DB명, Host/Endpoint, Schema, 연결 서버, 담당자, 공개범위) 관리 화면.
변경 사유: 테이블 정의서(B8)는 스키마/컬럼 정의이고, DB 접속 환경 정보는 보안 등급이 다릅니다. Host·Endpoint 등 민감 정보의 별도 마스킹 관리 및 접근 감사가 필요하여 분리 신설합니다.
신규 추가
신규
서버 정보 — B10 신규 추가
인프라 서버 환경 정보(서버명, 구분, Domain/IP, 용도, 연결 DB, 담당자, 공개범위) 관리 화면.
변경 사유: 서버 환경 정보도 SI 산출물의 일부입니다. 운영 이관 시 납품 패키지에 포함되어야 하며, IP·도메인 마스킹 처리와 역할 기반 열람 제어가 필요합니다.
신규 추가
신규
접근/보안 정보 — B11 신규 추가
계정·API Key·인증서·Token 등 보안 민감 정보 마스킹 관리 화면. 원문 열람 요청 → 사유 입력 → 승인 → 제한 시간 표시 → 자동 마스킹 → 감사로그 흐름 포함.
변경 사유: SI 프로젝트의 보안 정보(Secret Key, 인증서 지문, 접속 비밀번호)는 별도 보안 등급이 필요합니다. 감사로그와 연계한 Rotation 상태 관리로 납품 전 보안 체계를 수립합니다.
신규 추가
신규
오픈 시나리오 — C8 신규 추가
릴리즈 체크리스트 및 Go/No-Go 의사결정 관리 화면. 오픈 단계별 시나리오, 롤백 계획, 담당자·고객 확인자 지정, 예정일시·소요시간 포함.
변경 사유: 오픈 시나리오는 테스트/품질(C그룹) 마지막 단계 산출물입니다. Go/No-Go 의사결정을 시스템 내에서 추적하고, 리스크·롤백 여부를 공식 산출물로 관리할 필요가 있습니다.

서비스 루트 vs 프로젝트 내부 구분 원칙

서비스 루트 영역 6개 메뉴 · 프로젝트 선택 전

  • 사이드바(Left Navigation) 없음

  • 로그인, 프로필 설정, 구독 관리 등 계정 수준 화면
  • 프로젝트 목록에서 프로젝트를 선택하면 내부 진입
  • 프로젝트 내부 사이드바 58개 메뉴에 포함하지 않음 (절대 원칙)
  • stam/index.html 에서 구현 예정 (후속 1순위)

프로젝트 내부 영역 58개 메뉴 · App Shell 고정

  • 고정 Topbar (48px) — 프로젝트명·검색·알림·내보내기·사용자메뉴
  • 고정 Left Navigation (256px) — 58개 메뉴 트리 A~G 그룹
  • 본문 콘텐츠 영역만 화면별로 변경 (.page-content)
  • 화면마다 App Shell을 새로 디자인하지 않음
  • Project Overview Dashboard (A1)가 프로젝트 진입 첫 화면
  • stam/project/*.html 에서 구현 예정
