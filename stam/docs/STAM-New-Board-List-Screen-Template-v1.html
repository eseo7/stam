STAM New Board List Screen Template v1

신규 게시판 목록 화면을 만들 때 동일한 Board List 구조 · 동작 계약 · 검증 기준을 따르도록 강제하는 생성 템플릿 문서.

버전: v1 공통 JS: stam.board-list.js 기준 PR: #92 Controller · #105~#113 표준화 트랙 성격: 생성 템플릿
Related: Board List Contract v1 Bare Preview Controller Target Audit v1 WBS Structure Review v1 WBS Contract v1 Docs Index
Section 1 · Summary

신규 게시판은 기본적으로 공통 컨트롤러 window.STAMBoardList 를 사용한다.

특수 구조가 아니라면 화면별 선택(selection) / row-click / delete-count 로직을 직접 만들지 않는다. 공통 class + 공통 JS 한 줄 init 으로 row=active / checkbox=selected / 삭제 카운트 / 전체선택·indeterminate / Escape 분리를 모두 얻는다.

  • 적용 완료 사례: 기능정의서(#93) · Open Scenario(#106) · 메뉴/화면목록(#107) · 요구사항(#108).
  • 특수 구조 예외: 화면설계서(#109, 그룹/상태 rerender — 현행 유지) · WBS(#110~#112, 다중 tbody/Gantt/주입형 checkbox — 자체 계약).
  • 본 템플릿은 Board List Contract v1 의 적용 절차를 신규 화면 관점으로 정리한 것이다.

Section 4 · 필수 JS 초기화 예시
// stam.xx.js — board-list.js 가 먼저 로드된 상태
var listRoot = document.querySelector('[data-stam-board-list]');
var boardApi = null;
if (listRoot && window.STAMBoardList) {
  boardApi = window.STAMBoardList.init(listRoot, {
    deleteBtn: '#xx-del-btn',                  // 삭제 버튼 selector
    onRowActivate: function (row, api) {       // row click 직후 (이미 .is-active 부여됨)
      openDetailDrawer(row.getAttribute('data-id'));   // 화면별 상세 drawer open 함수 호출
    },
    onDelete: function (rows, api) {           // 삭제 버튼 클릭 (선택>=1 일 때만)
      // 실제 삭제 로직은 화면별 데이터 계층에서 — 본 init 에서는 호출만
    },
    // onClearActive, onSelectionChange, clearActiveOnCheckbox 는 선택적
  });
}

// drawer close / Escape 시 active 동기화
function closeDetailDrawer() {
  // scrim/drawer 닫기 (화면별 UI)
  if (boardApi) boardApi.clearActive();        // .is-active 만 해제 — selected 는 유지
}
API / option설명
STAMBoardList.init(root, opts)root([data-stam-board-list])에 row/checkbox/delete/header 위임 부착. api 반환.
onRowActivate(row, api)row click 직후 호출 — 여기서 상세 drawer open. (active 부여는 컨트롤러가 이미 처리)
onDelete(rows, api)삭제 버튼 클릭(선택≥1) 시 호출. 실제 삭제는 화면별 데이터 로직.
api.clearActive()drawer close / Escape 시 .is-active 만 해제(선택 유지).
api.getSelectedRows() / clearSelected() / refresh()선택 행 조회 / 선택 초기화 / DOM 재렌더 후 재계산.

* 컨트롤러가 document keydown Escape → clearActive 를 자체 처리한다. 화면별 Escape 핸들러는 drawer UI 닫기 + api.clearActive() 동기화만 추가하면 된다.

Section 5 · 공통 동작 계약
상호작용계약
row click클릭 행에 .is-active 부여(다른 행 active 해제) + 상세 drawer open. checkbox checked / delete count 변경 금지, .is-selected 부여 금지.
checkbox click행에 .is-selected + checked + delete count 갱신. drawer open 금지.
selected row click.is-selected.is-active 동시 상태 가능 — checkbox checked / count 유지하며 drawer open.
Escape / drawer close.is-active 만 해제. .is-selected · checked · delete count 는 유지.
header checkbox전체선택 / 전체해제 / 일부 선택 시 indeterminate (컨트롤러 자동).
Section 6 · 삭제 버튼 계약
선택 수버튼 상태라벨 (.stam-delete-label)
0개disabled삭제
1개enabled삭제 (1)
N개enabled삭제 (N)

* 라벨 갱신은 컨트롤러가 .stam-delete-label 내부 textContent 로 처리. 화면별로 직접 갱신하지 않는다. 실제 삭제 동작은 onDelete 콜백 안에서 화면별 데이터 로직으로.

Section 7 · 신규 게시판에서 금지할 것

✅ Do

  • 공통 class + STAMBoardList.init() 사용
  • 상세 drawer 만 onRowActivate 로 연결
  • 실제 삭제는 onDelete 콜백에서 화면별 처리
  • 시각은 stam.table-selection.css 에 위임
  • Bare Preview 와 육안 비교 + Playwright smoke

❌ Don't

  • 화면별 row-click 선택 로직 재구현 (row click 으로 .is-selected 부여 금지)
  • 화면별 checkbox / delete count 로직 재구현
  • 공통 CSS(table-selection) 복붙 / 화면별 재정의
  • stam.board-list.js 임의 수정
  • 공통 class 없이 독자 table selection 구조 생성
  • checkbox click 으로 drawer 열기 / row click 으로 checkbox 체크
Section 8 · 예외 처리 기준 (특수 구조)

아래와 같은 특수 구조는 평면형 STAMBoardList 직접 적용 대상이 아니다. Controller 직접 적용 전에 구조 검토 문서를 먼저 만든다.

  • 그룹 / 상태 기반 전체 rerender 구조 — 화면설계서처럼 선택 상태가 JS state(Set)이고 tbody 를 통째 재렌더하는 경우. (참고: Controller Target Audit v1 — 화면설계서 "특수 구조 · 현행 유지")
  • 다중 tbody / Gantt / JS 주입 checkbox / 특수 focus 구조 — WBS 처럼 그룹별 다중 <tbody>, 런타임 주입 checkbox, 22컬럼 가로스크롤·Gantt·focus 모드가 얽힌 경우. (참고: WBS Structure Review v1 / WBS Contract v1)

* 이 경우에도 상호작용 계약(Section 5·6)은 동일하게 충족하되, 구현은 자체 로직 + 화면 전용 계약 문서로 관리한다. 직접 적용/확장 설계 여부는 검토 문서 판정에 따른다.

Section 9 · QA 체크리스트
  1. 화면 status 200 / console error 0.
  2. row click → 클릭 행 .is-active / checkbox unchecked 유지 / .is-selected 미부여 / delete count 미증가 / drawer open.
  3. 다른 row click → 이전 행 .is-active 해제 · 새 행 .is-active.
  4. checkbox click → .is-selected / checked / 삭제 (1) / drawer 미오픈.
  5. selected row click → .is-selected.is-active 동시 / checked·count 유지 / drawer open.
  6. header checkbox → 전체선택 / 전체해제 / 일부선택 indeterminate.
  7. Escape / drawer close → .is-active 만 해제 / selected·checked·delete count 유지.
  8. delete count → 0개 삭제 disabled · 1개 삭제 (1) · N개 삭제 (N).
  9. search / filter smoke (목록 검색·필터 동작 확인).
  10. 금지 파일 변경 없음 (stam.board-list.js · 공통 CSS · 다른 화면 등).
Section 10 · PR 작성 기준
  • 일반 게시판 — Controller 적용 PR (HTML 공통 class + init() + 자체 선택/카운트 로직 제거). 변경 범위는 해당 화면 HTML/JS 로 한정.
  • 특수 게시판 — Controller 직접 적용 전 구조 검토 PR(docs) 을 먼저 만들어 판정(직접 적용 / 부분 적용 / 현행 유지 / 확장 설계). 그 후 적용/계약 PR.
  • docs-only / code PR 분리 — 문서(템플릿·계약·검토)와 제품 코드 변경은 별도 PR 로 분리한다.
  • 각 적용 PR 은 Bare Preview 비교 + Playwright smoke + merge 후 staging deploy 확인을 수행한다.
