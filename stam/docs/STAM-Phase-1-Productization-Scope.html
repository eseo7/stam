문서 성격 — 의사결정 문서 본 문서는 STAM을 정적 Preview / 문서 확장 중심 작업에서 실제 Auth / DB / Project / Permission 기반 1차 제품화로 전환하는 canonical scope 문서입니다. 실제 구현 지시서가 아니라, 이후 구현 PR(#295~#307)의 범위를 통제하는 결정 기준서입니다.

핵심 서사 · 1차는 전체 제품의 폭을 줄이고, Auth / Project / Permission / DB / 3개 핵심 산출물의 깊이를 실제 제품 수준으로 구현한다. · 폭은 줄이고, 깊이는 진짜로 판다. · 요구사항 / WBS / 화면설계서 3개만 Live 산출물로 둔다. · 기존 Preview와 문서는 버리지 않고 실제 제품 구현 기준으로 재활용한다.
01 전환 배경 — Why Phase 1 Now Decision
  • 기존에는 정적 Preview / 문서 / 가이드 / 템플릿 기준이 많이 쌓였다.
  • 전체 메뉴와 전체 산출물을 한 번에 풀스택 구현하는 것은 범위가 과도하다.
  • 따라서 1차 제품화는 산출물 폭을 3개로 줄이고, Auth / Project / Permission / DB / 저장 구조를 실제 제품 수준으로 만든다.
  • 기존 Preview와 문서는 버리지 않고 실제 제품 구현 기준서로 재활용한다.
전환 원칙 정적 Preview 확장은 중단하지 않지만, 실제 제품화 우선순위보다 앞서지 않는다. Preview·가이드·템플릿은 실제 CRUD·Auth·Rules 구현의 기준서로 재사용한다.

06 Auth 정책 — Google + projectMembers Gate Decision
결정 문구 · “로그인은 가볍게, 권한 구조는 진짜로 간다.” · “1차 베타는 Google 로그인 단일화 + projectMembers 이메일 기반 접근 게이트로 운영한다.”

Auth 결정 사항

  • Google 로그인만 사용
  • 공개 회원가입 없음
  • 이메일/비밀번호 로그인 없음
  • 카카오/네이버 로그인 없음
  • 초대받거나 등록된 이메일만 프로젝트 접근 가능
  • projectMembers.email 기준으로 접근 게이트 처리
  • 로그인 후 users 문서 생성/갱신
  • projectMembers.email이 일치하면 userId 연결 후 active 처리
  • 프로젝트가 없으면 Access Pending 화면 표시
  • 초대 이메일 발송 / 토큰 링크는 후속 범위
  • projectInvites는 1차 DB 대상이 아니라 후속 확장 후보로 둔다.
Google 로그인 › users 생성/갱신 › projectMembers.email 매칭 › 프로젝트 진입 · Active
