01 Overview — 문서 목적 목적

STAM 의 관리자 권한 관리 화면은 단순한 "사용자 설정 화면"이 아니라 보안 통제 화면이다. 누가 어떤 메뉴를 보고, 누가 개인정보를 unmask 할 수 있고, 누가 다운로드할 수 있는지 결정하는 곳이며, 여기서의 모든 행위는 admin_action_logs 에 append-only 로 기록된다. UI 와 DB / 감사로그가 함께 움직여야 정책이 살아 있다.

이 문서가 다루는 것
  • Security Guide v1 의 7 role · 3축 권한을 관리자 화면 단위로 풀이
  • 사용자 목록 / 상세 / 역할 변경 / 예외 부여 / 회수 / 승인 후보
  • 권한 매트릭스 (Role × Action × Scope) · 변경 이력 / audit log 연결
  • 위험 표시 (장기 미접속 + 높은 권한 / MFA 미설정 + ADMIN 이상 등 7종)
  • 모바일 — 조회 / 긴급 잠금 / 승인 후보 한정
  • 상태 — loading / empty / denied / change failed / audit write failed
  • Mock Screens A ~ I (PC 5종 + 모바일 + 상태 + Bad Example)
이 문서가 다루지 않는 것
  • stam/pages/**, stam/js/**, stam/css/** 직접 수정
  • 실제 DB 스키마 생성 / Firebase rules 수정 / API 구현
  • 실제 권한 middleware · approval workflow 코드
  • SSO / MFA / OAuth 구체 구현 (v2)
  • 위탁사 / 외부 제공 흐름 — Security Guide v2 영역
  • 전역 .tbtn 일괄 치환 · Firebase / package.json 변경
Security Guide v1 과의 관계
  • Role / Permission / Audit Log 모델은 모두 Security Guide 에서 상속
  • 본 문서는 그 모델을 운영자가 매일 만지는 UI로 풀어낸 것
  • UI 의 모든 필드는 Security Guide 의 DB 14 테이블 중 7개와 매핑됨 (S19 참조)
  • SUPER_ADMIN / SECURITY_ADMIN 의 운영 분리 원칙은 본 문서에서 화면 단계에서 강제
  • EXTERNAL role 의 개인정보 / unmask / download 차단도 화면 정책으로 명시
실제 구현 전 기준 문서
  • 본 문서는 실제 구현 전 시각 / 정책 기준이며 product 적용은 별도 Gate
  • 적용 순서: 본 가이드 승인 → DB schema 확정 → permission middleware → 사용자 목록 / 상세 → 변경 / 회수 → 매트릭스 / 이력 / 승인
  • 한 커밋에 화면 + 로직 + DB 마이그레이션 섞지 않는다
  • 충돌 시 Security Guide v1 정책이 우선
02 Permission Management Principles Principle
!최소 권한 원칙 (Least Privilege) — 기본은 "권한 없음". 필요한 만큼만 부여.
✓역할 기반 (RBAC) — 사용자에게 직접 권한 부여 X · role 통해 부여.
✓3축 분리 — Menu · Action · Scope. 어느 하나만 풀려도 사고.
!변경 = 사유 입력 필수 — 5자+ 의무. 빈값 / 공백 거부.
!변경 = admin_action_logs 기록 — append-only · 트랜잭션 안에서 처리.
!SUPER_ADMIN 본인 행위 예외 없음 — 본인 권한 변경도 기록 · 2인 승인 후보.
iSECURITY_ADMIN = 감사 전용 — 업무 데이터 read 권한 분리.
!EXTERNAL = 개인정보/unmask/download 기본 차단 — 화면에서 명시적 차단.
i권한 상승은 더 강한 확인 — UPGRADE diff 가 노출되고 추가 체크박스 의무.
✓임시 권한은 만료일 의무 — 만료 없는 임시 권한 금지.
i프론트 권한 숨김은 보안 아님 — API 권한 재검증 의무.
!audit log 저장 실패 시 변경 차단 — 트랜잭션 롤백 · 사용자 안내.

08 Role Change Flow Mock C

role 변경은 before / after diff 가 보이고, 권한 상승은 추가 체크박스 가 의무화된다. SUPER_ADMIN 부여 / 본인 role 변경은 2인 승인 후보 (v2).

· 메뉴 +3 (사용자 관리 / 감사로그 / 보안 정책)
· ACTION +3 (DELETE / EXPORT / ADMIN_MANAGE)
· scope: team → all
✓ 권한 상승 (UPGRADE) 임을 확인했습니다. ADMIN 은 모든 사용자 / 권한 / 정책을 변경할 수 있습니다.
감사로그 저장 실패 시 변경은 적용되지 않습니다.

권한 상승 (UPGRADE) — 추가 통제
  • "권한 상승임을 확인" 체크박스 의무
  • 변경 영향 요약 (메뉴 / action / scope diff) 자동 표시
  • SUPER_ADMIN 부여 시 2인 승인 (v2 후보)
  • 본인 role 변경 시 시스템 거부 또는 2인 승인 (v2)
  • 변경 후 즉시 적용 — 사용자 다음 요청부터 새 권한
권한 하강 (DOWNGRADE) — 일반 통제
  • before / after diff 만 표시 (체크박스 X)
  • 대상 사용자의 현재 세션 강제 종료 (옵션)
  • scope 축소는 별도 모달 — 어떤 프로젝트 / 팀 제외인지 명시
  • 완전 회수는 S10 Revoke flow 사용

  • v1 일반 변경 + v2 승인 워크플로우
  • PC + Mobile 동시 도입
핵심 주의
· UI 만 구현하고 API 권한 재검증을 빼면 보안 무력
· 권한 변경 + 감사로그 = 트랜잭션 · 부분 반영 금지
· SUPER_ADMIN / SECURITY_ADMIN 의 운영 분리는 PM 결정 후 진행
· DB schema 확정 전 화면 선구현 금지 (필드 충돌 위험)
· 매트릭스에서 직접 cell 클릭 → 변경은 사고 위험 — v2 도 신중
