STAM Batch Program Board Product Route / Shell Contract v1 docs-only

"배치프로그램 관리" 게시판을 실제 제품 화면으로 구현하기 전, 제품 라우팅 · 좌측 메뉴 · Shell · Drawer 계약을 문서로 먼저 확정하는 사전 계약서.
본 문서는 계약(Contract) 단계이며, 실제 라우팅·메뉴·page·JS/CSS·API/DB/스케줄러는 생성하지 않는다.

Section 02 · 계약 목적
  • 배치프로그램 관리 게시판이 실제 제품 메뉴와 화면에 들어가기 전 결정해야 할 라우팅·Shell·Drawer 기준을 확정한다.
  • 다음 구현 PR 에서 파일 위치 / 라우팅 / 메뉴 배치 / Shell 구조가 흔들리지 않도록 사전 계약으로 고정한다.

이번 PR 구현 금지선: 본 PR 은 docs-only 다. 실제 제품 메뉴 추가 · 라우팅 추가 · pages/**·js/**·css/** 생성 · nav-data.js/shell.js/topbar.js 수정 · firebase.json/.firebaserc/workflows 변경 · API/DB/Firestore/Storage/스케줄러/배치 실행 로직은 일절 하지 않는다. 본 문서의 route/메뉴 값은 모두 후보이며 확정·구현은 후속 PR + Gate 승인 후 진행한다.
Section 03 · 메뉴 배치 후보
후보위치장점단점 / 리스크
A시스템 관리 하위 운영/관리자 성격과 일치, 권한 통제 자연스러움 설계 산출물 흐름(요구사항·기능정의서·API)과 분리되어 추적성 약화
B산출물 관리 하위 기능정의서·API 정의서·WBS 와 같은 산출물 흐름에 자연 편입, 연결 산출물 추적 강점 "실행 대상"이라는 운영 성격이 산출물 그룹에서 다소 희석
C운영 관리 하위 실행주기·장애 대응 등 운영 관점과 일치 설계 단계 산출물 추적과 멀어짐, 설계기 사용자 접근성 저하
1차 권장안: 후보 B (산출물 관리 하위) — Definition/Preview 의 IA 후보(기능정의서 → API 정의서 → 배치프로그램 관리 → WBS)와 일치하고, 연결 산출물(요구사항·기능정의서·API) 추적성이 가장 강하다. 운영 관점(C)은 추후 운영 대시보드에서 별도 진입점으로 보완 가능.

Section 05 · Shell 계약
요소계약
STAM App Shell사용 — 기존 게시판과 동일한 App Shell 레이아웃 채택
Topbar사용 — 공통 Topbar 그대로, 검색/테마/알림 영역 임의 변경 없음
Left Navigation산출물 관리 그룹(후보 B)에 항목 노출. 노출 방식은 기존 nav 항목과 동일 패턴
목록 카드 영역본문 메인에 Board List(toolbar + 선택형 테이블) 배치
Drawer 영역우측 슬라이드 Drawer (Drawer Shell Contract) — 상세/등록/수정
브라우저 스크롤 원칙본문 목록 영역만 스크롤, Shell/Topbar/Left Nav 고정. Drawer body 내부 독립 스크롤
제품 화면 구현 시 기존 Shell/Topbar/Sidebar 렌더러(stam.shell.js·stam.topbar.js·stam.nav-data.js)를 임의 수정하지 않는다. 신규 게시판은 기존 Shell 을 "사용"할 뿐, Shell 자체를 변경하지 않는다.
Section 06 · Board / List 계약
  • 목록 우선 구조 — 진입 시 목록(List)이 기본. 상세는 Drawer 로 오버레이.
  • 검색 / 필터 / 상태 chip — toolbar 에 검색 input + 필터 버튼, 상태 chip 으로 빠른 필터.
  • 등록 버튼 — toolbar 우측 또는 헤더에 "등록" (등록 Drawer open).
  • 행 클릭 → 상세 Drawer — row click = .is-active → 상세 Drawer open (Board List Controller 계약).
  • 수정 — 상세 Drawer 내부 "수정" 진입 또는 별도 수정 Drawer (mode 전환).
  • 위험 액션 — 삭제·(향후)실행 등은 기본 숨김 또는 비활성. 운영반영 상태 삭제는 폐기 전환 우선.
Section 07 · Drawer 계약
Drawer / 영역계약
등록 Drawercreate mode — 필수/선택 필드 (Form Field Contract), custom select
수정 Draweredit mode — 등록과 동일 Form, 값 prefilled
상세 Drawerdetail mode — 읽기 표시 + "수정" 진입
실행 이력 영역상세 Drawer 내 영역 — MVP 에서는 Skeleton(표시 자리)만, 실데이터 없음
관련 산출물 영역연결 산출물(요구사항·기능정의서·API …) 링크/칩 표시
변경 이력 영역변경 이력 표시 영역 (후보)
댓글 / @멘션후속 확장 — MVP 제외
모든 Drawer 는 Board Drawer Shell Contract(root/scrim/head/body/foot · Escape · aria-modal)와 Form Field Contract 를 그대로 따른다.
Section 08 · MVP 포함 / 제외 범위

✅ MVP 포함

  • 목록 (List)
  • 등록 (Create Drawer)
  • 수정 (Edit Drawer)
  • 상세 (Detail Drawer)
  • 상태 표시 (상태 chip)
  • 실행 이력 표시 영역 Skeleton

❌ MVP 제외

  • 실제 배치 실행
  • 수동 실행 버튼 활성화
  • 스케줄러 연동
  • API 호출
  • DB 저장
  • 로그 수집
  • 알림 발송
실행/중지/스케줄러/API/DB 는 MVP 에서 명시적으로 제외한다. 실행 이력은 표시 자리(Skeleton)만 두고 실데이터 연동은 후속 단계(API/DB 설계 이후).
