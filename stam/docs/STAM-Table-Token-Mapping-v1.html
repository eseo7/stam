판정 근거 — Drawer Token Mapping v1에서 “리스트 row → Drawer 상세 → Full Detail”의 진입 흐름이 결정되었고, Layer Token Mapping v1에서 “좌측 Nav 위, Drawer 아래”의 stacking이 결정되었으며, Form Token Mapping v1에서 “Drawer · Full Detail · Dialog가 같은 입력 컴포넌트를 공유”라는 원칙이 확정되었다. 그러나 Table 자체의 density · column · header · sticky · scroll · row state · cell type · token 기준은 확정되지 않은 상태이다. 게시판마다 Table CSS가 따로 만들어지거나, 페이지 레벨 가로 스크롤이 발생하거나, 컬럼 커스텀이 CSS 하드코딩으로 처리되는 사고를 사전에 차단하기 위해 본 문서로 Table 매핑과 운영 원칙을 확정한다.

01 Overview — 문서 목적 목적

Drawer Token Mapping v1이 “리스트에서 한 건을 빠르게 보고 / 등록 / 수정하는 진입 UI”를 확정했다면, 본 Table Token Mapping v1은 그 앞 단계인 리스트 그 자체 — Basic List · Work Board · Artifact Board · WBS · Test Scenario · Defect · Server Info · Open Scenario · Custom Column · Compact · Wide Data Table의 structure · density · column · header · sticky · scroll · row state · cell type · token · board / drawer / full detail 흐름을 1:1로 매핑한다. STAM에서 Table은 “전체 산출물 게시판 · 요구사항 · 결함 · WBS · 테스트 · 서버 · 오픈 시나리오 등 모든 목록형 화면의 공통 기본 UI”이며, 모든 게시판에서 사용자(고객사 · 프로젝트 · 게시판 단위)는 표시 컬럼 · 순서 · width · 정렬 · 필터 · 내보내기 포함 여부를 커스텀할 수 있어야 한다. 기획자 · PM은 “어떤 게시판이 어떤 density / column 기준을 따르는지”, 디자이너는 “Table 내 영역 · 상태 · 셀 유형”을, 개발자는 “토큰명 · sticky / scroll 정책 · row → Drawer 흐름 기준”을 본 문서 하나로 읽고 승인할 수 있어야 한다.

02 Table Scope — 본 매핑이 다루는 Table 범위 범위

아래 11개 시나리오가 본 매핑의 대상이다. 일반 목록 / 작업 게시판 / 산출물 게시판 / WBS / 테스트 / 결함 / 서버 / 오픈 시나리오 / 커스텀 컬럼 / Compact / Wide Data Table을 포함한다. 게시판별로 시각·동작이 다르지 않으며, 한 가지 Table 컴포넌트가 density · column 설정으로 모든 화면을 커버한다.

본 문서가 다루지 않는 UI Calendar · Kanban · Gantt · Tree-only · Pivot · Excel-like inline grid는 본 v1 범위 밖이다. Drawer / Dialog 내부의 보조 목록은 Compact Table로 포함되며, “Drawer 안의 form”은 Form Token Mapping v1을 따른다. Toast · Tooltip · Pagination 컴포넌트 자체는 Component Guide / Layer Mapping v1에서 정의하며, 본 문서는 그 위치 · 토큰 사용 기준만 다룬다.

04 Table Density / Size Rule — row height · padding 기준 밀도

Table은 게시판마다 CSS를 새로 만들지 않고 density 토큰 / class 조합으로 row height · padding · font · icon size만 바꾼다. 일반 게시판은 normal, WBS / 테스트 / 오픈 시나리오처럼 컬럼이 많은 화면은 compact 또는 dense, 회의실 / 빔프로젝터 공유 화면은 large(readability mode)를 후보로 둔다.

  • WBS · Test · Open Scenario처럼 컬럼이 많은 화면은 compact 또는 dense 후보
  • 빔프로젝터 · 회의실 공유는 large(readability mode)를 별도 후보로 둠
  • 한 게시판에서 사용자 선택으로 density를 toggle 할지 / 게시판마다 고정할지 PM 결정
  • density는 row height · padding · font · icon size만 바꿈 — 색 · 토큰은 동일
제어 방식 원칙
  • token / class 조합으로만 제어 — 게시판마다 별도 CSS 작성 금지
  • density 변경은 --table-row-height · --table-cell-padding-x · --table-cell-padding-y 3개 token만 swap
  • JS · DOM 구조 변경 없이 density 전환 가능해야 함
  • density 토큰은 Component Guide의 size scale과 정합 — Form / Button과 일관
  • 한 화면에서 density가 row 별로 다르게 적용되지 않음 — table-level 단일 density
가독성 / 접근성 기준
  • dense는 12px 이하 폰트 금지 · 접근성 최소 기준 유지
  • compact / dense에서도 row click 영역은 row 전체 — 작은 row height에서도 hit 보장
  • large는 회의실 환경에서 5~7m 거리에서 읽을 수 있어야 함
  • density 전환 시 row 위치 / scroll 위치가 흔들리지 않도록 가급적 유지
  • density 전환은 페이지 단위가 아닌 게시판 단위 설정 우선
density 결정 우선순위
  • P0 — normal (기본) · 일반 게시판 즉시 사용
  • P0 — compact · WBS · Test · Defect 압축 화면
  • P1 — comfortable · Artifact Board 텍스트 위주
  • P1 — dense · Custom / Wide Data 분석 화면
  • P2 — large · 빔프로젝터 / 회의실 발표 모드 (Decision Needed)
Decision Needed — large(readability mode) 적용 범위 빔프로젝터 / 회의실 공유 화면에서 large mode를 별도로 둘지 / 사용자 “발표 모드” toggle로 둘지 PM 결정이 필요하다. 기본 후보는 Open Scenario · WBS 두 게시판에 한정 적용이며, ALL pages 적용은 Toolbar 위치 / Pagination 위치를 함께 검토해야 한다.

06 Sticky / Scroll Rule — sticky · 가로 / 세로 scroll 정책 스크롤

STAM Table의 scroll 정책은 단순하다. 좌측 Nav 고정 · 페이지 세로 scroll은 브라우저 기본 · 가로 scroll은 table container 내부에서만 발생 · 페이지 레벨 가로 scroll 금지. sticky header는 필요 시 table 내부 또는 page 기준 중 하나로 결정한다. sticky column은 후속 구현 대상이다.

PASS — 권장 scroll 패턴
좌측 Nav는 고정 · 페이지 세로 scroll은 브라우저 기본 · Wide Table은 container 내부 가로 scroll.
↕ page scroll · ↔ table-only scroll
FAIL — 금지 패턴
페이지 전체 가로 scroll · body 자체 세로 scroll · sticky가 table / page에서 임의 혼용.
⛔ page-level h-scroll
CONDITIONAL — sticky / pagination
sticky header / pagination 적용 범위는 PM 결정 — table 내부 or page 기준 중 하나로 통일.
≡ sticky header · ◐ sticky pagination

세로 scroll 원칙
  • 세로 scroll은 페이지 단위 브라우저 scroll만 사용
  • table body 자체 세로 scroll(고정 높이 container) 금지 — sticky / popup 충돌
  • row 수가 많으면 pagination 또는 virtualization(후속)으로 해결
  • row click → Drawer 흐름에서 scroll 위치는 유지 — Drawer 닫힘 후 같은 위치
  • filter 적용 시 scroll 위치는 최상단으로 리셋
가로 scroll 원칙
  • 가로 scroll은 table container 내부에서만 발생 · page-level 금지
  • 1440 · 1600 · 1920 가로 폭에서 페이지 레벨 가로 scroll 없음 — QA 필수
  • Wide Data Table은 container 내부 가로 scroll로 처리 · drawer 진입 시에도 잘리지 않음
  • scrollbar 자체도 토큰 — --table-scrollbar-bg · --table-scrollbar-thumb
  • 좌측 Nav · Drawer 진입 시 가로 scroll 보장 — page-level은 절대 금지
Sticky header · pagination 원칙
  • sticky header는 table 내부 또는 page 기준 중 하나로 통일 · 임의 혼용 금지
  • sticky pagination은 긴 list에서 권장 · scroll에 흔들리지 않음
  • sticky 영역은 Layer Mapping v1 z-index 토큰만 사용 · 임의 숫자 금지
  • sticky 영역의 그림자는 --table-sticky-shadow로 통일
  • sticky header / pagination이 Drawer / Dialog popup과 충돌하지 않음
popup 충돌 방지
  • Drawer · Dialog · Dropdown은 모두 portal 렌더링 — table scroll에 갇히지 않음
  • Drawer panel은 Table 위에 dim layer를 두고 row click을 차단
  • Drawer / Dialog 진입 시 페이지 가로 scroll 발생 금지
  • Drawer 닫힘 시 Table scroll 위치는 그대로 유지
  • filter / 컬럼 설정 dropdown은 Layer Mapping v1 --z-dropdown 사용
Decision Needed — Sticky Header / Pagination 기준 단일화 sticky header를 “table container 내부” 기준으로 잡을지 “page top” 기준으로 잡을지 PM 결정 필요. 기본 후보는 table 내부 sticky이며, Toolbar / Filter는 page top sticky로 분리. sticky pagination 적용 여부는 게시판별 row 수 분포 검토 후 결정 권장.

row state 우선순위 한 row가 여러 상태를 동시에 가질 수 있다 (예: hover + selected + warning). 우선순위는 error / warning (의미)이 배경, selected / active가 상호작용 표시, hover가 인터랙션 보조다. disabled / locked / readonly는 hover · selected · active를 차단한다. updated는 일정 시간 후 자동 해제되며 default로 돌아간다.
색만으로 의미 전달 금지 warning / error / success / locked / disabled는 반드시 색 + 라벨 또는 아이콘으로 함께 표현해야 한다. 다크 모드에서 색 차이가 약해질 수 있으며, 색약 사용자에게 정보가 전달되지 않는다. status badge(Section 08)와 row state는 서로 보완 — 둘 중 하나만으로 의미가 완성되지 않는다.

11 Accessibility / Interaction Rule — 접근성 · 키보드 · focus · dark 접근성

Table은 데이터 격자다. 시멘틱한 <table> 또는 ARIA role=table/row/cell을 사용하여 header와 cell의 의미를 연결하고, 키보드 row navigation · checkbox 접근성 · action button aria-label · focus ring을 보장한다. hover에만 의존하지 않으며, status / row state는 색만으로 의미를 전달하지 않는다.

Must의미 연결 원칙
  • 시멘틱 <table> 또는 ARIA role=table · row · columnheader · cell 사용
  • header cell과 body cell의 의미 연결 (scope=col · headers)
  • 정렬 가능 컬럼은 aria-sort=ascending / descending / none 표시
  • row 선택 상태는 aria-selected 또는 aria-checked 표시
  • row state(warning / error / success)는 aria-label로 의미를 명시
Keyboard키보드 nav 후보
  • row navigation 후보 — ↑/↓로 row 이동, Enter로 Drawer open (Decision Needed)
  • checkbox는 Space로 toggle · 헤더 checkbox는 전체 선택 / 해제
  • action button은 Tab으로 진입 · Enter / Space로 실행
  • 정렬 가능 header cell은 Enter / Space로 정렬 토글
  • Drawer open 시 focus는 Drawer 내부로 이동 · 닫힘 시 같은 row로 복귀
Focusfocus / hover 원칙
  • focus ring은 모든 interactive 요소에서 명확히 유지 (Button / Form Mapping v1)
  • row · action · checkbox · header cell 모두 focus ring 가짐
  • hover에만 의존하지 않음 — 키보드 / 터치 사용자도 동일 정보 접근 가능
  • focus ring 색은 Button / Form과 일관 — Table 전용 focus 색 추가 금지
  • focus 상태는 dark mode에서도 명확히 보임
Color색 / 대비 원칙
  • status badge / row state는 색 + 라벨 또는 아이콘으로 동시 전달
  • 색만으로 의미 전달 금지 — 색약 / 모노 인쇄 / 다크 모드 대비 보장
  • 모든 텍스트는 WCAG AA 대비 기준 충족
  • dark mode 대비 확보 — row state 색 토큰을 dark 별도 정의 (Section 09)
  • action button의 aria-label 필수 — 아이콘만 있는 버튼도 의미 전달
Decision Needed — 키보드 row navigation 정책 ↑/↓로 row 포커스 이동 시 Drawer를 자동 열지 / Enter로 명시적으로 열지 PM 결정 필요. 자동 열기는 빠르지만 의도치 않은 Drawer open · 잦은 API 호출 위험이 있다. 기본 후보는 “Enter로 명시 open · ↑/↓는 row focus 이동만”.
12 Do / Don't — 실무 체크리스트 규칙
Do — 권장 패턴
  • token 기반 구현 — 색 · 폭 · padding · row height 모두 --table-* 토큰만 사용
  • board / table 공통 구조 사용 — 모든 게시판이 같은 Table 컴포넌트 사용
  • 컬럼 설정과 visual token 분리 — 설정은 데이터, 토큰은 시각
  • compact / dense mode 기준 문서화 — Section 04 density 토큰 swap만으로 처리
  • 1440 / 1600 / 1920 가로 폭 QA — 페이지 레벨 가로 scroll 없음 확인
  • row click → Drawer · 같은 데이터 · 같은 컴포넌트 보장
  • status / row state는 색 + 라벨 동시 사용
  • light / dark 두 테마에서 동시 QA — row hover · selected · warning · error 대비 확인
  • sticky header / pagination 기준은 단일화 — 임의 혼용 금지
  • action / checkbox / link cell은 row click과 충돌 차단 (stopPropagation)
Don't — 금지 패턴
  • 화면마다 table CSS 따로 만들기 — 동일 컴포넌트가 게시판별로 분기되면 실패
  • 페이지 레벨 가로 scroll 만들기 — 좌측 Nav까지 잘림
  • 스프레드시트식 inline edit를 기본값으로 넣기 — v1 범위 밖
  • 컬럼 커스텀을 CSS 하드코딩으로 처리 — 데이터 모델로만 처리
  • sticky header / pagination 기준을 한 화면에 임의로 섞기
  • 전체 컴포넌트 CSS를 한 번에 구현 — Table만 제한 구현
  • row state · status badge에 색만으로 의미 전달
  • row click과 action / checkbox / link click이 동시 트리거되도록 두기
  • Table 전용 input · select · date picker 컴포넌트 만들기 (Form Mapping v1 사용)
  • z-index 임의 숫자 부여 — Layer Mapping v1 토큰만 사용
  • table body 자체 세로 scroll(고정 높이 container) — sticky / popup 충돌
  • density를 row 별로 다르게 적용 — table-level 단일 density만

Cursor는 Section 03 Structure · Section 04 Density · Section 06 Sticky/Scroll · Section 07 Row State · Section 08 Cell Type · Section 09 Token Mapping · Section 10–11 Rule만 코드로 옮기며, 새로운 토큰명 · 값을 임의로 추가하거나, 게시판마다 별도 변형 컴포넌트를 만들지 않는다. 데이터 구조 · 저장 API · 필드명 변경은 Table 디자인 작업 범위에 포함되지 않는다.
