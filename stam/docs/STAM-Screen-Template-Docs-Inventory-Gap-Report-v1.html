STAM Screen Template Docs
Inventory & Gap Report v1

이 문서는 후속 화면설계서(Screen Specification) 템플릿 문서를 새로 만들기 전에, 대화/기준상 언급된 문서가 실제 stam/docs/ repo에 존재하는지 전수 조사한 결과다. 이번 작업은 조사·정리·Gap Report 생성까지만 수행하며, 누락 문서를 새로 생성하지 않는다. 보고서 안의 모든 링크는 실제 존재하는 파일만 가리키며, 존재하지 않는 파일은 링크하지 않고 파일명 텍스트로만 표기한다. 기준 커밋은 main f1b43ff (PR #168 merge) 이다.

04

Gap Analysis

문서 체계의 빈칸

Detail Pack 과 Object Builder 버킷은 실재 문서가 0건이다. 상위 통합 가이드(System Master Guide)도 부재하여, 현재는 Baseline·Sample·Full-Stack Guide가 부분적으로 그 역할을 나눠 갖고 있다.

문서명 기준 불일치

“Screen Template” 명칭이 두 갈래로 혼재한다 — ① 화면설계서(Screen Specification) 계열, ② 게시판 목록(board list) 계열(New-Board-List-Screen-Template). 후속 파일명 확정 시 둘을 명확히 구분해야 중복/오인을 막는다.

Docs Index 링크 누락

실재 관련 문서 7건은 모두 이미 연결되어 있다(03·04·S00·Quick Find 섹션). 따라서 현재 공백은 링크가 아닌 콘텐츠 부재다. 본 Gap Report 카드 1개만 “04 화면설계서 템플릿 / Screen Template” 섹션에 추가한다.

실제 파일 vs 대화 기준 차이

대화 기준 9건 중 실재는 2건(22%)뿐이다. 누락 7건 중 3건은 기존 문서가 부분/전부 대체 가능하므로, 무조건 신규 생성보다 대체 가능 여부 우선 검토가 필요하다.

후속 작업 주의점

① 파일명·경로를 먼저 확정한 뒤 생성한다. ② Drawer Preview·Common Catalog는 기존 문서와 중복 위험이 있으니 통합/대체부터 판단한다. ③ board-list 와 화면설계서 네이밍을 분리한다. ④ 모든 신규는 docs-only 로 가능해야 하며 제품 파일·nav-data를 건드리지 않는다.

현재 기준 vs 보존/참고

본 보고서의 “실재 문서”는 현재 기준 문서다. Docs Index의 “06 보존/참고 전용” 그룹 문서(과거 QA·이력)는 기준 문서와 섞지 않는다.

05

Recommended Follow-up Order

P1 → P5

다음 작업자가 바로 후속 생성 프롬프트를 만들 수 있도록 우선순위와 판단 근거를 정리한다. 평가 관점: 화면설계서 기능 직접 필요 · Board Builder/템플릿 선택 연결 · Index 발견성 · 중복 위험 · docs-only 가능 여부.

순위후속 문서 (예상명)판단근거 / 비고
P1STAM-Screen-Template-System-Master-Guide-v1.html생성상위 통합 가이드 부재. 화면설계서 템플릿 체계의 진입점/SSOT 역할. 직접 필요·발견성 높음·중복 위험 낮음.
P2STAM-Screen-Template-Priority-30-Korean-Baseline-v1.html생성대체 문서 전무. 한국어 우선 30화면 기준선 → 템플릿 선택/적용에 직접 연결.
P3STAM-Screen-Template-Top-10-Detail-Pack-v1.html생성Detail Pack 버킷이 비어 있음. Priority 30의 상위 10화면 상세팩 → P2 이후 자연스러운 후속.
P4STAM-Screen-Template-Top-3-Drawer-Preview-v1.html생성 여부 결정Board-Content-Preview가 Drawer를 일부 커버. 신규 Preview가 필요한지(중복 회피) 먼저 결정.
P5STAM-Object-Builder-Preview.html · STAM-Common-Catalog-Preview.html보류 / 통합 판단Object Builder는 개념 정의부터 필요. Common Catalog는 Common-UI-Control-Center로 흡수 검토. 둘 다 즉시 생성보다 보류/통합 권장.
참고 — 표의 예상 파일명은 확정 경로가 아니다. 실제 생성 전 파일명/경로를 먼저 확정하고, 본 보고서의 Link Policy(06)를 따른다. Detail-Drawer-Result-Preview 는 Board-Content-Preview와 중복 가능성이 커서 별도 순위 없이 보류한다.
06

Link Policy

실재 파일만 링크본 보고서의 모든 href 는 stam/docs/ 에 실제 존재하는 파일만 가리킨다(01 인벤토리 7건).
없는 파일은 링크 금지누락 문서(02)는 링크하지 않고 예상 파일명을 코드 텍스트로만 표기한다.
임의 경로 생성 금지가상의 route·예상 경로를 확정 링크처럼 쓰지 않는다.
Index 추가도 실재 경로만Docs Index 카드 추가 시 실제 파일 경로(본 Gap Report)만 사용한다.
생성 전 경로 확정향후 문서 생성 전 파일명·경로를 먼저 확정한 뒤 링크를 만든다.
제품 파일 비연동App Shell/Left Nav/nav-data/저장 로직에 연결하지 않는 docs-only 링크만 사용한다.
