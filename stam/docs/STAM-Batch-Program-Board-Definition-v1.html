Section 01 · Summary

배치프로그램 관리 게시판이 필요한 이유

  • SI 프로젝트에서 배치(Batch) 프로그램은 정산·집계·연계·마감·통계 등 업무 핵심 흐름을 담당하지만, 요구사항·기능정의서·API 정의서·테스트 산출물에 흩어져 일관된 목록으로 관리되지 않는다.
  • 배치는 실행주기·실행시간·선후행 관계·실패 처리·재처리 기준이라는 온라인 기능과 다른 고유 속성을 가지며, 이를 별도 게시판으로 표준화해야 설계 누락과 오픈 사고를 줄일 수 있다.
  • 운영 이관(인수인계)·오픈 점검 시 "어떤 배치가 언제 어떤 순서로 돌고, 실패하면 어떻게 복구하는가"를 단일 화면에서 확인할 수 있어야 한다.

SI 프로젝트에서 배치가 다른 산출물과 연결되는 지점

연결 산출물배치와의 연결 지점
요구사항배치성 요구사항(정산·집계·마감 등)이 배치 프로그램으로 구체화되는 추적 출발점
기능정의서배치 기능 단위(기능 ID)와 배치 프로그램(배치 ID)의 매핑
API 정의서배치가 호출/제공하는 내부·외부 연계 API, 데이터 입출력 인터페이스
테스트 케이스 / 결함배치 단위 테스트·재처리 시나리오 검증, 발견 결함 추적
오픈 시나리오오픈 D-day 배치 수행 순서·점검 항목·롤백 기준
Section 02 · 게시판 목적
  1. 배치 프로그램 목록 관리 — 프로젝트 내 모든 배치를 ID·업무영역·담당자·상태로 단일 목록에서 추적.
  2. 실행주기 / 선후행 / 실패처리 / 재처리 기준 관리 — 스케줄(주기·시각), 선행·후행 의존성, 실패 시 알림·재시도·수동 재처리 정책을 명시.
  3. 운영 이관 및 오픈 점검 기준 확보 — 운영 이관 체크리스트, 오픈 시나리오 연동 점검, 장애 대응 절차의 단일 기준 제공.
본 게시판은 배치의 "정의·관리"를 목적으로 한다. 배치의 실제 실행/중지/스케줄링(스케줄러 연동, 잡 트리거)은 본 게시판의 범위가 아니며 Section 11(제외 범위)에 명시한다.
Section 03 · 메뉴 / IA 후보 위치

배치프로그램 관리는 설계 산출물 그룹 내, 기능정의서·API 정의서와 인접한 위치가 자연스럽다. 후보 순서는 다음과 같다.

  • 기능정의서
  • API 정의서
  • 배치프로그램 관리 ← 신규 후보
  • WBS
중요: 본 PR 에서는 stam.nav-data.js 나 실제 좌측 메뉴를 수정하지 않는다. 위 위치는 향후 nav-data 연결 단계(Section 10)에서 결정할 후보일 뿐이며, 이 문서는 IA 배치를 확정하지 않는다.

Section 09 · 권한 / 운영 고려사항
  • 등록/수정 권한 — 배치 설계 담당자·PL 이상에게 부여. 일반 열람자는 읽기 전용.
  • 삭제 제한 — 운영반영 이력이 있는 배치는 물리 삭제 대신 폐기 상태 전환을 원칙으로 한다(이력 보존).
  • 운영반영 상태 잠금 여부 — 운영반영 상태에서는 실행 정보·선후행 등 핵심 필드 변경 시 사유 입력 또는 별도 승인 절차를 고려(변경 통제).
  • 변경이력 필요성 — 실행주기·선후행·실패처리 정책은 오픈/장애와 직결되므로 변경이력(누가·언제·무엇을) 기록이 필요하다.
위 항목은 정책 후보이며, 실제 권한/잠금/이력 로직은 화면·백엔드 구현 단계에서 STAM 공통 권한·이력 정책 문서와 정합화하여 확정한다. 본 PR 에서는 정의만 한다.
Section 10 · 향후 구현 단계
  1. Definition 문서화 ← 본 PR (현재)
  2. Preview 문서화 — 목록/상세/등록/수정 정적 시안(Applied Preview)
  3. 실제 page 생성 — stam/pages/boards/ 하위 화면 + 공통 계약 적용
  4. nav-data 연결 — stam.nav-data.js 좌측 메뉴 등록 (Section 03 위치 확정)
  5. 테스트 / 검증 — Board List·Drawer·Form 회귀, staging 배포 검증

* 각 단계는 별도 PR 로 분리하며, 본 PR 은 1단계(Definition)만 수행한다.

Section 11 · 제외 범위 (본 PR)
