STAM Screen Template
System Master Guide v1

화면설계서(Screen Specification) 템플릿 시스템의 상위 기준 문서다. 공통 페이지 템플릿 기반 원칙, 12개 분류 체계, 사용자 흐름 9단계, 후속 문서 로드맵(P1–P5)을 하나의 문서로 정의한다. 이 문서는 참조 기준 가이드이며, 개별 템플릿 HTML과 상세 팩은 별도 후속 문서에서 관리한다. 제품 파일(App Shell, nav-data, 저장 로직, Firebase)은 수정하지 않는다.

01

템플릿 시스템 목표

핵심 목표

화면설계서를 처음부터 그리지 않는다. 공통 페이지 템플릿을 선택 → 수정하는 흐름으로 작업 시간을 줄이고, 프로젝트 간 화면 구조의 일관성을 확보한다.

적용 범위

STAM 화면설계서(B4 메뉴) 전 영역. List/Board, Detail, Form, Dashboard, Approval, Settings, Permission, Report, Wizard, Drawer/Modal, Empty/Error/Result 12개 유형을 포함한다.

작업 대상

신규 프로젝트의 화면 목록 정의 이후, 각 화면에 적합한 템플릿 유형을 선택해 필드·메타데이터를 채우고 검토/승인 흐름까지 연결한다.

제한 범위

이 문서는 기준 가이드다. 템플릿 HTML 생성, 게시판 설정, 저장 로직, Firebase, App Shell 수정은 별도 후속 작업으로 분리한다.

02

핵심 원칙 8가지

① 공통 페이지 템플릿 기반모든 화면설계는 검증된 12개 템플릿 카테고리 중 하나를 선택하는 것으로 시작한다. 처음부터 새 레이아웃을 만들지 않는다.
② 선택 → 수정 흐름템플릿 선택 후 프로젝트 맥락에 맞게 필드·레이아웃·권한 등을 수정한다. 템플릿 원본은 유지하고 복사본을 편집한다.
③ 필드 메타데이터 완전 기입화면 ID, 상태, 버전, 작성자, 담당 개발자, 연결 API, 권한 등급 등 15개 필드를 빠짐없이 기입한다.
④ 승인/검토 레이어 병행작성 완료 후 검토 요청 → 승인 처리 → 변경이력 기록 흐름을 필수로 거친다. 승인 없는 산출물은 납품 불가 상태로 표시된다.
⑤ 내보내기 연동 가능 구조최종 화면설계서는 PDF/Word/Excel 내보내기 및 고객 공유 링크 생성이 가능한 구조로 유지한다. 메타데이터 완전 기입이 선행 조건이다.
⑥ 모바일 / 좁은 화면 고려PC 테이블 기준으로 설계하되, 980px 이하에서 카드/전체보기 전환 흐름을 템플릿마다 명시한다.
⑦ Board Builder 보완 관계Board Builder는 게시판 구조 Preview 도구이며, 화면설계서 템플릿 시스템은 개별 화면의 설계 기준을 제공한다. 두 시스템은 경쟁하지 않고 보완 관계를 형성한다.
⑧ 실제 repo 경로 기준 링크문서 내 모든 링크는 실제 존재하는 파일만 참조한다. 아직 생성되지 않은 후속 문서는 파일명 텍스트로만 표기하고 href를 부여하지 않는다.
03

사용자 흐름 9단계

프로젝트 선택 → 내보내기 연결
1. 프로젝트 선택내 프로젝트 목록 또는 즐겨찾기에서 대상 프로젝트 진입
2. 화면설계서 진입B4 화면설계서 메뉴로 이동. 화면 목록 확인
3. 신규 화면 추가화면 목록에서 신규 화면 추가 액션 실행
4. 템플릿 유형 선택12개 카테고리 중 해당 화면 성격에 맞는 유형 선택
5. 메타데이터 기입화면 ID, 상태, 버전, 담당자, API, 권한 등 15개 필드 입력
6. 레이아웃 수정선택한 템플릿 기준으로 프로젝트 맥락에 맞게 필드·구역 수정
7. 검토 요청작성 완료 후 담당 검토자에게 검토 요청 전송
8. 승인 처리검토자 확인 후 승인 또는 반려. 반려 시 재수정 흐름으로 재진입
9. 내보내기 연결승인 완료 화면설계서를 PDF/Word 내보내기 또는 고객 공유 링크로 납품
04

템플릿 분류 체계 — 12개 카테고리

T01 · List / Board

레코드 목록, 게시판 목록. 필터·정렬·페이지네이션 포함. 가장 빈번하게 사용되는 유형.

T02 · Detail

단일 레코드 상세 보기. 탭/섹션 분리, 하위 항목 인라인 표시, 연결 산출물 표시.

T03 · Create / Edit Form

신규 작성 및 수정 폼. 필드 유형(텍스트, 날짜, 선택, 파일)별 기준 레이아웃 포함.

T04 · Search / Filter

고급 검색, 다중 필터 조합. 필터 칩, 초기화, 결과 수 표시 포함.

T05 · Dashboard

지표 카드, 차트, 요약 위젯. 프로젝트 대시보드(A1) 또는 산출물 현황 대시보드에 사용.

T06 · Approval

검토 요청 목록, 승인 상세, 승인/반려 액션, 승인 이력. D그룹 레이어와 연결.

T07 · Settings

프로젝트·게시판·계정 설정 화면. 섹션 분리, 저장/취소 버튼 기준 패턴.

T08 · Permission

역할/권한 관리, 멤버별 접근 제어, 산출물 접근권한. F3·F4 메뉴 설계 기준.

T09 · Report

테스트 결과, 결함 대시보드, 사용량 리포트. 출력/내보내기 기준 레이아웃 포함.

T10 · Wizard

단계별 입력 흐름(Step 1 → N). 진행 표시, 이전/다음 버튼, 중간 저장 패턴.

T11 · Drawer / Modal

우측 Drawer 상세, 확인 Modal, 빠른 편집 Panel. 메인 목록 위에 레이어로 표시.

T12 · Empty / Error / Result

데이터 없음 상태, 에러 화면, 처리 완료 결과 화면. 모든 유형에 공통 적용.

05

Priority 30 관계

문서 계층 구조

이 Master Guide → Priority 30 한국어 베이스라인 → Top 10 상세 팩 → Top 3 Drawer Preview 순서로 문서가 상세화된다. 각 후속 문서는 이 Master Guide의 원칙과 분류 체계를 기준으로 작성된다.

Priority 30 정의

STAM 화면설계서 시스템에서 가장 높은 빈도로 사용되는 상위 30개 화면 유형을 선별한 베이스라인이다. 12개 카테고리를 실제 빈도 기준으로 구체화한 목록이며, 후속 단계 문서로 생성 예정이다.

현재 상태: Priority 30 한국어 베이스라인(STAM-Screen-Template-Priority-30-Korean-Baseline-v1.html)은 아직 생성되지 않았다. 이 문서에서는 링크를 부여하지 않으며 파일명 텍스트로만 표기한다. 생성 완료 후 이 섹션에 링크를 추가한다.
06

Board Builder 관계

Board Builder 역할

게시판 만들기(F5-2) 단계에서 필드·목록·필터·미리보기 기준을 잡는 게시판 구조 Preview 도구다. 입력 결과는 localStorage 기준이며 Firestore/API/fetch 호출이 없는 상태를 유지한다.

→ Board Builder (pages/boards-v2/board-builder.html)

화면설계서 템플릿 역할

개별 화면의 설계 기준과 메타데이터 구조를 제공한다. Board Builder가 게시판 구조를 잡으면, 화면설계서 템플릿은 각 화면의 상세 설계를 담는다. 두 시스템은 경쟁하지 않고 단계적으로 보완 관계를 형성한다.

Board Builder 제품 파일(stam/pages/boards-v2/board-builder.html)은 이 문서 작업의 수정 범위에 포함되지 않는다. 문서 참조 링크만 제공한다.
07

화면설계서 제품 페이지 관계

화면설계서 제품 페이지

STAM 내 B4 메뉴인 화면설계서(stam/pages/boards/screen-specification.html)는 실제 데이터를 다루는 제품 화면이다. 이 Master Guide는 해당 제품 화면에서 사용할 템플릿 유형 분류·메타데이터·흐름 기준을 정의하는 참조 가이드이며, 제품 파일 자체는 수정하지 않는다.

→ 화면설계서 제품 (pages/boards/screen-specification.html)

기준 문서 역할

이 Master Guide는 화면설계서 제품에서 사용할 템플릿 유형 분류, 메타데이터 구조, 검토/승인 흐름 기준을 정의한다. 실제 제품 화면의 UI·저장 로직·Firebase 연동은 이 문서의 범위 밖이다.

화면설계서 템플릿 기준 HTML은 templates/STAM-Screen-Specification-Template-Baseline.html에 존재한다. 이 Master Guide는 해당 템플릿을 포함한 전체 시스템의 상위 기준을 제공한다.

09

권한 / 검토 / 이력

역할별 권한

관리자: 전체 생성·수정·삭제·승인
편집자: 화면 작성·수정·검토 요청
뷰어: 조회만 가능, 내보내기 제한

검토 / 승인 흐름

편집자가 검토 요청 → 검토자 확인 후 의견 등록 → 승인자가 승인 또는 반려. 반려 시 작성자에게 알림 발송 후 재수정 흐름으로 재진입.

변경이력

버전, 수정일, 수정자, 주요 변경 사항을 이력 테이블로 관리한다. 승인 완료 버전은 잠금 처리되며 신규 버전 생성으로만 수정 가능하다.

D그룹(협업/검토/승인) 공통 레이어 전체 구조는 Sitemap & IA Overview 05절을 참조한다: STAM-Sitemap-IA-Overview-v1.html#layer
10

내보내기 / 납품

내보내기 대상 형식

승인 완료 화면설계서에 한해 PDF, Word(.docx), Excel(.xlsx) 형식 내보내기를 지원한다. Markdown Export는 E5 메뉴에서 처리한다.

고객 공유 링크

납품 단계에서 고객 공유 링크(E3)를 생성해 외부 열람 권한을 부여할 수 있다. 승인 상태가 아닌 화면은 공유 링크 생성이 제한된다.

메타데이터 완전 기입 선행

내보내기 시 15개 필수 필드가 모두 채워진 화면만 내보내기 대상에 포함된다. 미기입 필드가 있으면 경고 표시 후 내보내기가 차단된다.

납품 이력 관리

내보내기 실행 시 E4 납품 이력에 날짜·담당자·형식·버전이 자동 기록된다. 재납품 시 이전 이력이 누적 보존된다.

11

모바일 / 좁은 화면 고려

PC 기준 설계

화면설계서는 PC 테이블/그리드 레이아웃을 기본으로 한다. 컬럼 수, 고정 컬럼, 가로 스크롤 기준을 각 템플릿별로 명시한다.

모바일 전환 기준

980px 이하에서 테이블형 → 카드형으로 전환하거나 가로 스크롤을 허용한다. 전환 방식은 T01 List/Board, T02 Detail, T06 Approval 유형에서 우선 정의한다.

필드 우선순위

모바일에서는 핵심 필드(상태, 이름, 날짜)만 우선 표시하고, 부가 필드는 '전체 보기' 확장 액션으로 제공한다.

Drawer / Modal 처리

T11 Drawer/Modal 유형은 모바일에서 전체 화면 전환을 원칙으로 한다. 우측 Drawer가 80% 이상 화면을 덮는 경우 Full Screen 모드로 대체한다.
