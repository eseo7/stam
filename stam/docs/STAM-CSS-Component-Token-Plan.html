CSS Component Token Plan — 설계 단계 Plan

이 문서는 실제 CSS 구현이 아닌 적용 계획이다. Component Guide v1(7종)과 UI Baseline Guide 기준을 실제 제품 CSS 파일 구조로 옮기기 전에, 공통 토큰·공통 클래스·CSS 파일 역할·적용 순서를 먼저 합의하기 위한 문서이다.

Source — Component Guide v1 · UI Baseline Guide Target — stam/css/** (이번 작업에서는 미수정)
01 Overview — 문서 목적 Plan

STAM Component Guide v1은 컴포넌트의 외형·상태·인터랙션을 정의했지만, 실제 제품 CSS 파일 구조와 1:1로 매핑되어 있지 않다. 화면별로 버튼·폼·테이블 스타일이 따로 작성되는 문제를 막기 위해, 가이드 → CSS 클래스 → CSS 파일까지 이어지는 단일 매핑 기준을 먼저 문서화한다.

이 문서가 다루는 것
  • Component Guide와 실제 CSS 사이의 연결 기준
  • 공통 CSS 클래스 네이밍 후보 (.stam-*)
  • CSS 파일별 역할 정의
  • 토큰 카테고리 목록
  • 실제 적용 순서, Do / Don't, Risk / QA
이 문서가 다루지 않는 것
  • 실제 CSS 파일 신규 작성 / 수정
  • 실제 JS 컴포넌트 구현
  • 제품 화면(WBS / Project Overview / Home) 수정
  • nav-data, IA 구조 변경
  • Firebase / GitHub Actions / package.json 변경
설계 단계 — 구현 단계 아님 이 문서를 근거로 stam/css/** 를 바로 수정하지 않는다. 본 계획 합의 → 별도 작업 단위로 분할 → 화면 단위 점진 적용 순서로 진행한다.

03 CSS File Responsibility — 파일 역할 정의 정의만 / 미수정
주의 — 이번 작업에서는 CSS 파일을 만들거나 수정하지 않는다 아래 표는 각 파일이 어떤 역할을 맡아야 하는가에 대한 합의안이다. 실제 분리·이동·신규 생성은 별도 작업 단위에서 수행한다.
CSS File역할
stam/css/stam.tokens.css 색상, 폰트, radius, shadow, spacing, z-index 등 기본 토큰. 모든 파일이 참조하는 단일 소스.
stam/css/stam.shell.css 앱 셸 — Topbar, Left Nav, Guide Shell 레이아웃 골격. 컴포넌트가 아닌 구조.
stam/css/stam.components.css 버튼, 폼, 드로어, 다이얼로그, 토스트 등 공통 컴포넌트 후보. .stam-* 네임스페이스의 공용 클래스가 위치할 자리.
stam/css/stam.tables.css 테이블, 목록, WBS compact / focus view 등 표 계열 공통 규칙 후보.
stam/css/stam.forms.css 신규 후보 폼 필드, 라벨, 상태(error / disabled / focused), validation 메시지 등 폼 전용 규칙 후보. 현재는 파일 없음 — 별도 작업에서 분리 여부 결정.
stam/css/stam.pages.css 제품 페이지 공통 레이아웃, 페이지 컨테이너, 공통 헤더 / 푸터 등.
stam/css/stam.docs.css 문서 / 가이드 화면 전용 스타일 (Docs Index, Component Guide 등 정적 문서).
stam/css/stam.ui-baseline-guide.css UI Baseline Guide 문서 화면 전용. 다른 화면에서 재사용하지 않음.
stam/css/stam.wbs.css WBS 화면 전용 확장. 공통 테이블 규칙을 오염시키지 않는다.
stam/css/stam.project-overview.css Project Overview 화면 전용 확장. 공통 컴포넌트로 흡수되기 전 임시 스타일은 이 파일에서만 유지.
stam/css/stam.home.css Home 화면 전용 스타일. 공통화 후보가 발견되면 components.css / pages.css로 승격.
승격 / 강등 규칙 (제안) 화면 전용 파일에서 같은 패턴이 2개 화면 이상 반복되면 stam.components.css 또는 stam.tables.css로 승격. 공통 파일의 규칙이 한 화면에서만 쓰이고 있으면 화면 전용 파일로 강등.

05 Token Categories — 토큰 카테고리 tokens.css 기준

stam.tokens.css가 단일 소스로 보유해야 할 토큰 카테고리. 현 시점에 신규 토큰을 추가하지는 않는다.

Color
brand · surface · border · text · status (info/warn/fail/pass) · light/dark 매핑.
Typography
S-Core Dream weight / size scale · line-height · letter-spacing.
Spacing
4pt 그리드 기반 gap·padding·margin 스텝.
Radius
--r-sm · --r-md · --r-lg 단계.
Shadow
card / popover / modal 3단계. 컬러 그림자 사용 안 함.
Border
--bd · --bd-s · --bd-f 기본 / hairline / focus.
Z-index
topbar · dropdown · drawer · dialog · toast 5단계 정수 토큰.
Motion
120ms / 160ms / 200ms ease-out. spring / overshoot 사용 안 함.
Layout width
Left Nav 폭 · Drawer 폭 · 페이지 최대 폭.
Table density
compact / default / focus view 행 높이.
Form height
Input / Select / Textarea 표준 높이 (28 / 32 / 40).
Button height
sm / md / lg 표준 높이. Icon-only 정사각 별도.
06 Implementation Order — 적용 순서 제안 제안

아래 순서는 권장 순서이며, 각 단계는 별도 작업 단위로 분할해서 진행한다. 한 단계 안에 화면 단위 점진 적용을 포함한다.

  1. 1
    토큰 점검tokens.css의 카테고리 빠짐 여부 확인. 누락 시에만 보강. 색·radius·shadow 신규 도입 없음.
  2. 2
    공통 클래스 설계S04 매핑 표 기준으로 .stam-* 공통 클래스 시그니처 확정. 화면 적용 없음 (드라이런).
  3. 3
    Button 적용.stam-btn* 적용. 한 화면씩 교체하며 깨짐 확인.
  4. 4
    Form 적용.stam-form / .stam-field / .stam-input 적용. validation 상태 우선 1개 화면에서 검증.
  5. 5
    Drawer · Dialog 적용Drawer가 테이블을 밀어내지 않는지 우선 확인. Dialog Focus Trap은 JS 단계에서 별도 처리.
  6. 6
    Table 적용.stam-table* 적용. WBS 전용 스타일은 stam.wbs.css 안으로만 한정.
  7. 7
    Topbar · Icon 적용.stam-topbar*, .stam-icon* 적용. 기존 .doc-nav / .tbtn은 호환 유지 후 정리.
  8. 8
    WBS / Project Overview 점진 적용화면 전용 파일에서 공통 클래스로 흡수 가능한 규칙을 단계별로 이동. 한 PR에 한 화면.
07 Do / Don't — 적용 원칙 규칙
Do
  • 기존 가이드 문서를 먼저 확인한다.
  • 공통 클래스와 화면 전용 클래스를 분리한다.
  • CSS 수정 범위를 작게 나눈다 (한 PR에 한 컴포넌트 / 한 화면).
  • 제품 화면 적용 전 문서 / Preview에서 먼저 확인한다.
  • S-Core Dream 폰트를 유지한다.
  • 토큰만으로 표현 가능한 값은 토큰으로 작성한다.
  • 라이트 / 다크 양쪽에서 검증한다.
Don't
  • 화면별로 버튼 / 폼 / 테이블 스타일을 새로 만든다.
  • 공통 CSS와 화면 전용 CSS를 한 파일에 섞는다.
  • WBS 전용 스타일을 전체 컴포넌트 기준으로 오염시킨다.
  • 실제 제품 화면을 한 번에 전체 변경한다.
  • 레이아웃 속성을 불필요하게 대량 수정한다.
  • Pretendard CDN, Google Fonts, Gulim, Dotum 등 외부 / 시스템 폰트를 사용한다.
  • 색·radius·shadow를 토큰 우회로 하드코딩한다.
08 Risk / QA — 위험 요소와 검증 기준 체크

Risk — 위험 요소

항목설명
기존 화면 깨짐공통 클래스 도입 시 기존 화면 선택자와 충돌. 한 화면씩 점진 적용 필요.
WBS 가로 스크롤 재발테이블 공통 규칙 변경으로 WBS Compact view 가로 스크롤이 다시 생길 위험.
Drawer가 테이블을 밀어냄Drawer 위치가 absolute / fixed가 아닐 경우 테이블 폭이 축소되어 레이아웃 변형 발생.
폰트 혼입버튼 / 폼 / 테이블 적용 시 외부 폰트 fallback이 끼어들 수 있음. S-Core Dream 강제 방어 필요.
Button 상태 불일치hover / active / disabled / focus-visible 상태가 variant마다 어긋날 위험.
Form validation 스타일 불일치has-error 상태에서 라벨·input·help 색이 일관되지 않을 위험.
Z-index 충돌Drawer / Dialog / Toast의 z-index 토큰이 정의되지 않으면 순서가 화면별로 달라짐.

QA — 검증 기준

항목확인 방법
변경 파일 범위git diff --stat로 한 작업 단위에서 의도한 파일만 바뀌었는지 확인.
가로 스크롤페이지 레벨 가로 스크롤 0. 테이블 내부 스크롤만 허용.
라이트 / 다크 모드두 테마 모두 본문 / 보조 / 비활성 텍스트 대비가 가이드 기준을 만족.
폰트DevTools Computed → font-family에 S-Core Dream만 노출. 외부 폰트 미사용.
Button 상태primary / secondary / ghost / danger × hover / active / disabled / focus-visible 매트릭스 확인.
Form 상태default / focus / disabled / has-error / read-only 5가지 상태 확인.
Table 상태정렬 / 빈 상태 / 로딩 / focus row / sticky header 확인.
Drawer 상태opened / closed / fullview 전환 시 본문 레이아웃 변형 없음.
콘솔 오류대상 화면 콘솔 0 error / 0 warning.
git diff금지 파일(README.md · .gitignore · Firebase 설정 · GitHub Actions · package.json) 미변경.
