STAM Form Token Mapping v1 Mapping Draft

본 문서는 실제 Form CSS 구현 작업이 아니라, Form / Select / Date Picker 컴포넌트가 사용할 토큰의 field × size × state 매핑 기준을 사전 확정하는 매핑 문서이다. Token Inventory v1에서 후보로 제안된 Form State 토큰을 Input · Select · Date Picker · Checkbox · Radio · Validation 축으로 풀어서 표로 정리하며, 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/**를 절대 수정하지 않는다.

Mapping Draft CSS Implementation Hold Drawer / Full Detail Sync Required Pre-Implementation No CSS Edit

판정 근거 — Token Inventory v1에서 Form State P0 토큰(--input-*, --dropdown-*, --calendar-*, --field-*)이 후보로 제안되었으나, Drawer 수정 화면과 Full Detail 수정 화면이 같은 Form 컴포넌트를 공유해야 한다는 동기화 기준, native select / date input 회귀 차단 기준, dropdown / calendar popup z-index 충돌 차단 기준이 확정되지 않은 상태이다. 본 문서로 매핑과 동기화 원칙을 확정해야 Form CSS 구현 단위에 진입할 수 있다.

01 Overview — 문서 목적 목적

CSS Token Readiness Check가 “전체 토큰 충분성”을, Token Inventory v1이 “신규 토큰 후보 목록”을, Button Token Mapping v1이 “Button variant × size × state” 매핑을 확정했다면, 본 Form Token Mapping v1은 “Input · Select · Date Picker · Checkbox · Radio · Validation이 실제로 어느 토큰을 어떻게 쓰는지” field × size × state 축으로 1:1 매핑을 확정한다. 특히 Drawer 수정 화면과 전체 상세 수정 화면에서 동일한 Form 컴포넌트를 사용해야 한다는 동기화 원칙을 본 문서에서 동결하여, 두 화면 사이의 시각·인터랙션 회귀를 차단한다. 기획자·PM도 “어떤 필드가 어떤 토큰을 참조하는지”를 한 화면에서 읽고 승인할 수 있어야 하며, 개발자는 후속 Form CSS 구현 단위에서 본 문서를 그대로 클래스·토큰 참조 기준으로 사용한다.

  • Form state 매트릭스 (default / hover / focus / active / disabled / readonly / error / success / warning / required / filled / empty)
  • Select · Dropdown · Date Picker · Calendar 인터랙션 규칙

주의 — Mapping은 코드 변경이 아니다 본 문서의 Candidate 토큰명·라이트/다크 값·size 후보값은 모두 매핑 기준 후보이며, 실제 토큰 추가·CSS 클래스 작성·Form 스타일 구현은 후속 Form CSS 구현 단위에서 코드 변경과 함께 확정한다. 본 문서만으로 stam.tokens.css · stam.components.css 등 어떤 CSS 파일도 수정하지 않는다.
핵심 원칙 — Drawer / Full Detail은 같은 Form을 본다 STAM의 수정 흐름은 리스트 → Drawer 수정 → 전체 상세 → Drawer 수정 복귀로 순환한다. Drawer 수정 화면과 Full Detail 수정 화면은 동일한 Input · Select · Date Picker 마크업과 init 함수를 공유해야 하며, 두 화면에서 같은 필드가 다르게 보이면 본 매핑 위반으로 간주한다. 상세 규칙은 Section 09에서 정의한다.
02 Form Scope — 본 매핑이 다루는 Form 범위 범위

아래 13개 Form 요소가 본 매핑의 대상이다. 시각 컴포넌트와 보조 요소(Label · Required · Helper · Validation)를 모두 포함한다.

Text Input
단일행 텍스트 입력. WBS · 산출물 · 인사 등 모든 수정 화면 기본.
Textarea
다중행 입력. 설명 · 메모 · 회의록. resize 정책 별도 정의.
Select Box
단일 선택. native select 기본 UI 노출 금지.
Custom Dropdown
검색 · 다중 선택 · 그룹 옵션 등. STAM 공통 dropdown 컴포넌트.
Date Picker
단일 날짜 · 기간. native date input 회귀 금지.
Calendar Popup
Date Picker 클릭 시 노출되는 popup. z-index · 다크모드 별도.
Checkbox
단일 · 다중 선택. 체크 · 미체크 · indeterminate 3 state.
Radio
단일 선택 그룹. 그룹 내 1개만 선택.
Search Field
목록 검색 · 명령어 검색. 좌측 아이콘 · 우측 클리어 버튼.
Validation
error / success / warning 메시지. 색·아이콘·텍스트 3원칙.
Field Label
필드 상단·좌측 라벨. for 속성 필수.
Required Mark
필수 표시 (* 등). 색 단독 의존 금지.
Helper Text
필드 하단 보조 설명. 에러 메시지와 영역 공유.
File / URL Ref
산출물 첨부 · 외부 링크. 파일명 · 링크 미리보기 영역 정의.
본 문서가 다루지 않는 Form 요소 Toggle Switch, Stepper(증감 컨트롤), Range Slider, Color Picker, Rich Text Editor, Combobox(타이핑 가능한 select), Multi-select Tag Input은 v1 범위 밖이며 후속 단계에서 별도 매핑한다. Tag · Chip · Badge는 Form 필드가 아니므로 별도 컴포넌트로 매핑한다.
03 Form Field Matrix — 필드별 역할 · 토큰 의존성 매트릭스

Field는 사용 위치(Usage)와 Drawer / Full Detail에서의 사용 원칙(Sync Rule)으로 정의한다. Required Tokens는 해당 필드 구현에 필수인 토큰 그룹이며, 상세 토큰명은 Section 08에서 풀린다.

Field Type Usage Required Tokens Interaction Risk Drawer / Full Detail Rule Decision Status
Text Input WBS Task명 · 산출물명 · 사용자명 등 단일행 텍스트. --input-bg · --input-border · --input-text · --input-placeholder focus ring 누락 시 키보드 사용자 손실. 동일 markup · class · init 사용. 폭만 컨테이너로 조절. 후보
Textarea 설명 · 메모 · 회의록 · 변경 사유. --input-bg · --input-border · --input-text · line-height 토큰 resize 정책 불일치 시 Drawer 폭 깨짐. resize 정책(vertical only)을 두 화면에서 동일하게 적용. 후보
Select 담당자 · 상태 · 우선순위 등 enum 단일 선택. --dropdown-* · --input-* (트리거) native 파란 option UI 노출 시 디자인 위반. 두 화면 모두 custom select 사용. native <select> 직접 노출 금지. 후보

Checkbox 다중 선택 · 약관 동의 · indeterminate 표시. --input-border · --stam (selected) · focus ring hit-area 부족 · selected 시 대비 부족. 두 화면 모두 동일 체크 마크 SVG · 동일 사이즈 사용. 후보

File / URL Ref 산출물 파일 첨부 · 외부 URL 참조. --input-* · 파일명 영역 · 액션 버튼 긴 파일명 overflow · 보안 정책 충돌. 두 화면 모두 동일 첨부 컴포넌트 · 동일 파일명 표시 규칙. 후보
필드 선택 우선순위 한 필드에 대해 “Drawer는 select, Full Detail은 input” 같은 분기는 금지한다. 같은 데이터 항목은 같은 필드 타입으로 통일하며, 화면에 따라 폭·표시 형식만 컨테이너로 조절한다. Select / Date Picker는 항상 STAM custom 컴포넌트를 사용한다.

09 Drawer / Full Detail Sync Rule — 동기화 원칙 Sync

STAM 수정 흐름의 핵심 원칙이다. 이 섹션을 지키지 않으면 Drawer 수정과 Full Detail 수정에서 동일 필드가 다르게 보이거나, 한쪽에서만 native select / date input으로 회귀하는 시각 사고가 발생한다.

리스트 → Drawer 수정 → 전체 상세 수정 → Drawer 수정 복귀 같은 Form 컴포넌트를 일관되게 본다
Must동일성 원칙
  • 같은 데이터 항목은 두 화면에서 같은 markup을 사용한다.
  • 같은 필드는 같은 class를 가진다. Drawer 전용 · Full Detail 전용 class 분기 금지.
  • 같은 select / date picker는 같은 init function으로 초기화된다.
  • 크기 · radius · padding · focus ring은 본 문서 Section 04 / 08 토큰을 동일 적용한다.
  • Light / Dark 테마는 두 화면에서 같은 토큰 값을 사용한다.
Must Not금지 패턴
  • Drawer 전용 select / date picker를 별도 구현하는 것 — 한쪽만 변경되면 다른 쪽 회귀.
  • Full Detail에서 native <select> · <input type="date">로 회귀하는 것 — 디자인 위반.
  • Drawer에서는 custom dropdown, Full Detail에서는 ad-hoc 옵션 리스트로 분리하는 것.
  • 같은 필드의 size를 Drawer만 dense, Full Detail은 md로 다르게 두는 것.
  • 같은 데이터 항목의 label · helper · required 표기를 두 화면에서 다르게 두는 것.
Bind이벤트 / 상태 바인딩
  • Drawer → Full Detail 전환 후에도 select · dropdown · date picker 이벤트 바인딩이 유지되어야 한다.
  • 두 화면에서 동일한 validation 규칙을 사용한다 — error · success · warning 분기 동일.
  • 필드 값 저장 형식은 두 화면에서 동일하다 — date는 ISO 8601 YYYY-MM-DD.
  • 키보드 동작(Tab · Enter · Space · ESC · ↑↓)이 두 화면에서 동일하다.
  • Drawer 내부 dropdown / calendar popup은 portal로 렌더링되어 Drawer overflow에 의해 clipping되지 않는다.
QA회귀 판정 기준
  • “Drawer에서 보이는 Form”과 “Full Detail에서 보이는 Form”의 스크린샷 비교 — 시각 차이 발견 시 실패.
  • Drawer → Full Detail → Drawer 흐름에서 같은 필드의 UI · 동작이 다르면 실패.
  • Full Detail에서 native select / date input이 1개라도 노출되면 실패.
  • Drawer에서 dropdown / calendar popup이 Drawer 폭으로 잘리면 실패.
  • Drawer · Full Detail 중 한쪽만 다크모드에서 깨지면 실패.
본 Sync Rule의 적용 범위 본 Rule은 “편집 가능한 Form 컴포넌트”에 한정한다. Drawer가 read-only 표시를 사용하고 Full Detail이 read-only 표시를 사용하는 경우, 둘 다 같은 read-only 컴포넌트를 사용해야 한다. Drawer 자체의 폭 · 위치 · 탭 구조는 Drawer Guide의 책임이며, 본 문서가 정의하지 않는다.
10 Accessibility / Interaction Rule — 필수 준수 필수
Label · Input 연결
  • <label for="...">로 input과 라벨 1:1 연결
  • label을 시각적으로 숨기는 경우 aria-label 또는 aria-labelledby 필수
  • placeholder만으로 label을 대체하지 않음
  • required field는 시각 표시 + aria-required="true" 병행
키보드 · 포커스
  • 모든 Form 필드는 focus-visible ring 필수 — --btn-focus-ring 공유 적용
  • native focus outline을 무작정 제거하지 않음 — 대체 ring이 있을 때만 제거
  • Tab 키 순서가 시각 순서와 일치 · 그룹 내 ↑↓ 이동 지원
  • Enter / Space로 select · checkbox · radio 활성화 가능
Error · Validation
  • 에러는 색 + 아이콘 + 텍스트 메시지 3원칙 — 색상 단독 의존 금지
  • 에러 메시지를 aria-describedby로 input과 연결
  • aria-invalid="true"를 error 상태에 부여
  • 한 필드 한 시점에 하나의 validation 상태만 유지
Disabled · Readonly
  • disabled / readonly는 시각으로 명확히 구분 — opacity 단독 처리 금지
  • disabled는 cursor:not-allowed + pointer-events: none
  • readonly는 텍스트 선택 · 복사 가능 유지
  • aria-disabled · readonly 속성 일관 적용
Date Picker A11y
  • 날짜 popup은 keyboard만으로 사용 가능해야 함 — ↑↓←→ 일 이동 · PageUp/Down 월 이동
  • icon-only 캘린더 트리거에 aria-label 필수
  • popup 오픈 시 focus를 popup 내부로 이동
  • ESC로 popup 닫힘 · trigger로 focus 복귀
Dark Mode 대비
  • 라이트 / 다크 모두 텍스트 대비비 4.5:1 이상 보장
  • focus ring은 다크에서 더 밝은 톤·높은 알파로 보강
  • disabled · placeholder도 다크에서 식별 가능해야 함
  • dropdown · calendar popup 배경과 텍스트 대비 별도 검수
Select / Dropdown / Calendar Z-Index
  • dropdown · calendar popup의 z-index는 임의 숫자가 아닌 layer token으로 관리
  • Topbar < Drawer < Dialog < Dropdown popup 순서 보장
  • Drawer 내부 dropdown / calendar는 portal로 렌더링 — Drawer overflow에 의해 clipping 금지
  • Dialog 내부 dropdown / calendar도 portal로 렌더링 — Dialog backdrop 위에 표시
11 Do / Don't — 구현 원칙 원칙
Do
  • 모든 색·radius·padding은 토큰 기반으로 구현 (--input-*, --dropdown-*, --calendar-* 등)
  • Drawer · Full Detail 모두 같은 Form 컴포넌트(markup · class · init)를 재사용
  • Select · Date Picker는 STAM 공통 init 함수로 초기화 — Drawer · Full Detail 모두 호출
  • Drawer · Full Detail · Light · Dark 4개 상태 동시 QA
  • focus ring은 --btn-focus-ring 단일 토큰을 Button과 공유
  • disabled는 토큰 기반 색상 + cursor:not-allowed 동시 적용
  • dropdown · calendar popup은 portal 렌더링 + layer token 기반 z-index
  • error는 색 + 아이콘 + 텍스트 3원칙 동반
Don't
  • raw color (#FFFFFF, #CBD5E1) 직접 사용 — 모두 토큰 경유
  • Drawer와 Full Detail에 별도 select · date picker 구현 — 한 컴포넌트로 통일
  • native <select> · <input type="date"> 기본 UI에 의존 — 디자인 위반
  • z-index를 임의 숫자(9999 등)로 땜질 — layer token 사용
  • disabled를 opacity만으로 처리 — 토큰 + cursor 차단 병행
  • 한 번에 Input · Select · Date Picker · Checkbox · Radio 모두 동시 CSS 구현 — Form 내에서도 단계 분리
  • placeholder로 라벨 대체 · 색상 단독으로 error 표시
  • 화면별·페이지별 Form CSS 별도 작성 — 공통 클래스 단일화

  • 제품 페이지(Dashboard · WBS · Boards)의 기존 input · select 회귀 확인
롤백 기준
  • Docs 또는 제품 화면에서 시각 회귀 1건 이상 발생 시 즉시 revert
  • 토큰명 충돌(기존 --stam-*, --btn-* 등) 발생 시 본 문서 v2로 재합의
  • Drawer · Full Detail 중 한 화면이라도 회귀 시 즉시 revert
  • 라이트 / 다크 중 한 테마에서만 깨질 경우 즉시 revert
  • native select · date input 회귀가 1건이라도 발견되면 즉시 revert
핸드오프 시 명시 사항 본 문서는 매핑 기준 문서이며, CSS·JS·페이지·자산 변경 권한을 포함하지 않는다. Cursor는 Section 08 Token Mapping 표와 Section 09 Sync Rule만 코드로 옮기며, 새로운 토큰명·값을 임의로 추가하지 않는다. Drawer 전용 · Full Detail 전용 분기 구현은 본 문서 위반으로 간주한다.
