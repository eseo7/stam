01 Purpose — 이 문서의 목적 Preview Baseline

이 문서는 STAM 의 핵심 제품 흐름인 요구사항 → 기능정의 → 메뉴/화면 → WBS → 화면설계서 → 테스트 시나리오 연결 상태를 한눈에 보는 Traceability Matrix UX 를 정의한다. Import Builder / Text Intake Helper 로 생성된 요구사항 Draft 가 이후 산출물과 어떻게 연결되고 어디가 누락됐는지를 보여주는 기준 화면이다.

  • 이 화면은 연결 상태 표시 · 누락 감지 · 생성 액션 제안까지만 담당한다.
  • 산출물을 자동으로 생성하거나 자동 수정하지 않는다.
  • Draft / Review Needed 요구사항은 사람이 승인하기 전까지 연결을 확정하지 않는다.
  • 이번 작업은 정적 UX Preview 이며 실제 제품 화면 / JS / CSS / API / DB / 연결 로직을 구현하지 않는다.
  • 이 Preview 는 후속 실제 구현의 기준 화면으로 사용한다.
정적 UX Preview 명시 Traceability Matrix 는 STAM 의 핵심 차별화 화면이지만, 이번 PR 은 정적 UX Preview 다. 실제 연결/생성/수정은 후속 구현에서 Product Screen Implementation Gate 를 적용해 진행한다.
02 Product Position — 왜 Traceability Matrix가 핵심인가 Differentiator

산출물 게시판은 각각 존재한다. 하지만 “요구사항이 실제로 기능·화면·WBS·테스트까지 연결됐는가”를 한 화면에서 확인하는 곳은 없다. Traceability Matrix 는 그 연결을 한 표로 모아 누락과 영향 범위를 드러낸다.

매트릭스가 없을 때
  • 요구사항과 산출물이 분리된 목록으로만 존재
  • 어떤 요구사항이 테스트까지 갔는지 알기 어려움
  • 변경 시 영향 범위를 수동으로 추적
  • 누락이 늦게 발견됨
매트릭스가 있을 때
  • 요구사항별 연결 상태를 한 행으로 확인
  • 누락(Missing) 셀을 즉시 식별
  • 변경 영향(Change Impact)을 한 화면에서 검토
  • 고객/내부 검토의 기준 화면으로 사용
핵심 차별화 Traceability Matrix 는 “산출물 모음”을 “연결된 한 사이클”로 바꾸는 화면이다(Project Cycle Automation 8장과 연결). 단, 연결을 표시·감지·제안할 뿐 자동 확정하지 않는다.
03 User Scenario — 실무 시나리오 Scenario
엑셀/텍스트로 요구사항 Draft 생성→ Matrix 에서 연결 상태 확인→ Missing 셀 식별→ 생성/연결 액션 제안 확인
사람이 검토·승인→ 연결 확정→ 추적성 갱신

PM/기획자는 Matrix 에서 “요구사항 12건 중 테스트까지 연결된 것은 7건, 기능정의 없는 것 2건, 변경 영향 있는 것 1건”과 같은 상태를 한눈에 본다. 시스템은 누락을 표시하고 생성 액션을 제안하지만, 실제 연결은 사람이 승인할 때 확정된다.

09 Missing Link Detection — 누락 감지 Detection

시스템은 연결이 없는 단계를 누락 상태로 표시하고 생성 액션을 제안한다. 자동 생성하지 않는다.

누락 유형표시제안 액션 (자동 생성 아님)
기능정의 없음Functional Definition = Missing기능정의 연결/생성 제안
메뉴/화면 없음Menu / Screen = Missing메뉴/화면 연결/생성 제안
WBS 없음WBS = MissingWBS 연결/생성 제안
화면설계서 없음Screen Specification = Missing화면설계서 연결/생성 제안
테스트 시나리오 없음Test Scenario = Missing테스트 연결/생성 제안
누락 ≠ 오류 Missing 은 자동 오류로 단정하지 않는다. 사람이 “아직 필요 없음 / 범위 외 / 후속 생성”을 판단한다. 제안은 “생성 액션 제안”까지이며 실제 생성은 사람 승인 후 별도 화면에서 진행한다.
10 Change Impact View — 변경 영향 Impact

요구사항이 변경되면 연결된 산출물에 영향 가능성을 표시한다. 실제 자동 수정은 하지 않으며, 상태는 Impact Review Needed 로 표시한다.

11 Draft / Review Needed Handling Human-in-loop
Draft 는 추적 완료가 아니다 Import Builder 또는 Text Intake Helper 에서 들어온 Draft 요구사항은 바로 추적 완료로 보지 않는다. Draft / Review Needed 상태에서는 연결 추천·생성 액션이 제한되거나 경고로 표시된다.

15 Relationship with Import Builder Mapping

Excel Import Builder 로 만든 요구사항 Draft 는 Matrix 의 한 행으로 들어온다. 들어온 직후 연결은 대부분 Missing 이며, reviewStatus 가 Review Needed 이므로 승인 전까지 연결 확정이 제한된다.

  • Import Builder 결과 = Draft Import → Matrix 행으로 표시
  • 표준 필드(title/description/priority 등)는 Row Anatomy 에 매핑
  • sourceType = Excel Import · 원본 메타(sourceFile/sheet/row) 연결
  • 승인 후 기능정의/화면/WBS/테스트 연결을 사람이 확정
기준 유지 Import Builder 결과는 확정 데이터가 아니라 Draft 에서 시작한다. Matrix 는 이를 “추적 완료”로 보지 않는다.
16 Relationship with Text Intake Helper Mapping

Text Intake Helper 로 만든 요구사항 Draft 도 Matrix 의 한 행으로 들어온다. 자유 텍스트는 AI 없이 의미 분석하지 않으며, Rule-based 블록 분리 + 사람 선택으로 만들어진 Draft 임을 그대로 유지한다(PR #227 기준과 충돌하지 않음).

  • Text Intake 결과 = Imported Draft / Review Needed → Matrix 행으로 표시
  • sourceType = Text Intake · extractionMode = Rule-based / Manual
  • 원본 메타(sourceText/sourceBlockId/라인 범위) 연결
  • 시스템은 원문을 “이해”하지 않으므로, 연결도 사람이 승인 후 확정
충돌 없음 “자유 텍스트를 AI 없이 자동 의미 분석한다”는 표현을 쓰지 않는다. Matrix 는 Text Intake 의 Draft 를 연결 상태로만 표시하며, 자동 추출/자동 분류는 후속 확장(AI 연동 후)으로만 다룬다.

  • 커버리지 리포트 / 통계
  • 버전 간 추적성 비교
  • 고객 제출용 추적성 문서 생성
20 Guardrails — 가드레일 Guardrails
  • 연결 상태를 표시하는 화면이지, 자동으로 모든 산출물을 생성하는 화면이 아니다.
  • Missing 상태를 자동 오류로 단정하지 않는다.
  • Out of Scope 요구사항은 누락으로 보지 않는다.
  • Draft 요구사항은 승인 전까지 연결 확정을 금지한다.
  • AI 자동 생성 / 자동 수정 표현을 쓰지 않는다 (후속 확장 / AI 연동 후로만 표기).
  • 사람 승인을 전제로 유지한다.
  • 자유 텍스트는 AI 없이 의미 분석하지 않는다 (Text Intake 기준 유지).
  • 실제 DB / API / Storage / 연결 로직은 이 단계에서 구현하지 않는다.
  • 제품 화면은 변경하지 않는다 (docs 전용 UX Preview).
