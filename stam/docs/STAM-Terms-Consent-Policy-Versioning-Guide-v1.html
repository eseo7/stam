01 Overview / Purpose 목적
STAM POLICY Guide 기준 검토 필요
약관/동의/고지 문서를 버전 중심으로 일관 관리하기 위한 STAM 문서 가이드

본 가이드는 회원가입, 정책 변경 공지, 재동의 요청, 철회 처리, 이력 조회를 하나의 버전 모델(MAJOR.MINOR)로 연결하기 위한 기준안입니다. 제품 구현 상세, 최종 DB 확정, 법률 자문 결과를 대체하지 않습니다.

03 필수 / 선택 / 고지 구분 구분
필수 (Required)
서비스 제공의 핵심 법적 근거가 되는 항목. 회원가입 완료 전 동의 확인이 필요합니다.
예: 이용약관, 개인정보처리방침 핵심 조항
선택 (Optional)
미동의해도 핵심 서비스는 이용 가능하나, 마케팅/부가 기능이 제한될 수 있습니다.
예: 마케팅 수신, 선택 쿠키
고지 (Notice)
동의가 아닌 공지 중심 전달 항목. 변경 사실 및 시행일을 추적 가능한 로그로 남겨야 합니다.
예: 운영 정책 변경 공지, 일반 안내
04 Version Model: MAJOR.MINOR Version
MAJOR재동의 필요 변경
사용자 권리/의무, 개인정보 처리 범위, 제3자 제공 범위 등 실질 영향이 큰 변경은 MAJOR를 증가시키고 재동의 절차를 트리거합니다.
예: 2.3 → 3.0
MINOR고지 중심 변경
문구 정비, 링크 수정, 오탈자 보정, 법령명 표기 보완 등 실질 동의 범위를 바꾸지 않는 변경은 MINOR를 증가시키고 고지로 처리합니다.
예: 2.3 → 2.4
Rule of Thumb: MAJOR/MINOR 판정 기준은 법무·개인정보보호·보안 책임자 검토가 필요합니다. 본 기준은 Guide 기준 / 검토 필요입니다.
05 Lifecycle: draft → published → superseded → archived Lifecycle
1draft
2published
3superseded
4archived
draft
작성/검토 단계. 사용자 노출 없음.
published
현행 버전. 가입/동의/고지 기준점.
superseded
신버전으로 대체된 과거본. 참조 가능.
archived
규정 보관기간 이후 아카이브. 운영 조회 제한.

12 Bad Example Section Anti-Pattern
지양해야 할 패턴
  • 버전 없이 본문을 덮어써서 동의 시점 추적이 불가능한 구조
  • MAJOR 변경인데 고지만 수행하여 재동의 근거가 사라지는 운영
  • 필수/선택/고지 구분 없이 체크박스를 단일 그룹으로 처리하는 UI
  • 정책 문서와 사용자 동의 이벤트를 동일 레코드로 섞어 저장하는 방식
  • 법률 검토 전 문구를 확정 문안처럼 배포하는 프로세스
13 DB Impact 후보 4개 DB Candidate
Table (Candidate) 역할 핵심 필드 예시 상태
policy_documents 정책 문서 메타(유형/소유/표시명) 관리 policy_type, document_key, owner_team 후보
policy_versions 버전 본문/상태/lifecycle 관리 major, minor, status, effective_at 후보
user_consents 사용자 동의 현재 상태(최신 포인터) user_id, policy_version_id, consent_state 후보
consent_events 동의/철회/재동의/고지확인 이벤트 로그 event_type, actor, occurred_at, ip_hash 후보
Important: 위 항목은 영향 분석용 후보이며, 최종 DB 스키마 확정안이 아닙니다. 본 문서는 DOCS ONLY 범위입니다.
14 법률 / ISMS-P 매핑 (Guide 기준) 검토 필요
법률 관점 매핑 (초안)
- 이용약관: 권리/의무/분쟁 관련 조항 변경 추적
- 개인정보처리방침: 처리 목적/항목/보유기간 변경 이력 관리
- 제3자 제공: 제공 대상/항목/기간 변경 시 MAJOR 후보
ISMS-P 관점 매핑 (초안)
- 정책 문서의 최신성, 변경관리, 고지 이력의 증적성 확보
- 동의/철회 이벤트 로그의 추적 가능성과 무결성 확보
- 운영자 콘솔의 권한 분리와 감사 흔적 연계 필요
