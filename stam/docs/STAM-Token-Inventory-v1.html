이 문서는 실제 CSS 구현 작업이 아니라, stam/css/stam.tokens.css에 반영할 STAM 공통 토큰의 목록·역할·값 후보를 사전 정리하는 인벤토리 문서이다. CSS Token Readiness Check에서 도출된 Gap·Decision 항목을 토큰 단위로 풀어서 표로 만든다. P0 항목(Shadow · Z-index · Border Focus · Button State · Form State)을 우선 확정하며, 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/assets/**를 수정하지 않는다.

판정 근거 — Color · Radius · Border color · Layout Width 등 v1 진입에 필요한 atomic 토큰은 정의되어 있어 인벤토리 작성이 가능한 상태이다. 단, Shadow · Z-index · Border Focus · Button State · Form State 등 P0 카테고리의 토큰명·라이트값·다크값이 본 문서로 확정되기 전까지 stam.tokens.css 수정·컴포넌트 CSS 구현·제품 화면 변경은 보류한다.

01 Overview — 문서 목적 목적

CSS Token Readiness Check가 “지금 토큰이 충분한가”를 점검했다면, 본 Token Inventory v1은 “구현 전에 추가·정의할 토큰의 목록과 값 후보가 무엇인가”를 표로 확정한다. 기획자·PM도 “어떤 토큰이 왜 필요한지”를 같은 화면에서 읽고 승인할 수 있어야 하며, 개발자는 후속 단계에서 본 문서를 그대로 stam.tokens.css 반영 기준으로 사용한다.

이 문서가 다루는 것
  • 현재 stam.tokens.css 토큰 인벤토리 분류
  • P0 카테고리 신규 토큰명·역할·값 후보
  • 컴포넌트별 토큰 사용 매핑
  • 라이트 / 다크 토큰 운영 원칙
  • 토큰 네이밍 규칙
  • 실제 CSS 구현 전 Hold Checklist
  • Cursor 핸드오프 기준
이 문서가 다루지 않는 것

  • stam/js/**, stam/pages/**, stam/index.html 변경
  • stam/assets/** 변경
  • Component Guide / UI Baseline / Token Plan 본문 수정

선행 문서와의 관계 · UI Baseline Guide — 폰트·레이아웃·Forbidden Rules 기준

주의 — Inventory는 코드 변경이 아니다 본 문서의 Candidate 토큰명·라이트/다크 값은 후보이며, 실제 토큰 추가·값 결정은 후속 작업 단위에서 코드 변경과 함께 확정한다. 본 문서만으로 stam.tokens.css를 수정하지 않는다.
02 Token Priority Summary — P0 / P1 / P2 우선순위

신규 토큰 카테고리를 우선순위 3단계로 나눈다. P0는 본 문서로 확정 후 즉시 CSS 구현 단위에 진입할 수 있어야 하며, P1·P2는 별도 인벤토리 단위에서 후속 확정한다.

P0컴포넌트 CSS 구현 전 필수
  • Shadow / Elevation — 카드 · 팝오버 · Drawer · Dialog · Focus
  • Z-index / Layer — Sticky · Topbar · Drawer · Dialog · Toast 위계
  • Border Focus — focus-visible ring · 두께 · 색 · offset
  • Button State — primary / secondary / danger × hover / active / disabled
  • Form State — input bg / border / placeholder / disabled / error / success
P1주요 컴포넌트 구현 전 확정
  • Table Density — row · header · cell padding · compact/default
  • Drawer Width / Overlay — 폭 · max · backdrop alpha
  • Dialog Size / Backdrop — sm/md/lg · backdrop alpha
  • Topbar Height / Gap — height · 좌우 gap · action 영역 폭
  • App Shell Layout Gap — 본문 padding · 사이드바 gap
  • Scrollbar / Sticky Header — sticky offset · scrollbar 두께
P2후속 확장
  • Motion / Transition — duration · easing · reduced-motion
  • Badge / Chip 상태 — info · success · warn · danger · neutral
  • Toast / Alert — 위치 · 폭 · z-index · 자동 닫기 시간
  • Empty State — placeholder bg · 텍스트 위계
  • Skeleton / Loading — base bg · shimmer alpha · duration
  • Mobile Breakpoint — bp-sm / bp-md / bp-lg 단계
우선순위 부여 기준 P0은 컴포넌트 CSS 구현의 “기본 골조”에 해당하는 토큰(없으면 구현 자체가 불가), P1은 “주요 컴포넌트(Drawer · Dialog · Table · Topbar)” 구현 전에 필요한 토큰, P2는 v1 안정화 이후 보강해도 무방한 토큰으로 분류한다.

06 Light / Dark Token Rule — 운영 원칙 원칙

라이트 / 다크는 동일 토큰 키를 유지하고, [data-theme="dark"] 블록에서 값만 재정의한다. 컴포넌트 CSS는 항상 토큰을 통해서만 색을 참조한다.

Light Theme
  • 기준 surface: --bg-base · --bg-sur · --bg-sur2
  • label / text: 단단한 hex 값 (#0F172A · #475569)
  • status: 단단한 hex bg/fg 쌍
  • focus ring: brand 색 + 30% alpha shadow
  • backdrop: rgba(15,23,42,.40) 후보
Dark Theme
  • 기준 surface: charcoal/navy (#13161D · #1C2030 · #222638)
  • label / text: alpha 기반 (rgba(230,232,240,.88))
  • status: alpha bg + 밝은 fg 쌍
  • focus ring: accent 색 + 40% alpha shadow
  • backdrop: rgba(0,0,0,.55) 후보
Do — 권장 운영
  • 같은 토큰명을 라이트·다크에서 유지한다.
  • [data-theme="dark"] 블록에서 값만 재정의한다.
  • 컴포넌트 CSS에서는 hex / rgba 직타이핑 금지, 항상 토큰 사용.
  • focus / ring / hover는 라이트·다크 모두 별도 확인한다.
  • native UI(select option · date input)는 라이트·다크에서 일관 외형이 되도록 별도 처리한다.
Don't — 금지
  • 다크 전용 신규 토큰명을 새로 만들지 않는다 (값만 재정의).
  • 컴포넌트 CSS에 raw hex / rgba를 직타이핑하지 않는다.
  • 다크 모드에서 native blue select option이 노출되도록 두지 않는다.
  • focus 색을 라이트에서만 검증하고 다크 확인을 생략하지 않는다.
  • 라이트 / 다크 간 토큰명을 분리(--bg-light / --bg-dark) 하지 않는다.
주의 — Native UI 회귀 Select option · Date input의 native UI가 다크 테마에서 파란색 박스로 노출되는 회귀가 가장 흔한 이슈다. 본 문서에서 native UI 제거 기준을 후속 작업 단위로 분리해 별도 매핑 문서에서 다룬다.
07 Naming Rule — 토큰명 규칙 규칙

토큰명은 “어디서 어떻게 쓰이는가”가 아니라 “무엇인가 / 어떤 역할인가”를 표현한다. 색 raw 이름·화면명 기반 이름은 사용하지 않는다.

권장 — 의미 기반 / 컴포넌트 별칭
  • --input-border-focus — 역할 기반
  • --btn-primary-bg-hover — 컴포넌트·variant·state
  • --shadow-dialog — 컴포넌트 elevation
  • --z-drawer — 레이어 역할
  • --ring-danger — 의미 기반 ring
  • --label-strong — 위계 기반 label
비권장 — raw color / 화면명 기반
  • --purple-500-input-line — raw color 직타
  • --wbs-row-blue — 화면명 기반
  • --home-card-shadow — 화면명 기반
  • --bg-light / --bg-dark — 테마 분리
  • --button1-bg — 의미 없는 숫자
  • --color1, --c-a — 추상 표기
네이밍 분류
  • Atomic — --bg-base, --bd, --t1, --brand · 화면 / 컴포넌트 무관
  • Semantic — --label-strong, --pass, --ring-focus · 역할 기반
  • Component Alias — --btn-primary-bg, --input-border-focus · 컴포넌트 전용
  • 화면 전용 토큰은 화면 CSS 파일에 한정해 정의하고 전역 토큰을 오염하지 않는다.
참조 우선순위
  • 컴포넌트 CSS → Component Alias 토큰 우선 사용.
  • Alias 토큰이 없는 경우만 Semantic 토큰 사용.
  • 화면별 CSS는 Atomic 토큰 사용 가능. 단 신규 의미가 필요하면 토큰 추가 검토.
  • raw hex / rgba 직타이핑 금지.
08 Implementation Hold Checklist — CSS 구현 보류 기준 Hold

아래 8개 항목이 본 문서 또는 후속 매핑 문서에서 확정되기 전까지 stam/css/** 토큰 추가·컴포넌트 CSS 구현·제품 화면 변경을 보류한다.

1
P0 토큰명 확정 Shadow · Z-index · Border Focus · Button State · Form State 5개 그룹의 토큰명을 본 문서로 확정한다.
2
라이트 값 후보 확정 P0 각 토큰의 라이트 테마 값이 후보 → 확정으로 전환되어 있어야 한다.
3
다크 값 후보 확정 동일 토큰의 다크 테마 값이 후보 → 확정으로 전환되어 있어야 한다.
4
Button 적용 범위 확정 Button variant / size / state별 토큰 매핑이 별도 Button Token Mapping 문서로 정리되어 있어야 한다.
5
Form 적용 범위 확정 Input · Select · Textarea · Validation 토큰 매핑이 Form Token Mapping 문서로 정리되어 있어야 한다.
6
Select / Date Picker native UI 제거 기준 확정 native option · native date 외형이 다크 모드에서 깨지지 않도록 제거 / 대체 기준을 후속 문서로 정리.
7
z-index 충돌 기준 확정 Topbar · Drawer · Dialog · Toast · Popover 5개 레이어의 z-index가 충돌 없이 확정.
8
Drawer / Dialog layer 기준 확정 Drawer < Dialog 위계, 동시 노출 시 backdrop · shadow · ESC 동작 기준이 정리되어 있어야 한다.
Hold 미해소 시 8개 조건 중 하나라도 미충족이면 stam.tokens.css 토큰 추가, stam.components.css 컴포넌트 CSS 구현, 제품 페이지 변경에 진입하지 않는다. 본 문서를 갱신하거나 결정 항목을 후속 작업 단위로 분리한다.
09 Cursor Handoff Notes — 후속 작업자 인계 기준 인계

Cursor 또는 다른 작업자에게 실제 CSS 구현을 넘길 때, 본 인벤토리를 그대로 입력 자료로 사용한다. 한 번에 전체 CSS를 구현하지 않는다.

Pre-Edit — 작업 진입 전
  • 한 번에 전체 CSS 구현 금지. 한 컴포넌트만 다룬다.
  • 첫 구현 대상은 Button 또는 Form 중 하나만.
  • 수정 허용 파일: stam/css/stam.tokens.css · stam/css/stam.components.css.
  • 금지 파일(stam/js/**, stam/pages/**, stam/index.html, stam/assets/**, Firebase / Actions / package*.json) 변경 금지.
  • 토큰 추가·변경 사유를 PR 또는 커밋 메시지에 명시한다.
  • git status clean 상태에서 시작한다.
Post-Edit — 작업 직후 QA
  • 배포 전 git diff --stat으로 변경 파일 범위 확인.
  • Firebase staging 배포 후 staging URL QA(stam-design-staging.web.app).
  • 라이트 / 다크 양 테마 QA.
  • 1440 / 1600 / 1920 가로폭에서 페이지 레벨 가로 스크롤 없음 확인.
  • native select / date input 회귀 QA — 다크에서 파란색 박스 노출 없음.
  • 금지 파일 변경 없음 — stam/css/** 외 파일 diff 0건 확인.
  • S-Core Dream 외 외부 폰트 호출 없음 — Google Fonts / Pretendard CDN / Gulim / Dotum 0건.
핸드오프 원칙 본 문서가 정한 범위 외 변경이 필요하면, 코드 수정 전에 본 문서를 먼저 갱신한다. 범위를 임의 확장하지 않는다. 본 문서의 판정(Inventory Draft)은 Section 08 Implementation Hold 8개 항목 해소를 전제로 한다.
10 Recommended Next Steps — 단계별 작업 순서 계획

CSS 구현은 한 번에 진행하지 않는다. 아래 6단계를 작은 단위로 끊어 진행하며, 각 단계는 작업 단위 1건에 대응한다.

1
Step 1 — Token Inventory v1 문서 리뷰 본 문서의 P0 후보 목록·라이트/다크 값을 PM·기획자·개발자가 함께 리뷰.
2
Step 2 — P0 토큰명 / 값 승인 Decision Status를 “후보 → 확정”으로 전환. 본 문서를 v1 confirmed 상태로 갱신.
3
Step 3 — Button Token Mapping 문서 작성 Button variant / size / state별 토큰 매핑을 별도 문서로 작성. CSS 구현 진입 전 마지막 게이트.
4
Step 4 — Button CSS 제한 구현 stam.tokens.css에 Button 관련 토큰만 추가 → stam.components.css에 Button 클래스만 정의. 다른 컴포넌트 동시 작업 금지.
5
Step 5 — Form / Select / Date Picker Token Mapping 작성 Button QA 통과 후, Form 계열 컴포넌트의 토큰 매핑을 별도 문서로 작성.
6
Step 6 — Form 계열 CSS 제한 구현 Input · Select · Textarea · Validation CSS를 토큰 기준으로 제한 구현. native UI 제거 기준 적용. 이후 Dialog / Drawer / Table 순서로 1개씩 확장.
속도 기준 한 컴포넌트 단위는 토큰 → 매핑 문서 → 클래스 → 화면 적용 → QA를 한 묶음으로 처리한다. 여러 컴포넌트 CSS를 동시에 손대지 않는다. 본 문서 갱신 없이 다음 단계로 임의 진입하지 않는다.
