STAM Button CSS QA Result Preview v1 QA Result Button CSS Foundation PASS Product Not Yet

본 문서는 Button CSS Implementation Preview v1이 시각 기준으로 고정한 결과와 commit e53abb3로 반영된 실제 Button CSS 제한 구현 결과를 비교하는 QA Result 문서이다. token · class · variant · size · state · regression · light/dark · staging 관점에서 통과 여부를 기록하며, 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/**를 단 한 줄도 수정하지 않는다.

판정 근거 — Button Token Mapping v1에서 정한 토큰값과 P0 Decision Approval Sheet v1의 결정이 stam.tokens.css·stam.components.css 두 파일에만 반영되었고, .stam-btn 공통 클래스가 variant 7종 / size 4종 / state 처리 / loading spinner / light·dark 토큰 대응까지 포함해 새로 추가되었다. 제품 페이지에 .stam-btn 치환은 아직 시작하지 않았다.

01 Overview — 문서 목적 목적

Button CSS Implementation Preview v1이 “구현 후 어떤 화면이 나와야 하는가”를 정적 시각 기준으로 고정했다면, 본 QA Result Preview는 “그 기준이 실제 구현 결과에서 충족되었는가”를 기록한다. commit e53abb3의 두 파일 diff(stam.tokens.css·stam.components.css)를 기준으로, 토큰·클래스·variant·size·state·회귀·라이트/다크·staging 8개 축에서 PASS / PASS WITH MANUAL CHECK / NOT APPLIED YET / FAIL을 부여한다. 본 문서는 다음 단계인 제품/Docs 일부 .stam-btn 적용 직전의 Gate 자료로 사용한다.

이 문서가 다루는 것
  • commit e53abb3의 token / class 추가 결과 QA
  • Button CSS Implementation Preview v1과의 시각 차이 점검
  • variant 7종 · size 4종 · state 7종 PASS 여부 매트릭스
  • 기존 .tbtn · .sbadge · .bcp-* 회귀 여부
  • light / dark 모두 token 분리 적용 여부
  • Staging URL · 배포 CSS 내 token / class 존재 확인
  • 제품 적용 가능 여부(Foundation Ready) 판정
  • 다음 단계 .stam-btn 첫 적용 후보 1개 선정 가이드
이 문서가 다루지 않는 것
  • stam/css/** 추가 토큰·클래스 작성·값 변경
  • stam/js/**, stam/pages/**, stam/index.html 변경
  • stam/assets/** 변경
  • Form · Dialog · Drawer · Table 등 다른 컴포넌트 QA
  • Firebase · GitHub Actions · package*.json 변경
  • 실제 제품 App Shell · 페이지에 .stam-btn 부착
Button CSS Implementation Preview v1과의 관계 · Preview v1의 Section 02~06이 “있어야 하는 시각 결과”라면, 본 문서는 그 시각 결과가 “실제로 보였는가”를 기록한다.
· Preview v1에서 사용된 격리 변수 --bcp-*는 본 구현에서 실제 토큰 --btn-*로 치환되었다.
· Preview v1에서 사용된 격리 클래스 .bcp-btn은 본 구현에서 공통 클래스 .stam-btn으로 치환되었다.
· 따라서 본 QA Result는 Preview v1을 좌측, 실제 구현된 .stam-btn 화면을 우측에 띄워 비교하는 방식을 권장한다.
주의 — QA Result는 코드 변경이 아니다 본 문서는 정적 HTML 문서 생성만 수행한다. stam.tokens.css·stam.components.css를 추가로 수정하지 않으며, 제품 페이지에 .stam-btn을 부착하지 않는다. CSS / JS / 제품 화면 변경이 필요하다면 별도 작업지시서로 분리한다.

State “focus-visible”에 PASS WITH MANUAL CHECK가 붙는 이유 CSS 단위 점검에서는 :focus-visible 분기가 정확히 작성되었지만, “마우스 클릭만으로 ring이 떠오르지 않는가”는 실제 staging 브라우저에서 키보드 Tab 이동 시 한 번 더 수동 점검을 권장한다. 본 결과는 PASS이며, MC는 가시성 검증 절차에 한해 적용된다.
Variant 결과 요약
  • Primary · Secondary · Danger는 색·hover·active·disabled까지 Preview v1과 완전 동일
  • Ghost · Text는 background transparent + hover 시 surface fill 패턴 동일
  • Icon-only는 size별 정사각형 width 차원 (24/28/32/40) 모두 일치
  • aria-label 필수성은 Class 자체가 아니라 사용처 가이드라인에서 강제
Size 결과 요약
  • xs / sm / md / lg 각각 height · padding · font-size · icon · gap 모두 Preview v1 표 그대로
  • border-radius는 모두 var(--r-md) = 8px
  • Form input(32–36px) 옆에서 md=32px 정렬은 staging 시각 점검 권장
State 결과 요약
  • default → hover → active 색 단계 흐름 동일
  • focus-visible 외곽 3px ring (--btn-focus-ring) 적용
  • disabled는 opacity 단독 아닌 토큰 기반 색 + pointer-events 차단
  • loading은 spinner + aria-busy 호환 구조 유지
Manual Check 항목
  • focus-visible — 키보드 Tab만으로 ring 떠오르는지 staging 점검
  • loading — 실제 클릭 시 중복 클릭 차단되는지 (CSS는 OK)
  • 대비비 4.5:1 — 본 문서는 토큰 기준 OK, 실제 색상 측정 별도 권장
