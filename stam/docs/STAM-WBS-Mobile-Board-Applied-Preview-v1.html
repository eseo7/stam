02 Mobile WBS IA Summary — 6 화면 구조 IA

모바일 WBS는 PC의 단일 테이블 화면을 다음 6개 모드 + 보조 2개로 분리한다. 기본 진입은 WBS List (카드형)이며, 검색 / 필터는 모두 List 위에서 동작한다. 상세 / 등록 / 수정은 Fullscreen 으로 분리되어 PC Drawer 와는 다른 UX 로 처리된다.

화면 표시 형태 역할 진입 경로
WBS List 카드형 리스트 모바일 기본 진입. 작업 카드 스크롤로 탐색 Nav drawer → WBS 작업 / 홈 카드 / 외부 링크
Search / Filter Inline expand + Bottom sheet 카드 리스트 위에서 검색 / 필터 조건 합성 Topbar 검색 icon · 필터 chip · Bottom sheet
Card Detail Fullscreen panel 한 작업 카드의 상세 / 댓글 / 이력 / 산출물 카드 탭
Create / Edit Fullscreen form 작업 등록 또는 기존 작업 수정 FAB + · Bottom sticky 작업 등록 · 상세 → 수정
Full Table View 가로 스크롤 테이블 (전용 화면) 모든 컬럼이 필요한 점검 / 검수 시 모드 전환 List 더보기 → 전체 테이블 보기
Nav Drawer 좌측 80% slide-in 전역 메뉴 진입 · 메뉴 검색 포함 Topbar LEFT 햄버거
Bottom Action / FAB Sticky / 부유 버튼 등록 / 필터 / 보기 모드 전환 등 화면 액션 화면 하단 sticky · 우하단 FAB
Tablet (참고) 2-pane 또는 모바일 룰 768–1023px 구간에서 모바일 룰을 우선 따른다 breakpoint 자동 분기
화면 분리 원칙
  • List는 항상 진입의 시작 — 다른 화면에서 List로 복귀할 수 있어야 한다
  • Detail / Create / Edit 는 Fullscreen 으로 별도 화면 취급
  • Full Table View 는 명시적 모드 전환 — 자동으로 진입하지 않는다
  • Nav Drawer 는 어디서든 호출 가능 (Topbar 햄버거 필수)
PC WBS 와의 차이
  • PC 단일 테이블 → 모바일 6 화면으로 분리
  • PC Right Drawer 상세 → 모바일 Fullscreen panel
  • PC Page Toolbar 액션 → Topbar / Bottom action / Bottom sheet 로 분산
  • PC 검색 input → 모바일 icon → inline expand

04 Mobile WBS Search — 검색 진입 패턴 패턴

WBS는 검색 빈도가 높은 화면이다. 모바일에서 검색은 topbar RIGHT의 검색 icon으로만 진입하며, 검색 활성 시 topbar 아래로 inline search bar 가 확장(A안)된다. 전역 검색은 별도 entry로 분리하고, 조건 검색은 Filter Bottom sheet 로 책임을 분리한다. WBS는 A안(Inline expand)를 1차 적용한다.

S3 안 비교 — Inline / Full overlay / Filter sheet
A Inline Search Expand 1차 적용
topbar 아래 36px search bar 가 펼쳐진다. 현재 화면(WBS) 컨텍스트를 유지하면서 작업명 · 담당자를 즉시 검색.
Pros
  • 화면 전환 없이 결과 확인 가능
  • topbar 아래 36px 만 차지 — content 영향 최소
  • 결과 수 · 초기화 버튼을 search bar 옆에 배치
Cons
  • 최근 검색 / 그룹 결과를 한 화면에 노출하기 어려움
B Full Search Overlay 전역 검색
전체 화면을 검색 모드로 덮는다. 최근 검색 / 카테고리별 결과 노출 가능. STAM 전체 (홈) 진입 검색에 적합. WBS 단독 검색에는 과한 화면 전환.
Pros
  • 최근 검색 / 자동완성 / 그룹별 결과 모두 노출
  • history 분리 명확 (back 으로 이전 화면)
Cons
  • WBS의 빠른 작업 탐색에는 과중
  • 화면 컨텍스트가 일시적으로 사라짐
C Bottom Sheet Search 검색에는 부적합
Bottom sheet 안에 search input 노출. 일반 검색에는 깊이가 어색 — 필터 / 정렬 조건에만 한정 사용.
Pros
  • 조건 선택(상태 / 담당자 등) 과 자연스럽게 결합
Cons
  • 키보드 + sheet 가 동시에 차지하는 화면이 크다
  • 1차 검색 진입으로는 비효율

05 Mobile WBS Filter — Chip + Bottom Sheet Filter

16 Cursor Handoff Notes — 향후 적용 후보 Handoff

17 Implementation Guardrails — 적용 금지 / 제한 Guard
금지 항목
  • PC 테이블을 모바일에 그대로 축소 표시
  • 페이지 레벨 가로 스크롤 발생
  • touch target 40px 미만
  • 숫자 notification badge — dot 만 허용
  • 전역 .tbtn 일괄 치환
  • 기능 로직 / 데이터 모델 변경
  • 본 가이드 적용을 단독으로 product 코드에 push
제한 / Gate 필요
  • 모바일 카드 모드 적용은 별도 Gate 후 진행
  • 먼저 CSS / 레이아웃 Preview → 데이터 렌더링 → 액션 순서
  • PC WBS 회귀 QA 통과 전 모바일 분기 머지 금지
  • Filter 차원 추가는 사업 / 권한 정책 결정 후
  • FAB / Bottom Nav 동시 적용은 Shell 정책 확정 후
권장 / 강제 기준
  • footer CTA 44px 권장 · 최소 40px
  • Topbar 48px · action 최대 3
  • Detail / Create / Edit 는 Fullscreen 강제
  • Bottom sheet 는 swipe / scrim tap / ESC 로 닫기
  • chip row 가로 스크롤 허용 — 페이지 hscroll 아님
  • S-Core Dream 폰트만 사용 — Google Fonts / Pretendard CDN / Gulim / Dotum 금지
미적용 항목 (본 문서 범위 외)
  • 모바일 카드 모드 실제 데이터 바인딩
  • 실제 product CSS / JS 적용
  • 모바일 알림 / push 정책
  • 가로모드 / 분할 / 접근성 추가 가이드
