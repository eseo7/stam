본 문서는 STAM 의 개인정보 · 고객정보 · 계정 · 권한 · 로그인 · 비밀번호 · 세션 · 감사로그 · 다운로드 · 보관/파기 정책을 화면 / 로직 / DB 설계 / 감사로그 저장 구조에 모두 반영하기 위한 기준 문서이다. 단순 UI 가이드가 아니라 Frontend · Backend · DB · 운영정책까지 일관되게 연결한다.

적용 범위 — 본 작업은 stam/docs/** 하위 1개 HTML 생성과 STAM-Docs-Index.html 카드 1개 추가만 수행한다. stam/pages/**, stam/js/**, stam/css/**, stam/index.html, stam/assets/**, .firebaserc, firebase.json, .github/workflows/**, package.json, _ds/**, uploads/**, *.zip, .env*, node_modules/**, dist/**, build/**, .firebase/** 는 한 줄도 수정하지 않는다. 01 Overview — 문서 목적 목적

STAM 은 PM 협업 · 일정 · 결함 · 산출물 등 고객 / 직원 / 외부 협력사의 개인정보가 일상적으로 흘러다닌다. 보안 가이드가 UI 문서로만 끝나면 — 화면에서는 마스킹 되어 있어도 API 가 원문을 그대로 내려주고, DB 에 평문이 저장되며, 감사로그가 없어 사후 조사도 불가한 상황이 일어난다. 본 문서는 그 분리를 막기 위해 화면 · 로직 · DB · 로그 · 운영정책을 한 줄로 묶는다.

이 문서가 다루는 것
  • Frontend / Backend / DB / 감사로그를 한 번에 연결한 정책 기준
  • 인증 · 비밀번호 · 세션 · 권한 · 마스킹 / 전체 보기 / 다운로드 정책
  • 개인정보 분류 4단계 (L0 ~ L3) 와 항목별 표시 / 권한 / 감사 정책
  • 14개 보안·감사·정책 DB 테이블 후보 — 목적 / 필드 / 인덱스 / 보존 / 권한
  • Frontend 필수 화면 12종 · 프론트 로직 / 백엔드 API 요구사항
  • 이상 행위 탐지 / 알림 정책 · ISMS-P 10 범주 매핑
  • UI Mock 8종 (Login · 비밀번호 · 마스킹 · 사유 입력 · 감사로그 · 권한 이력 · 파기 정책)
이 문서가 다루지 않는 것
  • stam/pages/**, stam/js/**, stam/css/** 직접 수정
  • 실제 DB 스키마 생성 / Firebase rules 수정 / 인덱스 생성
  • 실제 인증 / OAuth / SSO / MFA 코드 구현
  • 암호화 알고리즘 / 키 관리 / KMS 구체 구현
  • ISMS-P 인증 신청 / 컨설팅 / 실사 대응 (정책 매핑만)
  • 전역 .tbtn 일괄 치환 · Firebase / package.json 변경
왜 UI 가이드로 끝나면 안 되는가
  • 화면 마스킹만 있고 API 가 평문 → 개발자 도구로 평문 노출
  • 버튼 숨김만 있고 권한 미검증 → URL 직타 / 권한 우회 가능
  • 감사로그 누락 → 사후 조사 불가 · 책임 추적 불가
  • DB 평문 저장 → 유출 시 피해 즉시 · 법적 책임
  • 운영 정책 부재 → 보관 기간 무한 · 파기 미실행 · 휴면 미처리
실제 구현 전 기준 문서
  • 본 문서는 실제 구현 전 정책 / 설계 기준이며 product 적용은 별도 Gate
  • 적용 순서: 본 가이드 승인 → DB 설계 확정 → API Boundary 정의 → 프론트 UI 적용 → 감사로그 writer → 알림 / 운영
  • 한 커밋에 정책 + 코드 + DB 마이그레이션 섞지 않는다
  • 충돌 시 ISMS-P · 개인정보보호법 · 회사 보안 정책이 우선
02 Security Architecture Principles Principle

STAM 보안 설계의 7대 원칙. 모든 화면 · 로직 · DB · 정책 결정의 기준이 된다.

01 프론트 버튼 숨김은 보안이 아니다 권한 검증은 서버 · API · DB 에서 한다. 프론트는 UX 일 뿐.
02 개인정보는 기본 마스킹 목록 · 상세 · 수정 모든 화면의 default 가 마스킹. 전체 보기는 선택적 행위.
03 전체 보기 / 다운로드 / 출력은 사유 + 감사로그 조회 사유 입력 모달 → 감사로그 저장 성공 → 비로소 원문 노출.
04 관리자 행위도 모두 감사로그 권한 부여 · 비밀번호 초기화 · 정책 변경 · 감사로그 조회까지 전부 기록.
05 로그는 일반 사용자가 수정 / 삭제 불가 audit_logs 는 append-only. 삭제 권한은 별도 보존 정책 만료 자동 파기만.
06 보안 정책은 화면 · 로직 · DB 에 동시에 반영 정책 변경 시 3 계층 모두 갱신 — 한 곳만 바뀌면 정책 일관성 깨짐.
07 감사로그 저장 실패 = 작업 거부 로그 저장 실패 시 개인정보 전체 보기 / 다운로드 결과 반환 차단.

06 Authorization / Role Policy Auth

RBAC (Role-Based Access Control) 기반. menu permission · action permission · data scope permission 3축 분리. 특히 read / unmask / export 는 별도 권한.

Action Permission화면 내 액션 권한 — CRUDXcreate · read · update · delete · export · unmask · approve
Data Scope Permission볼 수 있는 데이터 범위본인 / 팀 / 프로젝트 / 부서 / 전사

07 Personal Data Classification Class

STAM 의 모든 데이터를 4단계로 분류한다. L0 일반 · L1 개인정보 · L2 민감 개인정보 · L3 고객 식별 / 정산. 화면 / 로직 / DB 모두 이 분류에 따라 처리.

L0 · 일반 공개 가능 / 비식별 개인을 식별하지 않는 일반 업무 데이터
업무명상태일정진척률프로젝트명
L1 · 개인정보 기본 마스킹 + 권한 read 개인 식별 가능 — 화면 기본 마스킹, 사유로 unmask
이름이메일전화번호부서직위
L2 · 민감 개인정보 강한 마스킹 + 사유 + 감사로그 건강 / 종교 / 노조 등 민감정보 (현재 STAM 에는 거의 없음)
주민번호생년월일여권번호외국인등록번호
L3 · 고객·정산 DB 암호화 + 다운로드 통제 고객 식별 / 계약 / 정산 — DB 저장 시 암호화, 다운로드 별도 권한
고객번호계약번호계좌번호주소정산금액
분류기본 표시전체 보기다운로드DB 저장
L0원문—로그평문
L1마스킹사유 + 로그사유 + 로그 + 권한평문 (필요 시 컬럼 암호화)
L2강한 마스킹사유 + 로그 + 추가 인증금지 또는 별도 승인암호화 강제
L3마스킹사유 + 로그사유 + 로그 + 권한 + 워터마크암호화 강제 + KMS 키
08 Masking / Unmasking Policy Masking

  • 1차: DB 14 테이블 schema 확정 (S13) → migration 작성
  • 2차: 권한 middleware + 감사로그 writer (BE)
  • 3차: 로그인 · 비밀번호 · 세션 화면 (FE)
  • 4차: 마스킹 컴포넌트 + unmask API + reason 모달
  • 5차: 다운로드 통제 + 워터마크
  • 6차: 관리자 화면 (감사로그 / 권한 / 정책)
  • 7차: 이상 행위 탐지 / 알림 (S17)

  • DB 마이그레이션 + 비즈니스 로직
  • 마스킹 + 권한 middleware
  • 감사로그 + 사용자 정책 변경
  • 관리자 화면 + 일반 사용자 화면
  • 알림 / 이상 행위 탐지 + 기본 로그 schema
