01 Purpose — 이 문서의 목적 Preview Baseline

이 문서는 자유 텍스트 형태의 요구사항(메모 · 회의록 · 메일 등)을 사람이 요구사항 Draft 로 만들기 쉽게 돕는 수동 보조 UX 를 정의한다. 이것을 “자동 분석 솔루션”처럼 과장하지 않는다. 시스템은 원문을 의미적으로 이해하지 않으며, Rule-based block split(빈 줄 / 번호 / 불릿 / 제목 기준)으로 블록을 나누고, 사용자가 선택 · 병합 · 분리 · 수정해 요구사항 Draft 를 만드는 흐름을 돕는다.

  • 흐름: 텍스트 업로드 → 원문 보존 → 문단/번호/불릿 기준 블록 분리 → 사용자가 선택/병합/분리/수정 → 요구사항 Draft 생성
  • 자동 생성 결과는 절대 확정 데이터가 아니다. 모든 결과는 Imported Draft / Review Needed 상태로 생성한다.
  • AI / NLP 기반 요구사항 자동 추출은 후속 확장 범위로만 표기한다(13장).
  • 텍스트 업로드만으로 새 게시판을 자동 생성하지 않는다. 새 게시판 생성은 사용자가 명시적으로 선택·승인할 때만 가능하다.
  • 이 Preview 는 후속 실제 구현의 기준 화면으로 사용한다.
반복 명시 이 화면이 만드는 것은 Draft(초안) 데이터이지 확정 데이터가 아니다. 사람이 검토하고 승인해야 실제 요구사항이 된다.
02 Scope Boundary — AI 없는 1차 범위 Boundary
1차 범위는 AI 없는 수동 보조다 AI 없이 자유 텍스트를 정확히 의미 분석하는 것은 1차 범위가 아니다. 시스템은 원문을 “이해”하지 않는다. 시스템이 할 수 있는 것은 줄바꿈 / 문단 / 번호 / 불릿 / 간단한 패턴 기준으로 블록을 나누고, 사용자가 요구사항으로 만들 수 있게 돕는 것뿐이다.
1차 범위 (이번 Preview)
  • 원문 보존
  • Rule-based block split
  • 사용자 선택 / 병합 / 분리 / 수정
  • 사용자 검토 후 요구사항 Draft 생성
  • source traceability
1차 범위가 아닌 것 (후속 확장)
  • 원문의 의미 이해
  • 요구사항 자동 확정
  • 정책 / 질문 / 제약 자동 분류 확정
  • AI 요구사항 자동 추출
  • 새 게시판 자동 생성
03 User Scenario — 실무 시나리오 Scenario

엑셀처럼 정형화되지 않은 요구사항도 흔하다. 회의록 · 메일 · 메모 텍스트를 그대로 붙여넣고, 사람이 골라 Draft 로 만든다.

회의록/메일/메모 확보→ STAM 에 붙여넣기/업로드→ 원문 그대로 보존→ Rule-based 블록으로 표시
기획자가 요구사항이 될 블록 선택→ 병합/분리/수정→ 요구사항 Draft 생성 (Review Needed)
핵심 시스템은 “이 문장이 요구사항이다”를 판단하지 않는다. 어떤 블록이 요구사항인지는 사람이 결정한다. 시스템은 블록을 보기 좋게 나눠주고, 선택한 블록을 Draft 로 옮기는 작업을 돕는다.
04 Text Intake Flow Overview — 단계 구성 Flow

Text Intake Helper 는 5단계로 구성한다. 모든 단계는 원문 보존과 사람 검토를 전제로 한다.

1 업로드/붙여넣기→ 2 원문 보존→ 3 Rule-based 블록 분리→ 4 선택/병합/분리/수정→ 5 요구사항 Draft 생성
단계화면 목적사용자 행동시스템 역할 (이해 아님)
1 업로드/붙여넣기텍스트 입력txt/md 업로드 또는 붙여넣기텍스트 받기
2 원문 보존원문 그대로 표시원문 확인line number / block id 부여
3 Rule-based 블록 분리블록 후보 표시분리 기준 선택빈 줄/번호/불릿/제목 기준 분리
4 선택/병합/분리/수정블록 가공선택·병합·분리·수정사용자 편집 반영
5 Draft 생성요구사항 초안 생성Draft 생성 확정Review Needed 상태로 생성
05 Step 1 — Text File Upload / Paste Text Step 1
입력 영역
  • 큰 텍스트 붙여넣기 영역 (Paste Text)
  • 파일 업로드 (Drag & Drop)
  • 붙여넣기 / 업로드 모두 동일하게 원문으로 취급
지원 입력
  • .txt — 우선 적용 범위
  • .md — 우선 적용 범위
  • pasted text — 우선 적용 범위
  • .docx / .pdf parsing — 후속 확장 범위
주의 이 화면은 실제 파일 parsing 구현이 아니라 Preview 기준이다. .docx / .pdf 추출은 후속 확장 범위로만 표기하며, 실제 파싱 / Storage 업로드 / AI API 호출은 수행하지 않는다.
06 Step 2 — Source Preview and Preservation Step 2

원문은 가공 전·후 항상 그대로 보존한다. 원문 패널은 전체 텍스트와 함께 라인 번호 / 블록 id / 원본 파일을 표시한다.

lineblockId원문 (preserved)
1blk-001[회의록] 2026-06-20 고객 킥오프
3blk-0021. 로그인은 2차 인증(OTP)을 지원해야 한다.
4blk-0032. 관리자 권한과 일반 권한을 구분한다.
6blk-004- 게시판 목록은 검색/필터가 필요함
7blk-005* (질문) 외부 연동 범위는 협의 필요
원문 보존 패널 구성
  • 원문 전체 (수정 없이 보존)
  • line number
  • paragraph / block id
  • source file
보존 원칙 사용자가 블록을 병합·분리·수정해도 원문은 변경하지 않는다. Draft 는 원문 블록을 참조(reference)하며, 원본 라인 범위와 block id 를 함께 보존한다.
07 Step 3 — Rule-based Block Split Rule-based
자동 의미 분석이 아니다 이 단계는 텍스트의 의미를 해석하지 않는다. 줄바꿈 / 번호 / 불릿 / 제목 같은 형식 패턴만으로 블록 경계를 제안한다. “요구사항인지”는 판단하지 않으며, 그 판단은 다음 단계에서 사람이 한다.
분리 기준패턴 예동작
빈 줄 기준연속 줄바꿈 / 빈 줄문단 단위 분리
번호 기준1. 2) (3)번호 항목 단위 분리
bullet 기준- * ·불릿 항목 단위 분리
제목/섹션 기준# 머리글 / [제목]섹션 경계 표시
분리 결과 (블록 후보)
  • blk-002 — “로그인은 2차 인증(OTP)…” (번호)
  • blk-003 — “관리자 권한과 일반 권한…” (번호)
  • blk-004 — “게시판 목록은 검색/필터…” (불릿)
  • blk-005 — “(질문) 외부 연동 범위…” (불릿)
사용자 조정
  • 분리 기준 on/off 토글
  • 잘못 나뉜 블록 병합
  • 한 블록을 여러 블록으로 분리
  • 블록 경계 직접 수정
08 Step 4 — Manual Select / Merge / Split Step 4

사람이 블록별로 액션을 선택한다. 시스템은 사용자의 선택을 반영할 뿐, 어떤 블록이 요구사항인지 자동으로 확정하지 않는다.

blockId원문 블록사용자 액션결과
blk-002로그인은 2차 인증(OTP)을 지원해야 한다요구사항 Draft 생성Draft 후보
blk-003관리자 권한과 일반 권한을 구분한다요구사항 Draft 생성Draft 후보
blk-004게시판 목록은 검색/필터가 필요함기존 Draft에 병합blk-002에 병합
blk-005(질문) 외부 연동 범위는 협의 필요메모로 보관요구사항 아님 / 메모
blk-006(머리말) 회의 참석자 명단무시제외
블록별 액션
  • 요구사항 Draft 생성
  • 기존 Draft에 병합
  • 블록 분리
  • 블록 병합
  • 메모로 보관
  • 무시
사람이 결정한다 질문 / 정책 / 제약처럼 보이는 블록도 시스템이 자동 분류·확정하지 않는다. 사용자가 “요구사항 / 메모 / 무시”를 직접 정한다.
09 Step 5 — Create Requirement Draft Step 5

선택한 블록으로 요구사항 Draft 를 만든다. 임시 제목은 첫 문장 기반 또는 사용자 입력이며, 대부분의 필드는 미지정으로 둘 수 있다.

필드값 / 기본값설명
title첫 문장 기반 임시 제목 또는 사용자 입력의미 분석 아님 · 첫 문장 절단
description원문 블록 기반원문 블록 텍스트 보존
priority미지정 가능사용자 검토 후 지정
status미지정 가능사용자 검토 후 지정
owner미지정 가능사용자 검토 후 지정
reviewStatusReview Needed항상 검토 필요로 생성
sourceTypeText Intake출처 = 텍스트 인테이크
extractionModeRule-based / ManualAI 아님 명시
Draft 데이터임을 명시 생성 결과는 확정 데이터가 아니라 Draft 다. title 은 의미 분석 결과가 아니라 첫 문장 기반 임시값이며, priority / status / owner 는 미지정으로 둘 수 있다. 확정은 사람 승인을 거친다.
10 Draft Result in Requirement Board Result

생성된 Draft 는 요구사항 게시판에 “초안 / 미검토 / Text Intake 출처” 상태로 들어간다. 원본 텍스트 위치가 함께 보존된다.

요구사항(임시 제목)상태출처extractionMode원본 라인
로그인 2차 인증(OTP) 지원Review NeededText IntakeRule-based / Manual3
관리자/일반 권한 구분Review NeededText IntakeRule-based / Manual4
상태 체계
  • Imported Draft — 초안 / 미검토
  • Review Needed — 검토 필요
  • Confirmed — 사람 승인 완료
상세 Drawer
  • 요구사항 상세에 source traceability 섹션 노출
  • 원본 파일 / 라인 범위 / block id 표시
  • 원문 텍스트(sourceText) 보존
게시판 자동 생성 없음 텍스트 업로드만으로 새 게시판을 자동 생성하지 않는다. 새 게시판 생성은 사용자가 명시적으로 선택·승인할 때만 가능하다.
11 Source Traceability — 원본 추적 메타 Traceability

Text Intake 로 생성된 모든 Draft 는 아래 메타를 보존한다. 원본 추적(source traceability)은 선택이 아니라 필수다.

메타의미예시
sourceFile원본 파일명 / 입력 출처kickoff_meeting_2026.txt
sourceText원문 블록 텍스트 (보존)로그인은 2차 인증(OTP)을…
sourceLineStart원본 시작 라인3
sourceLineEnd원본 끝 라인3
sourceBlockId블록 식별자blk-002
importBatchId인테이크 배치 식별자txt_20260626_01
createdBy생성 수행자기획자 A
createdAt생성 시각2026-06-26 15:10
역할 어느 파일 · 어느 라인 · 어느 블록에서 어떤 원문으로 만들어졌는지를 복원 가능하게 한다. 원문을 변경하지 않고 참조만 보존하므로, 후속 검토·수정 시에도 출처를 추적할 수 있다.

13 AI Requirement Extraction — 후속 확장 Follow-up Only
AI 자동 추출은 후속 확장 범위로만 표기한다 아래 항목은 이번 1차 범위에 포함하지 않는다. AI / NLP 연동 후 단계적 구현으로만 다루며, 이번 Preview 에서는 구현·확정하지 않는다.
  • 요구사항 후보 자동 추출 — 후속 확장
  • 정책 / 질문 / 제약 분류 — 후속 확장
  • 필드 후보 자동 채움 — 후속 확장
  • 기능정의 초안 생성 — 후속 확장
조건부 표현 원칙 “자동 분석”이라는 표현을 쓰더라도 반드시 “후속 확장” 또는 “AI 연동 후”로 제한한다. AI 연동 후에도 결과는 Draft 로 생성하고 사람 승인을 전제로 한다.
14 Guardrails — 가드레일 Guardrails
  • AI 없이 자유 텍스트를 정확히 의미 분석하는 것은 1차 범위가 아니다.
  • 시스템은 원문을 이해하지 않는다 (Rule-based 형식 분리만).
  • 자동 생성 결과는 절대 확정 데이터가 아니다.
  • 모든 결과는 Imported Draft / Review Needed 상태로 생성한다.
  • AI Requirement Extraction 은 후속 확장 범위로만 표기한다.
  • 텍스트 업로드만으로 새 게시판을 자동 생성하지 않는다.
  • 새 게시판 생성은 사용자가 명시적으로 선택·승인할 때만 가능하다.
  • 원문은 변경하지 않고 보존한다 (source traceability 필수).
  • “자동 분석으로 요구사항을 정확히 추출한다”는 과장 표현을 쓰지 않는다.
  • 실제 파일 parsing / AI API / Storage / DB / API 는 이 단계에서 구현하지 않는다.
15 Relationship with Excel Import Builder Mapping

Text Intake Helper 와 Excel Import Builder 는 둘 다 요구사항 게시판의 Draft 진입 경로이며, 입력 형태와 분리 방식이 다르다.
