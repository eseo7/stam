01 Overview 가이드
컴포넌트Drawer (Side Panel)
기본 폭콘텐츠 영역 약 40%
진입 트리거Row 클릭 · 등록 버튼
Full View 전환전체 보기 버튼

목적 및 역할

  • 목록 화면의 흐름을 유지한 채 상세·수정·등록을 빠르게 처리하는 보조 패널이다.
  • 산출물·게시판의 등록·수정·상세는 기본적으로 Drawer를 사용할 수 있다.
  • WBS, 요구사항정의서, 화면설계서, 테이블 정의서, DB 정보, 서버 정보, 오픈 시나리오 등 주요 게시판에 동일 기준을 적용한다.
  • 긴 입력·긴 상세는 Drawer보다 Full View(전체 화면 전환)가 적합하다. 이 기준은 전체보기 버튼으로 제공한다.

Dialog vs Drawer 비교

구분 Dialog Drawer
형태 화면 중앙 모달 화면 우측 슬라이드인 패널
적합한 사용 단순 확인·경고·위험 액션 확인 목록 행 상세·등록·수정
목록 유지 불가 (배경 잠금) 가능 (목록 일부 노출)
콘텐츠 길이 짧고 단순한 메시지 중간 길이 정보, 폼
Full View 전환 해당 없음 전체보기 버튼으로 전환
Baseline Guide 참조 Drawer 기준은 Baseline Guide S12 Drawer Standard에도 정의되어 있다. 이 가이드는 컴포넌트 구현 관점에서 보완한다.
02 Drawer Types 완료
유형 칩 사용 맥락 진입 적용 예시
Detail Drawer 상세 목록 행 클릭 후 요약 상세 확인 Row 클릭 WBS 작업 상세, 요구사항 상세
Create Drawer 등록 목록 흐름을 유지한 신규 등록 등록 버튼 산출물 등록, 서버 정보 등록
Edit Drawer 수정 상세에서 수정 모드 전환 (페이지 이동 없음) 상세 내 수정 버튼 WBS 작업 수정, 오픈 시나리오 수정
Review Drawer 검토 검토 의견, 승인 요청, 코멘트 확인 검토 버튼 또는 Row 클릭 고객 검토 의견 확인, 승인 요청
Attachment / URL Drawer 첨부 첨부파일과 URL 참조 관리 첨부 아이콘 또는 버튼 파일 설명, 공개범위, 내보내기 포함 여부
중첩 금지 Drawer 안에 또 다른 Drawer를 중첩하지 않는다. Drawer 안에 Dialog는 위험 액션 확인(삭제·반려)에만 제한적으로 사용한다.

04 Width / Layout Rules — 너비 및 레이아웃 완료
항목 기준 비고
기본 폭 콘텐츠 영역 약 40% 1440px 화면 기준 약 480–560px
최소 폭 360px 이 이하에서는 Full View 전환 우선
최대 폭 640px 복합 폼 · 긴 상세 시 확장 허용
위치 콘텐츠 영역 위 Overlay 좌측 메뉴를 밀어내지 않음
테이블 레이아웃 테이블 폭 유지 Drawer가 테이블을 밀거나 줄이지 않음
가로 스크롤 콘텐츠/테이블 영역 기준 페이지 전체 가로 스크롤 금지
세로 스크롤 Body 영역만 허용 Header · Footer는 flex-shrink: 0으로 고정
좁은 화면 Full View 전환 우선 뷰포트 < 960px 이하 고려
Overlay 방식 Drawer는 좌측 메뉴(Sidebar)와 상단 Topbar를 제외한 콘텐츠 영역 위에만 올라온다. 좌측 메뉴 너비(--sw: 256px)는 항상 유지된다.

07 Form Rules — 폼 구성 기준 완료
Form Guide 참조 필드·유효성·라벨·에러 메시지 기준은 Form Guide를 따른다.
  • Drawer 내부 폼은 Field Group으로 나눈다. 관련 필드는 섹션 제목과 함께 묶는다.
  • 긴 폼(필드 10개 초과, 복잡한 에디터 포함)은 전체보기 전환을 제공한다.
  • 오류 메시지는 해당 필드 바로 아래에 표시한다. 상단 모아 표시는 사용하지 않는다.
  • 저장 중 Loading 상태를 제공한다. Primary 버튼에 스피너를 표시한다.
  • Dirty 상태(입력값 변경 후 미저장)는 이탈 확인 대상이다. 닫기·ESC 시 Dialog 확인을 띄운다.
  • 읽기 전용 필드는 입력 스타일이 아닌 텍스트 표시로 구분한다.
08 Usage Rules — 사용 규칙 완료
상황권장 컴포넌트이유
단순 확인 / 경고 Dialog 짧고 즉각적인 응답이 필요한 경우
목록 행 상세 확인 Drawer 우선 목록 맥락을 유지하면서 내용 확인
복잡한 작성 / 긴 입력 Full View 우선 좁은 패널에서 긴 폼 작성은 UX 저하
상세 → 수정 전환 같은 Drawer 내 mode 전환 별도 페이지 이동 없이 연속성 유지
위험 액션 확인 Drawer 내 Dialog 삭제·반려는 Drawer 위에 Dialog로 이중 확인
Drawer 중첩 금지 Drawer 안에 Drawer는 구조적으로 허용하지 않음
  • 프로젝트 핵심 게시판(WBS·요구사항·화면설계서·테이블정의서·DB·서버정보·오픈시나리오)은 동일 Drawer 패턴을 따른다.
  • 빈 탭·빈 섹션이 있더라도 탭/섹션 자체는 항상 표시하고 빈 상태 문구로 안내한다.
09 Do / Don't 완료
Do
  • 목록 맥락을 유지하면서 상세를 보여준다
  • 전체보기 전환 버튼을 Footer 좌측에 제공한다
  • 상세 → 수정 모드를 같은 Drawer 안에서 자연스럽게 전환한다
  • Footer 버튼 위계를 명확히 구분한다 (Primary / Secondary / Danger)
  • 긴 내용은 Section Group으로 나눈다
  • Dirty 상태에서 닫기 시 이탈 확인 Dialog를 띄운다
  • Header와 Footer는 항상 고정하고 Body만 스크롤한다
Don't
  • 모든 화면을 억지로 Drawer 안에 넣는다
  • Drawer가 테이블·목록 레이아웃을 밀어내게 만든다
  • Drawer 안에 Drawer를 중첩한다
  • 긴 폼을 좁은 Drawer에 그대로 넣는다 (전체보기 없이)
  • 삭제·반려를 Dialog 확인 없이 즉시 실행한다
  • 닫기(×)와 취소 버튼의 의미를 혼동해서 사용한다
  • 좌측 메뉴나 Topbar 위에 Drawer를 올린다

11 Accessibility — 접근성 완료
항목기준
Drawer 열림 시 Focus Drawer 내 첫 번째 인터랙티브 요소(닫기 버튼 또는 첫 입력 필드)로 자동 이동
닫기 버튼 라벨 aria-label="닫기" 또는 aria-label="Drawer 닫기" 명시
ESC 닫기 ESC 키로 Drawer 닫기 지원. Dirty 상태에서는 이탈 확인 Dialog 표시
키보드 이동 순서 Drawer 열림 중 Tab 이동은 Drawer 내부로 제한(Focus Trap). 닫힘 시 진입 트리거 요소로 복귀
Overlay 구분 Scrim 영역은 aria-hidden="true". 본문 영역은 inert 또는 aria-hidden으로 비활성화
Screen Reader 제목 Drawer 컨테이너에 role="dialog" + aria-labelledby로 제목 연결
Dirty 확인 Dialog 닫기·ESC 시 Dirty 상태라면 Dialog 확인 필수. Dialog Guide 기준 적용
색상 단독 상태 표현 금지 색상만으로 상태를 구분하지 않는다. 상태 칩·아이콘·텍스트를 함께 사용한다
12 QA Checklist 완료
Drawer가 필요한 상황인가? Dialog / Full View가 더 적합하지 않은가?
기본 폭 기준(콘텐츠 영역 약 40%)이 유지되는가?
테이블·목록 레이아웃을 밀어내지 않는가?
전체보기 버튼이 필요한 경우 Footer 좌측에 제공되는가?
상세 → 수정 전환이 같은 Drawer 내부에서 이루어지는가?
Footer 버튼 위계가 올바른가? (Primary · Secondary · Danger)
Dirty 상태 닫기 시 이탈 확인 Dialog가 표시되는가?
Header · Footer는 고정되고, Body 영역만 스크롤되는가?
닫기 버튼에 접근성 라벨(aria-label)이 있는가?
S-Core Dream 폰트가 유지되는가? (Pretendard·Gulim·Dotum 금지)
Drawer 중첩이 없는가?
가로 스크롤이 페이지 전체에 발생하지 않는가?
