01 Overview 가이드
컴포넌트 Button
CSS 위치 stam.components.css
Variant Primary / Secondary / Ghost / Danger / Icon / Text
Size Large / Medium / Small / Icon-only

목적

  • 버튼은 사용자 액션의 우선순위를 구분하는 컴포넌트다. 화면 내에서 어떤 액션이 주요하고 어떤 것이 보조적인지를 Variant와 Size로 표현한다.
  • STAM 전체 화면에서 사용할 버튼 기준을 이 가이드에서 정의한다. 화면마다 임의로 새로운 버튼 스타일을 생성하는 것을 금지한다.
  • 모든 버튼은 이 가이드에 정의된 Variant · Size · State 조합만 사용해야 한다.
현재 구현 상태 .tbtn (Topbar 전용 버튼)이 stam.components.css에 정의되어 있다. 제품 화면용 Primary / Secondary / Ghost / Danger 버튼 클래스는 이 가이드 확정 후 stam.components.css에 추가한다.

Button vs Link — 무엇을 버튼으로 만들 것인가

  • 화면 이동만 발생하면 link(<a> / navigation action)로 표현한다. URL이 바뀌고 뒤로가기로 되돌릴 수 있다면 link다.
  • 데이터 상태가 바뀌면 button으로 표현한다. 저장 · 등록 · 수정 · 삭제 · 검토 요청 · 승인 · 반려 · 내보내기 · 닫기 · 취소처럼 결과가 발생하는 액션은 모두 버튼이다.
  • 결과가 큰 액션(저장 · 삭제 · 승인 등)은 우선순위(Variant)와 확인 절차(Dialog 또는 명시적 라벨)를 함께 둔다.
  • “링크처럼 보이는 버튼”이 필요하면 Text Button을 사용한다. 텍스트 흐름 안에서는 link, 카드/footer 등 독립 영역에서는 Text Button을 권장한다.
02 Button Hierarchy — 버튼 위계 완료

Variant 정의

Variant 역할 우선순위 사용 예시
Primary 주요 저장·생성·확정 액션. 화면의 핵심 목표를 완료하는 단일 CTA. 최상위 저장, 생성, 등록, 승인 요청, 내보내기 생성
Secondary Primary와 함께 노출되는 보조 액션. Primary 다음 우선순위. 상위 취소, 목록으로, 미리보기, 임시저장
Tertiary / Ghost 낮은 우선순위의 보조 액션. 강조 없이 노출. 보조 더보기, 상세 보기, 필터 초기화
Danger 삭제·반려·취소·권한 해제 등 되돌릴 수 없는 위험 액션 전용. 경고 삭제, 반려, 초대 취소, 공유 해제
Icon Button 레이블 없이 아이콘 단독으로 동작하는 정사각형 버튼. 보조 / 도구 검색 초기화, 닫기, 접기·펼치기, 새로고침
Text Button 문장 안 또는 카드 하단의 낮은 강조 링크형 액션. 최하위 자세히 보기, 가이드 보기

04 Button State — 인터랙션 상태 완료

상태 정의

상태 설명 원칙
Default 기본 노출 상태 Variant별 정의된 색상·테두리 적용
Hover 마우스 커서가 버튼 위에 올라간 상태 배경색을 한 단계 어둡게. scale 변환 없음. 120ms ease-out.
Active / Pressed 클릭·탭 진행 중인 상태 Hover보다 한 단계 더 어두운 배경. 160ms ease-out.
Focus 키보드 Tab으로 포커스된 상태 반드시 outline: 2px solid var(--brand) 표시. 키보드 접근성 필수 요건.
Disabled 액션 불가 상태 opacity 단독 사용 금지. pointer-events: none + disabled attribute 함께 적용. 이유를 tooltip으로 제공 권장.
Loading 비동기 처리 진행 중 상태 스피너 표시 + 텍스트 숨김. 중복 저장 방지를 위해 클릭 비활성화. 저장·등록·생성 등 확정 액션에 반드시 적용.

05 Usage Rules — 사용 규칙 완료

위계 원칙

  • 한 화면의 Primary 버튼은 가능하면 1개만 둔다. 복수의 Primary는 위계를 무너뜨린다.
  • 저장·등록·승인 같은 확정 액션은 Primary를 사용한다.
  • 삭제·반려·권한 해제는 반드시 Danger를 사용한다. 일반 Primary로 표시하지 않는다.
  • 취소 버튼은 Primary보다 강하면 안 된다. Secondary 또는 Ghost를 사용한다.

배치 원칙

  • 테이블 행 안에서는 Small 또는 Icon Button을 우선 사용한다. Medium 이상은 행 밀도를 해친다.
  • 버튼 그룹 내 gap은 최소 8px. Drawer·Dialog footer 기준 권장 gap은 8px.
  • 폼·Drawer footer는 오른쪽 정렬, Empty State CTA는 가운데 정렬을 원칙으로 한다.

텍스트 원칙

  • 버튼 텍스트는 동사형으로 작성한다. (저장, 생성, 삭제 ← 가능 / 저장하기, 삭제 완료 ← 금지)
  • 버튼 텍스트는 가능하면 5자 이내로 짧게 작성한다.
  • 버튼 색상만으로 의미를 전달하지 않는다. 텍스트 또는 아이콘으로 보완한다.
  • 아이콘 단독 버튼은 반드시 title 또는 aria-label을 지정해야 한다.
06 Do / Don't 완료
Do — 이렇게 사용한다
  • 주요 액션은 명확한 동사형 텍스트로 작성한다
  • 위험 액션(삭제, 반려, 해제)은 Danger 스타일을 사용한다
  • Loading 상태로 비동기 처리 중 중복 클릭을 방지한다
  • 버튼 그룹 내 Variant 우선순위를 명확히 구분한다
  • 버튼 사이 gap을 일관되게 유지한다
  • 아이콘 버튼에 aria-label 또는 title을 반드시 지정한다
  • 키보드 Focus 상태가 시각적으로 보이도록 유지한다
Don't — 이렇게 하지 않는다
  • 한 화면에 Primary 버튼을 여러 개 나열하지 않는다
  • 삭제 버튼을 일반 Primary로 표시하지 않는다
  • Disabled 상태를 opacity 단독으로 애매하게 처리하지 않는다
  • 아이콘만 있고 의미 라벨이 없는 버튼을 사용하지 않는다
  • 화면마다 새로운 임의 버튼 스타일을 생성하지 않는다
  • 버튼 텍스트를 지나치게 길게 쓰지 않는다
  • 색상만으로 버튼 상태·의미를 전달하지 않는다

08 Accessibility — 접근성 기준 완료
  • 버튼 텍스트는 맥락 없이도 의미를 알 수 있어야 한다. "클릭", "여기" 같은 모호한 텍스트는 금지한다.
  • 아이콘 단독 버튼은 반드시 aria-label 또는 title을 지정해야 한다. 스크린 리더가 해당 라벨을 읽는다.
  • :focus-visible 스타일이 있어야 한다. outline: 2px solid var(--brand)를 기준으로 적용한다. 마우스 클릭 시에는 숨기고 키보드 포커스 시에만 표시한다.
  • Disabled 버튼은 키보드·마우스 양쪽 모두 비활성으로 처리해야 한다. pointer-events: none과 disabled attribute를 함께 적용한다.
  • 색상만으로 상태를 구분하지 않는다. Danger는 색상 + 텍스트(삭제, 반려)로, Loading은 스피너 + aria-label로 보완한다.
  • 버튼의 최소 터치 영역은 44×44px 이상을 권장한다. Small 버튼은 패딩 또는 투명 영역으로 44px를 충족시킨다.
주의 outline: none 또는 outline: 0 전역 설정은 키보드 사용자의 접근성을 완전히 차단한다. :focus-visible 셀렉터로 마우스·키보드를 분리 처리할 것.

10 Usage by Area — 화면 맥락별 사용 기준 1차 상세 기준

영역별 원칙

영역 Primary 사용 주요 액션 예 주의
Topbar 전역 액션 최소화 새로고침, 테마 전환, 프로젝트 전환 위험(삭제 · 권한 변경) 액션 배치 금지. 전역 성격이 아닌 액션은 toolbar로 옮긴다.
Page Toolbar 등록 또는 핵심 실행 1개 등록, 조회, 필터, 내보내기 Primary는 1개만. 보조 액션은 Secondary / Ghost로 정렬한다.
Drawer Footer 우측 하단 Primary 1개 저장, 검토 요청, 승인 / 취소, 임시저장 Primary는 오른쪽 끝. Secondary · Cancel은 Primary 앞에 둔다. 상세 보기 footer의 [전체 보기] [수정]은 Ghost · Secondary 위계로 표현한다.
Dialog Footer 확인 액션 1개 [취소] [확인] / [취소] [삭제] 순서는 취소 → 확인. 위험 액션은 Danger + “삭제” 같이 결과를 포함한 라벨을 쓴다.
Table Row Action 행별 Primary 두지 않음 보기, 수정, 더보기 행별 액션은 과도하게 두지 않는다. 삭제는 더보기 메뉴 또는 확인 단계로 분리하는 것을 권장한다.
Empty State 첫 행동 Primary 가능 요구사항 등록, 화면설계서 작성 비어 있는 화면의 “시작 행동”은 Primary CTA로 강조한다. 텍스트는 무엇이 만들어지는지 명확히 한다.
Mobile / Narrow Width 화면당 Primary 1개 저장, 등록, 검토 요청 터치 영역 최소 44px. 버튼 2개 이상이면 stack 또는 줄바꿈 허용. Destructive 액션을 하단 고정 CTA 단독으로 배치하지 않는다.

11 Label / Copy — 버튼 문구 기준 1차 상세 기준

한국형 업무 시스템 문구 원칙

  • 동사형 짧은 문구로 작성한다. 결과가 명확해야 한다.
  • 위험 액션은 대상 포함을 허용한다. (예: “요구사항 삭제”, “권한 제거”)
  • 검토 / 승인 계열은 상태 변화가 명확해야 한다. (예: “검토 요청”, “승인”, “반려”)
  • 같은 화면에서 “확인”이라는 단어가 저장 / 삭제 / 닫기 등 여러 의미로 반복되지 않게 한다.
좋은 문구
  • 저장
  • 등록
  • 수정
  • 검토 요청
  • 승인 / 반려
  • 삭제 / 요구사항 삭제
  • 내보내기 / 다운로드
  • 전체 보기 / 닫기 / 취소
  • 다시 시도
피해야 할 문구
  • OK / Submit / Go / Click
  • 처리 / 실행
  • 확인하기 같은 모호한 과장 표현
  • 같은 화면에서 의미가 다른 “확인” 반복
  • “하기”, “해주세요” 같은 종결 표현
  • 아이콘만 두고 라벨 없음 (대체 설명 부재)
  • 색상으로만 의미 전달 (예: 빨간 “확인”)
