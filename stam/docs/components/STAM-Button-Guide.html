상태: Detail Draft 2026-06-29

이 문서는 1차 기준 초안입니다. STAM 화면 작업 시 버튼 판단 기준으로 참고하십시오. 현재 기준은 1차 초안이며, 향후 실제 화면 리뷰 결과를 반영하여 순차 보완합니다.

01 목적 Detail Draft
  • STAM 화면 전반에서 버튼 사용 기준을 통일한다.
  • Cursor / Claude / Codex가 STAM 화면 작업 시 버튼 유형·문구·위계를 판단하는 기준 문서로 사용한다.
  • 기존 공통 CSS/JS/DOM 기준을 설명한다. 새 버튼 클래스·토큰·DOM 구조를 제안하지 않는다.
  • 새 화면을 만들거나 기존 화면을 수정할 때, 이 문서를 먼저 확인한 뒤 버튼 배치를 결정한다.
02 버튼 유형 Detail Draft
  • Primary Button — 화면의 핵심 완료 액션에 사용한다. 화면당 원칙적으로 1개만 배치한다. 예: 등록, 저장, 적용.
  • Secondary Button — 보조 액션 또는 Primary와 병렬로 배치되는 선택지에 사용한다. 예: 취소, 닫기, 이전.
  • Ghost / Text Button — 강조가 필요 없는 인라인 액션·툴바 보조 액션에 사용한다. 시각 무게를 최소화해야 할 때 선택한다. 예: 초기화, 전체 보기, 필터 해제.
  • Danger Button — 삭제·초기화 등 되돌릴 수 없는 위험 액션에 사용한다. 반드시 Dialog 확인 단계와 함께 배치한다. 예: 삭제, 일괄 삭제, 강제 초기화.
  • Icon Button — 아이콘만으로 구성된 버튼이다. 반드시 aria-label을 지정하여 의미를 명확히 한다. 아이콘 의미가 맥락 없이 불명확한 경우 사용하지 않는다.
03 사용 위치별 기준 Detail Draft
  • 목록 상단 주요 액션 — Primary 1개를 상단 우측에 배치한다. 보조 액션은 Secondary 또는 Ghost로 좌측에 배치한다. 예: 등록 (Primary) / 내보내기·필터 (Secondary 또는 Ghost).
  • 툴바 보조 액션 — Secondary 또는 Ghost를 사용한다. 아이콘 + 텍스트 조합을 권장한다. 예: 필터, 열 설정, 초기화, 내보내기.
  • 드로어 하단 액션 — 취소(Secondary) → 저장/등록(Primary) 순서를 유지한다. 위험 액션이 포함된 경우 Danger를 별도 좌측에 배치한다. 버튼 순서와 레이아웃은 전 화면 일관되게 유지한다.
  • 테이블 행 액션 — Ghost 또는 Icon Button을 사용한다. 행 내 버튼 수는 최소화한다 (원칙 3개 이하). 예: 상세, 수정, 삭제.
  • 위험 액션 — Danger Button을 사용하고 반드시 Dialog 확인 단계를 거친다. 목록 상단에 Danger를 직접 배치하지 않는다.
  • 빈 상태(Empty State) 액션 — Primary 1개만 배치한다. 예: 등록하러 가기, 전체 보기, 새로 만들기.
04 버튼 문구 기준 Detail Draft
  • 사용자가 할 행동으로 작성한다. 상태 설명이 아닌 동작 중심 문구를 사용한다.
  • 한국형 업무 문구를 사용한다. 권장 표현: 등록, 저장, 수정, 삭제, 취소, 닫기, 적용, 초기화, 전체 보기, 확인, 이전, 다음.
  • 영어·혼합 표현을 남발하지 않는다. 예: Save → 저장 / Cancel → 취소 / Delete → 삭제. 대외 서비스 연동 등 고유명사가 필요한 경우에 한해 예외를 허용한다.
  • "확인" 단독 사용에 주의한다. "확인"은 어떤 행동인지 불명확할 수 있다. 맥락이 분명한 경우에만 사용하고, 그렇지 않으면 구체적 행동 문구로 대체한다. 예: "확인" → 저장, 적용, 닫기 등.
  • Danger 버튼 문구는 위험성을 명확히 드러낸다. 예: "삭제", "일괄 삭제", "초기화" — 부드럽게 포장하지 않는다.
05 우선순위 기준 Detail Draft
  • Primary는 화면당 원칙 1개. 한 화면에 Primary가 2개 이상 보이면 위계가 무너진다. 여러 완료 액션이 동시에 노출되어야 하는 경우, 우선 순위를 재검토한다.
  • 저장 / 등록 / 적용 → Primary. 사용자의 핵심 완료 의도를 담은 버튼이다.
  • 취소 / 닫기 / 이전 → Secondary 또는 Ghost. 이탈·복귀 액션은 강조하지 않는다.
  • 삭제 / 초기화 등 위험 액션 → Danger. 문맥에 따라 신중한 Secondary 처리도 허용하나, Danger가 기본 기준이다. 위험 액션을 Primary처럼 강조하지 않는다.
  • 동급 선택지가 나열될 경우 → Secondary 동등 배치. 예: 드로어 하단의 취소 + 저장 중 저장만 Primary.
06 Do Detail Draft
  • 기존 공통 버튼 클래스를 사용한다. 임의 스타일 오버라이드를 금지한다.
  • 화면별 Primary 액션을 1개로 유지한다.
  • 버튼 문구는 사용자가 할 행동으로 작성한다.
  • 아이콘 버튼에는 반드시 aria-label을 지정한다.
  • 드로어 하단 버튼 순서를 일관되게 유지한다 (취소 → 저장/등록).
  • Danger 버튼 사용 시 Dialog 확인 단계를 거친다.
  • Ghost 버튼은 툴바·인라인 등 강조가 불필요한 위치에 한정해 사용한다.
07 Don't Detail Draft
  • 화면마다 새 버튼 스타일을 만들지 않는다.
  • 색상만으로 버튼 의미를 전달하지 않는다.
  • Primary 버튼을 한 화면에 여러 개 배치하지 않는다.
  • 삭제·위험 버튼을 일반 Primary처럼 강조하지 않는다.
  • 아이콘만 있고 의미가 불명확한 버튼을 사용하지 않는다 (aria-label 없는 Icon Button 금지).
  • Disabled 버튼에 클릭 이벤트를 처리하지 않는다.
  • 인라인 배경색·크기 오버라이드를 추가하지 않는다.
  • 취소/닫기 버튼을 과하게 강조하지 않는다.
08 Cursor 작업 지시 기준 Detail Draft
버튼 작업 전 필수 확인 지시문
아래 내용을 Cursor / Claude / Codex에게 버튼 관련 작업 지시 시 함께 전달한다.
  • 버튼을 추가·수정할 때는 반드시 STAM-Button-Guide.html을 먼저 확인한다.
  • 새 CSS 클래스, 새 JS, 새 토큰을 생성하지 않는다. 기존 공통 버튼 클래스만 사용한다.
  • Primary 버튼은 해당 화면에 이미 몇 개 있는지 먼저 파악한다. 추가 시 1개 초과 여부를 확인한다.
  • 삭제·초기화 액션에는 Danger 클래스를 사용하고, Dialog 확인 단계가 구현되어 있는지 확인한다.
  • Icon Button을 추가할 때는 aria-label이 포함되어 있는지 확인한다.
  • 드로어 하단 버튼 순서는 취소 → 저장/등록 순을 유지한다.
09 QA 체크리스트 Detail Draft
  • Primary 버튼이 화면당 1개 중심인지 확인한다.
  • 저장 / 등록 / 적용 버튼이 Primary 클래스로 명확히 적용되어 있는지 확인한다.
  • 취소 / 닫기 버튼이 과하게 강조되지 않았는지 확인한다 (Secondary 또는 Ghost).
  • 삭제 / 위험 액션이 Danger 클래스로 구분되어 있는지 확인한다.
  • 라이트 / 다크 모드 전환 시 버튼 대비가 충분한지 확인한다.
  • 좁은 화면에서 버튼 줄바꿈·고정 footer가 깨지지 않는지 확인한다.
  • 키보드 Tab 이동 시 :focus-visible 포커스 링이 표시되는지 확인한다.
  • Icon Button에 aria-label이 모두 지정되어 있는지 확인한다.
  • Disabled 버튼에 불필요한 클릭 이벤트가 처리되지 않는지 확인한다.
  • 드로어 하단 버튼 순서가 전 화면 일관되게 적용되어 있는지 확인한다.
