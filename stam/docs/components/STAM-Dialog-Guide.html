01 Overview 가이드
컴포넌트 Dialog (Modal)
Type Confirm · Warning · Danger · Approval · Information
CSS 위치 stam.components.css
상태 가이드 확정 v1

목적

  • Dialog는 사용자의 흐름을 일시적으로 멈추고 판단을 요구하는 컴포넌트다. 현재 작업을 보류하고 사용자가 명확하게 응답한 뒤에야 다음 단계로 진행할 수 있다.
  • STAM에서 Dialog는 확인, 경고, 결정, 승인/반려처럼 결과가 중요한 흐름에서만 사용한다. 단순 상세 보기나 입력 화면에는 Drawer 또는 Full View를 우선 검토한다.
  • Dialog 안에는 판단에 필요한 최소한의 정보만 담는다. 내용이 길어지면 Dialog가 아니라 별도 화면으로 분리한다.
Dialog 남발 금지 모든 알림과 안내를 Dialog로 처리하지 않는다. 확인 없이도 진행할 수 있는 가벼운 알림은 Toast · Snackbar로 대체한다. Dialog는 사용자의 판단이 반드시 필요한 상황에만 연다.

04 Button Rules — 버튼 규칙 완료
Button Guide 참조 Dialog 안의 버튼은 STAM Button Guide에서 정의한 Variant · Size · State 기준을 동일하게 따른다. Dialog Footer에는 Medium 크기 버튼을 기본으로 사용하며, 좁은 공간에서는 Small을 허용한다.

Dialog Footer 버튼 위계

역할 Variant 배치 위치 텍스트 원칙
주요 확인 액션 Primary Footer 가장 오른쪽 나가기, 승인 요청, 변경 확인, 확인
취소 / 닫기 Secondary 또는 Ghost Primary 왼쪽 취소, 닫기
삭제 / 반려 / 권한 해제 Danger Footer 가장 오른쪽 삭제하기, 반려하기, 공유 해제

버튼 원칙

  • Dialog 안 Primary 버튼은 1개만 둔다. 복수의 Primary는 사용자의 판단 우선순위를 무너뜨린다.
  • 삭제·반려·권한 해제는 Primary가 아닌 반드시 Danger 버튼으로 표시한다.
  • 위험 액션의 버튼 텍스트는 구체적으로 작성한다. 나쁜 예: "확인" — 좋은 예: "삭제하기", "반려하기", "공유 해제"처럼 동작을 명확히 표현한다.
  • 취소 버튼은 Primary보다 강조되면 안 된다. Secondary 또는 Ghost를 사용한다.
  • Information Dialog (단순 안내)는 Primary 단독 버튼("확인")만 사용한다.
05 Usage Rules — 사용 규칙 완료

사용 기준

  • 단순 상세 보기에는 Dialog를 사용하지 않는다. 상세 내용은 Drawer 또는 Full View로 열어야 한다.
  • 긴 입력 폼은 Dialog보다 Drawer 또는 Full View를 사용한다. Dialog 높이가 뷰포트 70%를 넘으면 별도 화면으로 분리한다.
  • 삭제·반려·권한 해제는 반드시 확인 Dialog를 거친다. 버튼 클릭 한 번으로 바로 실행되면 안 된다.
  • Dialog 제목은 사용자가 해야 할 판단을 명확히 보여준다. "알림", "확인" 같은 모호한 제목 사용을 금지한다.
  • 본문은 액션의 결과와 영향 범위를 설명한다. 되돌릴 수 없는 경우 이 점을 반드시 명시한다.

제한 사항

  • 닫기 버튼(✕)과 취소 버튼의 기능은 동일하게 동작해야 한다. 둘 다 Dialog를 닫고 진행 중인 액션을 취소한다.
  • Dialog 위에 Dialog를 중첩하지 않는다. 중첩이 필요한 구조라면 플로우를 재설계한다.
  • 모바일에서는 화면 폭에 맞게 안전 여백(최소 16px)을 유지한다. Dialog 너비는 화면 폭 - 32px 이하로 제한한다.
  • Overlay(배경 Dim) 클릭으로 Dialog를 닫는 것은 Confirm · Warning · Danger · Approval 유형에서 허용하지 않는다. 사용자가 명확히 취소를 선택해야 한다.
06 Do / Don't 완료
Do — 이렇게 사용한다
  • 위험 액션에 명확한 제목과 영향 설명을 제공한다
  • 버튼 텍스트를 구체적으로 작성한다 (삭제하기, 반려하기)
  • Dialog가 열릴 때 포커스를 첫 번째 버튼으로 이동한다
  • Esc 키로 Dialog를 닫을 수 있도록 처리한다
  • Overlay로 배경과 Dialog를 시각적으로 구분한다
  • 승인/반려 사유 입력 필요 여부를 유형별로 명확히 구분한다
Don't — 이렇게 하지 않는다
  • 모든 알림을 Dialog로 처리하지 않는다
  • 긴 폼을 작은 Dialog 안에 넣지 않는다
  • 삭제·반려 버튼을 Primary로 표시하지 않는다
  • "확인/취소"만 반복해서 의미를 모호하게 만들지 않는다
  • Dialog를 연속으로 중첩하지 않는다
  • 배경 클릭 한 번으로 위험 Dialog를 닫히게 하지 않는다

08 Accessibility — 접근성 기준 완료
기준 설명 적용 범위
role="dialog" Dialog 컨테이너에 role="dialog"를 지정해 스크린 리더가 모달 영역임을 인식하게 한다. 모든 Dialog
aria-modal="true" aria-modal="true"를 함께 적용해 배경 콘텐츠를 스크린 리더가 무시하도록 처리한다. 모든 Dialog
제목·설명 연결 aria-labelledby로 제목 요소를 연결하고, 본문이 있으면 aria-describedby로 본문도 연결한다. 모든 Dialog
Focus Trap Dialog가 열리면 포커스가 Dialog 내부로 이동하고, Tab 키가 Dialog 내 요소 사이에서만 순환해야 한다. Dialog 닫힘 시 원래 포커스 위치로 복원한다. 모든 Dialog
Esc 닫기 Esc 키로 Dialog를 닫을 수 있어야 한다. Esc는 취소(닫기)로만 동작하며, 위험 액션이 실행되어선 안 된다. 모든 Dialog
실수 방지 문구 Danger Dialog 본문에 "되돌릴 수 없습니다"와 같은 경고 문구를 포함한다. 색상만으로 위험 상태를 표현하지 않는다. Danger Dialog
색상 의존 금지 버튼 색상(Danger = 빨간색)만으로 위험 여부를 판단할 수 없도록, 텍스트("삭제하기", "반려하기")로도 반드시 의미를 전달한다. 모든 Dialog
주의 outline: none 전역 처리는 키보드 사용자의 포커스 경로를 완전히 차단한다. Dialog 안의 버튼도 :focus-visible 기준으로 포커스 링이 시각적으로 표시되어야 한다.
