01 Overview 가이드
컴포넌트 Form Elements
포함 유형 Create · Edit · Search · Inline · Approval · Attachment
CSS 위치 stam.components.css
상태 가이드 확정 v1

목적

  • STAM Form Guide는 산출물 등록/수정, 검색/필터, 승인/반려, 첨부/URL 입력, 프로젝트 설정 등 전체 화면에서 일관된 입력 폼 기준을 제공한다.
  • 입력 폼은 데이터 품질과 업무 흐름을 결정하는 핵심 컴포넌트다. 잘못 설계된 폼은 입력 오류, 데이터 누락, 승인 지연의 원인이 된다.
  • 화면마다 임의 폼 스타일을 만들지 않는다. 이 가이드에서 정의한 필드 구조, 레이아웃, 검증 기준을 공통으로 따른다.
  • Button Guide / Dialog Guide 기준을 Form과 함께 적용한다. 저장·취소 버튼 위계, 이탈 확인 Dialog는 각 가이드를 참조한다.
현재 구현 상태 현재 WBS·프로젝트 개요 화면의 필터 입력 요소는 각 화면 CSS에 직접 정의되어 있다. 공통 폼 컴포넌트 정규화는 이 가이드 확정 후 stam.components.css에 추가한다.
02 Form Types — 폼 유형 정의 완료

유형 정의

Type 목적 배치 위치 사용 예시
Create Form 신규 데이터 등록 Drawer 또는 Full View 산출물 등록, 프로젝트 생성
Edit Form 기존 데이터 수정 Drawer 또는 Full View 요구사항 수정, WBS 작업 수정
Search / Filter Form 목록 검색과 조건 필터 목록 상단 Toolbar 영역 상태, 담당자, 기간, 키워드 검색
Inline Form 짧은 단일 입력 목록 행 내 또는 하단 댓글 입력, 간단 메모
Approval Form 승인/반려/검토 의견 입력 Dialog 또는 Drawer 반려 사유, 승인 의견
Attachment / URL Form 첨부파일과 URL 참조 입력 Drawer 탭 내 섹션 파일 설명, URL명, 공개범위
폼 유형과 화면 선택 기준 긴 Create / Edit Form은 작은 Dialog에 넣지 않는다. Dialog 높이가 뷰포트 70%를 넘으면 Drawer 또는 Full View로 전환한다. Approval Form은 내용이 짧으면 Dialog에, 긴 사유 입력이 필요한 경우 Drawer에 배치한다.
03 Field Structure — 필드 구성 요소 완료

구성 요소

# 요소 역할 필수 여부
① Label 필드의 의미를 명확히 전달하는 텍스트. for 속성으로 input과 연결한다. Placeholder를 Label 대체로 사용하지 않는다. 필수
② Required Mark 필수 항목임을 표시하는 * 마크. Label 우측에 배치. 색상(#EF4444)과 텍스트를 함께 제공한다. 필수값에 필수
③ Input / Select / Textarea 실제 입력이 이루어지는 컨트롤 영역. Default · Focus · Filled · Error · Disabled · Readonly 상태를 처리한다. 필수
④ Placeholder 입력 전 힌트 텍스트. Label이 있을 때에만 보조 목적으로 사용. 입력 후 사라진다. Label을 대체하지 않는다. 권장
⑤ Help Text 업무 기준이나 입력 조건을 보충 설명하는 텍스트. 필드 아래에 위치. 선택
⑥ Error Message 유효성 실패 시 필드 바로 아래에 표시. 사용자가 직접 고칠 수 있는 문장으로 작성한다. 오류 시 필수
⑦ Counter 입력 글자수 현황. Textarea 우측 하단에 표시. 현재/최대 형식. 선택
⑧ Field Group 연관된 필드를 묶는 컨테이너. 구분선 또는 배경색으로 그룹을 시각화한다. 선택
⑨ Action Area 저장/취소 버튼 영역. Drawer footer 또는 Form 하단. Button Guide 기준을 따른다. 필수

04 Field Types — 필드 유형 정의 완료
유형 사용 위치 사용 예 주의사항
Text Input 단행 텍스트 — 이름, 제목 산출물명, 프로젝트명, URL 최대 글자수를 사전에 정의한다
Textarea 다행 텍스트 — 설명, 의견 산출물 설명, 반려 사유, 작업 메모 resize는 vertical only. 글자수 카운터 권장
Select 단일 선택 — 상태, 분류 상태(진행중/완료), 우선순위, 담당자 옵션명은 목록 필터와 용어 통일
Multi Select 복수 선택 — 태그, 참여자 관련 WBS 항목, 참여자 지정 선택 항목이 많아지면 태그 형태로 표시
Checkbox 다중 선택 — 공개범위, 옵션 내보내기 포함, 고객 공개 단독 사용 시 Toggle과 구분해 사용
Radio 단일 선택 — 2~5개 고정 옵션 우선순위(높음/중간/낮음), 공개범위 옵션이 5개 초과하면 Select로 전환
Date Picker 날짜 선택 — 기간, 마감일 시작일, 마감일, 검색 기간 날짜 형식은 YYYY-MM-DD로 통일. 직접 입력 허용
File Upload 첨부파일 등록 산출물 파일 첨부, 참고 자료 허용 확장자, 최대 파일 크기를 Help Text로 안내
URL Input 외부 링크 입력 Confluence, Jira, GitHub 링크 URL 형식 유효성 검사 필수. 프로토콜 포함 입력 안내
Search Input 목록 검색 키워드 입력 산출물 검색, 담당자 검색 검색 아이콘(돋보기)을 Prefix로 배치
Toggle 즉시 적용되는 ON/OFF 전환 알림 ON/OFF, 공개 여부 저장 버튼 없이 즉시 반영되는 경우에만 사용
Readonly Field 수정 불가 정보 표시 생성일, 생성자, 승인자 ID Disabled와 시각적으로 구분. 텍스트 복사는 허용
05 Layout Rules — 폼 배치 기준 완료

배치 원칙

  • 기본은 세로형 Label + Field 구조다. Label은 필드 상단에 고정하고 좌측 정렬 배치를 우선한다.
  • 관련 필드는 Field Group으로 묶는다. 그룹 제목(Group Title)과 구분선으로 섹션을 시각화한다.
  • 필수 입력값은 Label 옆에 *로 명확히 표시한다. 폼 상단에 "* 는 필수 항목" 안내를 추가할 수 있다.
  • 검색/필터 영역은 Compact하게 구성한다. Label 없이 Placeholder와 아이콘으로 대체 가능한 유일한 예외 영역이다.
  • 긴 입력 폼은 Dialog보다 Drawer 또는 Full View를 우선한다. Dialog 높이가 뷰포트 70%를 넘으면 분리한다.
  • 2열 이상 배치는 충분한 폭(최소 600px 이상)에서만 사용한다. 연관성이 높은 필드(시작일/종료일)끼리 묶는다.
  • 모바일(뷰포트 폭 640px 미만)에서는 1열로 전환한다.
  • 저장/취소 버튼은 Button Guide 기준을 따른다. Drawer footer 또는 Form 하단 우측 정렬로 배치한다.
검색/필터 레이아웃 예외 Search / Filter Form에서는 Label 없이 Placeholder + 아이콘으로 필드를 표현할 수 있다. 단, 스크린 리더를 위한 aria-label은 반드시 각 컨트롤에 직접 지정해야 한다.
06 Validation Rules — 유효성 검사 기준 완료

상태 정의

상태 시각 처리 발생 조건 메시지 위치
Required Label 옆 * 붉은색 표시 필수 필드 지정 시 항상 표시 Label 영역
Invalid 테두리 --fail 색상, 배경 미세 붉은 틴트 형식 오류, 필수값 누락, 중복 입력 필드 바로 아래 오류 메시지
Warning 테두리 --warn 색상, 경고 아이콘 권장값 범위 초과, 업무 주의 사항 필드 바로 아래 경고 메시지
Success 테두리 --pass 색상, 체크 아이콘 URL 유효성 확인, 중복 체크 통과 필드 바로 아래 (간결하게)
Readonly 배경 --bg-sur2, 커서 default 수정 권한 없음, 시스템 자동 입력 값 —
Disabled 전체 opacity 0.4, 커서 not-allowed 조건 미충족, 폼 제출 중 —
Loading 버튼 Loading 스피너, 입력 필드 비활성 저장 요청 중, 비동기 유효성 확인 중 버튼 내 스피너
Dirty / Unsaved 헤더 또는 탭에 미저장 표시 (선택) 초기값 대비 변경사항이 존재 이탈 시 Confirm Dialog 연동

검증 원칙

  • 오류 메시지는 필드 바로 아래에 표시한다. 상단 에러 박스에 모아서 표시하지 않는다.
  • 오류 메시지는 사용자가 직접 고칠 수 있는 문장으로 작성한다. 나쁜 예: "필수 항목입니다" → 좋은 예: "산출물명을 입력해 주세요."
  • 저장 실패(서버 오류)와 입력 오류(클라이언트 유효성 실패)를 구분해 처리한다.
  • 필수값 누락은 저장 시점에 명확히 안내하고 첫 번째 오류 필드로 포커스를 이동한다.
  • Dirty 상태에서 이탈할 때는 Confirm Dialog를 사용한다. Dialog Guide의 Confirm 유형을 따른다.
07 Usage Rules — 사용 규칙 완료

입력 설계 규칙

  • Placeholder를 Label 대체로 사용하지 않는다. Placeholder는 입력 후 사라지므로 의미를 전달하는 Label을 항상 제공한다.
  • 필수값과 선택값을 명확히 구분한다. 필수 필드에는 * 표시를 유지하고 선택 필드에는 "(선택)" 텍스트를 추가할 수 있다.
  • Select 옵션명은 업무 용어 기준으로 통일한다. 예: "진행중", "완료", "검토중" — 화면마다 다른 표현을 사용하지 않는다.
  • 날짜/상태/담당자 필드는 목록 필터와 같은 용어를 사용한다. 폼과 필터가 다른 용어를 쓰면 사용자가 혼란을 겪는다.
  • 저장 버튼은 입력 영역 하단 또는 Drawer footer에 둔다. 폼 중간이나 여러 위치에 분산시키지 않는다.
  • 삭제/반려 같은 위험 액션은 Form 내부에서 직접 실행하지 않는다. 반드시 Dialog로 이중 확인한다. Dialog Guide의 Danger / Approval 유형을 따른다.
  • 파일/URL 입력은 공개범위와 내보내기 포함 여부를 함께 고려한다. 관련 설정 필드를 동일 Form 내에 배치한다.
08 Do / Don't 완료
Do — 이렇게 사용한다
  • Label을 명확하게 작성한다
  • Help Text로 업무 기준과 입력 조건을 보완한다
  • 오류 메시지를 해당 필드 근처에 표시한다
  • 필수값과 선택값을 구분해 표시한다
  • 저장 중 Loading 상태를 버튼에 제공한다
  • 변경사항이 있을 때 이탈 시 Confirm Dialog를 연다
Don't — 이렇게 하지 않는다
  • Placeholder만으로 필드 의미를 전달하지 않는다
  • 너무 많은 필드를 한 화면에 몰아넣지 않는다
  • 긴 폼을 작은 Dialog에 넣지 않는다
  • 오류 메시지를 상단에만 몰아서 표시하지 않는다
  • 필드마다 다른 용어를 사용하지 않는다
  • 저장 버튼을 여러 위치에 남발하지 않는다

10 Accessibility — 접근성 기준 완료
기준 설명 적용 범위
label 연결 <label for="id">로 label과 input을 반드시 연결한다. 시각적 Label이 없는 필드(검색/필터)는 aria-label을 사용한다. 모든 필드
필수값 표시 * 시각 표시만으로 충분하지 않다. aria-required="true"를 함께 제공하고 스크린 리더에서 "필수" 문구가 인식될 수 있도록 처리한다. 필수 필드 전체
오류 메시지 role="alert" 또는 aria-live="polite"로 오류 메시지를 스크린 리더가 인식하도록 처리한다. 오류 상태 input에는 aria-invalid="true"와 aria-describedby로 오류 메시지를 연결한다. 오류 상태 필드
focus-visible 유지 outline: none 전역 처리를 하지 않는다. :focus-visible로 키보드 포커스 링이 시각적으로 표시되어야 한다. 모든 인터랙티브 요소
키보드 이동 순서 Tab 키 이동 순서가 시각적 레이아웃과 일치해야 한다. 2열 배치 폼에서는 왼쪽 → 오른쪽 → 다음 행 순서를 유지한다. 모든 폼
Disabled vs Readonly disabled 필드는 폼 제출에 포함되지 않는다. 값을 유지하면서 수정만 막으려면 readonly를 사용한다. 두 상태를 시각적으로 구분한다. Disabled / Readonly 필드
색상 의존 금지 오류/성공/경고 상태를 색상만으로 구분하지 않는다. 아이콘 + 색상 + 텍스트 메시지를 함께 제공한다. 모든 상태 표시
주의 Search / Filter Form에서 Label 없이 Placeholder만 사용할 경우, aria-label을 반드시 각 input에 직접 지정해야 한다. Placeholder 텍스트는 스크린 리더에서 Label로 인식되지 않는다.

13 Label / Placeholder / Help / Error — 문구 기준 1차 상세 기준

원칙

  • Label은 항상 의미가 명확해야 한다. 입력 대상이 무엇인지 한 눈에 알 수 있게 작성한다.
  • Placeholder는 Label을 대체하지 않는다. Placeholder는 입력 예시 또는 형식만 보여주고, 의미는 Label이 책임진다.
  • Help text는 입력 규칙 · 예시 · 제약을 설명한다. 짧고 구체적으로 쓴다.
  • Error message는 “무엇이 잘못되었는지 + 어떻게 고치는지”를 모두 알려준다. 단순 “오류”는 금지.
  • 같은 화면에서 “입력하세요”만 반복하는 placeholder는 의미 없는 노이즈다.
좋은 Label 예
  • 화면명
  • 담당자
  • 완료 예정일
  • 검토 요청 사유
  • 권한 그룹
  • 요구사항 ID
피해야 할 Label 예
  • 값
  • 내용
  • 입력
  • 선택
  • 기타
  • 정보

Error message 예

상황 좋은 문구 피해야 할 문구

14 Required / Optional — 필수 · 선택 표시 기준 1차 상세 기준
  • 필수값은 명확히 표시한다. (별표 또는 “필수” 라벨 등 한 가지 방식을 일관되게 사용)
  • Optional 표시를 모든 필드에 남발하지 않는다. 대부분이 선택값인 화면이라면 필수만 표기한다.
  • 필수 / 선택 표시 방식은 한 화면 / 한 가이드 안에서 한 가지로 일관되어야 한다.
  • 저장 버튼 disabled만으로 필수값 누락을 설명하지 않는다. 어떤 필드가 비었는지 즉시 알 수 있어야 한다.
  • 스크린리더가 필수 상태를 인지할 수 있게 한다 (예: aria-required 또는 label 보조).
핵심 원칙 “필수값 누락은 저장 실패가 아니라 입력 시점에 알린다.” 사용자가 저장 버튼을 눌렀을 때 비로소 알게 되는 패턴은 피한다.
15 Mobile / Narrow Width — 모바일 기준 1차 상세 기준
  • 430px 이하에서는 1열 stack을 우선한다. 좁은 폭에서 2열 / 3열을 억지로 유지하지 않는다.
  • Label과 input이 겹치거나 잘리지 않도록 line-height와 폭을 확보한다.
  • select · date · 숫자 input 등 가상 키보드와 상호작용하는 컨트롤은 터치 영역을 충분히 확보한다 (권장 44px).
  • 오류 메시지가 화면 밖으로 밀리지 않도록 한다. 긴 메시지는 줄바꿈을 허용한다.
  • 하단 고정 저장 버튼은 모바일 가상 키보드에 가려질 수 있다. Drawer 또는 sticky footer의 동작 기준을 함께 점검한다.
  • 긴 form은 섹션 단위로 묶고, 상단에 진행 위치를 인지할 수 있는 보조 정보를 둔다.
