문서 성격 — Decision / Design Baseline 본 문서는 PR #294에서 확정한 1차 제품화 범위를 실제 Auth · Project · Member 구조로 이어가기 위한 데이터 구조 설계 기준서이며, PR #296 (Firebase Auth / Firestore / Rules baseline) 착수 전에 반드시 확정되어야 할 논리 구조를 정의합니다.

핵심 문구 “1차에서는 보안을 과도하게 구현하지 않는다. 다만 나중에 ISMS-P에 준하는 보안·감사·접근통제 체계를 붙일 수 있도록 DB와 권한 구조는 확장 가능하게 설계한다.”
01 문서 목적 — Why This Guide Decision
  • PR #294에서 확정된 1차 제품화 범위를 실제 Auth / Project / Member 구조로 이어가기 위한 설계 문서이다.
  • 이번 PR에서는 구현하지 않고, PR #296 ~ #298 구현 기준을 정의한다.
  • Google 로그인 + projectMembers 이메일 게이트 구조를 데이터 모델로 정리한다.
  • 나중에 권한 · 감사 · 보안 정책을 붙일 때 DB를 갈아엎지 않도록 확장성 기준을 잡는다.
이 문서의 효력 범위 본 문서에서 정의한 엔티티 · 필드 · 상태 · 흐름 · Rules 판단 기준은 PR #296 ~ #298 구현 PR의 canonical 설계 기준으로 사용된다. 구조 변경이 필요한 경우 별도 결정 PR을 통해 본 문서를 먼저 갱신한 뒤 구현을 진행한다.
02 PR #294에서 상속받는 결정 — Inherited Decisions Inherited
  • 1차 Live 산출물은 요구사항 / WBS / 화면설계서 3개.
  • Google 로그인만 사용.
  • 공개 회원가입 없음.
  • projectMembers.email 기반 접근 게이트.
  • projectInvites는 후속 후보.
  • users, tenants, projects, projectMembers는 1차 데이터 구조에 포함.
  • tenantId는 후속 멀티테넌시 / 보안 확장을 위해 고려.
  • Owner / Admin / Editor / Viewer role 사용.
  • active project member만 project-scoped data 접근 가능.
  • Viewer write 차단.
  • 세밀한 권한 엔진은 후속.
03 1차 Auth 정책 — Google + projectMembers Gate Decision
결정 문구 · “로그인은 가볍게, 권한 구조는 진짜로 간다.” · “1차에서는 초대 메일 / 토큰을 구현하지 않고, Admin이 이메일을 등록하는 방식으로 접근을 제어한다.”

결정

  • Google 로그인 단일.
  • 이메일/비밀번호 로그인 없음.
  • 카카오/네이버 로그인 없음.
  • 공개 가입 없음.
  • 로그인한 Google 계정이 projectMembers.email에 등록되어 있어야 프로젝트 접근 가능.
  • 등록된 프로젝트가 없으면 Access Pending 화면.
  • 접근 권한이 없는 프로젝트 URL 접근 시 Access Denied 화면.

Email 정규화 기준

매칭 규칙 projectMembers.email과 Google 로그인 이메일 매칭은 정규화된 이메일 기준으로 처리한다.
  • email은 저장 / 비교 시 소문자 기준으로 정규화한다.
  • 앞뒤 공백을 제거한다.
  • emailNormalized 필드 사용을 권장한다.
  • Google 로그인 후 users.email과 projectMembers.emailNormalized를 비교한다.

필드 후보

  • email
  • emailNormalized

주의

  • 이메일 대소문자 차이로 접근 실패가 발생하지 않아야 한다.
  • Gmail의 점(.) 처리나 plus alias 정책은 1차에서 자동 병합하지 않는다.
  • 1차에서는 “소문자 + trim” 정규화까지만 기준으로 둔다.
  • 고급 이메일 alias 처리 정책은 후속 보안 / 계정 정책에서 별도 결정한다.
04 보안 / 확장성 기준 — Extensible Without Rebuilds Decision
원칙 1차에서는 보안 기능을 과도하게 구현하지 않는다. 다만 후속 보안 · 감사 · 접근통제 확장을 위해 DB와 권한 구조는 확장 가능하게 잡는다.
  • 모든 핵심 데이터는 projectId 기준으로 격리 가능해야 한다.
  • 향후 멀티테넌시와 조직 단위 보안 확장을 위해 tenantId 구조를 고려한다.
  • 사용자 / 멤버 / 권한 구조는 users, projects, projectMembers를 기준으로 분리한다.
  • projectMembers에는 role, status, email, emailNormalized, userId, joinedAt, removedAt 등 향후 권한 회수와 감사에 필요한 필드를 확장 가능하게 둔다.
  • 핵심 산출물에는 createdAt, createdBy, updatedAt, updatedBy 등 기본 audit 필드를 둔다.
  • 실제 감사로그, 관리자 행위 로그, ISMS-P 체크리스트, 개인정보 보관 / 파기 정책, 다운로드 통제, 세션 보안 등은 후속 보안 거버넌스 트랙으로 분리한다.

11 로그인 후 접근 게이트 흐름 — Two-Layer Judgement Decision
필수 분리 원칙 한 사용자는 여러 프로젝트에 동시에 속할 수 있고, 프로젝트마다 projectMembers.status가 다를 수 있다. 따라서 로그인 직후 전역 판정과 특정 프로젝트 접근 판정을 분리한다. 단일 멤버십 상태를 가정한 흐름을 만들지 않는다.

11-1. 로그인 직후 전역 판정

  1. 사용자가 Google 로그인
  2. users 문서 생성 / 갱신
  3. Google 계정의 정규화된 이메일 기준으로 projectMembers.emailNormalized 또는 정규화된 email 일치 항목을 모두 조회
  4. 조회된 멤버십 중 status === active인 프로젝트가 하나 이상 있으면 프로젝트 목록을 표시
  5. status === pending인 멤버십이 있고 이메일이 일치하면 userId를 연결하고 active 전환 후보로 처리
  6. active 전환 후 active 프로젝트가 하나 이상 생기면 프로젝트 목록을 표시
  7. active 프로젝트가 하나도 없으면 Access Pending 화면을 표시
Google 로그인 › users 생성/갱신 › projectMembers.emailNormalized 전체 조회 › pending → active 전환 › active 프로젝트 목록 or Access Pending
주의 · 로그인 직후에는 특정 프로젝트 하나의 removed 상태 때문에 전체 사용자를 Access Denied 처리하지 않는다. · Access Pending은 “로그인은 되었지만 접근 가능한 active 프로젝트가 없음”을 의미한다. · 공개 가입처럼 보이면 안 되며, 프로젝트 접근은 등록된 이메일 기반으로만 허용한다.

11-2. 특정 projectId URL 직접 접근 판정

  1. 현재 사용자 확인
  2. 접근하려는 projectId 확인
  3. 해당 projectId의 projectMembers에서 현재 사용자 userId 또는 정규화된 email 매칭
  4. 멤버십이 없으면 Access Denied
  5. 멤버십이 removed이면 Access Denied
  6. 멤버십이 pending이면 Access Pending 또는 참여 처리 흐름으로 이동
  7. 멤버십이 active이면 프로젝트 진입 허용
  8. 이후 role 기준으로 read / write / action 가능 여부를 판단
projectId URL 진입 › 해당 projectMembers 조회 › status 판정 › Access Denied / Pending / 진입 (role 판정)
주의 · Access Denied는 전역 사용자 상태가 아니라 “특정 projectId 접근 권한 없음”에 대한 판정이다. · 한 프로젝트에서 removed 상태여도 다른 프로젝트에서 active라면 사용자는 active 프로젝트 목록에 접근할 수 있어야 한다. · 프로젝트 접근은 항상 projectId와 해당 프로젝트의 projectMembers 기준으로 판단한다.

13 데이터 격리 기준 — projectId / tenantId Decision
  • 모든 project-scoped 데이터는 projectId를 가져야 한다.
  • 멀티테넌시 확장을 고려하는 데이터는 tenantId를 가져야 한다.
  • 사용자가 접근할 수 있는 프로젝트 목록은 projectMembers 기준으로 계산한다.
  • 직접 URL 접근 시에도 projectMembers 확인이 필요하다.
  • projectId 없는 산출물 데이터는 1차 구현에서 허용하지 않는다.
