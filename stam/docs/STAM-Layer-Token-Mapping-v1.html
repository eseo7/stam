STAM Layer Token Mapping v1 Mapping Draft

본 문서는 실제 Layer / z-index CSS 구현이 아니라, Topbar · Sticky Header · Dropdown · Popover · Calendar · Drawer · Dialog · Toast · Global Loading이 서로 겹칠 때 어떤 순서로 보여야 하는지를 토큰 기반으로 사전 확정하는 매핑 문서이다. Dropdown / Select / Date Picker popup이 Drawer 또는 Full Detail 화면에서 뒤로 깔리거나 잘리는 사고를 차단하기 위한 기준을 정의하며, 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/**를 절대 수정하지 않는다.

Mapping Draft CSS Implementation Hold Layer Collision Prevention Pre-Implementation No CSS Edit

판정 근거 — Form Token Mapping v1에서 “Drawer 내부 dropdown / calendar popup은 portal로 렌더링되어야 하며 z-index는 layer token으로 관리한다”는 원칙이 동결되었으나, 실제 layer token명·값·순서가 확정되지 않은 상태이다. Drawer 내부 select가 Drawer panel 뒤에 깔리는 사고, Dialog 내부 calendar가 Dialog backdrop 아래로 사라지는 사고, Full Detail에서 sticky header가 dropdown을 가리는 사고를 사전에 차단하기 위해 본 문서로 layer 매핑과 충돌 방지 원칙을 확정한다.

01 Overview — 문서 목적 목적

CSS Token Readiness Check가 “전체 토큰 충분성”을, Token Inventory v1이 “신규 토큰 후보 목록”을, Button / Form Token Mapping v1이 “컴포넌트별 토큰 매핑”을 확정했다면, 본 Layer Token Mapping v1은 “화면에 떠 있는 여러 컴포넌트가 어느 순서로 겹쳐야 하는지” z-index 축으로 1:1 매핑을 확정한다. STAM의 수정 흐름은 리스트 → Drawer 수정 → Full Detail 수정 → Drawer 복귀로 순환하며, 그 사이사이에 Topbar · Sticky Table Header · Dropdown · Calendar · Toast · Global Loading이 동시에 떠 있을 수 있다. 본 문서는 이 모든 레이어가 충돌하지 않도록 token 기반 z-index 체계를 정의하며, Dropdown · Select · Date Picker popup이 Drawer / Full Detail 전환에서 사라지거나 뒤에 깔리는 사고를 차단한다. 기획자·PM은 “어떤 화면에서 어떤 레이어가 위에 떠야 하는지”를 한 화면에서 읽고 승인할 수 있어야 하며, 개발자는 후속 Layer CSS 구현 단위에서 본 문서를 그대로 토큰 정의 기준으로 사용한다.

주의 — Mapping은 코드 변경이 아니다 본 문서의 Candidate 토큰명·Suggested Value·Used By는 모두 매핑 기준 후보이며, 실제 토큰 추가·CSS 클래스 작성·z-index 부여는 후속 Layer CSS 구현 단위에서 코드 변경과 함께 확정한다. 본 문서만으로 stam.tokens.css · stam.components.css 등 어떤 CSS 파일도 수정하지 않는다.
핵심 원칙 — popup은 owning context보다 위에 떠야 한다 STAM의 select / date picker popup은 Drawer · Dialog · Full Detail · 일반 페이지 어느 곳에서 열리더라도 owning context(자신을 띄운 컨테이너)보다 위에 떠야 하며, overflow hidden 부모에 잘리지 않아야 한다. 이를 위해 popup은 portal 렌더링 + layer token 기반 z-index를 사용한다. 상세 규칙은 Section 06–08에서 정의한다.
02 Layer Scope — 본 매핑이 다루는 레이어 범위 범위

아래 14개 레이어가 본 매핑의 대상이다. 시각적 컴포넌트 레이어(Topbar · Dropdown · Drawer · Dialog · Toast)와 보조 레이어(Sticky Header · Backdrop · Global Loading)를 모두 포함한다.

Base Content
페이지 본문 · 카드 · 표 · 일반 컨텐츠. 모든 레이어의 기준점.
z: 0
Sticky Table Header
WBS · 산출물 · 인사 등 테이블의 sticky 헤더. 스크롤 시 고정.
z: 100
Sidebar
STAM App Shell 좌측 네비게이션. 페이지 스크롤과 무관하게 고정.
z: 200
Topbar
STAM App Shell 상단바 (48px). Sticky · 검색 · 프로필 영역 포함.
z: 300
Drawer Backdrop
Drawer 오픈 시 본문 dim layer. 클릭 시 Drawer close.
z: 1000
Drawer
우측 또는 좌측 Drawer panel. 수정·미리보기·요약 화면.
z: 1100
Dialog Backdrop
Dialog 오픈 시 본문 dim layer. Drawer 위까지 덮음.
z: 1200
Dialog
Confirm · Alert · 작은 입력 모달. Drawer 위에서 오픈 가능.
z: 1300
Dropdown
Select / Custom Dropdown / Menu / Context Menu popup.
z: 1500
Select Option Panel
Select trigger 클릭 시 옵션 패널. Dropdown과 동일 layer.
z: 1500
Date Picker Calendar
Date Picker 트리거 클릭 시 calendar popup. Dropdown과 동일 layer.
z: 1500
Popover / Tooltip
툴팁 · 보조 popover · 헬프 박스. Dropdown과 동일 layer.
z: 1500
Toast
알림 · 성공 · 에러 · 진행 메시지. 모든 panel 위에서 알림.
z: 2000
Global Loading Overlay
전체 화면 로딩 dim + 스피너. Toast와의 우선순위는 본 문서에서 결정.
z: 3000
본 문서가 다루지 않는 레이어 Print 모드 전용 overlay, iframe 내부 자체 z-index, 외부 SaaS 위젯(채팅 · 분석 등) layer는 v1 범위 밖이며 후속 단계에서 별도 매핑한다. CSS position: fixed를 사용하지 않는 일반 본문 컴포넌트는 모두 Base Content layer로 간주한다.
03 Layer Order Principle — 기본 순서 원칙 원칙

STAM은 “일반 화면 stack order”와 “owning context 내부 popup 예외” 두 원칙을 동시에 사용한다. 두 원칙이 충돌하는 경우 popup은 항상 owning context보다 위에 떠야 한다는 원칙이 우선한다.

①Default Stack Order 기본 화면 z-index 순서 (아래 → 위)
3000Global Loading Overlay--z-global-loading
2000Toast--z-toast
1500Dropdown · Popover · Calendar (Portal)--z-dropdown · --z-popover · --z-calendar
1300Dialog--z-dialog
1200Dialog Backdrop--z-dialog-backdrop
1100Drawer--z-drawer
1000Drawer Backdrop--z-drawer-backdrop
300Topbar--z-topbar
200Sidebar--z-sidebar
100Sticky Table Header--z-sticky
0Base Content--z-base
기본 순서 원칙
  • Base Content < Sticky Header < Sidebar < Topbar < Drawer (Backdrop → Panel) < Dialog (Backdrop → Panel) < Dropdown / Popover / Calendar < Toast < Global Loading 순서 보장
  • 같은 layer 내에서 backdrop은 panel보다 항상 아래(--z-drawer-backdrop < --z-drawer, --z-dialog-backdrop < --z-dialog)
  • Sticky Header는 본문보다 위지만 Topbar보다 아래 — Topbar 아래로 스크롤되어 들어가지 않아야 함
  • Sidebar는 본문보다 위 · Topbar보다 아래 (Topbar가 Sidebar 위로 가로지름)
Owning Context 예외 원칙
  • Drawer 안에서 열리는 Dropdown / Calendar / Popover는 Drawer panel보다 위에 떠야 함
  • Dialog 안에서 열리는 Dropdown / Calendar / Popover는 Dialog panel보다 위에 떠야 함
  • Full Detail (페이지 본문) 화면에서 열리는 Dropdown / Calendar / Popover는 sticky header · topbar보다 위에 떠야 함
  • 이를 위해 popup은 --z-dropdown을 단일 token으로 사용하며, Dialog 위 layer로 설정한다 (1500 > 1300)
  • popup은 반드시 portal 렌더링 — Drawer · Dialog의 overflow: hidden 부모에 갇히지 않아야 함
Drawer ↔ Dialog 우선순위
  • Drawer가 열린 상태에서 Dialog가 열리면 Dialog가 우선 (1300 > 1100)
  • Dialog backdrop은 Drawer panel보다 위로 덮을 수 있음 (1200 > 1100)
  • Dialog 닫힘 시 focus는 Drawer 내부 원래 요소로 복귀
  • 두 모달이 동시에 떠 있을 때 ESC는 Dialog만 닫고 Drawer는 유지
Toast / Global Loading 원칙
  • Toast는 Dropdown · Dialog · Drawer 모두 위에 표시 (2000)
  • Global Loading은 Toast보다도 위에 표시하는 것을 기본 후보로 둠 (3000)
  • 단 Loading 중에도 Toast 알림이 보여야 하는 시나리오 발생 시 Loading < Toast로 재합의 필요 — Decision Needed
  • Toast는 자동 dismiss · Loading은 작업 종료 시 dismiss — 두 layer가 동시에 떠 있는 시간이 짧아야 함
z-index 임의 숫자 금지 원칙 STAM CSS·JS 어디에서도 z-index: 9999, z-index: 100 같은 임의 숫자 사용을 금지한다. 모든 z-index는 본 문서의 layer token (--z-base ~ --z-global-loading) 13종 중 하나만 사용한다. 새로운 layer가 필요한 경우 본 문서 v2로 재합의하여 token을 추가한 뒤에만 사용 가능하다.
04 Layer Token Candidate Table — z-index 토큰 후보 P0 매핑

Token Inventory v1의 Layer P0 후보를 풀어 정리한다. Suggested Value는 후속 단계에서 확정되며, 본 문서에서는 후보값으로 둔다. 13개 토큰은 STAM 전역에서 사용 가능한 모든 z-index 값을 망라한다.

Token Candidate Layer Role Suggested Value Used By Collision Risk Priority Decision Status
--z-base Base 0 페이지 본문 · 카드 · 표 · 일반 영역 본문 내 임의 z-index: 1이 sticky header를 뚫는 경우 P0 후보
--z-sticky Sticky 100 Table sticky header · WBS · 산출물 · 인사 표 Topbar가 본문 위로 스크롤될 때 sticky header가 Topbar를 덮음 P0 후보
--z-sidebar Shell 200 STAM App Shell 좌측 네비게이션 Sidebar가 Topbar 위로 올라가 검색·프로필 UI를 가림 P0 후보
--z-topbar Shell 300 STAM App Shell 상단바 (48px) Topbar가 Drawer · Dialog · Dropdown 위로 노출 P0 후보
--z-drawer-backdrop Backdrop 1000 Drawer 오픈 시 본문 dim layer Drawer panel 위까지 backdrop이 덮어 panel이 가려짐 P0 후보
--z-drawer Drawer 1100 Drawer panel 본체 (우측 / 좌측) Drawer 위에서 dropdown 열렸을 때 dropdown이 Drawer 뒤로 깔림 P0 후보
--z-dialog-backdrop Backdrop 1200 Dialog 오픈 시 본문 dim layer (Drawer 위까지) Dialog backdrop이 Dialog panel 위까지 덮어 panel이 가려짐 P0 후보
--z-dialog Dialog 1300 Dialog panel · Confirm · Alert · 작은 입력 모달 Dialog 내부 dropdown / calendar가 Dialog 뒤로 깔림 P0 후보
--z-dropdown Popup 1500 Select · Custom Dropdown · Menu · Context Menu popup 임의 z-index: 9999로 Toast를 뚫고 올라감 P0 후보
--z-popover Popup 1500 Tooltip · Help popover · 보조 안내 Dropdown과 다른 값으로 분리되면 동시 노출 시 stacking 모호 P1 후보
--z-calendar Popup 1500 Date Picker Calendar popup Dropdown보다 낮게 두면 Date Picker 위에서 Dropdown이 calendar를 가림 P0 후보
--z-toast Toast 2000 알림 · 성공 · 에러 · 진행 메시지 Dropdown / Dialog 보다 낮게 두면 모달 위에서 toast가 안 보임 P0 후보
--z-global-loading Loading 3000 전체 화면 로딩 dim + 스피너 Toast와의 상하 결정 — 본 문서에서 Decision Needed P1 결정 필요
Suggested Value 설계 원칙 인접 layer 간 최소 100 간격을 두어 후속 신규 layer 추가 여지를 남긴다 (예: --z-dropdown과 --z-toast 사이 1500–2000 영역에 새 layer 추가 가능). 모든 값은 정수이며 음수는 사용하지 않는다 (음수 z-index는 부모 stacking context 외부로 빠지는 사고를 유발). 토큰은 stam/css/stam.tokens.css에 정의하며, 컴포넌트는 var(--z-*) 만 참조한다.
Decision Needed — Global Loading vs Toast “Loading 중에도 사용자에게 진행 메시지를 toast로 보여야 한다”는 시나리오와 “Loading은 전체 화면을 막아야 한다”는 시나리오가 충돌한다. 본 문서는 기본 후보로 --z-global-loading: 3000 > --z-toast: 2000을 제안하지만, “Loading dim 위에 toast가 떠야 한다”가 PM 결정으로 채택되는 경우 두 토큰 값을 swap한다 (Loading: 1900 / Toast: 2000). 본 결정은 Section 11 Gate Checklist 항목으로 다룬다.

06 Drawer / Dropdown Collision Rule — Drawer 내부 popup 충돌 방지 규칙

STAM 수정 흐름의 가장 큰 위험 — Drawer 수정 화면에서 Select / Date Picker popup이 Drawer panel 뒤에 깔리거나 panel overflow에 잘려서 사라지는 사고를 차단한다.

MustDrawer 내부 popup 원칙
  • Drawer 안의 Select Dropdown은 Drawer panel보다 위에 노출되어야 함 (--z-dropdown 1500 > --z-drawer 1100)
  • Drawer 안의 Date Picker Calendar는 Drawer panel보다 위에 노출되어야 함 (--z-calendar 1500 > --z-drawer 1100)
  • Drawer 안의 Tooltip / Popover도 동일하게 Drawer panel보다 위에 표시
  • popup은 반드시 portal 렌더링 — document.body 직속에 mount되어 Drawer의 stacking context에 갇히지 않아야 함
  • popup의 위치 계산은 trigger 요소 기준으로 매 frame 재계산 (Drawer 내부 scroll · resize 시에도 따라옴)
Must NotDrawer 충돌 금지 패턴
  • Drawer panel overflow: hidden 부모 안쪽에 popup을 직접 자식으로 렌더링하는 것 — 잘림 사고
  • Drawer 내부에서 popup의 z-index를 임의 숫자(z-index: 5, z-index: 9999)로 부여 — 다른 모달과 충돌
  • popup이 Drawer backdrop보다 아래로 깔리는 것 — backdrop dim에 popup이 가려져 안 보임
  • popup이 Drawer 밖 content 위로 너무 올라가 Dialog까지 뚫고 가는 것 — Dialog가 열렸을 때 dropdown이 Dialog 위에 노출되면 안 됨
  • Drawer 내부 scroll 영역에 popup이 잘려 사라지는 것 — popup은 viewport 기준 위치이며 Drawer scroll에 따라 흔들리지 않아야 함
Layerz-index 적용 순서
  • Drawer Backdrop --z-drawer-backdrop (1000) — 본문 dim
  • Drawer Panel --z-drawer (1100) — Drawer 본체
  • Drawer 내부 Dropdown / Calendar --z-dropdown / --z-calendar (1500) — Drawer panel 위
  • Drawer 위에서 Dialog 열림 → Dialog Backdrop --z-dialog-backdrop (1200) → Dialog Panel --z-dialog (1300)
  • Dialog 내부 Dropdown / Calendar --z-dropdown (1500) — Dialog panel 위 (같은 단일 token 재사용)
QA회귀 판정 기준
  • Drawer 수정 화면에서 Select 클릭 → 옵션 패널이 Drawer 본문 위에 보임 → PASS
  • Drawer 수정 화면에서 Date Picker 클릭 → calendar popup이 Drawer 본문 위에 보임 → PASS
  • popup이 Drawer panel 뒤에 가려 안 보이면 FAIL
  • popup이 Drawer 내부 scroll에 의해 잘리거나 흔들리면 FAIL
  • popup이 Drawer backdrop 아래로 깔리면 FAIL
  • popup이 Dialog 위로 올라가면 FAIL (Dialog가 동시에 열려 있을 때만 해당)
portal 렌더링은 layer 매핑의 전제 조건이다 본 Rule은 popup이 portal로 렌더링됨을 전제로 한다. Drawer panel 자식 DOM에 popup을 직접 두는 구현은 stacking context 격리에 의해 본 z-index 매핑이 무효화된다. popup 컴포넌트는 mount 시 document.body 또는 별도 portal root에 append되어야 하며, position은 fixed로 viewport 좌표를 사용한다.
07 Dialog / Dropdown Collision Rule — Dialog 내부 popup 충돌 방지 규칙

Dialog(Confirm · Alert · 입력 모달)에서 Select · Date Picker가 사용될 때, popup이 Dialog 뒤로 깔리거나 Dialog backdrop 아래로 사라지는 사고를 차단한다.

MustDialog 내부 popup 원칙
  • Dialog 내부 Select Dropdown은 Dialog panel 위에 노출되어야 함 (--z-dropdown 1500 > --z-dialog 1300)
  • Dialog 내부 Date Picker Calendar는 Dialog panel 위에 노출되어야 함 (--z-calendar 1500 > --z-dialog 1300)
  • Dialog 내부 popup도 Drawer와 동일하게 portal 렌더링 사용
  • popup이 Dialog backdrop보다 위에 표시 (--z-dropdown 1500 > --z-dialog-backdrop 1200)
  • popup이 Toast보다는 아래로 유지 (--z-dropdown 1500 < --z-toast 2000)
Must NotDialog 충돌 금지 패턴
  • Dialog panel 자식 DOM에 popup을 직접 두어 stacking context에 가두는 것
  • Dialog 위에서 dropdown이 panel 뒤로 깔리는 것 — confirm 화면에서 옵션 선택 불가
  • Dialog backdrop이 dropdown 위로 덮어 클릭이 backdrop으로 흡수되는 것
  • popup이 Dialog 자체보다 위로 너무 올라가 Toast까지 뚫는 것
  • Dialog 열린 상태에서 dropdown 외부 클릭 시 Dialog까지 함께 닫혀버리는 것 (popup만 닫혀야 함)
OrderDialog ↔ Drawer 동시 노출
  • Drawer가 열려 있을 때 Dialog가 열리면 Dialog가 Drawer보다 위 (1300 > 1100)
  • Dialog backdrop은 Drawer panel 위까지 덮음 (1200 > 1100) — Drawer dim 처리
  • Dialog가 열려 있을 때 Drawer는 인터랙션 차단
  • Dialog 내부에서 열리는 Dropdown도 동일 --z-dropdown (1500) — Drawer · Dialog 모두 위에 표시
  • Dialog 닫힘 시 focus는 Drawer 내부 원래 trigger로 복귀
FocusFocus Trap 호환성
  • Dialog는 focus trap을 사용하나, popup이 portal로 body 직속에 mount되어 trap 외부에 위치할 수 있음 — 이 경우 popup이 열리는 동안 Dialog focus trap을 일시 확장
  • popup 닫힘 시 focus는 popup trigger로 복귀 — Dialog 자체로 빠지지 않음
  • ESC 키는 popup이 열려 있으면 popup만 닫고 Dialog는 유지 → 두 번째 ESC에서 Dialog 닫힘
  • Tab 이동 순서가 layer 순서와 충돌하면 안 됨 — Dialog 내부 → popup 옵션 → 다시 Dialog 내부 순으로 유지
Toast는 Dialog · Dropdown 모두 위에 떠야 한다 저장 성공 / 실패 toast가 Dialog · Dropdown 위에 보이지 않으면 사용자가 결과를 인지하지 못한다. 따라서 --z-toast는 항상 Dialog · Dropdown · Drawer · Topbar보다 큰 값을 유지한다. 단, Global Loading dim 상태에서는 toast 노출 정책을 Section 04 Decision Needed 항목에서 결정한다.
08 Full Detail / Page Content Rule — 전체 상세 / 페이지 본문 layer 규칙 규칙

Drawer가 닫힌 “Full Detail 수정 화면”과 일반 페이지 본문(Dashboard · WBS · 산출물 · 인사)에서 Select / Date Picker popup이 sticky header · topbar 뒤로 깔리거나 native UI로 회귀하는 사고를 차단한다.

MustFull Detail layer 원칙
  • Full Detail 화면에서 Select · Date Picker는 STAM custom 컴포넌트를 사용 — native UI 회귀 금지 (Form Token Mapping v1 동일)
  • Full Detail 내 dropdown / calendar는 Topbar · Sticky Header보다 위에 떠야 함 (--z-dropdown 1500 > --z-topbar 300 > --z-sticky 100)
  • 같은 popup 컴포넌트가 Drawer · Dialog · Full Detail에서 동일한 z-index token (--z-dropdown)을 사용
  • 페이지 본문 overflow와 popup layer는 충돌하지 않아야 함 — popup은 portal로 viewport에 mount
  • 테이블 내부 인라인 셀 편집 시 열린 dropdown / calendar도 sticky header를 가리지 않고 위에 표시
Must NotFull Detail 금지 패턴
  • Full Detail에서 Select·Date Picker가 native <select>·<input type="date"> UI로 회귀하는 것 — 디자인 위반 + layer 매핑 무효
  • Full Detail 내 dropdown이 sticky header 뒤에 깔리는 것 — 페이지 상단 옵션이 안 보임
  • Full Detail 내 dropdown이 Topbar 뒤에 깔리는 것 — 화면 상단 영역 옵션이 안 보임
  • 페이지 본문의 임의 overflow: hidden이 popup을 잘라먹는 것 — popup은 portal로 회피
  • 테이블 내부 / Drawer 내부 / Full Detail 내부에서 서로 다른 layer rule을 사용하는 것 — 모두 동일 token 사용
Same레이어 일관성 원칙
  • 테이블 셀 인라인 편집 → 같은 dropdown 컴포넌트 → 같은 --z-dropdown 사용
  • Drawer 내부 form → 같은 dropdown 컴포넌트 → 같은 --z-dropdown 사용
  • Full Detail 내 form → 같은 dropdown 컴포넌트 → 같은 --z-dropdown 사용
  • Dialog 내부 form → 같은 dropdown 컴포넌트 → 같은 --z-dropdown 사용
  • 네 화면 모두에서 popup의 시각·동작·layer가 동일하게 보장된다 (Form Token Mapping v1의 Sync Rule과 정합)
QA회귀 판정 기준
  • Full Detail에서 Select 클릭 → 옵션 패널이 sticky header · Topbar 위에 보임 → PASS
  • Full Detail에서 native select / date input UI가 1개라도 노출되면 FAIL
  • Drawer → Full Detail 전환 후 같은 필드의 dropdown이 다른 모양으로 보이면 FAIL
  • 테이블 인라인 편집 시 dropdown이 sticky header 뒤로 깔리면 FAIL
  • 페이지 본문 overflow에 popup이 잘리면 FAIL
Form Token Mapping v1과의 정합 본 Rule은 Form Token Mapping v1 Section 09 “Drawer / Full Detail Sync Rule”의 layer 측면을 구체화한다. “두 화면에서 같은 dropdown / calendar 컴포넌트가 같은 시각·동작을 보장한다”는 원칙은 layer 측면에서 “두 화면에서 같은 --z-dropdown token을 사용한다”로 풀린다. Form Mapping v1의 portal 렌더링 원칙이 본 문서의 layer 매핑이 작동하기 위한 전제 조건이다.
09 Overlay / Backdrop Rule — dim layer 정책 규칙

Drawer · Dialog · Loading의 dim layer(backdrop)와 panel 본체의 관계를 정리한다. backdrop은 항상 panel보다 아래, base content보다는 위에 있다.

Overlay Type Required Token Position vs Panel Click Behavior Stack Behavior Decision Status
Drawer Backdrop --z-drawer-backdrop (1000) Drawer panel 아래 (1000 < 1100) 클릭 시 Drawer close (옵션) Topbar · Sidebar · Sticky · Base 위 / Dialog · Dropdown 아래 후보
Dialog Backdrop --z-dialog-backdrop (1200) Dialog panel 아래 (1200 < 1300) 클릭 시 Dialog close 또는 차단 (variant 별) Drawer · Topbar 위까지 덮음 / Dropdown · Toast 아래 후보
Global Loading Overlay --z-global-loading (3000) 모든 panel · backdrop · popup 위 클릭 차단 — pointer-events 모두 막음 Toast와의 상하는 본 문서 Decision Needed에서 결정 결정 필요
Modal Dim Layer 위 3종 중 해당 — — Drawer / Dialog / Loading 별 자체 backdrop 사용 — 별도 layer 신설 금지 후보
Click-outside Close Layer — popup 자체와 동일 layer · 단 pointer-events: auto는 popup만 활성 외부 클릭 → popup close backdrop처럼 시각 dim을 두지 않고 투명 click-trap만 사용 후보
backdrop 기본 정책
  • backdrop은 panel보다 항상 아래 z-index
  • backdrop은 base content보다 항상 위 z-index
  • Dialog backdrop은 Drawer panel보다 위에 올 수 있음 (1200 > 1100) — 의도된 동작
  • backdrop dim 색은 토큰 기반 (별도 컬러 토큰) — z-index와 별개로 정의
  • backdrop은 panel과 같이 mount / unmount되어 stale dim 잔존 금지
click outside / close-trap 정책
  • popup(Dropdown · Calendar · Popover)은 시각 dim 없이 transparent click-trap을 사용
  • click-trap은 popup과 같은 layer 또는 한 단계 아래 — pointer-events로 외부 클릭만 잡음
  • 외부 클릭 → popup만 close, 같은 stack의 Dialog · Drawer는 닫히지 않음
  • ESC 키 우선 처리 — 가장 위 popup → Dialog → Drawer 순으로 닫힘
Loading Overlay 정책
  • Loading은 모든 panel · backdrop · popup · toast 위에 두는 것이 기본 후보
  • Loading 중에는 pointer-events 모두 차단 · 키보드 입력도 차단(ESC 제외 검토)
  • Loading dim 색은 backdrop과 다르게 — 사용자 인지 가능한 톤
  • Loading 노출 시 page scroll lock — body overflow hidden
  • Toast와의 상하는 본 문서 Section 04 Decision Needed에서 PM 결정
Modal dim 단일화 원칙
  • “dim layer”라는 별도 token을 새로 만들지 않음 — Drawer / Dialog / Loading의 자체 backdrop을 사용
  • Drawer 위 Dialog가 떠 있을 때 dim은 Dialog backdrop으로 통합 — 이중 dim 금지
  • dim 강도 변경이 필요한 경우 backdrop의 색 토큰을 조정 (z-index와 별개)
  • 임시 dim이 필요한 경우 (예: drag preview overlay) 본 문서 v2에서 신규 token 합의
backdrop의 클릭 동작은 variant 별로 다르다 Drawer backdrop은 클릭 시 Drawer close (기본). Dialog backdrop의 클릭 동작은 Dialog variant 별로 다르며 (Confirm: 차단 / Light Dialog: close), 본 layer 매핑은 클릭 동작은 다루지 않는다. 본 문서는 z-index 순서만 정의하고, 클릭 동작 정책은 Drawer Guide / Dialog Guide에서 정의한다.
10 Do / Don't — 구현 원칙 원칙
Do
  • 모든 z-index는 layer token 기반으로 구현 (--z-base ~ --z-global-loading 13종만 사용)
  • popup이 열리는 owning context를 항상 고려 — Drawer · Dialog · Full Detail · 테이블 내부 모두 같은 token 사용
  • popup은 반드시 portal 렌더링 — document.body 직속에 mount하여 stacking context 격리 회피
  • Drawer · Dialog · Full Detail · Light · Dark 5개 상태 동시 QA
  • light / dark에서 overlay 투명도(backdrop · dim)가 모두 충분히 보이는지 확인
  • overflow hidden 부모에 popup이 갇히지 않는지 확인 — popup은 viewport 기준 fixed position
  • backdrop은 panel보다 아래, base content보다 위로 유지
  • Dialog가 Drawer 위에서 열릴 때 Dialog backdrop이 Drawer까지 dim 처리
Don't
  • z-index 숫자(9999, 100, 1)를 컴포넌트 CSS에 직접 하드코딩
  • 컴포넌트마다 임의 z-index를 부여 — 단일 token 13개 외 추가 금지
  • popup을 native <select> · <input type="date">에 맡겨 OS 기본 UI로 회귀
  • Drawer · Dialog 안쪽 popup이 panel 뒤에 깔리게 두기 (portal 미사용 시 발생)
  • overflow hidden으로 popup을 잘라먹기 — Drawer · Dialog · Full Detail에서 자주 발생
  • 전체 CSS를 한 번에 수정 — layer 구현은 token 추가 또는 단일 layer 문제만 제한 수정
  • Drawer 전용 dropdown layer와 Full Detail 전용 dropdown layer를 분리 — 같은 token 사용
  • backdrop을 panel과 같은 z-index 또는 위로 두기 — panel이 backdrop에 가려짐

  • git diff로 허용 파일 외 변경 없음 확인
  • staging 배포 후 URL(https://stam-design-staging.web.app/)에서 QA 진행
롤백 기준
  • Docs 또는 제품 화면에서 layer 회귀 1건 이상 발생 시 즉시 revert
  • 토큰명 충돌(기존 --stam-*, --btn-*, --input-* 등) 발생 시 본 문서 v2로 재합의
  • Drawer · Dialog · Full Detail 중 한 화면이라도 popup이 깔리면 즉시 revert
  • light / dark 중 한 테마에서만 깨질 경우 즉시 revert
  • native select · date input 회귀가 1건이라도 발견되면 즉시 revert
  • 임의 z-index 숫자 사용이 발견되면 즉시 revert
핸드오프 시 명시 사항 본 문서는 매핑 기준 문서이며, CSS·JS·페이지·자산 변경 권한을 포함하지 않는다. Cursor는 Section 04 Token Mapping 표와 Section 06–09 충돌 / overlay rule만 코드로 옮기며, 새로운 토큰명·값을 임의로 추가하지 않는다. Drawer · Dialog · Full Detail 중 한쪽만 위해 별도 layer를 분기 구현하는 것은 본 문서 위반으로 간주한다.
