문서 성격 — Infra Baseline / Decision 본 문서는 PR #297의 Firebase Auth · Firestore · Rules baseline 기준서이며, 실제 Auth / CRUD 구현이 아니라 “붙이기 전에 보안 기본문과 설정 기준을 먼저 잡는다”는 성격의 infra 결정 문서입니다.

핵심 문구 “1차에서는 보안을 과도하게 구현하지 않는다. 다만 나중에 ISMS-P에 준하는 보안·감사·접근통제 체계를 붙일 수 있도록 Auth / Firestore / Rules 구조는 확장 가능하게 설계한다.”
01 문서 목적 — Why This Baseline Decision
  • PR #294 / #295 결정을 실제 구현으로 잇기 위한 infra baseline을 정의한다.
  • 실제 Auth / CRUD를 구현하지 않고, firebase.json에 Firestore 경로만 연결하고 deny-by-default Rules를 baseline으로 둔다.
  • PR #298 (projectMembers email gate + project-scoped read) 구현 전 기준을 정리한다.
  • PR #306 (role hardening + Viewer write block + action-level enforcement) 로 넘길 범위를 명시한다.
이 문서의 효력 본 문서의 결정 문구는 PR #298 / #306 구현 PR의 canonical 기준으로 사용된다. 변경이 필요할 경우 별도 결정 PR로 본 문서를 먼저 갱신한 뒤 구현을 진행한다.

Auth 정책 상속 (PR #295 §03)

  • 로그인한 Google 계정이 projectMembers.email에 등록되어 있어야 프로젝트 접근 가능.
  • 등록된 프로젝트가 없으면 Access Pending 화면.
  • 접근 권한이 없는 projectId URL 접근 시 Access Denied 화면.
  • Email 매칭은 emailNormalized (소문자 + trim) 기준.
  • Gmail dot / plus alias 자동 병합은 1차에서 하지 않는다.
08 Firestore Rules deny-by-default 원칙 — This PR's Rules Baseline
기본문 원칙 · 모든 read / write는 기본적으로 거부된다. · 이번 PR에서 특정 컬렉션의 read / write를 열지 않는다. · 임시로라도 allow read, write: if true를 넣지 않는다. · signedIn()만으로 전체 read를 여는 규칙을 넣지 않는다. · 실제 접근 게이트는 PR #298에서 projectMembers 기반으로 구성한다.

이번 PR의 firestore.rules 요약

  • rules_version = '2'.
  • helper signedIn()는 정의만 두고 이번 PR에서 사용하지 않는다. PR #298에서 projectMembers 조회와 조합해서 활용한다.
  • catch-all match /{document=**} { allow read, write: if false; }로 전면 거부.
  • 주석에 후속 PR (#298 · #306) 범위를 명시한다.

허용되지 않는 패턴

  • allow read, write: if true;
  • allow read: if signedIn(); (전체 read 개방)
  • 임의 컬렉션 (users · projects · projectMembers · requirements · wbsItems · screenSpecs · screenFields · screenActions · artifactLinks)에 대한 read/write 개방
  • 테스트를 위한 개발용 open 규칙

13 보안 / 감사 확장 원칙 — Extensible Without Rebuilds Principle
원칙 1차에서는 보안 기능을 과도하게 구현하지 않는다. 다만 나중에 ISMS-P에 준하는 보안 · 감사 · 접근통제 체계를 붙일 수 있도록 Auth / Firestore / Rules 구조는 확장 가능하게 설계한다.
  • Rules는 deny-by-default로 시작한다. 필요한 곳만 최소로 연다.
  • helper 함수 (signedIn() 등)는 정의만 두고 필요 시점부터 사용한다.
  • 실제 감사로그 · 관리자 행위 로그 · ISMS-P 체크리스트 · 개인정보 보관 / 파기 정책 · 다운로드 통제 · 세션 보안은 후속 보안 거버넌스 트랙으로 분리한다.
  • secret / service account / API key는 절대 커밋하지 않는다. 필요 시 GitHub Secrets / Firebase Functions config로만 다룬다.
  • 임시 개발용 open 규칙은 존재 자체를 금지한다. staging에도 두지 않는다.
