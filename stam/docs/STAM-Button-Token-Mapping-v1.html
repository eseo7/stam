STAM Button Token Mapping v1 Mapping Draft

본 문서는 실제 Button CSS 구현 작업이 아니라, Button 컴포넌트가 사용할 토큰의 variant × size × state 매핑 기준을 사전 확정하는 매핑 문서이다. Token Inventory v1에서 후보로 제안된 Button State 토큰을 Button 변형·사이즈·상태 축으로 풀어서 표로 정리하며, 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/**를 절대 수정하지 않는다.

Mapping Draft CSS Implementation Hold Pre-Implementation No CSS Edit

판정 근거 — Token Inventory v1에서 Button State P0 토큰(--btn-primary-*, --btn-secondary-*, --btn-danger-*, --shadow-focus)이 후보로 제안되었으나, 실제 CSS 구현 시 필요한 size 조합·icon-only/text/ghost 변형·loading/selected 상태 처리 기준이 확정되지 않은 상태이다. 본 문서로 매핑을 확정해야 Button CSS 구현 단위에 진입할 수 있다.

01 Overview — 문서 목적 목적

CSS Token Readiness Check가 “전체 토큰 충분성”을, Token Inventory v1이 “신규 토큰 후보 목록”을 정리했다면, 본 Button Token Mapping v1은 “Button 컴포넌트가 실제로 어느 토큰을 어떻게 쓰는지” variant × size × state 축으로 1:1 매핑을 확정한다. 기획자·PM도 “어떤 버튼이 어떤 토큰을 참조하는지”를 한 화면에서 읽고 승인할 수 있어야 하며, 개발자는 후속 Button CSS 구현 단위에서 본 문서를 그대로 클래스·토큰 참조 기준으로 사용한다.

이 문서가 다루는 것

  • Button size 기준 (xs / sm / md / lg) 후보값
  • Button state 매트릭스 (default / hover / active / focus / disabled / loading / selected)
  • variant × state → 토큰 1:1 매핑 표
  • P0 Button State 토큰 의존성 정리
  • 접근성 · 인터랙션 규칙
  • 실제 CSS 구현 전 Gate Checklist
  • Cursor 핸드오프 기준

02 Button Scope — 본 매핑이 다루는 버튼 범위 범위

아래 8개 버튼 형태가 본 매핑의 대상이다. 각 형태는 variant(시각적 우선순위)와 별개로 사용 맥락이 정의된다.

Primary
주요 저장 · 등록 · 확정 · 제출 액션. 화면당 1개 원칙.
Secondary
취소 · 보조 액션. Primary와 함께 dialog 풋바·폼 푸터에 사용.
Ghost
낮은 우선순위 액션. Toolbar · 메뉴 내부 · 텍스트 인근 인라인.
Danger
삭제 · 위험 액션. Confirm Dialog 주요 CTA로 한정.
Icon
아이콘 단독. 검색 · 닫기 · 더보기 · 캘린더 등.
Text
링크성 보조 액션. 인라인 텍스트 톤 유지.
Disabled
변형이 아닌 상태이지만, 클릭 차단·focus 가능 여부 매핑 대상.
Loading
변형이 아닌 상태. 중복 클릭 방지·스피너 영역 매핑 대상.
본 문서가 다루지 않는 버튼 FAB(Floating Action Button), Segmented Button Group, Split Button, Menu Button(드롭다운 트리거)은 v1 범위 밖이며 후속 단계에서 별도 매핑한다. Tag / Chip / Badge는 Button이 아니므로 별도 컴포넌트로 매핑한다.
03 Button Variant Matrix — 변형별 역할 · 토큰 의존성 매트릭스

Variant는 시각적 우선순위(Visual Priority)와 사용 위치(Usage)로 정의한다. Required Tokens는 해당 variant 구현에 필수인 토큰 그룹이며, 상세 토큰명은 Section 06에서 풀린다.

Secondary 취소 · 보조 액션. Primary와 한 쌍으로 사용. High --btn-secondary-bg · --btn-secondary-bd · --btn-secondary-text Ghost와의 시각 차이 모호 시 사용자 혼동 후보

07 Accessibility / Interaction Rule — 필수 준수 필수
키보드 · 포커스
  • 모든 Button은 focus-visible ring 필수 — --btn-focus-ring 적용
  • native focus outline을 무작정 제거하지 않음 — 대체 ring이 있을 때만 제거
  • Tab 키 순서가 시각 순서와 일치
  • Enter / Space 키로 활성화 가능
Disabled
  • 색상만 낮추지 않고 cursor:not-allowed로 인터랙션 차단
  • pointer-events: none 또는 aria-disabled="true"로 click 차단
  • opacity 단독 처리 금지 — 토큰 기반 disabled 색상 사용
  • focus 가능 여부 결정: 사용자에게 사유 안내 필요 시 focusable 유지
Loading
  • 중복 클릭 방지 — pointer-events 차단 또는 onClick guard
  • aria-busy="true" 명시
  • 스피너 영역은 텍스트와 겹치지 않게 별도 영역 확보
  • 최소 300ms 이상의 작업에만 loading 표시
Icon-only Button
  • aria-label 필수 — 시각 텍스트가 없으므로
  • 최소 hit-area 28×28 (sm) 이상 권장
  • tooltip 동반 시에도 aria-label 별도 제공
  • 아이콘만으로 의미 불명확하면 텍스트 동반 형태로 대체
Hover 의존 금지
  • hover에만 의존한 정보 노출 금지 — 터치 디바이스에서 불가
  • 중요 액션은 default 상태에서 인지 가능해야 함
  • Ghost · Icon Button도 default에서 클릭 가능 영역이 보여야 함
Dark Mode 대비
  • 라이트 / 다크 모두 텍스트 대비비 4.5:1 이상 보장
  • focus ring은 다크에서 더 밝은 톤·높은 알파로 보강
  • disabled는 다크에서도 명확히 인지 가능해야 함
  • Primary · Danger 모두 라이트/다크 후보값으로 분리 정의
Danger 분리 원칙
  • Danger Button은 일반 Primary와 색·focus ring 모두 분리 — 빨강 계열 단독
  • 한 화면에 Primary · Danger를 동시에 두지 않음 (Confirm Dialog 예외)
  • Danger의 hover · active는 반드시 더 어두운 빨강 — 다른 hue로 이동 금지
  • Danger ring은 --btn-danger-ring으로 분리 — 일반 --btn-focus-ring 미사용
08 Do / Don't — 구현 원칙 원칙
Do
  • 모든 색·radius·padding은 토큰 기반으로 구현 (--btn-*, --r-md 등)
  • variant × size × state 조합 일관성 유지 (한 variant가 모든 size·state를 지원)
  • icon 동반 시 gap 토큰 또는 size별 표 기준값 일관 적용
  • 라이트 / 다크 동시 QA — 토큰 추가 시 두 테마 값 함께 정의
  • focus ring은 --btn-focus-ring 단일 토큰 사용 (Danger만 별도)
  • disabled는 토큰 기반 색상 + cursor:not-allowed 동시 적용
Don't
  • raw color (#FFFFFF, #5451E8) 직접 사용 — 모두 토큰 경유
  • 화면별·페이지별 Button CSS 별도 작성 금지 — 공통 클래스 단일화
  • native outline 무작정 none 처리 금지 — 대체 ring 없으면 유지
  • disabled를 opacity만으로 처리 금지 — 토큰 + cursor 차단 병행
  • 한 번에 Form · Dialog · Drawer · Table 모두 동시 CSS 구현 금지 — Button만 우선
  • Danger를 Primary 대체로 사용 금지 — Confirm 액션에만 한정
09 Implementation Gate Checklist — CSS 구현 진입 전 점검 Gate

아래 항목이 모두 확정될 때까지 stam.tokens.css · stam.components.css에 Button 관련 토큰 추가·클래스 구현을 진입하지 않는다.

01 Button Token Mapping v1 본 문서 승인 PM · 디자인 · 개발 3자 합의 — variant · size · state 매트릭스 확정
02 P0 Button State 토큰명 확정 --btn-primary-* · --btn-secondary-* · --btn-danger-* · --btn-focus-ring 네이밍 동결
03 라이트 / 다크 후보값 확정 Section 06 표의 Light / Dark Candidate 값 PM 승인
04 Button Guide와 충돌 없음 확인 기존 Button Guide variant · size · state 정의와 정합
05 수정 허용 CSS 파일 범위 확정 stam/css/stam.tokens.css · stam/css/stam.components.css 두 파일만 허용 — 별도 지정 시까지 다른 CSS 미수정
06 Button만 제한 구현 Form · Dialog · Drawer · Table 클래스는 본 단위에서 절대 수정하지 않음
07 동시 수정 금지 컴포넌트 확인 Form / Dialog / Drawer / Table 매핑은 별도 후속 매핑 단위에서 진행
08 기존 문서·페이지 회귀 QA 기준 정의 Docs 문서 내 .tbtn · .sbadge 등 기존 버튼 영향 범위 사전 확인
10 Cursor Handoff Notes — 구현 핸드오프 기준 핸드오프

Section 09 Gate가 통과되면, 아래 기준으로 Cursor에게 Button CSS 구현 단위를 넘긴다.

구현 대상 · 파일 범위
  • 첫 구현 대상은 Button만 — Form / Dialog / Drawer / Table 동시 진입 금지
  • 수정 허용 파일: stam/css/stam.tokens.css · stam/css/stam.components.css (다른 CSS는 별도 지정 시까지 금지)
  • stam/js/** · stam/pages/** · stam/index.html · stam/assets/** 수정 금지
  • Firebase / GitHub Actions / package*.json 변경 금지
변경 검증
  • git diff로 stam.tokens.css · stam.components.css 외 변경 없음 확인
  • 금지 파일 tracking 발생 시 즉시 revert
  • 새 토큰명은 본 문서 Section 06의 Token 컬럼과 동일하게 사용
QA 기준
  • 라이트 / 다크 QA 동시 — 양 테마에서 대비비·focus ring 가독성 확인
  • Docs Index · Token Inventory · Component Guide 등 기존 Docs 문서 회귀 확인
  • 1440 · 1600 · 1920 가로 폭에서 페이지 레벨 가로 스크롤 없음
  • 제품 페이지(Dashboard · WBS · Boards)의 기존 .tbtn 회귀 확인
롤백 기준
  • Docs 또는 제품 화면에서 시각 회귀 1건 이상 발생 시 즉시 revert
  • 토큰명 충돌(기존 --go-bg, --stam-* 등) 발생 시 본 문서 v2로 재합의
  • 라이트 / 다크 중 한 테마에서만 깨질 경우 즉시 revert
핸드오프 시 명시 사항 본 문서는 매핑 기준 문서이며, CSS·JS·페이지·자산 변경 권한을 포함하지 않는다. Cursor는 Section 06 Token Mapping 표만 코드로 옮기며, 새로운 토큰명·값을 임의로 추가하지 않는다.
