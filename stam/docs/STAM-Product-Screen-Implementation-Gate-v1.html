01 Overview — 이 Gate의 목적 1차 기준
문서명STAM Product Screen Implementation Gate v1
대상제품 화면 작업 (구현 통제 기준)
상태1차 기준

목적

  • Component Guide 7종(Button · Icon · Dialog · Form · Drawer · Table · Topbar)은 설명서이고, 이 문서는 제품 화면 구현 통제 기준(Gate)이다.
  • Claude Code가 제품 화면을 수정할 때 Component Guide를 임의로 무시하거나 “감”으로 구현하지 못하도록 강제한다.
  • 제품 화면 수정 전 반드시 본 Gate의 매핑 / Intake / Completion Report / Merge Gate를 따른다.
  • 화면 작업에 공통 적용한다: WBS · 요구사항 · 메뉴/화면목록 · 화면설계서 · 기능정의 · 프로그램목록 · API 명세 · 정책 · 테이블정의 · DB 정보 · 서버 정보 · 접근보안 · 프로젝트 Overview · 설정/권한.
핵심 원칙 “Component Guide는 무엇을 어떻게 보여줄 것인가의 설명서다. 이 Gate는 제품 화면을 만들 때 그 설명서를 반드시 따르도록 강제하는 운영 기준이다.”
02 Scope — 적용 / 제외 대상 1차 기준

적용 대상 (제품 화면 작업)

  • WBS
  • 요구사항
  • 메뉴/화면목록
  • 화면설계서
  • 기능정의
  • 프로그램목록
  • API 명세
  • 정책
  • 테이블정의
  • DB 정보
  • 서버 정보
  • 접근보안
  • 프로젝트 대시보드 / Overview
  • 설정 / 권한 / 산출물 접근권한

제외 대상 (Gate 미적용)

  • 단순 문서 오탈자 수정
  • Preview 문서만 수정하는 작업
  • Firebase / 배포 인프라 작업
  • nav-data 구조 변경 전용 작업
적용 대상 화면을 수정하는 모든 PR은 본 Gate를 통과해야 한다. 제외 대상은 Gate 없이 진행 가능하지만, 화면에 시각적 영향을 주는 부수 변경이 섞이면 즉시 Gate 적용.
03 Mandatory Component Guide Mapping 1차 기준

제품 화면 수정 전 반드시 작성해야 하는 매핑

화면 요소 반드시 참조할 Guide
상단 화면명 / 주요 액션 / 프로젝트 컨텍스트 Topbar Guide
목록 / 컬럼 / 정렬 / 필터 / 검색 / 상태 chip Table Guide
버튼 위계 (Primary / Secondary / Ghost / Danger) Button Guide
아이콘 버튼 / 행 액션 / hit area / aria-label Icon Guide (+ Button Guide)
등록 / 수정 / 상세 / 검토 패널 Drawer Guide
입력 필드 / validation / error / required / optional Form Guide
삭제 / 반려 / 권한 제거 / 위험 확인 Dialog Guide
전체보기 / Focus View / 고밀도 테이블 Table Guide + UI Baseline Guide (S10)
모바일 / 좁은 화면 (430 / 768) Topbar + Table + Drawer + Form Guide의 Mobile 섹션
매핑이 없는 화면 요소는 작업을 시작하지 않는다. 매핑이 누락된 채로 “일단 구현부터” 진행하면 Component Guide 위반 가능성이 높다.
04 Product Screen Work Intake Gate 1차 기준

제품 화면 작업 시작 전 Claude Code가 반드시 보고할 항목

항목 설명
작업 대상 화면예: WBS / 요구사항 / 화면설계서 …
실제 repo 파일 경로`stam/pages/...` / 관련 JS / CSS 경로
수정 허용 파일이 작업에서 건드릴 수 있는 파일 명시
수정 금지 파일건드리면 안 되는 파일 명시 (nav-data / Shell / Firebase 등)
적용할 Component Guide 목록S03 매핑 결과를 화면 기준으로 압축
예상 영향 범위본 화면 외 다른 화면에 영향이 있는지
제품 로직 변경 여부저장 / 삭제 / 상태 전이 / 권한 검사 등 로직 영향
저장 / 삭제 / API / localStorage / Firestore 영향 여부데이터 영향 명시
모바일 영향 여부430 / 768 영향 여부
다크모드 영향 여부토큰·테마 깨질 가능성
예외 필요 여부S09 Exception Policy 적용 여부
Intake 보고가 부족하면 사용자(QA)는 작업 시작을 보류시킬 수 있다. 시작 후 변경 범위가 바뀌면 즉시 재보고한다.
05 Implementation Rules — 구현 중 지켜야 할 규칙 1차 기준

핵심 규칙

  • Component Guide와 다르게 구현해야 할 경우 코드 수정 전에 사유를 보고하고 승인받는다.
  • 기존 공통 App Shell / Left Nav / Topbar 구조를 깨지 않는다.
  • 화면별로 필터 / 검색 / 버튼 / Drawer / 상태 chip을 임의 패턴으로 새로 만들지 않는다. 공통 기준을 따른다.
  • WBS에서 정한 Table / Filter 기준은 다른 목록 화면에도 확장 가능한 형태여야 한다.
  • 제품 화면에서 임의 inline style / 신규 local style / 외부 CDN 추가 금지.
  • 모바일에서 전체 페이지 가로 스크롤이 생기면 실패로 본다. 가로 스크롤은 테이블 컨테이너 내부에서만 허용된다.
  • 다크모드 토큰을 깨면 실패로 본다. 색을 직접 박지 않고 토큰을 사용한다.
  • 위험 액션(삭제 · 반려 · 권한 제거)은 즉시 실행하지 않고 Dialog Guide 기준을 따른다.
  • 권한이 없는 액션은 hide-by-default. 필요 시 disabled + 이유 안내 기준을 따른다.
  • 저장 / 삭제 / API / 상태 전이 등 로직 변경은 Intake 보고에 포함된 범위 안에서만 진행한다.

07 PR Completion Report Gate 1차 기준

Claude Code 완료 보고에 반드시 포함해야 할 항목

  • 변경 파일 (정확한 경로)
  • 변경 요약 (무엇이 / 왜 / 어디서)
  • 적용한 Component Guide 목록
  • Component Guide별 준수 여부 (Topbar / Table / Drawer / Form / Dialog / Button / Icon)
  • 예외 여부 + 사유 (S09)
  • 제품 로직 변경 여부
  • 저장 / 삭제 / API / localStorage / Firestore 영향 여부
  • CSS / JS / HTML 변경 범위
  • 모바일 확인 (430 / 768)
  • 1366 확인
  • Light / Dark 확인
  • 신규 `<style>` / 신규 inline `style=` / 신규 JS / 외부 CDN 여부
  • 콘솔 red error 여부
  • Preview URL (또는 인프라 blocked 사유)
  • 사용자 QA 전 Draft 유지 여부
08 Merge Gate — merge 전 확인 기준 1차 기준
  • changed_files 수치와 실제 변경 파일 확인
  • 허용 파일 외 변경 없음 확인
  • Component Guide 매핑 보고 완료 (S03)
  • 제품 로직 변경 여부 명시
  • Preview success 또는 인프라 blocked 사유 분리
  • 사용자가 승인하기 전 ready 전환 / merge 금지
  • ready 전환 및 squash merge는 Claude Code가 실행, 사용자(지민)는 QA 판단만 수행
인프라(예: Firebase OAuth `Premature close`) 사유로 Preview / Staging deploy가 실패하면 “인프라 blocked”로 분리 보고. 코드/문서 변경이 정상이면 사용자 판단으로 merge 진행 가능.
09 Exception Policy — 예외 처리 1차 기준

예외 허용 조건

  • 기존 구조상 Component Guide와 완전히 맞출 수 없는 경우
  • 기술 부채로 인해 별도 단계화가 필요한 경우
  • 모바일 / 접근성 / 성능 상 Guide와 다르게 처리해야 하는 경우

예외 보고 형식 (필수)

  • 어떤 Guide와 다른가
  • 왜 다른가
  • 대안은 무엇인가
  • 후속 보정이 필요한가 (있다면 후속 PR 항목으로 기록)
  • 사용자 승인 여부
예외는 “사후 통보”가 아니라 “사전 승인”이다. 코드를 다 바꾼 다음에 “예외였습니다”라고 보고하는 패턴은 거부된다.
10 WBS First Application Rule 1차 기준
  • WBS는 첫 적용 기준 화면이다. 필터 / 검색 / 정렬 / 상태 chip / 행 액션 / 전체보기 / Focus View 기준을 Table Guide 기준으로 먼저 정리한다.
  • WBS에서 확정한 공통 목록 패턴은 요구사항 · 메뉴/화면목록 · 화면설계서 · 기능정의 · 프로그램목록 등 다른 목록 화면에 그대로 확장한다.
  • WBS만 특수하게 “WBS Special”로 끝내지 않고, 추출한 공통 기준은 Table Guide / Drawer Guide / Filter 기준으로 환원한다.
  • WBS 작업에서 새로 발견한 “공통화 후보”는 후속 PR 항목으로 기록한다.
WBS가 “특수 화면”이면 다른 목록 화면이 또 “특수 화면”이 된다. 본 Gate는 WBS를 공통 기준의 출발점으로 본다.
