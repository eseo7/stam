01목적 / 원칙Candidate
STAM CORE DB v0.1 Candidate Guide 기준 / 검토 필요 Soft Delete First
STAM Core DB 논리 구조 v0.1 — 6개 게시판 게시판별 테이블 + import 이력 + 링크/변경

본 문서는 STAM 산출물 데이터의 논리 DB 구조(테이블/컬럼/상태/삭제 정책)를 정의한다. 요구사항 가져오기(Excel/CSV/TSV import)로 생성되는 6개 게시판 데이터를 게시판별 테이블에 저장하며, 통합 artifacts 단일 테이블 구조는 사용하지 않는다.

  • Local DB도 DB로 취급한다. 사용자가 직접 초기화하지 않는 한 데이터를 보존한다.
  • 자동 clear 금지, indexedDB.deleteDatabase() 금지, 자동 seed/샘플 자동 삽입 금지.
  • 게시판 화면/메뉴는 이미 존재한다. 데이터 row 는 사용자가 등록하거나 import 해야 생긴다.
  • 삭제는 물리 삭제가 아니라 soft delete(status=deleted)가 기본이다. hard delete 는 v0.1 범위 밖.
02Local DB ↔ 실DB 관계원칙
“Local DB는 실험용이지만 논리 구조는 실DB와 맞춘다.” 물리 저장소(브라우저 IndexedDB vs 실DB)는 다를 수 있어도 테이블 분리 · 컬럼 의미 · 상태값 · 삭제 정책 같은 논리 구조는 동일해야 한다. 그래야 프로토타입에서 검증한 흐름이 실DB로 이어진다.

08soft delete 정책No Hard Delete
  • 모든 주요 테이블은 soft delete 기준. 삭제 시 status = deleted + deletedAt / deletedBy / deleteReason 기록.
  • list 기본 조회에서 deleted 는 제외한다(includeDeleted 옵션으로만 포함).
  • 물리 삭제(hard delete)와 deleteDatabase() 는 v0.1 에서 구현하지 않는다.
  • “초기화”는 사용자가 직접 실행할 때만 동작하는 명시적 동작이며, 자동으로 데이터를 비우지 않는다.
