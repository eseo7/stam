01 Hero / Purpose 목적
STAM POLICY Candidate Guide 기준 / 검토 필요 Not Undo
STAM 데이터 변경의 추적성과 복구 가능성 — 정책 경계 정의

본 가이드는 STAM에서 발생하는 데이터 변경에 대한 추적성(Change History)과 복구 가능성(Recovery)의 정책 경계를 정의한다. 사용자가 Ctrl+Z로 즉시 되돌리는 형태의 Undo / Revision 기능을 약속하는 문서가 아니며, 변경이력 / Field Diff / 복구 요청 / 관리자 복구 / Snapshot 후보를 정책 기준으로 정리한 가이드이다.

02 Policy Summary Summary
Policy Key 정의 주요 책임 상태
Change History 대상 엔티티의 변경 시점·주체·이벤트 유형을 시계열로 기록하는 정책 "누가 언제 무엇을 바꾸었는가"의 추적 근거 확보 Candidate
Field Diff 변경 직전값과 변경 직후값을 필드 단위로 비교 표현하는 정책 변경의 범위와 영향을 사람이 읽을 수 있는 형태로 제공 Candidate
Recovery Request 사용자가 과거 시점 또는 특정 변경 이전 상태로의 복구를 신청하는 정책 요청 접수 · 검토 · 승인 / 반려의 절차적 처리 Candidate
Admin Recovery 승인된 복구 요청에 대해 관리자 권한으로 복구 작업을 수행하는 정책 권한 / 감사 / 사유 기록을 포함한 처리 Candidate
Snapshot Candidate 특정 시점의 엔티티 상태를 보관 단위로 후보 정의 — 확정 구현 아님 복구·감사·증적 후보 자원으로 평가 대상 Candidate
Not Undo 본 가이드는 단일 클릭 되돌리기 / 자동 롤백을 약속하지 않는다는 명시 정책 제품 기대치 관리 · UI 카피 가드레일 Hard Limit
03 Scope — 포함 / 미포함 범위
포함 (In Scope)
  • 변경이력(Change History) 기록 정책의 기준
  • Field Diff 표현의 정책 기준
  • 복구 요청(Recovery Request) 절차의 정책 기준
  • 관리자 복구(Admin Recovery) 처리의 정책 기준
  • Snapshot의 후보 정의 (Candidate)
  • 권한 / 보안 / 약관 / 삭제 / 결제 / 감사 대상의 별도 엄격 처리 원칙
  • 버튼 더블클릭 등 동일 행위 중복 요청에 대한 차단 원칙
미포함 (Out of Scope)
  • Ctrl+Z 형태의 단일 클릭 되돌리기 기능 구현 약속
  • 자동 롤백(Auto Rollback) 약속
  • 모든 필드 / 모든 시점에 대한 복구 보장
  • DB 테이블 / 컬럼 / ERD의 확정
  • 구체적 인덱스 / 파티셔닝 전략 확정
  • 제품 화면 / WBS / Project Overview / nav-data 수정
  • API 코드 / Firebase Functions / Firestore rules 변경
  • 법무 / 보존기간 / ISMS-P 통제 항목의 최종 확정
04 Terminology 용어
용어 정의 주의
Change Record 1건의 변경 행위에 대응하는 이력 레코드 후보 엔티티 + 시점 + 주체 + 이벤트 유형
Field Diff 변경 전/후 값을 필드 단위로 비교한 표현 민감 필드는 마스킹 후보
Recovery Request 특정 시점/특정 변경 이전으로의 복구를 신청하는 사용자 액션 즉시 복구 아님 · 검토 후보
Admin Recovery 승인된 요청을 관리자가 권한으로 처리하는 작업 감사로그 필수 후보
Snapshot 특정 시점의 엔티티 상태 보관 단위 — 후보로만 다룸 확정 구현 아님
Concurrent Save 동일 대상에 대해 동시에 발생하는 여러 저장 요청 빠른 저장·늦은 저장 모두 이력 후보
Double-Click 동일 버튼을 짧은 시간 내 중복 클릭한 행위 동일 행위의 중복 요청 · 차단 대상
Strict-Handling Domain 권한 / 보안 / 약관 / 삭제 / 복구 / 결제 / 감사 대상 영역 일반 정책과 별도 엄격 처리
05 What This Guide Does Not Promise Anti-Promise
본 가이드가 약속하지 않는 것
  • 단일 클릭 되돌리기 (One-click Undo) 기능 제공
  • 자동 롤백 (Auto Rollback) 동작
  • 모든 엔티티의 모든 필드에 대한 변경이력 보장
  • 모든 시점에 대한 복구 보장
  • 실시간 복구 / 즉시 복구 보장
  • Snapshot의 주기 · 보존기간 · 저장소 확정
  • 법무 · 보존기간 · ISMS-P 통제 항목의 최종 확정 문구
  • DB 테이블 / 컬럼 / ERD / 인덱스의 확정
Note: 본 가이드는 정책 경계만 정의한다. 실제 기능 범위·시한·우선순위는 별도 Product / Engineering 결정으로 정의한다.
06 Change History Policy Policy
기록 단위
엔티티 단위 + 시점 + 주체 + 이벤트 유형(생성·수정·상태전이 등)을 1건의 Change Record 후보로 기록한다.
동시성 (일반 업무 저장)
동일 대상에 대해 빠른 저장과 늦은 저장이 모두 변경이력 후보가 될 수 있다. 최종값은 늦은 저장 기준이 될 수 있다.
중복 요청
버튼 더블클릭은 동일 행위의 중복 요청으로 보고 차단 대상이다. 중복 차단된 시도는 이력으로 남기지 않거나 단일 이력으로 통합 후보로 다룬다.
엄격 처리 영역
권한 / 보안 / 약관 / 삭제 / 복구 / 결제 / 감사 대상의 변경은 별도 엄격 처리한다. 일반 업무 저장 규칙을 그대로 적용하지 않는다.
조회 기본 정렬
시간 내림차순(최신 우선)을 기본 후보로 한다. 권한·필터 정책은 별도 Permission / Audit 가이드를 따른다.
보관 기간 — 미확정
엔티티 유형별 보관기간은 본 문서에서 확정하지 않는다. Retention 정책은 후속 가이드 대상이다.
07 Field Diff Policy Policy
표현 단위
필드 단위 비교 표현을 기본으로 한다. 변경된 필드만 노출하는 모드와 전체 노출 모드를 후보로 분리한다.
민감 필드 마스킹
개인정보·결제·인증 관련 필드는 마스킹 후보이다. 마스킹 기준은 Security / Privacy Guide를 따른다.
대용량 텍스트
긴 텍스트 필드는 요약/Diff 영역 표시를 분리 후보로 한다. 전체 보기는 권한 필터 후 제공 후보.
결정형 필드
상태·단계·승인 등 결정형 필드는 일반 텍스트 Diff와 시각적으로 구분한다(예: 색·뱃지).
08 Recovery Request Policy Policy
  • 복구 요청은 즉시 복구가 아니다. 접수 → 검토 → 승인/반려 절차를 거치는 후보 흐름으로 정의한다.
  • 요청 시 사유를 필수 후보로 한다. 사유는 감사 자료로 활용된다.
  • 요청 대상 시점은 변경이력 상의 특정 Change Record 또는 사용자 지정 시점을 후보로 한다.
  • 승인된 요청만 Admin Recovery 처리 대상이 된다.
  • 반려된 요청은 사유와 함께 사용자에게 안내한다.
  • 본인 외 타인의 데이터에 대한 요청은 권한 정책에 의해 제한된다.
09 Admin Recovery Policy Policy
  • 관리자 복구는 승인된 Recovery Request에 기반해서만 수행한다.
  • 처리 시 대상 엔티티 · 복구 기준 시점 · 처리자 · 사유 · 결과를 감사로그 후보로 기록한다.
  • 처리 결과는 신규 Change Record로 기록한다. 과거 이력을 덮어쓰지 않는다.
  • 엄격 처리 영역(권한·약관·삭제·결제·감사)의 복구는 별도 승인 라인을 둘 수 있다.
  • 복구 후 검증 단계를 후보로 한다(차이가 의도된 결과인지 사용자에게 확인 요청).
  • 실패한 복구 시도도 감사 대상이다.
10 Snapshot Candidate Policy Candidate Only
Important: Snapshot은 확정 구현이 아닌 후보(Candidate)로만 표현한다. 주기 · 보존기간 · 저장소 · 트리거는 본 문서에서 확정하지 않는다.
후보 트리거
상태 전이, 승인 완료, 마감 처리, 대량 변경 직전 등을 후보 트리거로 검토한다.
후보 저장 단위
엔티티 단위 · 도메인 묶음 단위 · 사용자 단위 등 후보 단위를 비교 평가한다.
후보 보존 기간
법령·약관·운영 기준의 영향을 받는다. 본 문서에서 확정하지 않는다.
복구 연계
Snapshot은 Recovery 처리의 후보 자원으로 평가된다. Snapshot이 없는 복구는 Change Record + Field Diff 재구성을 후보로 한다.
11 Permission / Security Considerations Security
  • 변경이력 조회는 권한 기반으로 제한한다. 본인 데이터 / 팀 데이터 / 관리자 데이터 범위는 Permission Policy Guide를 따른다.
  • 민감 필드는 Field Diff 표현 시 마스킹 후보로 처리한다.
  • 관리자 복구 처리에는 처리자 식별 · 사유 · 대상 · 결과를 감사로그 필수 후보로 한다.
  • 권한 변경 자체의 이력은 일반 변경이력과 별도 엄격 처리 영역에 둔다.
  • 약관·동의 변경 이력은 Terms / Consent / Policy Versioning Guide의 정책을 우선 적용한다.
  • 삭제·결제·감사 대상 변경은 별도 보존·접근 정책의 영향을 받는다.
12 Product Impact Impact
변경이력 진입점 후보
상세 화면 / 행 메뉴 / Topbar 보조 메뉴 등을 진입점 후보로 평가한다.
복구 요청 UI 후보
사유 입력 · 대상 시점 선택 · 영향 안내 · 확인 단계를 갖춘 모달/페이지 후보.
관리자 복구 처리 화면 후보
요청 리스트 · 상세 · 승인/반려 · 처리 결과 · 감사로그 진입의 후보 흐름.
감사 로그 후보
Admin Recovery 처리 결과의 감사로그 항목 추가 후보.
Save / Conflict 정책 연동 후보
동시 저장 · 더블클릭 차단 · 마지막 쓰기 우선 정책과의 연결 후보.
권한 UI 영향 후보
변경이력 보기 · 복구 요청 · 관리자 복구 권한 매핑 후보.
QA 영향 후보
변경이력 정합성 · 더블클릭 차단 · 복구 처리 결과 검증 시나리오 후보.
Note: 위 항목은 모두 후보이며, 실제 화면 도입은 별도 Product 결정으로 정의한다. 본 문서로는 제품 화면 / WBS / Project Overview / nav-data를 수정하지 않는다.
13 DB Impact Candidates DB Candidate
Table (Candidate) 역할 후보 핵심 필드 후보 상태
changeHistory 엔티티 단위 변경이력 인덱스 entity_type, entity_id, occurred_at 후보

(index) 대상 식별자 + 발생 시각 인덱스 후보 entity_ref + occurred_at 후보
Important: 위 항목은 영향 분석용 후보이며, 최종 DB 스키마 / Firestore collection / 인덱스 확정안이 아니다. 본 문서는 DOCS ONLY 범위이다.
14 Document QA Checklist QA
