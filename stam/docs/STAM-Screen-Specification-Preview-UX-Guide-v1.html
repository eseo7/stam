화면설계서 미리보기 UX 고도화 기준안
Editor "미리보기" 버튼을 눌렀을 때 나오는 Preview UX의 세 가지 대안을 비교하고, STAM 화면설계서 저장·검토 흐름에 가장 적합한 방식을 정의한다.

1
현재 미리보기 문제 요약
mockup이 목업 수준에 그침. 현재 Preview는 WF 블록을 단순 선 박스로만 표현해 검토자가 화면 구조를 파악하기 어렵다. 번호 마커가 있지만 정보가 빈약하다.
Description 정보가 없음. 표시 조건, 권한, 액션/연결, 데이터 출처 등 검토에 필요한 정보를 미리보기에서 확인할 수 없다.
저장/검토요청 액션이 분리됨. 미리보기에서 문제를 발견해도 닫고 → Editor로 돌아가 → 수정하는 흐름이 명확하지 않다. 미리보기 안에서 저장/검토요청이 가능해야 한다.
필수 미작성 항목 표시 없음. 검토 요청 전 필수 항목이 누락됐을 때 경고를 주는 기능이 없다.
화면 메타 정보가 없음. 화면 ID, 작성자, 최종 수정일, 상태가 표시되지 않아 검토자 입장에서 맥락 파악이 어렵다.

추천안 및 이유
STAM 최종 추천
C안 — Review Sheet Preview (Topbar 아래 고정 오버레이)
화면설계서 미리보기의 1차 목적은 "검토자에게 전달될 산출물이 어떻게 보이는가"를 작성자가 먼저 확인하는 것이다. 단순 overlay보다 실제 검토 문서처럼 보이는 구조가 UX 완성도가 높다. Topbar는 유지해 STAM 도구 맥락을 유지하고, Topbar 아래 전체를 Review Sheet가 덮는다.
1
산출물 미리보기가 목적. 저장·검토요청 전 확인용이므로 문서 형식이 적합하다. 검토자가 볼 화면과 동일한 구조로 미리 확인해야 한다.
2
번호 마커 + Description이 한 화면에. 좌측 mockup과 우측 Description 패널이 함께 보여야 번호 연동이 의미 있다. A안과 같은 2열 구조를 C안도 채택한다.
3
필수 항목 미작성 경고가 검토요청 직전에 필요. "저장 후 검토요청" 전에 필수 Description이 비어있는 항목을 footer에서 경고해야 검토 품질이 높아진다.
4
Topbar 유지 = STAM 맥락 유지. 완전한 fullscreen(A안)보다 Topbar를 유지한 상태가 "STAM 도구 안에서의 검토"임을 자연스럽게 전달한다.
4
C안 — 화면 구성 와이어프레임 (인터랙티브)
사용 방법. 좌측 블록 또는 우측 Description 항목을 클릭하면 번호 연동 강조가 활성화됩니다. "필수 누락" 탭에서 누락 상태를 확인하세요.

스크롤 이동 구현 기준

scrollIntoView 사용 금지. STAM 환경에서 scrollIntoView는 예기치 않은 페이지 점프를 일으킬 수 있다. 대신 각 패널(mockup col, desc col)의 scrollTop을 직접 계산해 이동한다.

계산식: targetEl.offsetTop - panelEl.offsetTop - 40 (40 = 여유 padding). 결과가 0 미만이면 0으로 클램프한다.

애니메이션: panelEl.scrollTo({ top: value, behavior: 'smooth' }) 사용. smooth 스크롤이 지원되지 않는 환경에서는 panelEl.scrollTop = value fallback.

이유. S21 계약은 본문 max-width: 1560px; padding: 0 32px을 요구한다. Preview View는 2열 레이아웃(mockup + description)이 뷰포트 전체를 사용해야 양쪽 패널에 충분한 공간이 확보된다. Editor Workbench와 동일한 예외 사유가 적용된다.

예외 범위. .prev-ui 및 그 하위 전체. Topbar와 STAM GNB Sidebar는 유지한다.

등록 표기 제안. LAYOUT_EXCEPTION: PREVIEW_REVIEW_SHEET — "화면설계서 Preview는 2열 Review Sheet 구조로 full-width를 사용한다."
