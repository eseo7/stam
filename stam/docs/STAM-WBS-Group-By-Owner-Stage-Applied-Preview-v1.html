STAM WBS Group By Owner / Stage Applied Preview v1 Preview Owner · Stage · Status · Risk Product Not Yet

본 문서는 STAM WBS 에서 업무를 담당자별 · 단계별 · 상태별 · 지연/위험별로 묶어 보는 Group By 화면 기준을 확정한다. 작업자용 WBS, 모바일 카드형 WBS, 발표 모드 3종 컨텍스트에서 공통 사용된다. 기본 WBS 목록을 대체하지 않으며, 사용자가 명시적으로 Group View 에 진입할 때만 활성화된다.

적용 범위 — 본 작업은 stam/docs/** 하위 1개 HTML 생성과 STAM-Docs-Index.html 카드 1개 추가만 수행한다. stam/pages/**, stam/js/**, stam/css/**, stam/index.html, stam/assets/**, .firebaserc, firebase.json, .github/workflows/**, package.json, _ds/**, uploads/**, *.zip, .env*, node_modules/**, dist/**, build/**, .firebase/** 는 한 줄도 수정하지 않는다.

01 Overview — 문서 목적 목적

WBS 의 기본 목록은 "한 줄 한 줄의 업무"를 본다. 하지만 실제 운영에서는 "이 업무가 누구한테 얼마나 몰려있나", "어느 단계가 막혀 있나", "위험 / 지연 업무는 누구 책임이 큰가" 같이 덩어리로 보고 싶어진다. Group By 보기는 이 시야를 채운다. 일반 WBS · 모바일 WBS · 발표 모드 3종 컨텍스트 모두에서 동일 모델로 작동한다.

이 문서가 다루는 것
  • Group By 의 목적과 작업자용 / PM용 / 발표용 차이
  • v1 지원 모드 4종 — 담당자 · 단계 · 상태 · 지연/위험
  • Entry 패턴 — segmented + Group By dropdown · 모바일 bottom sheet
  • Group Header — name · count · progress · risk badge · 접기/펼치기
  • Body Layout — PC group + table / Mobile group + card / Presentation group + 큰 글자
  • Filter / Sort 와의 상호작용 · empty group 정책
  • 담당자 미지정 · 단계 미지정 그룹 처리
  • Mock Screens 9종 (A ~ I — D PC Status Group 추가) · Good / Bad 비교 · 3 컨텍스트 비교표
이 문서가 다루지 않는 것
  • stam/pages/**, stam/js/**, stam/css/** 직접 수정
  • WBS / 결함 / 테스트 IA · 컬럼 라인업 자체 변경
  • 실제 group 렌더링 로직 / 데이터 모델 · 별도 Gate
  • 요구사항 ID 기준 group (v2 후보 — v1 범위 밖)
  • 다단 group (group of group) · 별도 Gate
  • 전역 .tbtn 일괄 치환 · Firebase / package.json 변경
3 컨텍스트와의 관계
  • 작업자용 WBS — 담당자 / 단계 group 으로 자기 업무 파악
  • PM 회의 / 발표 모드 — 담당자 호명, 단계별 진척, 위험/지연 그룹 우선
  • 모바일 WBS — group accordion + card list · bottom sheet 로 group 선택
  • WBS Presentation Mode (v1) 의 "담당자별 보기"는 본 가이드의 Owner Group View 를 발표 density 로 확장한 것
  • WBS Mobile Board (v1) 의 카드 리스트 패턴을 group accordion 안에 그대로 재사용
실제 구현 전 디자인 기준 문서
  • 본 문서는 실제 구현 전 시각 기준이며 product 적용은 별도 Gate
  • 적용 순서: 본 가이드 승인 → group 데이터 모델 정합 → PC group render → Mobile accordion → Presentation density 합류
  • Group By 는 데이터 렌더링 로직 변경이므로 한 커밋에 PC / Mobile / Presentation 을 섞지 않는다
  • 충돌 시 Data Table Guide / WBS Presentation Mode / WBS Mobile Board 가 우선
02 Group By Purpose Purpose

Group By 는 단순한 정렬과 다르다. 정렬은 "줄 순서"만 바꾸지만, Group By 는 덩어리의 크기 · 균형 · 위험도를 한 번에 보여준다.

사용자주요 모드달성하려는 것
작업자 (Owner)담당자별 (자기) / 단계별자기 일의 양 · 마감 · 이번 주 우선순위 파악
팀 리드담당자별 / 위험·지연팀원 간 작업 분배 불균형 발견, 차단된 일 식별
PM단계별 / 상태별 / 위험·지연프로젝트 진행 단계 균형, 의사결정 대기 / 차단 발견
회의 진행자 (발표)담당자별 / 위험·지연담당자 호명, 위험 우선 토론, 다음 액션 결정
외부 보고 (옵션)단계별 / 상태별전체 진척률 · 단계 완료율 빠른 요약
Group By 가 달성해야 하는 것
  • 담당자별 업무량 / 위험 비중 한눈에 — 작업 분배 불균형 즉시 식별
  • 단계별 진행 상태 — 어느 단계가 막혀 있나
  • 지연 / 위험 업무 집중 확인 — 회의 우선 안건
  • 회의 중 담당자 호명 및 즉시 액션 결정
  • group header 만 봐도 0.5초 안에 "이 그룹의 상태" 파악
Group By 가 절대 해선 안 되는 것
  • 기본 목록을 강제로 group view 로 덮어쓰기
  • filter 무시한 group count 표시 (사용자 혼란)
  • 담당자 미지정 / 단계 미지정 그룹 누락 (데이터 손실 인식)
  • empty group 을 그대로 노출해 화면을 채우기
  • dim 처리로 두 가지 그룹 상태를 한 화면에 섞기
03 Group By Modes Mode

v1 에서 지원하는 모드는 4종 — 사용 빈도가 가장 높고 데이터 모델이 단순한 것부터 시작한다. 요구사항 ID / 마감 주차 / 우선순위 group 은 v2 이후 후보.

담당자별 v1
avatar + 이름 group · 작업자 / PM / 회의 모두 사용. 미지정 group 포함.
단계별 v1
기획 → 디자인 → 퍼블 → 개발 → QA → 오픈준비 group. 프로젝트 진행 순서 정렬.
상태별 v1
대기 · 진행 · 보류 · 완료 group. (지연 · 위험은 별도 flag — 상태와 분리)
지연 / 위험 v1
위험 / 지연 / 마감 임박 / 차단 / 확인 필요 group. PM 회의 빠른 보기.
요구사항 ID v2
REQ-### group. 요구 → 업무 추적성. 데이터 정합화 필요로 v2 후보.
마감 주차 v2
이번 주 / 다음 주 / 다음 달 group. 일정 정합화 필요로 v2 후보.
우선순위 v2
P0 · P1 · P2 group. priority 필드 정의 후 v2.
차단 이유 v2
의사결정 대기 / 외부 의존 / 인력 부족 등. 차단 분류 정의 후 v2.
v1 vs v2 — 분리 기준 v1 4종은 모두 이미 WBS 데이터에 존재하는 필드이므로 frontend 만 정합화하면 동작한다. v2 후보는 별도 데이터 필드 정의 / 정합화가 선행되어야 하므로 분리한다.
04 Entry Pattern UX

Group View 는 자동 진입하지 않는다. 사용자가 명시적으로 선택해야 활성화된다. 진입 지점은 view mode segmented + Group By dropdown 조합. 모바일에서는 bottom sheet.

기존 WBS 검색 / 필터와의 분리
· Group By 는 기존 목록 / 검색 / 필터를 대체하지 않는다. 별도 축이다.
· 적용 순서 — 검색 + 필터 → 결과 row → Group By 로 묶음. 검색/필터가 먼저, group 이 나중.
· 기본 WBS 목록 모드는 항상 유지 — 사용자가 segmented 에서 "목록" 으로 언제든 복귀 가능.
· 검색 / 필터의 UX (검색 input · filter drawer · chip) 은 group view 에서도 그대로 노출. group 만 추가 컨트롤.
· group 진입과 동시에 filter 가 변경되거나, filter 변경 시 group 이 해제되는 동작은 금지.

09 Status Group View — 상태별 보기 Mode

상태별 그룹은 4개 — 대기 · 진행 · 보류 · 완료. 지연 / 위험은 상태가 아니라 별도 flag 이므로 상태 그룹 안에서도 위험/지연 row 강조는 유지한다. (지연/위험만 보고 싶다면 S10 Risk / Delay Group 사용.)

그룹정렬 순서설명
대기 (todo)1아직 시작 전 · 우선순위 낮음
진행 (progress)2가장 큰 그룹 — 현재 작업 중. 지연/위험 flag 있으면 상위 정렬
보류 (hold)3의사결정 / 외부 대기 · 회의 안건 후보
완료 (done)4마지막 그룹 · 기본 접힘 후보
상태 vs 위험·지연 — 관계
  • 상태 = 4종 (대기 · 진행 · 보류 · 완료) — 업무 라이프사이클
  • 위험 / 지연 = flag — 모든 상태에 부수적으로 붙음
  • 예: "진행 + 지연 3일", "보류 + 위험 고" 가능
  • 상태별 그룹에서도 row 단위 강조 유지 (left bar + bg)
  • 위험/지연만 모아 보고 싶으면 S10 Risk / Delay Group 사용
상태 그룹의 함정
  • 완료 그룹이 누적되면 다른 그룹이 묻힘 — 기본 접힘 후보
  • 지연 flag 가 있는 진행 업무가 상태 그룹에서 묻힐 위험 — row 강조 유지 필수
  • 대기 그룹이 길어지면 우선순위 정렬 후 상위 N개만 노출 고려
  • "보류"는 사용 빈도 낮을 수 있어 0건이면 자동 숨김

17 Good / Bad Comparison 비교
Good
  • 그룹 헤더에 count + 위험 비중 표시
  • 접기/펼치기 가능 · 완료 그룹 기본 접힘
  • filter 후 count 가 정확히 갱신
  • 모바일에서는 accordion + chip bar
  • 발표 모드에서는 큰 글자 + 핵심 count
  • 담당자 미지정 / 단계 미지정 그룹 항상 표시
  • group sort + row sort 분리
  • empty group 기본 숨김
Bad
  • 단순 정렬만 있고 그룹 헤더 없음 (Bad Example)
  • count 없음 — PM 이 직접 세야 함
  • 위험 / 지연이 묻혀 보임 (row 강조 없음)
  • 모바일에서 긴 테이블 그대로 유지
  • empty group 까지 모두 노출 — 화면 낭비
  • dim 처리로 두 그룹 상태가 한 화면에 섞임
  • filter 무시한 group count (사용자 혼란)
  • group dropdown 만 있고 segmented 없어 "현재 모드 불명"

실제 구현 전 주의 — 순서
  • 1차: group 데이터 모델 정합 — owner / stage / status / risk 필드 표준화
  • 2차: PC group view (header + body) — list 와 분리된 render path
  • 3차: filter + group count 정합화
  • 4차: Mobile accordion + bottom sheet
  • 5차: Presentation density 합류 (별도 Gate)
한 커밋에 섞지 말 것
  • PC group view + Mobile group accordion
  • group view + filter 로직 변경
  • group view + presentation density
  • group sort + row sort 동시 도입
  • group by + 컬럼 라인업 변경
Security / Privacy Guide 기준 충돌 후속 확인 — 별도 Gate
  • 담당자명 / 사용자명 노출 정책은 Security / Privacy Guide (S07 Personal Data Classification · S08 Masking) 기준과 충돌 여부 확인 필요
  • 외부 사용자 / 협력사 (EXTERNAL role) 권한에서는 담당자 정보 마스킹 또는 표시 범위 제한 (예: 박*원) 가능성 검토
  • Owner Group 의 group key 자체가 "이름" 인 경우 — EXTERNAL role 에게는 그룹 라벨까지 마스킹 / 익명화 (Owner 1 · Owner 2) 후보
  • Group By 구현 전 권한 / 개인정보 노출 정책을 먼저 확인 후 진행 — 1차 그룹 모델 정합 단계에서 검토
  • group count / progress 자체는 비식별 통계이므로 외부 사용자에게도 노출 가능 (별도 검토)
  • 참고 문서 — Security / Privacy / ISMS-P Architecture & DB Policy Guide v1
권한 × Group View — 정책 체크 매트릭스
  • SUPER_ADMIN / ADMIN — 모든 그룹 / 모든 라벨 원문 노출 OK
  • PM — 본인 프로젝트 그룹 원문 노출 · 다른 프로젝트는 scope 차단
  • MEMBER — 팀 / 본인 그룹 원문 · 외부 팀원 이름은 정책 결정 필요
  • VIEWER — 그룹 라벨 노출은 OK · unmask / export 차단
  • EXTERNAL — 담당자 이름 마스킹 또는 익명화 후보 · 그룹 count 만 노출하는 패턴 검토
  • 모든 그룹 view 의 row click → detail panel 은 Security Guide 의 read-only panel 정책 상속
