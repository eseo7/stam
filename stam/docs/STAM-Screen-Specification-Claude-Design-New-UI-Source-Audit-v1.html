01

Executive Summary

결론 확정
현재 제품은 “신규 UI 전체 적용본”에 매우 근접
  • 새 화면설계서 작성 진입은 Wizard 4단계 만 노출 (구 template view 경로 제거됨)
  • Editor는 3-Panel (Component Library + Wireframe Canvas + Inspector 4탭) 적용
  • Preview는 55/45 Mockup + Description + Check Point 패널 적용 (PR #188 + #198)
  • PR #199 sticky footer 보정 완료, PR #177 starter card는 #181에서 제거
다만 다음 한 가지 잔여 차이 존재
  • PR #197 OPEN — Inspector 패널 너비를 274px 고정에서 clamp(274px, 27vw, 440px)로 보정 (2줄 CSS)
  • 1366px 이하에서 Inspector가 좁아 사용자가 “옛 UI 같다”고 느낄 가능성 가장 큰 영역
  • 병합 후 List 페이지 chrome (header/strip/toolbar/table/drawer)은 신규 UI 범위가 아니라는 점도 인식 필요

Keep / Absorb / Replace Decision

판단
PR #199 Sticky Footer Fix — 유지토큰 정의 누락 (var(--bg) → var(--bg-base)) 해결. 회귀 위험 없음.
PR #198 Annotated Preview — 유지escHtml + Check Point + fallback. Preview UX Guide 및 Visual Annotation 문서와 일치.
PR #177 Starter Card — 이미 흡수/제거PR #181에서 Starter Card와 ss-starter* CSS 모두 제거되어 Wizard 진입으로 통합. 추가 조치 없음.
구 domain preset (서비스 유형 기반)유지. 구조형 Top-10 카탈로그와의 매핑은 후속(PR-F)에서 별도 판단.
기존 editor / preview 기능유지. 신규 UI 가 이미 동일 renderer 에 반영되어 별도 재배치 불필요.
List 페이지 chrome유지. 신규 UI 범위 외. 사용자 요청 시 별도 PR-G 로 분리.

Product Migration Scope

잔여 범위

Scope A — Inspector 너비 보정

PR #197 머지. CSS 2줄. 즉시 효과 큼.

Scope B — Editor 빈 상태 / 캔버스 상단 정리

wfBlocks.length === 0 일 때 placeholder 텍스트를 빈 상태 CTA 로 보강.

Scope C — Inspector 입력 활성화

표시/권한·액션/연결·데이터/검증 탭의 readonly 컨트롤을 편집 가능하게 전환. Check Point 카운터 실작동.

Scope D — Library 드래그 인서트

현재 click-to-append 지원, drag-from-library 추가.

Scope E — Template 변경 시 데이터 보존

editor-change-template 시 block 데이터 유지 여부 확인 다이얼로그.

Scope F — Top-10 카탈로그 매핑

구조형 Top-10 카탈로그와 현재 4-템플릿 preset의 매핑 정책 정리.

Scope G — 저장 / Board Builder 연계

In-memory 저장을 실제 영속화로 전환할지 별도 결정. 본 audit 범위 외.

Scope H — List 페이지 chrome (선택)

사용자가 원할 때만 별도 PR. 본 audit 의 신규 UI 정의 범위 외.

Recommended PR Split

후속 PR 제안

PR-A · Inspector clamp() 보정 (= PR #197 머지)

  • 목적: Inspector 너비 1366px 비례 보정
  • 예상 파일: stam/css/stam.screen-specification.css (2줄)
  • 금지: JS / HTML / 저장 / 데이터
  • 검증: 1366 / 1024 / 768 / 430px 비례 확인
  • 리스크: 매우 낮음
  • merge/deploy: 사용자 승인 후

PR-B · Editor 빈 상태 / Canvas top

  • 목적: wfBlocks.length === 0 시 빈 상태 CTA
  • 예상 파일: js renderer 1곳 + CSS
  • 금지: 저장 / Firestore / nav-data / board-field-schema
  • 리스크: 낮음
  • merge/deploy: 사용자 승인 후

PR-C · Inspector 탭 입력 활성화

  • 목적: 4탭 readonly 컨트롤 → 편집 가능
  • 예상 파일: js renderer + CSS
  • 금지: 저장 로직 / Firestore / API
  • 리스크: 중간 (Check Point 카운터 연계)
  • merge/deploy: 사용자 승인 후

PR-D · Library drag-from-library

  • 목적: 드래그-드롭 인서트 추가
  • 예상 파일: js + CSS
  • 금지: 저장 / 데이터 모델
  • 리스크: 중간
  • merge/deploy: 사용자 승인 후

PR-E · Template 변경 보존 다이얼로그

  • 목적: editor-change-template 시 block 데이터 유지 옵션
  • 예상 파일: js
  • 금지: 저장 / Firestore
  • 리스크: 중간
  • merge/deploy: 사용자 승인 후

PR-F · Top-10 카탈로그 매핑

  • 목적: 4-템플릿 preset ↔ Top-10 카탈로그 매핑 정책

  • 리스크: 큼 (데이터 모델 결정)

URL Policy

중요
Firebase Hosting public root는 stam/ 입니다.
  • repo 경로의 leading stam/ 은 staging URL 에서 제거합니다.
  • 예: repo stam/pages/boards/screen-specification.html → URL https://stam-design-staging.web.app/pages/boards/screen-specification.html
  • /stam/pages/... 형태의 URL 은 잘못된 경로 입니다.
  • 완료 보고 / 문서 / PR 본문에서 QA URL 에 /stam/ 를 붙이지 않습니다.

Audit Conclusion

결론
  • 현재 제품은 “Claude Design 신규 UI 전체 적용본”의 실질적 완성 상태에 도달해 있습니다. Wizard·Editor·Preview 모두 신규 UI 가 적용되어 있고, 구 진입 경로는 제거되었습니다.
  • “클디 신규 UI 원본”은 별도 prototype HTML 이 아니라 PR 시리즈 #181 → #187 → #188 → #189 → #190 → #191 의 merge 결과 자체입니다. 별도 claude-design/newui 브랜치는 repo 내 존재하지 않습니다.
  • 다음 실제 이식 PR 의 최우선 후보는 PR-A (PR #197 머지) — Inspector 너비 보정으로, 사용자가 “옛 UI 같다”고 느낄 가능성이 가장 큰 차이를 해소합니다.
  • 승인 필수 영역: Scope C / D / E / F / G / H — 본 audit 결과 후 별도 승인 필요.
  • 제품 변경은 본 audit 에서 진행하지 않습니다. merge/deploy 는 사용자 승인 전까지 보류합니다.
