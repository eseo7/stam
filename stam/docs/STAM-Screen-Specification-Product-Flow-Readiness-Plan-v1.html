Screen Specification
Product Flow Readiness Plan v1

실제 화면설계서 제품 페이지(stam/pages/boards/screen-specification.html)와 참조 JS/CSS의 현재 상태를 조사하고, P1~P6 문서 기준과의 Gap을 정리한 진단·계획 문서다. 제품 파일을 바로 고치기 전에, 후속 제품 보정 작업을 안전하게 쪼개기 위한 기준을 제공한다. 이 문서는 제품 기능을 구현하거나 제품 파일을 수정하지 않는 docs-only readiness plan이다. P1~P6(Master Guide · Priority 30 · Top 10 Detail Pack · Integration Blueprint · Selector UX Flow · Top 3 Drawer Preview) 다음 단계인 P7이다. 기준 커밋은 main d392497 (PR #175 merge)이다.

00

왜 바로 수정이 아니라 Readiness Plan인가

문서 목적

현재 제품의 실제 구조·동작 범위를 조사하고 P1~P6 기준과의 Gap을 정리해, 후속 제품 보정 PR을 안전한 단위로 나눈다.

바로 수정의 위험

화면설계서 페이지는 App Shell·드로워·목록·mock 데이터·다중 View가 얽혀 있다. 무계획 수정은 회귀·데이터/권한 로직 위험이 크다. 먼저 진단·분류가 필요하다.

P1~P6과의 관계

P1~P6은 템플릿·연결·선택·드로워 기준을 정의했다. 이 문서는 그 기준을 실제 제품에 적용하기 위한 준비 점검이다.

이 문서의 성격

실제 Selector·Drawer·Board Builder·화면설계서 제품 기능을 구현하지 않는다. 제품 파일 무변경 진단·계획 문서다. merge/deploy는 사용자 승인 전 대기한다.

핵심 발견: 화면설계서 제품 페이지는 이미 템플릿 선택 View · 편집기(Wireframe+Form) · 미리보기 View · 상세/등록 드로워를 갖추고 있으나, JS 주석 기준 "mock only — no localStorage / Firestore / DB"이며 템플릿 분류 체계가 P2/P3의 구조형 Top 10(T01~T27)과 다른 도메인형 프리셋(bran딩/커머스/업무/커뮤니티)으로 되어 있다. Board Builder 연계(P4)는 제품에 아직 연결되어 있지 않다. 단계 범위는 "1차 범위 / 우선 적용 / 후속 단계" 표현을 사용한다.

Current Flow Map

1
목록 진입

화면설계서 목록(표) 표시. 검색·필터·페이징 동작(mock).

2
"템플릿으로 작성" → Template View

switchView('template')로 풀페이지 템플릿 선택 화면 전환.

3
템플릿 선택 → Editor View

도메인형 프리셋 선택 후 편집기(좌 Wireframe + 우 Form) 전환. draft 모델 구성.

4
Preview View

draft 기준 초안 미리보기(mock, "실제 데이터 아님" 표기).

5
목록 행 클릭 → 상세 드로워

개요/연결정보/검토이력 탭. 우측 드로워 표시.

6
저장 / 취소 / 내보내기

버튼 존재하나 mock. 실제 저장·검토요청·내보내기 동작 없음(검토요청 비활성).

현재 흐름은 UI/뷰 전환은 동작하나 영속 저장·데이터·권한 판정·Board Builder 연계는 없음이다. P5 Selector 흐름(진입점→카드→필드소스→미리보기→초안)과 부분 일치하되, Field Source Selection과 구조형 Top 10 카드는 미구현이다.

05

Safe Change Classification

A. Docs-only

  • 문서 링크 보정 · 설명 문구 보정
  • QA 기준 추가
  • 위험 낮음 — 이번 문서 범위

B. UI-only / Static Preview

  • 템플릿 카드 표시 · 미리보기 영역 UI
  • 드로워 레이아웃 정리 · 버튼/라벨/배지 보정
  • 주의 — 정적, 저장 로직 무관

C. Product UI with Mock Data

  • 제품 파일에서 정적 샘플 기반 selector 표시
  • 저장 없이 preview-only 동작
  • 주의 — 기존 mock 흐름 회귀 점검

D. Data / Logic Sensitive

  • 실제 저장 · Firestore · API · fetch · localStorage 로직
  • 권한 판정 · 실제 배포/생성
  • 금지(이 단계) — 별도 승인·별도 PR
후속 제품 보정은 A → B → C 순으로 진행하고, D는 분리·보류한다. 각 PR은 단일 분류에 머무르게 해 회귀 범위를 좁힌다.

  • 스크롤 들썩임 없음
  • 가로 스크롤 컨테이너 한정
핵심: 현재 stam.screen-specification.css에 @media 브레이크포인트가 없어 768px/430px 전환이 정의되어 있지 않다. 모바일 보정(PR-F)은 CSS @media 추가 위주로 하고 JS 로직은 건드리지 않는다.
