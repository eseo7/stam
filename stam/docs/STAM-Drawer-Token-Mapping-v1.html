STAM Drawer Token Mapping v1 Mapping Draft

본 문서는 실제 Drawer CSS 구현이 아니라, Create · Edit · Detail · Review · Approval · Comment · Filter Drawer와 Full Detail 전환의 width · header · body · footer · backdrop · shadow · scroll · transition · 내부 Form · Full Detail handoff 기준을 토큰 기반으로 사전 확정하는 매핑 문서이다. Drawer는 등록·수정·상세의 공통 진입 UI이며, Dialog보다 한 layer 아래에 위치한다. Drawer 내부 Select · Date Picker · Dropdown은 Drawer panel보다 위에 떠야 한다. 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/**를 절대 수정하지 않는다.

Mapping Draft CSS Implementation Hold Full Detail Handoff Required Pre-Implementation No CSS Edit

판정 근거 — Layer Token Mapping v1에서 “Drawer는 Dialog 아래, Dropdown은 Drawer 위”의 z-index 순서가 동결되었고, Form Token Mapping v1에서 “Drawer 내부 Select · Date Picker는 Full Detail과 같은 컴포넌트를 portal로 mount”라는 원칙이 동결되었으나, Drawer 자체의 width · header · footer · backdrop · scroll · transition · Full Detail handoff 기준은 확정되지 않은 상태이다. Drawer에서 “전체 보기”로 넘어간 Full Detail이 다른 컴포넌트로 보이거나, Drawer 내부 popup이 panel 뒤에 깔리는 사고를 사전에 차단하기 위해 본 문서로 Drawer 매핑과 운영 원칙을 확정한다.

01 Overview — 문서 목적 목적

Layer Token Mapping v1이 “화면에 떠 있는 모든 컴포넌트의 z-index 순서”를 확정했다면, 본 Drawer Token Mapping v1은 그 안에서 동작하는 Create · Edit · Detail · Review · Approval · Comment · Filter Drawer 그리고 Full Detail 전환의 width · structure · state · token · backdrop · scroll · transition · footer · button 우선순위를 1:1로 매핑한다. STAM에서 Drawer는 “리스트에서 한 건을 빠르게 보고 / 등록하고 / 수정하는 공통 진입 UI”이며, Full Detail은 “Drawer가 좁을 때 본문에서 같은 데이터를 크게 보는 경로”다. 두 화면은 같은 Form · Button · Layer 기준을 따라야 하며, 서로 다른 UI로 분기 구현하지 않는다. 기획자·PM은 “어떤 Drawer가 어떤 시나리오에서 떠야 하는지”, 디자이너는 “Drawer 안 영역과 폭 규칙”을, 개발자는 “토큰명 · backdrop · transition · Full Detail handoff 기준”을 본 문서 하나로 읽고 승인할 수 있어야 한다.

주의 — Mapping은 코드 변경이 아니다 본 문서의 Candidate 토큰명·Suggested Value·Used By는 모두 매핑 기준 후보이며, 실제 토큰 추가·CSS 클래스 작성·z-index 부여·JS 구현은 후속 Drawer CSS / JS 구현 단위에서 코드 변경과 함께 확정한다. 본 문서만으로 stam.tokens.css · stam.components.css · 어떤 JS 파일도 수정하지 않는다.
핵심 원칙 — Drawer · Full Detail은 같은 컴포넌트를 쓴다 Drawer 안의 등록·수정·상세 화면과 Full Detail 화면은 같은 Form · 같은 Select · 같은 Date Picker · 같은 Button · 같은 validation을 사용한다. “Drawer 안에서는 작은 폼”, “Full Detail에서는 다른 폼”이라는 식의 분기 구현은 금지된다. Drawer가 좁아 답답할 때 사용자가 “전체 보기”로 Full Detail에 넘어가도 입력값·UI 상태가 그대로 유지되어야 한다.
02 Drawer Scope — 본 매핑이 다루는 Drawer 범위 범위

아래 11개 시나리오가 본 매핑의 대상이다. 등록(Create) · 수정(Edit) · 상세(Detail) · 검토(Review) · 결재(Approval) · 댓글(Comment) · 필터(Filter) Drawer 7종과 Full Detail Handoff, 그리고 Drawer 내부에서 발생하는 Form / Select·Date Picker / Footer Actions를 포함한다.

본 문서가 다루지 않는 UI Dialog · Modal · Confirm · Alert · Toast · Bottom Sheet는 Drawer와 시각·동작이 다르므로 본 문서 범위 밖이다 (Dialog Token Mapping v1 별도). Drawer에서 발생한 confirm / 삭제 / 위험 액션은 Drawer 위에 Dialog로 띄우며, Drawer 안에 inline confirm을 만들지 않는다. Tooltip · Popover는 Drawer 내부 popup 시나리오에서만 본 문서가 다루며, 자체 정의는 Form / Layer Mapping v1을 따른다.

04 Drawer Width / Layout Rule — 폭 · viewport · scroll 기준 폭/레이아웃

Drawer의 기본 폭은 컨텐츠 영역 기준 약 40%로 잡는다. 본문 60%를 그대로 두고 우측 40%에 Drawer가 슬라이드 진입한다. 1440 · 1600 · 1920 가로폭에서는 각각 권장 폭 후보를 제시하며, 좁은 화면(< 1280px)에서는 폭이 커지거나 fullscreen으로 fallback한다. Drawer 내부만 scroll되어야 하며 페이지 전체 scroll과 충돌하지 않는다.

width 선택 원칙
  • 기본 후보는 컨텐츠 영역 기준 약 40% — 본문 60% 보존
  • Drawer 폭은 토큰 --drawer-width 1개로 통제 · 시나리오별 분기 금지
  • 최소 폭 --drawer-width-min · 최대 폭 --drawer-width-max로 viewport 양 끝 보호
  • Filter Drawer는 본 폭의 75% 정도로 좁힘 옵션 검토 (Decision Needed)
  • 한 화면에서 두 Drawer가 동시에 떠 다른 폭으로 분기되지 않음 — 1 Drawer 1 width
scroll / overflow 원칙
  • body의 내용이 길면 body 내부 scroll — header · sticky footer는 고정
  • 페이지 전체 scroll과 충돌 금지 — Drawer open 시 body { overflow: hidden }
  • scrollbar gutter 보정 — Drawer open 시 layout shift 방지
  • 가로 scroll 금지 — content가 넘치면 줄바꿈 · ellipsis · Full Detail 권장
  • body overflow가 내부 popup을 잘라먹지 않음 — popup은 portal 렌더링 (Section 09)
responsive fallback
  • viewport < 1280px → Drawer 폭이 480px 이하로 좁아짐
  • viewport < 1024px → fullscreen 모드로 강제 전환 (radius 0 · width 100vw)
  • fullscreen 모드에서도 header · sticky footer 고정 · body만 scroll
  • safe-area-inset-top / bottom 보정 — iOS notch · home indicator
  • 좁은 화면의 긴 form은 Drawer로 강행하지 않고 Full Detail 또는 별도 페이지로 이동
Full Detail 필요도 기준
  • Drawer 폭 기준 form column 수 ≤ 2 → Drawer 충분
  • form column 3 이상 또는 표 · 좌우 비교 → Full Detail 권장
  • 좁은 viewport(< 1440)에서 긴 form → Full Detail 권장
  • 첨부 / 미리보기 / 우측 도움말 패널이 필요 → Full Detail 본 흐름
  • Detail Drawer는 readonly이므로 Full Detail 필요도 ▼ · Edit Drawer는 ▲
Decision Needed — fullscreen fallback breakpoint Responsive fullscreen 진입 breakpoint를 1024px으로 잡을지 1280px으로 잡을지 PM 결정이 필요하다. 기본 후보는 1024px이며, 1280px 미만에서는 폭이 좁아지지만 fullscreen으로 빠지지 않는다. Filter Drawer의 좁힘 옵션과 함께 일괄 결정 권장.

07 Drawer / Form Sync Rule — Drawer 내부 Form 일관성 규칙

Drawer 안에서 사용되는 Input · Select · Date Picker는 Form Token Mapping v1과 같은 컴포넌트 · 같은 토큰을 사용한다. Drawer 전용 native UI 회귀는 금지되며, 같은 컴포넌트가 Drawer · Full Detail · Dialog 세 화면에서 동일한 시각·동작·layer로 보장된다.

MustForm Sync 원칙
  • Drawer 내부 Input은 Form Token Mapping v1의 --input-* 토큰 그대로 사용
  • Drawer 내부 Select는 Full Detail · Dialog와 동일한 STAM custom select 컴포넌트 사용
  • Drawer 내부 Date Picker도 동일 — calendar popup은 portal로 mount
  • Drawer 내부 form layout · label · helper · error 토큰 모두 Form Mapping v1 재사용
  • field 간격은 --drawer-field-gap · 섹션 간격은 --drawer-section-gap만 사용
Must NotForm Sync 금지 원칙
  • Drawer 전용 native <select> · <input type="date"> 사용 금지
  • Drawer 전용 input · select 변형 컴포넌트를 별도로 만들지 않음
  • Drawer 내부 popup이 panel 뒤에 깔리는 것 → 실패 (portal 미적용)
  • Drawer 내부 scroll 영역에 popup이 잘리는 것 → 실패 (overflow 사고)
  • Drawer 내부 form에서 Full Detail과 다른 padding / gap 사용 → 실패
Layer같은 popup token 재사용
  • Drawer 내부 Select Dropdown → --z-dropdown (1500)
  • Drawer 내부 Date Picker Calendar → --z-calendar (1500)
  • Drawer 내부 Tooltip / Help popover → --z-popover (1500)
  • 같은 popup 컴포넌트가 Drawer · Full Detail · Dialog · 테이블 인라인에서 모두 같은 z-index 사용
  • 화면별 popup 시각 / 동작 / layer가 동일하게 보장됨
QA회귀 판정 기준
  • Drawer 내부 Select 클릭 → 옵션 패널이 Drawer 본문 위에 보임 → PASS
  • Drawer 내부 Date Picker 클릭 → calendar popup이 Drawer 본문 위에 보임 → PASS
  • popup이 Drawer panel 뒤에 깔리면 FAIL
  • Drawer 안에 native select / date input이 1개라도 노출되면 FAIL
  • Drawer 안 input이 Full Detail · Dialog 안 input과 다르게 보이면 FAIL
  • Drawer 내부 scroll에 popup이 잘리면 FAIL (portal 미적용)
Form Token Mapping v1과의 정합 본 Rule은 Form Token Mapping v1 “Drawer / Full Detail Sync Rule”의 Drawer 측면 구체화다. “Drawer · Full Detail · Dialog 세 화면에서 같은 input · select · date picker 컴포넌트가 같은 시각·동작·layer를 보장한다”가 단일 원칙이다. Form Mapping v1이 portal infrastructure와 token을 정의하고, Drawer Mapping v1은 그 위에서 동작하는 Drawer shell · footer · button · transition 기준을 정의한다.

09 Drawer Layer / Overlay Rule — z-index 순서 / 내부 popup 규칙

Layer Token Mapping v1의 “Drawer는 Dialog 아래, Dropdown은 Drawer 위”의 원칙을 Drawer 측면에서 구체화한다. Drawer 내부 Select · Date Picker · Tooltip · Menu는 portal 렌더링으로 Drawer panel 뒤에 깔리지 않아야 한다.

Mustz-index 순서 원칙
  • Drawer backdrop은 base content / Topbar / Sidebar / Sticky 위까지 덮음 (--z-drawer-backdrop 1000)
  • Drawer panel은 Drawer backdrop 위 (--z-drawer 1100 > --z-drawer-backdrop 1000)
  • Drawer 내부 Dropdown · Calendar · Popover는 Drawer panel 위 (--z-dropdown 1500 > --z-drawer 1100)
  • Dialog는 Drawer 위 (--z-dialog 1300 > --z-drawer 1100)
  • Dialog backdrop도 Drawer panel 위까지 덮음 (--z-dialog-backdrop 1200 > --z-drawer 1100)

  • Drawer CSS에 z-index: 9999, z-index: 100 같은 임의 숫자 사용 금지
  • 모든 z-index는 Layer Token Mapping v1의 13개 token (--z-base ~ --z-global-loading) 중 하나만 사용
  • Drawer panel 안에 overflow: hidden이 있으면 내부 popup이 잘림 — portal로 회피
  • Drawer 내부 Select · Date Picker를 자식 DOM에 mount하면 stacking context에 갇혀 layer 매핑 무효
  • backdrop의 z-index를 panel과 같거나 위로 두면 panel이 backdrop에 가려짐
Portalportal 렌더링 원칙
  • Drawer 내부 popup(Dropdown · Calendar · Popover · Menu)은 portal 렌더링 — document.body 또는 별도 portal root에 mount
  • popup은 position fixed · viewport 좌표 기준으로 계산
  • Drawer 내부 scroll · resize 시 popup 위치 재계산
  • Drawer 자체와 popup은 별개 DOM tree에 있어도 사용자 시각으로는 같은 layer로 보임
  • Drawer · Full Detail · Dialog 모두 같은 portal root를 사용 — 일관성 보장
DialogDrawer 위 Dialog · Toast 정책
  • Drawer가 열려 있을 때 Dialog가 열리면 Dialog가 Drawer 위 (1300 > 1100)
  • Dialog backdrop은 Drawer panel까지 dim 처리 (1200 > 1100) — Drawer 인터랙션 차단
  • Drawer 안에서 발생한 confirm / 삭제 / 위험 액션은 Drawer 위에 Dialog로 띄움
  • Drawer 안에 inline confirm을 만들지 않음 — Dialog Mapping v1 재사용
  • Toast / Global Loading은 Drawer · Dialog 모두 위 (Toast 2000 · Loading 3000 후보) — Dialog 문서의 정책과 충돌하지 않음
overflow hidden으로 내부 popup을 잘라먹지 않는다 Drawer panel · body · form area · section group 어디에도 popup을 잘라먹는 overflow: hidden이 적용되면 안 된다. scroll이 필요한 경우 body에만 한정해 적용하며, popup은 portal로 viewport에 mount되어 Drawer의 overflow 경계와 무관하게 표시된다. 이 원칙은 Form / Dialog Token Mapping v1의 portal 렌더링 원칙과 동일 — 세 문서는 같은 portal infrastructure를 전제한다.
10 Drawer Button / Footer Rule — sticky footer 액션 정책 규칙

Drawer Sticky Footer의 버튼 variant · 정렬 · 우선순위 · loading · disabled를 Button Token Mapping v1과 동일 토큰으로 정의한다. Primary / Secondary / Ghost / Danger 4종이 Footer 단일 정책을 따른다.

  • 기본은 우측 정렬 — Secondary 좌 / Primary 우
  • “전체 보기” Ghost는 좌측 분리 — 우측 액션과 시각 분리
  • 위험 액션(삭제 · 반려)은 좌우 분리 — 좌측 Ghost, 우측 Cancel + Danger
  • Footer에는 최대 4개 버튼 — 5개 이상은 menu로 묶음
  • Primary와 Danger는 동시에 표시하지 않음 — 위험 액션은 Primary를 Danger로 대체
loading / disabled 원칙
  • Primary 클릭 → 즉시 saving 상태 진입 → 중복 클릭 차단
  • saving 중 Secondary · Ghost는 disabled · 또는 enable 유지 (Cancel 가능 시)
  • Danger 액션은 Dialog Mapping v1의 Danger Confirm으로 보호 — 중복 클릭 차단 필수
  • Form validation 미통과 시 Primary disabled · Enter 키 trigger 차단
  • disabled 버튼은 cursor not-allowed · 클릭 시 시각 변화 없음
시나리오별 footer 구성
  • Create — “전체 보기” + “취소” + “등록” (Primary)
  • Edit — “전체 보기” + “취소” + “저장” (Primary)
  • Detail — “전체 보기” + “닫기” + “수정” (Primary)
  • Review / Approval — “전체 보기” + “반려”(Danger) + “승인”(Primary)
  • Filter — “초기화”(Ghost) + “취소” + “적용” (Primary)
중복 클릭 방지
  • Primary 클릭 즉시 saving 진입 — 다음 클릭은 무시
  • 저장 / 등록 / 승인 모두 idempotency key 또는 disabled로 차단
  • API 응답까지 Primary loading 유지
  • 응답 후 saved → toast → close 또는 detail mode 전환
  • error → Primary 라벨 “다시 시도” 옵션 · 자동 close 금지
Button Token Mapping v1과의 정합 본 Rule은 Button Token Mapping v1의 5종 variant (Primary / Secondary / Danger / Ghost / Text)를 Drawer Sticky Footer에 1:1 매핑한다. Drawer 전용 button 변형을 만들지 않으며, 모든 버튼은 Button Mapping v1 토큰만 사용한다. footer 자체의 padding · height · gap만 Drawer 토큰 (--drawer-padding · --drawer-footer-height · --drawer-section-gap)으로 정의한다.
11 Accessibility / Interaction Rule — ESC · backdrop · dirty · focus A11y

Drawer는 본문을 dim 처리하고 사용자 집중을 부분 강제하는 UI이므로 ESC · backdrop click · dirty 확인 · focus trap · 초기 focus · 복귀 focus · 다크모드 대비가 모두 정의되어야 한다.

스크린리더 heading Drawer 시작 알림 Header Title을 H레벨로 단일 노출 · Meta Area는 보조 텍스트 모든 variant 공통 후보
focus outline 유지 focus 시각 표시 모든 focusable 요소에 visible outline 유지 — outline 제거 금지 모든 variant 공통 후보
다크모드 대비 light / dark 대비비 backdrop · panel · text · border 모두 light / dark 두 테마에서 WCAG AA 이상 모든 variant 공통 후보
ESC / backdrop click variant
  • Detail / Filter — ESC close · backdrop click close (dirty 없음)
  • Create / Edit — ESC · backdrop click 시 dirty 있으면 confirm
  • Review / Approval — ESC close · backdrop click 차단 옵션 (실수 dismiss 방지)
  • Comment — ESC close · backdrop click close (편집 중이면 confirm)
  • saving 중 — ESC · backdrop click 모두 차단 옵션 (race condition)
dirty 확인 dialog
  • dirty 시 close trigger → Confirm Dialog (Dialog Mapping v1)로 “변경사항이 저장되지 않았습니다”
  • 버튼: “계속 작성” · “나가기” — Primary는 “계속 작성” (안전 액션)
  • “나가기” 선택 시 Drawer close · 입력값 폐기
  • “계속 작성” 선택 시 Drawer 유지 · focus 복귀
  • “전체 보기” 경로에서는 “이동” / “취소” · “이동” 시 입력값 보존하며 Full Detail로 이동 (Section 08)
focus trap 세부 원칙
  • open 시 trap 활성 — close 시 즉시 해제
  • Drawer 내부 popup이 portal로 body 직속에 mount된 경우 trap을 일시 확장
  • popup 닫힘 시 focus는 popup trigger로 복귀 (Drawer 자체로 빠지지 않음)
  • Drawer 위에서 Dialog 열림 시 Dialog만 trap 활성 · Drawer는 inert
  • Dialog close 시 Drawer trap 복귀 · Drawer 내부 원래 focus로 복귀
스크롤 / body lock
  • Drawer open 시 body scroll lock — body { overflow: hidden }
  • scrollbar gutter 보정 — layout shift 방지
  • Drawer close 시 즉시 lock 해제
  • Drawer 위에서 Dialog 열렸을 때 body lock은 한 번만 적용
  • iOS Safari touch scroll 차단 정책 검토 필요 — Dialog Mapping v1과 동일
Decision Needed — role="dialog" vs role="region" · keyboard tab order의 Close 포함 (1) Drawer panel의 aria role을 role="dialog"로 둘지 role="region"으로 둘지 결정 필요. 기본 후보는 role="dialog" + aria-modal="true". (2) Header의 Close (X) 버튼을 keyboard tab order에 포함할지 제외할지 결정 필요. 기본 후보는 “포함” (접근성 우선).
12 Do / Don't — 구현 원칙 원칙
Do
  • Drawer · Full Detail 컴포넌트 기준 통일 — 같은 데이터 / 같은 컴포넌트 / 같은 검증
  • token 기반 구현 — --drawer-* 15종 토큰만 사용
  • Form · Button · Layer · Dialog mapping 재사용 — Drawer 전용 변형 금지
  • Drawer 내부와 Full Detail 동시 QA — 같은 form · 같은 popup 검증
  • 1440 / 1600 / 1920 가로 폭 QA — width 권장 후보 검증
  • Drawer 내부 popup은 portal 렌더링 — Drawer panel 위에 표시
  • dirty 상태에서 close · backdrop click · 전체 보기 시 confirm — 입력값 보호
  • Drawer 위에서 confirm / 삭제 / 위험 액션은 Dialog로 띄움 — Drawer 안 inline 금지
  • Primary 클릭 후 즉시 saving 진입 — 중복 클릭 차단
  • light · dark 동시 QA — backdrop · panel · shadow 양 테마에서 명확
Don't
  • Drawer 전용 Form을 따로 만들기 — Form Mapping v1 컴포넌트 재사용
  • Full Detail에서 native <select> · <input type="date">로 회귀 — STAM custom 컴포넌트 사용
  • z-index 숫자(9999, 100, 1)를 임의 지정 — Layer Mapping v1 token만 사용
  • overflow hidden으로 popup 자르기 — portal 렌더링 강제
  • 저장 로직 / 데이터 필드명을 디자인 작업에서 변경 — Drawer는 UI 레이어만
  • 전체 Drawer variant를 한 번에 구현 — 하나씩 제한 구현 (첫 구현은 layout 또는 handoff)
  • Drawer와 Full Detail이 서로 다른 UI로 보이게 분기 구현
  • dirty 상태에서 확인 없이 close · 전체 보기로 입력값 손실
  • focus outline 제거(outline: none) — 접근성 위반
  • Drawer 안에서 confirm을 inline으로 처리 — Dialog Mapping v1 사용
