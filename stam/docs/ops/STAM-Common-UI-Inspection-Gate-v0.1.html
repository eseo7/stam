01 Common UI Inspection Gate 목적 v0.1
문서명STAM Common UI Inspection Gate v0.1
대상공통 UI / 공통화 PR merge 전 코드 검사
검사 스크립트scripts/qa/audit-common-ui-contract.mjs
  • “비슷하게 보이는 것”과 “실제 공통화”를 구분한다. screenshot만으로는 shell class 일부 적용을 공통화 완료로 오판할 수 있다.
  • PR #253 공통화 정의를 코드 검사 기준으로 연결한다.
  • UI PR에서 source review + screenshot QA만으로 부족한 경우, 공통 class / 공통 JS renderer 사용 여부를 검사한다.
  • PR #250처럼 공통 shell/class 일부만 적용하고 “공통화 완료”로 오판하는 상황을 막는다.
범위
본 Gate는 기준·contract 스크립트·baseline 기록만 정의한다. Detail Drawer 공통 CSS/JS 생성, 제품 UI 코드 전환, class명 변경은 이후 PR에서 별도로 진행한다. v0.1은 공통화 완료 판정 도구가 아니다.
02 적용 대상

검사 필수

  • Drawer / Modal / Popover
  • Detail Drawer 본문
  • Footer button area
  • Custom select
  • Form controls
  • Board list / filter / toolbar
  • 공통 component CSS/JS 변경
  • 요구사항정의서 / 메뉴화면 / WBS / 화면설계서 / 기능정의 등 주요 board 화면

검사 생략 가능

  • docs-only PR
  • ops log / schema-only PR
  • 제품 UI에 영향 없는 문서 / infra PR
03 검사 관점
# 관점 설명
1 공통 CSS 파일 로드 stam.drawer.css, stam.custom-select.css, stam.form-controls.css 등 HTML link 여부
2 공통 class 사용 stam-drawer, stam-drawer-head/body/foot, stam-btn, stam-cs 등
3 공통 JS renderer STAM.customSelect, 향후 STAM.detailDrawer
4 화면별 prefix class 잔존 rq-, msv2-, wbs-, ssv2-, fn- 등
5 detail layout 직접 생성 화면별 JS가 innerHTML / template string으로 detail layout을 직접 생성하는지
6 신규 전용 class 증가 baseline 대비 화면별 전용 class / renderer 직접 생성이 늘어나는지
7 실사용 2화면 이상 공통화 완료 주장 시 최소 2개 이상 실제 화면이 같은 class/function을 써야 함
04 용어 판정 (PASS / WARN / FAIL)
PASS
  • 같은 stam-* class + 같은 STAM.* renderer 사용
  • 2개 이상 화면에서 실사용 확인
  • legacy prefix 감소 또는 공통 renderer로 대체
  • “공통화 완료” 문구는 이 조건을 만족할 때만 사용
WARN
  • legacy class 잔존 (현재 baseline 상태)
  • shell만 공통, detail body는 화면별 전용 class
  • renderer 미공통 (innerHTML 직접 생성)
  • 공통 CSS 일부만 link
  • v0.1 baseline: legacy 존재만으로 전체 FAIL 처리하지 않음
FAIL
  • baseline 대비 신규 전용 detail class 증가
  • 제품 UI PR에서 screenshot 없이 merge
  • 공통화 완료 허위 주장 (PASS 조건 미충족)
  • 공통 파일 없이 화면별 CSS/JS 복제 추가
  • contract 대상 파일 누락 (failOnMissingFile)
05 현재 baseline (솔직한 현황 기록) WARN

미완료
Detail Drawer 공통 CSS (stam.detail-drawer.css) / JS renderer (STAM.detailDrawer)는 아직 미완료. absence는 baseline WARN이며 즉시 FAIL이 아니다.
06 운영 원칙
  • 현재 legacy를 모두 즉시 FAIL 처리하지 않는다. v0.1은 현황 기록 + 증가 방지가 목적이다.
  • baseline 대비 전용 class / renderer 직접 생성이 증가하면 FAIL 후보로 본다.
  • 공통화 PR은 기존 legacy 감소 또는 공통 사용 증가가 있어야 한다.
  • 검사 결과는 human review와 함께 판단한다.
  • 자동 검사는 보조 장치이며 최종 판단은 PR Audit Log에 기록한다.
  • 자동 생성 report는 tmp/**에만 저장하고 repo tracked report로 커밋하지 않는다.

Screenshot QA Gate와의 관계

  • Screenshot QA Gate: 시각적 PASS/FAIL (Playwright + human eye)
  • Common UI Inspection Gate: 코드 contract PASS/WARN/FAIL (class / renderer / legacy)
  • 두 Gate를 함께 통과해야 “공통화 완료” 주장이 가능하다.
07 스크립트 사용법

CLI

  • node scripts/qa/audit-common-ui-contract.mjs --help
  • node scripts/qa/audit-common-ui-contract.mjs --dry-run — 대상/규칙만 출력
  • node scripts/qa/audit-common-ui-contract.mjs --format text --fail-on none — baseline WARN, exit 0
  • node scripts/qa/audit-common-ui-contract.mjs --format json --fail-on none
  • node scripts/qa/audit-common-ui-contract.mjs --out tmp/common-ui-audit.json — JSON report (tmp only)

옵션

옵션설명
--config <path>contract JSON 경로
--format text|json출력 형식
--out <path>JSON report (tmp/** only)
--fail-on warn|fail|noneexit code 정책 (기본 none)
--dry-run파일 스캔 없이 규칙만 출력
