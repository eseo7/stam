01 STAM에서 공통화의 정의 v0.1
문서명STAM Commonization Definition v0.1
대상UI 공통화 판정 기준
상태운영 기준 (v0.1)

공통화란

PASS 정의
  • 여러 화면이 같은 stam-* CSS class와 같은 STAM.* JS renderer를 실제로 사용하고,
  • 화면별 JS는 데이터/config만 넘기는 상태이다.
  • 공통 부품이 기준이고, 요구사항정의서 · 메뉴화면 · 화면설계서 · WBS가 그 공통 부품을 써야 한다.
이 문서의 범위
본 문서는 공통화의 정의와 판정 기준만 고정한다. 상세 Drawer 공통 CSS/JS 생성, 제품 UI 코드 전환, staging 배포는 이후 PR에서 별도로 진행한다.
02 공통화가 아닌 것
아래는 공통화가 아니다
  • 같아 보이는 것 — 시각적으로 비슷해 보여도, 공통 class/renderer를 공유하지 않으면 공통화가 아니다.
  • 같은 색상/간격을 복사한 것 — 토큰 값이나 spacing을 화면별로 복사·붙여넣기한 것은 공통화가 아니다.
  • rq-*를 보고 msv2-*를 만든 것 — 기준 화면의 패턴을 다른 prefix로 복제한 것은 패턴 복제다.
  • 요구사항정의서가 기준 화면으로 남아 있는 상태 — 특정 화면이 SSOT(기준)로 남아 있고 다른 화면이 그것을 흉내 내면, 최종 공통화가 아니다.
03 용어 구분
패턴 보정 기준 화면(예: 요구사항정의서)을 보고 비슷하게 맞춘 것. 시각·구조를 맞추지만 공통 renderer/CSS 계약이 아직 없을 수 있다.
공통 shell 적용 stam-drawer 같은 껍데기 공통 class를 사용하는 단계. 본문·필드 렌더링은 아직 화면별일 수 있다.
공통화 같은 CSS + 같은 JS renderer + 같은 HTML 구조 계약을 실제 2개 이상 화면이 사용하는 것. 화면별 JS는 config/data만 전달한다.
용어 CSS JS renderer 화면 수 판정
패턴 보정 화면별 / 복제 화면별 HTML 생성 1~N 공통화 아님
공통 shell 적용 stam-drawer 등 shell 본문은 화면별 2+ 부분 단계
공통화 공통 stam-* class 공통 STAM.* renderer 2+ PASS

05 공통화 PASS 조건

공통화 PR이 merge되기 위해 충족해야 하는 조건

  • 공통 CSS 파일 존재 — 대상 UI에 대한 공통 stylesheet가 repo에 있고, 화면별 복제가 아니다.
  • 공통 JS renderer 존재 — STAM.* 네임스페이스의 공통 renderer/function이 존재한다.
  • 대상 화면 2개 이상 — 같은 공통 class/function을 실제로 사용한다.
  • 화면별 JS는 config/data만 전달 — HTML 구조를 직접 만들지 않는다.
  • 전용 detail class 신규 생성 금지 — rq-*, msv2-* 같은 화면 전용 class를 새로 만들면 공통화가 아니다.
  • Playwright screenshot QA 증빙 — 시각 회귀·패턴 일치를 screenshot으로 검증한다.
  • PR Audit Log 기록 — stam/ops/pr-audit-log.jsonl에 본 PR을 기록한다.
체크리스트 요약
  • □ 공통 CSS 파일
  • □ 공통 JS renderer
  • □ 2개 이상 화면 실사용
  • □ 화면별 JS = config/data only
  • □ 전용 detail class 미생성
  • □ Playwright screenshot QA PASS
  • □ PR Audit Log 기록
