STAM Data Table / Full Table Responsive Guide v1 Guide PC · Tablet · Mobile Product Not Yet

본 문서는 STAM의 WBS · 작업요청 · 테스트 · 결함 · 산출물 등 테이블형 화면에서 공통 적용할 Data Table / Full Table View 기준을 확정한다. PC는 브라우저 세로 스크롤을 기본으로 하되 페이지 가로 스크롤은 금지한다. 모바일은 카드 리스트를 기본으로 하되 사용자가 선택했을 때만 "전체 테이블 보기" 모드를 제공한다.

01 Overview — 문서 목적 목적

STAM은 WBS, 결함, 테스트, 작업요청, 산출물 목록 등 테이블형 화면이 사용자 활동의 중심이다. 테이블 화면은 스크롤 정책 · 컬럼 우선순위 · 밀도(density) · 모바일 전환의 4가지 축에서 화면별로 편차가 생기기 쉽고, 그 결과 같은 제품 안에서 조작감이 일관되지 않는다. 본 문서는 PC / Tablet / Mobile 3종 분기에서 데이터 테이블이 공통으로 따라야 할 기준을 시각화한다.

이 문서가 다루는 것
  • PC Table Layout Policy — Left Nav / Topbar 고정, browser V-scroll 사용
  • Horizontal Scroll Policy — 페이지 전체 H-scroll 금지, container 내부만 허용
  • Column Priority — P0 / P1 / P2 / P3 4단 우선순위와 viewport 별 노출
  • Table Density — Comfortable / Compact / Presentation 3종
  • Header / Cell / Row Interaction · Toolbar · Empty · Pagination
  • Mobile Card List vs Full Table View · Tablet 절충안
  • WBS 빔 프로젝터 대응 — Presentation Mode
  • Mock Screens 8종 — PC Good / PC Hscroll / PC Bad / Mobile Card / Mobile Full / Tablet / Presentation / State
이 문서가 다루지 않는 것
  • stam/pages/**, stam/js/**, stam/css/** 직접 수정
  • stam/index.html, stam/assets/**, _ds/**, uploads/** 변경
  • WBS / 결함 / 테스트 / 산출물 보드 IA · 컬럼 라인업 자체 변경
  • row virtualization · 대용량 stream · server-side pagination 구현
  • 전역 .tbtn 일괄 치환 · 기능 로직 변경
  • Firebase · GitHub Actions · package.json 변경
테이블 기준이 필요한 이유
  • WBS / 결함 / 테스트 / 작업요청은 모두 다열 데이터이며 viewport 폭에 민감
  • 화면 마다 컬럼 정책이 다르면 사용자는 항상 다시 학습해야 함
  • PC에서 가로 스크롤이 페이지 전체에 발생하면 Left Nav가 같이 밀려 IA 파괴
  • 모바일에서 PC 테이블을 그대로 축소하면 텍스트가 8 ~ 9px 까지 줄어 가독 불가
  • WBS는 빔 프로젝터로 회의 중 보는 경우가 있어 별도 Presentation Mode 가 필요
실제 구현 전 디자인 기준 문서
  • 본 문서는 구현 전 시각 기준이며 product 적용은 별도 Gate
  • 적용 순서: 본 가이드 승인 → Table Token Mapping 정합 → PC scroll 보정 → Mobile card / full 분기
  • PC table scroll 보정과 mobile card mode 구현은 한 커밋에 섞지 않는다
  • 충돌 시 UI Baseline Guide / Mobile Shell Guide / Table Token Mapping 이 우선

03 PC Table Layout Policy Policy

PC에서 데이터 테이블은 Left Nav 고정 + Topbar 고정 + Content 영역 V-scroll 구조를 기본으로 한다. 브라우저 세로 스크롤은 사용하지만, 페이지 전체 가로 스크롤은 어떤 경우에도 금지한다. 테이블이 viewport 보다 넓을 경우 가로 스크롤은 table container 내부에서만 발생한다.

PC 기본 정책 — Good
  • Left Nav · Topbar 항상 고정 (sticky / fixed)
  • Content 영역은 min-width:0; overflow:hidden 로 폭 제약 받음
  • Browser V-scroll = 페이지 세로 흐름의 기본
  • Table container 가 Content 폭을 초과하면 container 안에서만 H-scroll
  • Shell 은 절대 overflow-x 를 켜지 않는다
PC 금지 — Bad
  • Body / Shell / Content 에 overflow-x:auto 또는 min-width:1280px
  • Left Nav 가 가로 스크롤에 같이 밀리는 구조
  • Table 이 Content 보다 넓은데 wrapper 없이 그대로 흐르도록 둠
  • 모든 row 를 한 화면에 보여주기 위해 zoom-out 강요
  • 스크롤바 위치가 Topbar 위 또는 페이지 최상단
APC Good — Container H-scroll page 는 흐르지 않음

04 Horizontal Scroll Policy Rule

가로 스크롤은 오직 한 곳에서만 발생한다 — table container. 그 외에는 어떤 컨테이너도 overflow-x:auto 를 켜지 않는다.

허용 — Content 내부
  • Table container — overflow-x:auto
  • Chip Row (필터 chip) — overflow-x:auto; scrollbar-width:none
  • Drawer Tab 가로 스크롤 (수가 많을 때만)
  • Carousel / 가로 카드 리스트 (별도 인디케이터 필수)
금지 — Page-level
  • body, html, .app-shell 에 overflow-x:auto
  • min-width:1280px 같은 고정 폭으로 페이지 자체를 늘림
  • Left Nav 가 같이 밀리는 구조
  • Topbar 가 viewport 밖으로 빠지는 구조

  • Presentation Mode + 기본 테이블 변경
19 Implementation Guardrails Rule
!페이지 전체 가로 스크롤 금지 — body, html, .app-shell 에 overflow-x:auto 사용 금지.
!Left Nav 가 가로 스크롤에 밀리는 구조 금지 — 어떤 viewport 에서도 IA가 깨지면 안 됨.
!PC table 과 Mobile card mode 한 커밋에 섞지 말 것 — 각각 별도 PR.
!Mobile 에서 PC table 을 기본 화면으로 사용 금지 — 기본은 카드.
✓Full table view 는 사용자가 선택한 경우에만 진입 — 자동 전환 X.
✓Touch target 40px 미만 금지 — Compact row 안에서도 hit area 보정.
!전역 .tbtn 일괄 치환 금지 — 도큐 nav 전용 클래스. 제품 버튼은 분리.
!기능 로직 변경은 별도 Gate — Sort / Filter / Search 동작 정의 변경은 본 가이드 범위 밖.
!대용량 데이터 / row virtualization 은 별도 Gate — 본 가이드는 정적 시각 기준.
iRow 필터링 — dim 이 아니라 제거 방식이 기본 — dim 은 비교 / 검토 모드에서만.
iPresentation Mode 는 별도 feature flag 후보 — 기본 진입에 노출하지 않음.
iP3 컬럼은 사용자가 "모든 컬럼" 토글했을 때만 노출 — 기본 진입 비노출.
