STAM CSS Token Readiness Check Check v1

이 문서는 실제 CSS 수정 작업이 아니라, stam/css/stam.tokens.css 기준으로 STAM 공통 컴포넌트 구현에 필요한 토큰이 충분한지 점검하기 위한 사전 점검 문서이다. Component Guide 7종 v1과 CSS Component Token Plan을 기준으로, 12개 토큰 카테고리와 7종 컴포넌트 관점에서 토큰의 누락 / 부분 / 결정 필요 항목만 정리한다. 본 문서로는 CSS · JS · 페이지 · Firebase 설정을 수정하지 않는다.

Verdict — Ready with Notes Pre-Implementation Readiness Only No CSS Edit

판정 근거 — Color · Radius · Border · Layout Width 등 v1 진입에 필요한 토큰은 정의되어 있어 문서화·기준 합의는 가능한 상태이다. 단, Shadow · Z-index · Motion · Button H · Form H 등 컴포넌트 구현 직전에 반드시 필요한 토큰은 Missing이므로, 본 문서의 Gap List를 기준으로 Implementation Hold 항목을 먼저 해소한 뒤 CSS 수정에 진입한다.

01 Overview — 문서 목적 목적

CSS Component Token Plan으로 컴포넌트별 토큰 매핑 방향이 정리된 이후, 실제 CSS를 수정하기 직전 단계에서 토큰 정의 자체가 준비되어 있는지를 확인한다. 이번 단계의 결과는 코드가 아니라 Gap List와 신규 토큰 후보 목록이다. 본 점검을 통과하지 못한 카테고리는 CSS 수정 대상에서 제외하고, 후속 작업 단위에서 다시 다룬다.

이 문서가 다루는 것
  • 12개 토큰 카테고리 준비 상태 점검
  • Button · Form · Drawer · Dialog · Table · Topbar · Icon 관점의 토큰 수요
  • Ready / Partial / Missing / Needs Decision 분류
  • 실제 CSS 수정 진입 전 Gate 조건
  • 우선순위(P0 / P1 / P2) 권고
이 문서가 다루지 않는 것
  • stam/css/** 토큰 정의 / 클래스 정의 변경
  • stam/js/**, stam/pages/**, stam/index.html 변경
  • Component Guide / UI Baseline / Token Plan 본문 수정
  • QA Checklist · Firebase Staging 설정 작업
  • 신규 컴포넌트 / 신규 폰트 / 신규 외부 의존성 추가
주의 — Readiness Check는 결정 문서가 아니다 본 문서의 Recommendation / Priority는 제안이며, 실제 토큰 추가·이름·값은 후속 작업 단위에서 코드 변경과 함께 확정한다. 본 문서만으로 stam/css/**를 수정하지 않는다.
02 Current Strengths — 현재 강점 강점

CSS 수정 작업 진입 전 이미 정리되어 있는 자산을 확인한다. 아래 항목은 별도 보강 없이 그대로 활용 가능한 기준이다.

FTS-Core Dream 폰트 기준
자체 서빙 폰트 1종으로 전역 통일.
  • --font 토큰이 html · body · UI 요소 · 표 · 코드 모두에 강제 상속된다.
  • 외부 폰트 의존 없음 — Pretendard CDN / Google Fonts / Gulim / Dotum 미사용.
  • 라이트 / 다크 양 테마 모두 S-Core Dream 단일 패밀리로 호환.
TH라이트 / 다크 테마 기준
[data-theme="dark"] 블록에서 동일 키 재정의.
  • Surface · Border · Text · Status · Label · Group 모두 라이트 / 다크 쌍이 정의되어 있다.
  • 다크 테마는 alpha 기반 정의로 시각 위계를 유지한다.
  • 테마 토글은 js/stam.theme.js 단일 진입점이며 문서 / 제품 양쪽에서 재사용한다.
DX문서 연결 기준
Docs Index 중심 문서 그래프.
  • UI Baseline Guide · Component Guide 7종 · CSS Component Token Plan과 정합 가능한 상태이다.
  • 본 Readiness Check 문서는 Token Plan과 실제 CSS 수정 사이의 Pre-Implementation Gate로 동작한다.
  • Staging Deployment QA Result / Preview 문서로 배포 흐름까지 연결된다.
RD문서화 가능 상태
코드 변경 없이 토큰 기준선을 작성할 수 있다.
  • 토큰 카테고리별 현황을 정적 보고로 정리할 수 있다.
  • CSS 수정 전 결정 항목을 Gap List · Implementation Hold 항목으로 분리할 수 있다.
  • Cursor / 후속 작업자에게 허용 / 금지 범위를 본 문서로 인계할 수 있다.
요약 — Strength는 Gate 통과의 충분 조건이 아니다 본 섹션의 강점은 문서화 / 합의 가능 상태까지를 보장한다. 컴포넌트 CSS 구현 진입에는 별도로 Section 06 Gap Summary와 Section 10 Implementation Gate 조건을 모두 충족해야 한다.
03 Target Token File — 기준 파일 기준 고정

아래 한 파일을 기준으로 토큰 정의 상태를 점검한다. 토큰 추가 / 정리는 후속 작업에서 동일 파일을 대상으로 한다.

구분경로
기준 파일 stam/css/stam.tokens.css
관련 문서 STAM UI Baseline Guide · STAM Component Guide · STAM CSS Component Token Plan · Button / Form / Drawer / Dialog / Table / Topbar / Icon Guide
점검 범위 토큰 정의 존재 여부와 의미 기준 적정성만 확인. 토큰 값 변경 · 신규 클래스 정의는 대상 아님.
참고 — 토큰 정의 위치 원칙 모든 디자인 토큰은 stam/css/stam.tokens.css 에만 정의한다. 화면별 CSS(stam.home.css, stam.wbs.css 등)에 임시 px / hex 값을 남발하지 않는다. 라이트 / 다크는 동일 토큰 키를 [data-theme="dark"] 블록에서 함께 정의한다.
04 Token Category Checklist — 12 카테고리 점검 체크

현재 토큰 파일 기준으로 12개 카테고리의 준비 상태를 점검한다. 컬럼 의미: Required For = 어디서 쓰는가 · Current Token Presence = 현재 정의 상태 · Gap = 누락 / 부족 항목 · Recommendation = 보강 방향.

Category Required For Current Token Presence Gap Recommendation Priority
Color Brand · Label · Status · Badge · Group A–G Ready
--brand, --accent, --navy, label 6단계, status 6쌍, group 7색 정의됨. 		컴포넌트 의미(primary / secondary / ghost / danger) 전용 키 없음. 토큰은 충분하나 별칭 부족. 기존 토큰 재사용 + 컴포넌트 별칭(--btn-primary-bg 등) 신설. P1
Typography 전역 폰트 · 본문 / 라벨 / 캡션 위계 Partial
--font 정의됨. body 기준 13px / 1.5만 고정. 		사이즈 / 라인하이트 / 자간 단계 토큰 없음. 화면별 px 직타이핑 다수. UI Baseline Guide 기준 단계명(--fs-body, --fs-label, --lh-tight 등) 후보 정리. P1
Spacing 카드 / 폼 / 테이블 패딩 · gap Partial
--content-pad-x, --grid-gap 등 콘텐츠 레이아웃 토큰만 존재. 		4 / 8 / 12 / 16 / 24 / 32 스케일 토큰 없음. gap, padding이 화면별 직타이핑. 의미 기반(--sp-1 ~ --sp-7 또는 --gap-tight/regular/loose) 후보 정리. P1
Radius 버튼 · 인풋 · 카드 · 칩 · 모달 Ready
--r-sm, --r-md, --r-lg 3단계 정의됨. 		pill(999px) / 모달용 큰 radius 별도 키 없음. 필요 시 --r-pill, --r-xl 1–2개 보강. 현재로도 v1 진입 가능. P2
Shadow 카드 · 팝오버 · Drawer · Dialog Missing
토큰 정의 없음. 화면별 box-shadow 직타이핑. 		elevation 단계 부재. Drawer / Dialog overlay 그림자 기준 없음. 3단계(--sh-card / --sh-pop / --sh-modal) 신설 후보. P0
Border 카드 · 인풋 · 구분선 · focus ring Ready
--bd, --bd-s, --bd-f 정의됨. 		두께 / focus 두께 / 점선 등 패턴 토큰 없음. --bd-f는 색만 정의. focus-visible 두께·색·offset 별칭(--ring, --ring-offset) 후보. P0
Z-index Topbar · Drawer · Dialog · Sticky header Missing
토큰 정의 없음. 화면별 숫자 임의 사용. 		레이어 위계 기준 부재. Drawer / Dialog overlay 순서 불명확. 5단계(--z-base / --z-sticky / --z-topbar / --z-drawer / --z-dialog) 신설. P0
Motion hover · focus · drawer open · dialog open Missing
duration · easing 토큰 정의 없음. 		상태 전이 시간 / 곡선이 화면별로 다름. prefers-reduced-motion 기준 부재. --dur-fast/normal, --ease-out 2–3개 + reduced-motion 가이드. P1
Layout Width 콘텐츠 본문 · Focus 모드 · 사이드바 Ready
--content-max, --content-focus-max, --sw 정의됨. 		반응형 breakpoint 토큰 없음. 모바일 / 태블릿 폭 정의 부재. media query 기준 폭(--bp-md, --bp-lg) 후보. 우선순위 낮음. P1
Button Height Button 사이즈 sm / md / lg Missing
토큰 정의 없음. Component Guide의 사이즈 정의가 코드에 반영되지 않음. 		화면별 button 높이 임의값 우려. 키보드 hit area 기준 부재. 3단계(--btn-h-sm/md/lg) + padding 별칭 신설. P0
Form Height Input · Select · Textarea · Field gap Missing
토큰 정의 없음. Form Guide 기준값이 코드에 반영되지 않음. 		입력 높이가 Button과 정렬되지 않을 위험. 라벨 / help / error 간격 기준 부재. --field-h, --field-pad-x, --field-gap 신설. Button 높이와 정합 유지. P0
Table Density Row · Header height · Cell padding · Sticky Partial
--th-fg, --row-hover만 존재. 		row 높이 / cell padding / density(compact, default, comfortable) 토큰 없음. 2–3단계 density(--row-h-compact/default, --cell-pad-x) 후보. P1
참고 — 우선순위 부여 기준 P0은 컴포넌트 구현 진입 자체를 막는 항목, P1은 구현 직후 반드시 정합화해야 하는 항목, P2는 v1 안정화 이후 보강해도 무방한 항목으로 분류한다.
05 Component Token Needs — 컴포넌트별 토큰 수요 수요

7종 컴포넌트가 v1 구현 시점에 필요한 토큰을 정리한다. 신규 토큰 명은 후보이며, 실제 이름은 후속 작업에서 확정한다.

BTButton
기준: STAM Button Guide v1 · 사이즈 sm / md / lg · variant 4종 · state 4종.
  • height — --btn-h-sm/md/lg
  • padding — --btn-pad-x-sm/md/lg
  • radius — --r-md 재사용
  • font size — Typography 카테고리 토큰 재사용
  • border — --bd-s 재사용 + --btn-bd 별칭
  • variant 색상 — --btn-primary/secondary/ghost/danger-bg/fg/bd
  • state — hover / focus-visible / disabled / loading 별칭
FMForm
기준: STAM Form Guide v1 · Input / Select / Textarea · label · help · error.
  • input height — --field-h (Button md와 정합)
  • field gap — --field-gap
  • label size — Typography 라벨 토큰 재사용
  • help text — --t3 + 캡션 토큰 재사용
  • error color — --fail 재사용 + --field-error-bd 별칭
  • disabled / readonly bg — --bg-sur2 재사용 또는 --field-disabled-bg
DRDrawer
기준: STAM Drawer Guide v1 · 우측 슬라이드 · 폼 / 상세 보기.
  • width — --drawer-w
  • max-width — --drawer-w-max
  • overlay — --overlay-bg (공용)
  • z-index — --z-drawer
  • header / footer height — --drawer-hdr-h, --drawer-ftr-h
  • shadow — --sh-modal 또는 별도 키
DGDialog
기준: STAM Dialog Guide v1 · 확인 / 경고 / 위험.
  • backdrop — --overlay-bg 공용
  • z-index — --z-dialog (Drawer보다 상위)
  • width — --dialog-w-sm/md/lg
  • radius — --r-lg 또는 --r-xl 후보
  • shadow — --sh-modal
  • danger state — --fail 재사용 + Button danger 토큰
TBTable
기준: STAM Table Guide v1 · Header / Body / Sticky · 가로 스크롤 컨테이너.
  • row height — --row-h-compact/default
  • header height — --row-h-header
  • cell padding — --cell-pad-x, --cell-pad-y
  • sticky z-index — --z-sticky
  • density — compact / default / comfortable 옵션
  • horizontal scroll container — --tbl-min-w 별칭 또는 overflow wrapper 기준
TPTopbar
기준: STAM Topbar Guide v1 · 좌측 타이틀 / 브레드크럼 · 우측 액션.
  • height — --topbar-h (기존 토큰 재사용)
  • action button size — Button sm 토큰 재사용
  • gap — Spacing 토큰 재사용
  • breadcrumb / title spacing — --topbar-gap 후보
  • z-index — --z-topbar
ICIcon
기준: STAM Icon Guide v1 · stroke 기반 · currentColor.
  • size scale — --ico-12/16/20/24 후보 (4단계)
  • stroke width — --ico-stroke (기본 1.6 / 1.8)
  • icon button hit area — Button sm 높이와 정합 (--btn-h-sm)
  • alignment gap — 라벨 / 아이콘 사이 gap, Spacing 토큰 재사용
네이밍 원칙 컴포넌트 별칭 토큰은 항상 의미 기반(--btn-primary-bg) 으로 정의하고, 값은 가능하면 atomic 토큰(--brand, --bg-sur 등)을 참조한다. 화면별 CSS에서 atomic 토큰 직접 사용은 허용하나, 컴포넌트 클래스 정의에서는 별칭 토큰 우선 사용한다.
06 Gap Summary — 준비 상태 요약 / Risk 요약

12개 카테고리 점검 결과를 4단계로 요약한다. Missing / Needs Decision 항목은 CSS 수정 진입 전 반드시 해소한다.

Ready
  • Color (atomic / semantic 토큰)
  • Radius (3단계)
  • Border (color 3종)
  • Layout Width (content / focus / sidebar)
Partial
  • Typography (전역 폰트만 정의)
  • Spacing (콘텐츠 토큰만 존재)
  • Table Density (header fg / row hover만 존재)
Missing
  • Shadow elevation 단계
  • Z-index 위계
  • Motion (duration / easing)
  • Button Height 3단계
  • Form Height / Field gap
Needs Decision
  • 컴포넌트 별칭 토큰 네이밍 규칙
  • focus-visible 표현 방식(ring vs border)
  • Drawer / Dialog overlay 색 / alpha
  • Table density 기본값(compact / default)
  • 반응형 breakpoint 토큰화 여부
해소 순서 권장 Missing 항목 → Needs Decision 항목 → Partial 항목 순으로 해소한다. Ready 카테고리는 별도 작업 없이 그대로 사용한다.
07 Priority Recommendation — 우선순위 권고 권고

실제 CSS 수정 시 다루는 순서. P0는 v1 진입 전 필수, P1은 v1 직후, P2는 v1 안정화 이후 보강.

P0v1 진입 전 필수
  • z-index 체계 (--z-sticky / --z-topbar / --z-drawer / --z-dialog)
  • Button / Form / Table 기본 높이 (Button md와 Input 높이 정합)
  • focus-visible 기준 (ring 두께 · 색 · offset)
  • Drawer / Dialog overlay 기준 (배경 alpha · z-index · shadow)
P1v1 직후 정합화
  • Table density 기준 (compact / default / comfortable)
  • Motion 기준 (--dur-fast/normal, --ease-out, reduced-motion)
  • Responsive width 기준 (--bp-md/lg · 가로 스크롤 임계점)
  • Typography 단계 토큰 (사이즈 · 라인하이트 · 자간)
  • Spacing 스케일 토큰 (4 / 8 / 12 / 16 / 24 / 32)
P2v1 안정화 이후 보강
  • Semantic color 확장 (variant 별칭 세분화)
  • Icon size scale 세분화 (12 / 16 / 20 / 24 · stroke 단계)
  • Radius 보강 (pill · 모달 전용)
  • Group 색상 활용 가이드 정리
08 Recommended Next Steps — 단계별 작업 순서 계획

CSS 구현은 한 번에 진행하지 않는다. 아래 5단계를 작은 단위로 끊어 진행하며, 각 단계는 작업 단위 1건에 대응한다.

1
Step 1 — Token Inventory 확정 P0 카테고리(Shadow · Z-index · Border focus · Button H · Form H)부터 신규 토큰 후보를 별도 작업 단위로 확정한다.
2
Step 2 — Component별 Token Mapping 확정 CSS Component Token Plan과 Section 05 Component Token Needs를 결합해 컴포넌트 별칭 토큰 매핑을 확정한다.
3
Step 3 — Button Token Mapping 작성 Button 한 컴포넌트만 대상으로 height · padding · variant · state 매핑 표를 확정한다. 다른 컴포넌트 동시 작업 금지.
4
Step 4 — Button CSS만 제한 구현 stam.tokens.css에 Button 토큰만 추가 → stam.components.css에 Button 클래스만 정의. 페이지별 CSS는 다음 단계에서.
5
Step 5 — QA 후 Form / Dialog / Drawer / Table 순서로 확장 Button QA(라이트·다크 · 1440 / 1600 / 1920 가로 스크롤 · 콘솔 오류 · Staging 200) 통과 후 동일 순서로 컴포넌트 1개씩 확장. 한 번에 1 컴포넌트.
속도 기준 한 컴포넌트 단위는 토큰 → 클래스 → 화면 적용 → QA를 한 묶음으로 처리한다. 여러 컴포넌트 CSS를 동시에 손대지 않는다. 본 문서 갱신 없이 다음 단계로 임의 진입하지 않는다.
09 Do / Don't — 작업 원칙 원칙
Do — 권장
  • 기존 토큰을 먼저 재사용한다.
  • 새 토큰은 의미 기반 이름(--btn-primary-bg)으로 추가한다.
  • 화면별 임의 값 추가 전 공통 토큰을 먼저 확인한다.
  • 라이트 / 다크 토큰 쌍을 함께 고려한다.
  • S-Core Dream 자체 서빙 폰트를 유지한다.
Don't — 금지
  • 화면별 CSS에 임시 px 값을 남발하지 않는다.
  • WBS / Project Overview 전용 토큰을 전역 토큰으로 오염하지 않는다.
  • 버튼 / 폼 / 테이블 높이를 화면마다 다르게 정의하지 않는다.
  • z-index를 임의 숫자로 추가하지 않는다.
  • Pretendard CDN · Google Fonts · Gulim · Dotum 등 외부 폰트를 추가하지 않는다.
10 Implementation Gate — CSS 수정 진입 조건 Gate

아래 7개 조건을 모두 만족할 때 실제 stam/css/** 수정을 시작한다. 하나라도 미충족이면 본 문서 단계로 회귀.

1
Token gap 확인 완료 12개 카테고리 점검 결과가 본 문서에 반영되어 있다.
2
필요한 신규 토큰 목록 확정 P0 항목의 신규 토큰 이름·키·라이트/다크 값이 후속 작업 단위로 확정되어 있다.
3
Button부터 작은 범위로 적용 첫 적용 대상은 Button. 한 컴포넌트 단위로 토큰화 → 클래스 정의 → 화면 적용 순서.
4
변경 파일 범위 제한 stam.tokens.css + stam.components.css 우선. 페이지별 CSS는 후속 단계.
5
라이트 / 다크 QA 신규 / 변경 토큰은 두 테마에서 모두 확인. 누락된 dark 키 없음.
6
가로 스크롤 QA 페이지 레벨 가로 스크롤이 발생하지 않는다. Topbar / Drawer 적용 시 z-index 확인.
7
git diff 검증 변경 파일이 허용 범위 내인지 git diff --stat 으로 확인. 금지 파일 변경 없음.
Gate 미충족 시 7개 조건 중 하나라도 미충족이면 CSS 수정에 진입하지 않고, 본 Readiness Check 문서를 갱신하거나 새 결정 항목을 후속 작업 단위로 분리한다.
11 Cursor Handoff Notes — 후속 작업자 인계 기준 인계

Cursor 또는 다른 작업자에게 CSS 구현 작업을 넘길 때 동일 기준을 유지하기 위한 항목. 본 문서가 합의된 허용 / 금지 범위의 기준이 된다.

Pre-Edit — 작업 진입 전
  • 본 문서를 기준으로 허용 / 금지 범위를 확인한다.
  • 수정 대상은 stam/css/stam.tokens.css · stam/css/stam.components.css 우선.
  • 금지 파일(stam/js/**, stam/pages/**, stam/index.html, stam/assets/**, Firebase 설정 / GitHub Actions / package*.json) 변경 금지.
  • 토큰 추가 / 변경 사유를 PR 또는 커밋 메시지에 명시한다.
  • 한 번의 작업은 한 컴포넌트만 다룬다.
  • git status clean 상태에서 시작한다.
Post-Edit — 작업 직후 QA
  • git diff --stat으로 변경 파일 범위를 확인한다.
  • 라이트 / 다크 양 테마 시각 QA.
  • 1440 / 1600 / 1920 가로폭에서 페이지 레벨 가로 스크롤 없음 확인.
  • Docs Index · Component Guide 링크 정합 확인.
  • Firebase Staging 배포 후 URL 200 · CSS 적용 · 콘솔 오류 없음 확인.
  • S-Core Dream 외 외부 폰트 호출 없음 확인.
Cursor 핸드오프 원칙 본 문서가 정한 범위 외 변경이 필요하면, 코드 수정 전에 본 문서를 먼저 갱신한다. 범위를 임의 확장하지 않는다. 본 문서의 판정(Ready with Notes)은 Section 06 Gap Summary의 Missing / Needs Decision 항목 해소를 전제로 한다.
