STAM Decision Needed Register v1 Decision Register

본 문서는 Button · Form · Layer · Drawer · Dialog · Table 6종 Mapping 문서에 분산된 Decision Needed / Pending / TBD 항목을 단일 Register로 통합 정리한 것이다. 실제 CSS · JS · 제품 화면 구현 전 PM · 디자인 · 개발이 결정해야 할 항목을 우선순위(P0 / P1 / P2)와 Owner · Due Point · Status별로 한 곳에서 추적한다. 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/**를 절대 수정하지 않는다.

Decision Register Implementation Gate PM Review Required Pre-Implementation No CSS Edit

판정 근거 — 6종 Mapping 문서(Button / Form / Layer / Drawer / Dialog / Table)는 각각 “Decision Needed · Suggested · Hold · Candidate · TBD” 항목을 본문 안에 분산 보관하고 있다. 구현 단위가 분리되어 있어, Cursor에 작업을 넘길 때 PM이 결정한 항목과 결정하지 않은 항목이 섞여 그대로 CSS · JS에 박힐 위험이 있다. 본 Register는 그 위험을 차단하기 위해 “이 항목이 결정되지 않으면 어떤 컴포넌트 CSS도 시작할 수 없다”는 Gate 기준표 역할을 한다.

01 Overview — 문서 목적 · Mapping 문서와의 관계 목적

STAM은 현재 7종 Component Guide v1과 6종 Token Mapping v1(Button · Form · Layer · Drawer · Dialog · Table)이 완료된 상태이다. 각 Mapping 문서는 “토큰 후보 · Suggested Value · 사용처”까지는 정리했지만, 일부 항목은 “PM 결정 필요”, “디자인 결정 필요”, “v2로 이연”, “구현 단위에서 확정” 상태로 남아 있다. 본 Register는 그 결정 항목만 6종 문서에서 뽑아 하나의 우선순위표로 정리한다. 기획자 · PM은 “지금 무엇을 결정해야 구현이 시작되는가”, 디자이너는 “시각/상호작용 의사결정이 필요한 항목”, 개발자는 “구현 전에 막혀 있는 Gate 항목”을 본 문서 하나로 확인할 수 있어야 한다.

이 문서가 다루는 것
  • Button / Form / Layer / Drawer / Dialog / Table 6종 Mapping의 미결정 항목 통합
  • P0 Critical · P1 Before Component CSS · P2 Later 우선순위 분류
  • Decision Item별 Options · Recommended Option · Impact · Owner · Due Point · Status
  • Recommended Decision Set — PM 승인 후보안
  • Implementation Gate — 어떤 결정 없이 어떤 구현이 시작 가능한가
  • Cursor Handoff Notes — 구현 위임 시 따라야 할 결정 게이트
  • Decision Status Legend — Pending · Recommended · Approved · Deferred · Blocked · Needs PM / Design / Dev
이 문서가 다루지 않는 것
  • stam/css/** 토큰 추가 / 값 변경 / 컴포넌트 클래스 작성
  • stam/js/** 컴포넌트 동작 / 라우팅 / 상태 관리 구현
  • stam/pages/**, stam/index.html, stam/assets/** 변경
  • 각 Mapping 문서 본문 수정 — Register는 발췌·통합만, 원본은 변경하지 않는다
  • Firebase / GitHub Actions / package*.json / _ds/** / uploads/** 변경
  • Production / live 배포 의사결정 — 별도 운영 문서 책임
  • 이미 Approved 항목의 재논의 — 결정된 토큰 · 값은 본 문서에서 다시 흔들지 않는다

주의 — Register는 결정이 아니다 본 문서의 “Recommended Option”은 디자이너 관점에서의 제안이며, PM이 Approved 처리하지 않은 항목은 어떤 항목이라도 CSS / JS 구현에 반영하지 않는다. 결정이 끝난 항목은 각 Mapping 문서 본문에 반영하는 것이 정식 절차이며, 본 Register는 항상 “결정 전 단계”만을 정리한다.
금지 — Register만 보고 코드 변경 금지 본 문서는 등록부이지 명세가 아니다. P0 Pending 항목이 하나라도 남아 있는 영역의 컴포넌트 CSS · JS는 시작하지 않는다. Cursor 핸드오프 시 “Register P0 모두 Approved”가 전제 조건이며, 그렇지 않은 상태로 구현을 시작하면 본 Register의 의미가 사라진다.

11 Implementation Gate — 구현 착수 조건 게이트

컴포넌트 CSS / JS 구현은 아래 6개 조건이 모두 충족되었을 때만 시작한다. 하나라도 미충족이면 시작하지 않는다.

01
P0 Decision 전부 Approved S03의 10개 항목 Status가 모두 Approved인가
02
6종 Mapping 문서 존재 및 최신 Button / Form / Layer / Drawer / Dialog / Table Mapping v1 모두 v1 유지
03
Docs Index 링크 정상 Docs Index에 Decision Register v1 카드 1개 추가 · 깨진 링크 없음
04
수정 허용 파일 범위 확정 예: stam/css/stam.tokens.css, stam/css/stam.components.css만 등 — 작업마다 명시
05
첫 구현 단위 1개만 선택 Button CSS 또는 Drawer Layout 중 하나만 — 동시 진행 금지
06
Git diff / Staging QA 기준 확정 staging 배포 후 URL / 콘솔 / light · dark / 가로 스크롤 QA 패스 기준 명시
금지 — Gate 미통과 상태 구현 P0 Pending 항목이 1건이라도 남아 있는 영역의 컴포넌트 CSS / JS 구현은 시작하지 않는다. 예: Layer P0(Toast vs Loading)이 Pending이면 Drawer / Dialog / Toast 어느 컴포넌트도 z-index를 확정할 수 없으므로 구현 보류.
12 Cursor Handoff Notes — 구현 위임 기준 핸드오프

Cursor에게 구현을 위임할 때 항상 본 Register와 함께 다음 7개 규칙을 명시한다.

Cursor 작업 규칙 — 진입 조건
  • Decision Register에서 해당 영역 P0 Pending 항목이 있으면 구현 금지
  • 첫 구현은 Button CSS 또는 Drawer Layout 중 하나만 선택
  • Button / Form / Drawer / Dialog / Table을 동시 구현 금지
  • CSS / JS 수정 허용 파일을 작업마다 다시 지정 — 누적 명시 금지
Cursor 작업 규칙 — 배포 / QA
  • 배포 전 git diff로 변경 파일·범위 확인
  • Staging 배포 후 URL / 콘솔 / light·dark / 가로 스크롤 QA 4종 패스
  • Production / live 배포 금지 — staging 까지만
  • React dist 배포 금지 — 정적 HTML 기준 유지
핸드오프 메시지 템플릿 “이번 작업: {component}의 CSS. 수정 허용: {file list}. 금지: 그 외 모든 파일. 사전 확인: STAM Decision Needed Register v1의 {area} 결정 P0 모두 Approved 여부. Staging URL / 콘솔 / light·dark / 가로 스크롤 QA 패스 후 보고.”
