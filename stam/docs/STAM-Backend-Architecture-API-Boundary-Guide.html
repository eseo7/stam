01 문서 개요 — Overview 아키텍처
문서명 Backend Architecture & API Boundary Guide
기준 루트 stam/
버전 v1.0
작성일 2026-06-08

목적

  • Frontend · Service Layer · Repository/API Client · API Server · DB · Storage · Batch · Export · Audit Log 각 레이어의 책임 경계를 사전에 명확히 정의한다.
  • 단기 Firebase 사용 범위와 장기 상용 SaaS 전환 기준을 사전 확정하여 재설계 없이 점진적 이관이 가능한 구조를 유지한다.
  • 정적 Preview · UX 검증 단계에서도 Firebase 종속적 구조가 되지 않도록 Interface 기반 추상화 기준을 제시한다.
Firebase 비종속 원칙 Firebase를 사용하더라도 Firebase SDK를 Frontend 또는 Service Layer에서 직접 호출하지 않는다. 모든 Firebase 접근은 Repository / API Client Layer에서만 허용하며, 상위 레이어는 추상화된 Interface만 참조한다. 이 원칙이 지켜져야 장기 SaaS 전환 시 상위 레이어 코드 수정 없이 Repository 구현체만 교체하여 이관을 완료할 수 있다.
이 문서의 적용 범위 이 문서는 설계 기준 문서이며, 실제 API 서버 · DB · Firebase Functions 구현 코드를 포함하지 않는다. Claude Design(정적 HTML)으로 생성된 아키텍처 가이드이므로 실행 가능한 런타임 코드는 제공하지 않는다.

03 레이어 책임 정의 — Layer Responsibilities 기준 고정
레이어 책임 허용 금지
Frontend UI 렌더링, 사용자 인터랙션 처리, 폼 입력 수집, 화면 상태 관리 Service Layer 호출, 로컬 UI 상태, 라우팅 Firebase SDK 직접 호출, DB 직접 접근, 비즈니스 로직 포함, 보안 규칙 우회
Service Layer 비즈니스 로직, 입력 Validation, 데이터 변환, 에러 핸들링, 권한 흐름 제어 Repository 호출, 도메인 모델 조작, 복합 트랜잭션 조율 Firebase SDK 직접 호출, UI 컴포넌트 직접 참조, HTTP 직접 호출
Repository / API Client 데이터 소스 추상화, Interface 구현, 요청/응답 변환, 캐싱 전략 Firebase SDK 호출 (단기), HTTP API 호출 (장기), 데이터 직렬화 비즈니스 로직 포함, UI 상태 참조, 복수 도메인 조합 처리
API Server 인증/인가, 비즈니스 규칙 서버 검증, DB CRUD, Export/Batch 실행, Audit Log 기록 권한 검사, DB 트랜잭션, 외부 API 연동, 결제 처리 Frontend 렌더링 로직 포함, 클라이언트 상태 관리, 과도한 응답 가공
DB / Storage 데이터 영속화, 파일 저장, 쿼리 실행 인덱스 최적화, 스토리지 접근 정책 설정 비즈니스 로직 (DB 트리거 최소화), 클라이언트 직접 접근 (보안 규칙 외)
Batch / Export Worker 대용량 비동기 처리, 비동기 Export(PDF/Excel), 스케줄 작업, 알림 발송 서버 사이드 렌더링, 외부 서비스 Push, 큐 소비 실시간 응답 요구 작업, 동기 Frontend 요청 직접 처리
Audit Log 모든 변경/삭제/권한 변경 이력 불변 기록, 규정 준수 타임스탬프 · 사용자 · 변경 내용(diff) 기록 로그 수정·삭제, 클라이언트 단에서 직접 기록, 개인정보 원문 저장
04 단기 전략 — Firebase Current

사용 목적

  • 정적 Preview · UX 검증 · 샘플 데이터 테스트 단계에서의 비용 최소화 인프라
  • 인증 · 저장소 · 호스팅을 최소 설정으로 빠르게 구성하여 기능 시나리오 검증
  • 운영 SaaS 전환 전 UI/UX 완성도 확보 플랫폼

사용 허용 범위

AuthFirebase Auth
이메일 / Google 로그인, UID 기반 사용자 식별. 토큰 발급은 Firebase 위임.
DBFirestore
프로젝트 · 태스크 · 댓글 샘플 데이터 저장. 스키마는 장기 RDB 구조와 동일하게 설계.
StoreStorage
첨부파일 · 이미지 업로드. 경로 규칙은 장기 S3/GCS 구조와 동일하게 유지.
HostHosting
정적 HTML Preview 배포. CDN 기반. 커스텀 도메인 연결 가능.
FnFunctions (후보)
경량 서버 로직이 필요한 경우 한정. 복잡한 비즈니스 로직은 포함 금지.

비용 최소화 기준

  • Firestore 읽기 횟수 최소화 — 불필요한 실시간 Listen 지양, 필요 시 once() 사용
  • Storage 용량 관리 — 테스트용 파일 주기적 정리, 대용량 파일 업로드 금지
  • Functions 콜드 스타트 최소화 — 빈번한 조회는 Repository 계층의 캐시 · 읽기 최적화 · API 캐시 전략으로 처리한다. Frontend에서 Firestore를 직접 호출하는 방식으로 대체하지 않는다.
  • Spark 플랜 한도 내 운영 — 초과 예상 시 즉시 이관 계획 수립
Firebase 단계 구현 원칙 Firestore Security Rules를 비즈니스 로직 대용으로 사용하지 않는다. 권한 체계 · 승인 흐름 · 결제 · Audit Log는 Frontend 또는 Firestore Rules 단독으로 구현하지 않는다. Firebase 단계에서 서버 책임 영역의 검증이 필요한 경우, Cloud Functions를 통해 서버 책임 영역으로 분리한다. 장기적으로는 API Server + PostgreSQL 구조로 전환 가능성을 유지한다.
05 장기 전략 — SaaS Architecture Planned

아키텍처 컴포넌트

컴포넌트 역할 기술 후보 책임 범위
API Server 모든 비즈니스 로직의 서버 진입점 Node.js / Go · Cloud Run · ECS 인증 검증, 권한 확인, 데이터 CRUD, 응답 직렬화
PostgreSQL 정형 데이터 영속화 · 관계형 쿼리 PostgreSQL · Supabase · Cloud SQL 프로젝트 · 멤버 · 태스크 · 권한 · 댓글 · 감사로그 스키마
Storage 파일 · 첨부 · Export 결과물 저장 AWS S3 / GCS · CDN 연동 업로드 경로 정책, 서명 URL 접근 제어, 라이프사이클 관리
Batch Worker 스케줄 기반 대용량 처리 Cloud Scheduler · Celery · Bull 통계 집계, 알림 발송, 만료 처리, 데이터 정리
Export Worker 비동기 문서 생성 (PDF / Excel) Puppeteer · ExcelJS · Headless Chrome Export 요청 큐잉, 렌더링, Storage 업로드, 완료 알림
Audit Log 변경 이력 불변 기록 · 규정 준수 PostgreSQL append-only 테이블 · BigQuery 모든 CUD 작업 자동 기록, 사용자 · 타임스탬프 · diff 저장
Payment 구독 · 플랜 과금 처리 Stripe · 토스페이먼츠 플랜 업그레이드, 결제 실패 처리, 영수증 발행, Webhook 수신
서버 전담 기능 원칙 권한 검사 · 승인 흐름 · 감사 로그 · Export · 결제는 반드시 API Server에서 처리한다. 클라이언트(Frontend)는 UI 표현만 담당하며, 이 기능들의 최종 판단을 내리지 않는다.
06 API Boundary Rules — 경계 규칙 기준 고정

클라이언트 vs 서버 책임 경계

Frontend / Client 담당
UI 상태 관리 (로딩 · 에러 · 포커스)
폼 입력 수집 및 1차 형식 Validation
화면 라우팅 및 전환 처리
API 응답을 화면에 맞게 표현 가공
낙관적 업데이트 (UX 즉시 반영)
로컬 캐시 및 페이지네이션 상태
API Server 담당
인증 토큰 검증 (JWT / Firebase ID Token)
권한 · 역할 기반 접근 제어 (RBAC)
비즈니스 Validation (중복 · 규칙 · 한도)
승인 흐름 상태 전환 결정
Audit Log 자동 기록
Export / Batch 작업 큐 등록
결제 처리 및 플랜 상태 관리

경계 위반 금지 패턴

금지 패턴 설명 올바른 방향
Client-side 권한 판단 Frontend에서 역할 값만 읽어 특정 기능 표시/숨김으로 접근 제어를 대체 API Server에서 모든 요청마다 권한 검사. UI는 표현만 담당.
Service Layer Firebase 직접 접근 Service Layer에서 Firebase SDK를 직접 import하여 Firestore 문서 저장 Repository Layer를 통해서만 Firestore에 접근.
Client-side Audit Log Frontend에서 직접 감사 로그 컬렉션에 기록 API Server 모든 CUD 핸들러에서 미들웨어로 자동 기록.
결제 로직 Client 노출 결제 키 · 플랜 결정 로직을 Frontend JS에서 처리 결제는 Server Webhook + API Server에서만 처리.
Firestore 직접 Export Frontend에서 Firestore 데이터를 직접 읽어 CSV/Excel 생성 Export Worker를 통해 비동기 생성 후 서명된 URL로 다운로드.

08 이관 체크리스트 — Migration Checklist Planned
Firebase → SaaS API Server 이관 시 아래 항목을 순서대로 점검한다. Repository Layer만 교체되면 상위 레이어 수정 없이 전환이 완료되어야 한다.

이관 전 준비 사항

Repository 추상화 검증
모든 Firebase 호출이 Repository Layer에만 존재하는지 확인
Service Layer에 Firebase import 없음 확인
Frontend에 Firebase import 없음 확인
Repository Interface가 명시적으로 정의되어 있음
데이터 스키마 검증
Firestore 컬렉션 구조가 PostgreSQL 테이블 스키마와 매핑 가능
관계형 ID 참조 방식이 Firestore 단계에서도 동일하게 사용됨
타임스탬프 포맷 통일 (ISO 8601)
파일 Storage 경로 규칙이 S3/GCS와 동일
