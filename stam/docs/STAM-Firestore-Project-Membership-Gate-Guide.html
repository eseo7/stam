문서 성격 — Decision / Rules Design Baseline 본 문서는 PR #298의 canonical 기준서이며, Firestore layout 최종 선택과 active project member read gate Rules의 설계 기준을 정한다. 실제 로그인 UI / SDK / CRUD 구현은 여전히 이번 PR의 범위가 아니다.

01 문서 목적 — Why This Gate Decision
  • PR #295 · #297에서 정의한 논리 구조를 실제 Firestore Rules 접근 판정으로 잇는다.
  • Firestore layout을 project subcollections로 확정한다.
  • active project member만 project-scoped read가 가능하도록 read gate만 연다.
  • write는 이번 PR에서 열지 않는다. role 기반 write는 PR #306 hardening으로 이월한다.
  • Auth UI / Firebase SDK / CRUD 구현은 이번 PR에서 다루지 않는다.
이 문서의 효력 본 문서의 결정 문구는 PR #306 role hardening과 이후 Auth bootstrap PR의 canonical 기준으로 사용된다. 변경이 필요하면 별도 결정 PR로 본 문서를 먼저 갱신한 뒤 구현을 진행한다.

06 Firestore layout 최종 선택 — Project Subcollections Final
최종 결정 1차 제품화 기준 layout은 project subcollections로 확정한다. top-level requirements / wbsItems / screenSpecs 구조는 채택하지 않는다.

확정 구조

경로목적
users/{uid}Firebase Auth 사용자 프로필
tenants/{tenantId}후속 조직 확장 골격 (1차 UI 없음)

projects/{projectId}/artifactLinks/{linkId}산출물 연결

채택 이유

  • projectId가 경로에 들어가므로 데이터 격리가 명확하다.
  • Rules에서 project-scoped read/write 검증이 단순하다.
  • active member 확인을 projects/{projectId}/members/{uid}로 고정할 수 있다.
  • Viewer write block / role hardening을 후속 PR #306에서 공통 함수로 확장하기 쉽다.

top-level 미채택 이유

  • document마다 projectId 필드가 정확히 있어야 하고 Rules에서 매번 확인 필요.
  • membership gate가 get(projectMembers/…) 형태로 복잡해질 수 있음.
  • collection별 룰이 병렬로 늘어남.
07 projects/{projectId}/members/{uid} 기준 — Canonical Membership Contract

문서 ID 기준

  • 문서 ID = Firebase Auth UID (request.auth.uid).
  • 따라서 “로그인한 사용자가 이 프로젝트의 active member인지”는 get(members/{uid})로 결정된다.
  • 이 경로가 존재하지 않거나 status !== "active"면 Rules는 read를 거부한다.

필수 필드 기준 (application/data contract)

필드타입내용
tenantIdstring소속 tenant
projectIdstring소속 projectId (경로와 일치)
userIdstringFirebase Auth uid와 동일
emailstringGoogle auth email 원문
emailNormalizedstringlower + trim, app/data contract
rolestringowner · admin · editor · viewer
statusstringpending · active · removed
createdAt / createdBytimestamp / string기본 audit
updatedAt / updatedBytimestamp / string기본 audit
Rules는 이 필드 중 아래 조합만 사용 · 존재: exists(memberPath) · status == "active" · userId == request.auth.uid (경로와 이중 매칭) · projectId == projectId (경로와 이중 매칭) · email / emailNormalized는 Rules 판정에 사용하지 않는다.
08 email / emailNormalized의 역할 Contract
  • email — Google auth 원문 저장 (표시 · 감사용).
  • emailNormalized — 소문자 + trim, 초대 · 등록 · 검색 · bootstrap 기준.
  • 이 두 필드는 member 문서에 보관된다.
  • 실제 Rules gate는 UID 기반이며, email 문자열을 Rules에서 직접 비교하지 않는다.
필수 문구 PR #298은 projectMembers.emailNormalized 구조를 보존하지만, Rules의 실제 접근 판정은 email 문자열 직접 비교가 아니라 Firebase Auth UID와 active member doc을 기준으로 한다. email 기반 최초 연결 / 초대 / 등록 흐름은 다음 Auth bootstrap PR에서 구현한다.

emailNormalized 정규화 기준

  • 소문자 + trim().
  • Gmail의 dot(.) 처리 / plus alias 자동 병합은 1차에서 하지 않는다.
  • 고급 alias 정책은 후속 보안 / 계정 정책에서 별도 결정.
09 Rules helper 설계 — Gate Composition Rules

이번 PR에 도입된 helper

helper정의
signedIn()request.auth != null
uid()request.auth.uid
memberPath(projectId)/databases/$(database)/documents/projects/$(projectId)/members/$(uid())
hasMemberDoc(projectId)signedIn() && exists(memberPath(projectId))
memberData(projectId)get(memberPath(projectId)).data
isActiveProjectMember(projectId)hasMemberDoc && userId == uid() && projectId == projectId && status == "active"
canReadProject(projectId)isActiveProjectMember(projectId)
왜 $(database) 완전 경로인가 Firestore Rules에서 다른 문서를 참조할 때는 /databases/$(database)/documents/… 형태의 완전 경로가 필요하다. 변수 바인딩은 반드시 $(variable) 형태로 표기한다.
단락 평가 (short-circuit) hasMemberDoc()이 false이면 memberData()는 호출되지 않는다 (Firestore Rules v2 && 단락 평가). 존재하지 않는 문서를 get()하지 않도록 반드시 exists()가 먼저 참인지 확인한 뒤 필드 비교로 진입한다.

11 write 비개방 원칙 — All Writes Still Closed Deny All Writes
이번 PR에서 write는 열지 않는다. · allow create, update, delete: if false; 를 모든 project-scoped 경로에 명시. · users/{uid}도 write=false. · tenants/{tenantId}도 read/write=false. · catch-all match /{document=**}도 allow read, write: if false로 유지.
  • Viewer write block은 “write가 열려 있을 때” 의미가 있는 규칙이므로, 이번 PR에서는 별도 표현이 필요 없다.
  • role별 write 허용 (Owner / Admin / Editor)와 상태 변경 조건은 PR #306에서 layer로 추가한다.
  • 임시 개발용 open 규칙은 절대 삽입하지 않는다 (staging에도 두지 않는다).

13 Rules are not filters — Client Query Discipline Principle
필수 원칙 Firestore Rules는 필터가 아니다. 사용자가 접근 권한 없는 projectId 전체를 query 하면 Rules는 “자동으로 걸러주지 않고” 전체를 거부한다. 따라서 클라이언트는 반드시 접근 가능한 projectId 기준으로 query해야 한다.
  • 단일 문서 read: doc('projects/{projectId}').get() — Rules는 canReadProject(projectId)만 확인.
  • subcollection query: collection('projects/{projectId}/requirements').where(...) — 마찬가지로 canReadProject(projectId)만 확인.
  • “내 프로젝트 목록”을 얻으려면 별도 index / query 전략이 필요하다 — 이는 후속 Auth bootstrap PR에서 결정한다.

collectionGroup query — 이번 PR 기준

PR #298 기준 · PR #298에서는 collectionGroup query를 제품 기준으로 열지 않는다. · 1차 기준은 projects/{projectId}/... 하위 subcollection 직접 query다. · collectionGroup query가 필요하면 후속 PR에서 별도 Rules 패턴, 인덱스, query discipline을 함께 확정한다. · 무제한 collectionGroup은 금지한다.
  • 1차 클라이언트 query 규율: 반드시 projectId가 경로에 명시된 subcollection query만 사용한다.
  • “모든 프로젝트를 가로지르는 조회”가 필요해 보이더라도, 이번 PR에서는 각 projectId별로 반복 query하거나 별도 UI PR 결정을 기다린다.
  • 후속 결정 항목: collectionGroup이 열릴 대상 collection, membership 검증을 표현할 Rules 패턴 (get()/exists() 경로), 필요한 복합 인덱스, 무제한 조회 방지를 위한 query 제약.
14 PR #306 role hardening handoff — What Comes Next #306 Input

Role hardening · write 개방

  • Owner / Admin / Editor / Viewer role helper (isRole('owner') 등) 도입.
  • projects/{projectId}/requirements · wbsItems · screenSpecs 등에 allow create, update 개방 (Editor 이상).
  • Viewer write 명시적 차단.
  • 권한표 OR 항목 최종 확정 (PR #295 §12 · 확정 전 더 제한적인 쪽 유지).
  • collection별 세부 hardening: 필드 단위 write 제한, 상태 변경 조건, 담당자 기반 조건 등.
  • UI 액션 제어와 Rules를 일관되게 매칭.

members read / list scope 재검토

  • projects/{projectId}/members/{memberUid} read / list scope 재검토.
  • 본인 member doc read와 전체 member list read를 분리할지 결정한다.
  • emailNormalized는 Rules 판정용이 아니며, UI 노출 대상도 별도 정책으로 제한한다.
  • removed / pending member 가시성은 owner / admin 중심으로 후속 결정한다.
주의 본 PR (#298)에서는 위 항목을 구현하지 않는다. role hardening은 별도 결정 PR과 UI PR을 거쳐 #306에서 landing 한다. 현재 match /members/{memberUid}는 active member면 get/list 모두 허용하는 최소 read gate만 열려 있으며, 본인 doc vs 전체 list 분리, removed / pending 가시성 정책은 #306에서 확정한다.
