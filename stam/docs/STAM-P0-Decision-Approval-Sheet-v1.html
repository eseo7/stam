STAM P0 Decision Approval Sheet v1 Approval Candidate

본 문서는 STAM Decision Needed Register v1의 P0 Critical Decisions 10개 항목을 실제 CSS · JS 구현 전 PM 승인 절차로 사용할 수 있도록 승인 후보(Approval Candidate) 형태로 정리한 문서이다. 각 항목별 Recommended · Approved Candidate · Implementation Note를 명시하고, 본 문서가 PM Sign-off를 받기 전에는 어떤 컴포넌트 CSS · JS 구현도 시작하지 않는다. 본 문서로는 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/**를 절대 수정하지 않는다.

Approval Candidate P0 Gate Implementation Ready After Sign-off No CSS Edit No JS Edit

판정 근거 — Decision Needed Register v1은 6종 Mapping 문서의 미결정 항목을 통합·분류한 등록부이다. 그중 P0 Critical 10건은 “결정되지 않으면 컴포넌트 CSS · JS 구현이 시작될 수 없는” 항목이며, 각각 디자이너 Recommended 후보가 부여되어 있다. 본 Approval Sheet는 그 10건의 Recommended 후보를 “PM이 결재할 단일 후보안(Approved Candidate)”으로 정리하고, 승인 직후 첫 구현 단위로 진입 가능한 게이트 조건과 구현 순서를 명시한다.

01 Overview — 문서 목적 · Decision Register와의 관계 목적

STAM은 현재 6종 Token Mapping v1(Button · Form · Layer · Drawer · Dialog · Table)과 Decision Needed Register v1까지 완료된 상태이다. Decision Register v1은 미결정 항목 52건을 P0 / P1 / P2로 분류했고, 그중 P0 Critical 10건이 구현 게이트로 지정되어 있다. 본 Approval Sheet는 그 10건을 PM 승인을 위한 단일 시트로 압축한다. 각 항목별 Recommended Option은 디자이너 제안이며, PM이 “Approved Candidate”란을 검토·결재하면 본 문서는 그대로 구현 게이트 통과 증빙으로 사용된다.

이 문서가 다루는 것
  • Decision Register v1의 P0 Critical 10건만 발췌 · 승인 후보 정리
  • P0 항목별 Approved Candidate · Reason · Implementation Impact · Owner · Status
  • Implementation Gate Result — 승인 시 어떤 구현 단위가 열리는가
  • Recommended First Implementation — Button → Drawer → Form → … 순서 권장
  • Cursor Handoff Rules — 승인 후 구현 위임 시 따라야 할 규칙
  • Risk Notes — 승인안 채택 시 남는 위험과 후속 결정 의존성
이 문서가 다루지 않는 것
  • stam/css/** 토큰 추가 / 값 변경 / 컴포넌트 클래스 작성
  • stam/js/** 컴포넌트 동작 / 라우팅 / 상태 관리 구현
  • stam/pages/**, stam/index.html, stam/assets/** 변경
  • P1 / P2 항목 — 본 문서 범위 아님(Decision Register에서 별도 추적)
  • Decision Register / Mapping 문서 본문 수정 — Approval Sheet는 발췌·요약만
  • Firebase / GitHub Actions / package*.json / _ds/** / uploads/** 변경
  • Production / live 배포 의사결정 — 별도 운영 문서 책임
선행 / 참조 문서 · Decision Needed Register v1 — P0 / P1 / P2 미결정 항목 등록부
· CSS Token Readiness Check — 12 카테고리 토큰 준비상태
· Token Inventory v1 — P0 / P1 / P2 토큰 후보
· Button Token Mapping v1 · Form · Layer · Drawer · Dialog · Table Mapping v1
· STAM P0 Decision Approval Sheet v1 (현재 문서)
주의 — Approval Sheet는 결재안이지 결재 자체가 아니다 본 문서의 “Approved Candidate”는 디자이너가 PM에게 제출하는 단일 후보안이다. PM Sign-off가 끝나지 않은 항목은 어떤 항목이라도 CSS · JS 구현에 반영하지 않는다. 결재 완료 후에는 Decision Register v1과 각 Mapping 문서 본문의 해당 항목 Status를 Approved로 갱신해야 한다.
금지 — Approval 전 구현 본 문서가 Approval Candidate 상태인 한 stam/css/**, stam/js/**, stam/pages/**, stam/index.html, stam/assets/** 어디에도 손대지 않는다. P0 10건 중 단 1건이라도 Pending 상태로 남아 있으면 컴포넌트 CSS · JS 구현은 시작하지 않는다.

05 Implementation Gate Result — 결재 후 구현 진입 조건 게이트

본 Approval Sheet의 P0 10건이 모두 Approved 처리되었을 때 비로소 컴포넌트 CSS · JS 구현이 가능해진다. 단, 진입은 단일 컴포넌트 단위로만 허용한다.

01
P0 10건 전부 Approved 본 시트 S03의 Status 컬럼이 모두 Approved가 되었는가
02
Decision Register v1 동기화 Decision Register v1의 동일 항목 Status가 Approved로 갱신되었는가
03
6종 Mapping 문서 본문 반영 Button / Form / Layer / Drawer / Dialog / Table Mapping v1 본문에 결재 결과가 반영되었는가
04
수정 허용 파일 범위 확정 예: stam/css/stam.tokens.css, stam/css/stam.components.css만 등 — 작업마다 명시
05
첫 구현 단위 1개만 선택 Button CSS 또는 Drawer Layout 중 하나만 — 동시 진행 금지
06
Form / Select / Date Picker는 공통 init 확정 후 진입 P0-05 initStamFormControls(rootElement) 시그니처가 Dev에서 확정된 다음에만 진입
OPT승인 후 첫 구현 후보 둘 중 하나만 선택
Option A — Button CSS 제한 구현
  • 영향 범위가 작고 회귀 위험이 낮음
  • variant · size · state · icon-only · focus ring만 적용
  • 수정 허용 — stam/css/stam.tokens.css, stam/css/stam.components.css 중 일부
  • JS 변경 금지 · Drawer / Dialog / Form 영향 없음
Option B — Drawer Layout 제한 구현
  • P0-02 (1280 breakpoint) · P0-03 (전체 보기 버튼 위치) 결과를 1차 반영
  • Layout · 폭 · 헤더/푸터 구조만 적용
  • Form / Select / Date Picker 구현은 본 단계에 포함하지 않음
  • JS는 close · 전체 보기 진입만 — 라우터 의존 금지(P0-04)
금지 — Gate 미통과 상태 구현 P0 Pending 항목이 1건이라도 남아 있는 한 컴포넌트 CSS · JS 구현은 시작하지 않는다. 특히 Layer P0(P0-01)이 Pending이면 Drawer · Dialog · Toast 어느 컴포넌트도 z-index를 확정할 수 없으므로 모든 구현 보류.

Button을 먼저 두는 이유
  • variant / size / state 토큰 외 다른 컴포넌트에 미치는 영향이 거의 없음
  • 회귀 발생 시 staging에서 빠르게 롤백 가능
  • 다른 컴포넌트(Drawer / Form / Dialog)의 액션 영역에 공통 기반이 됨
Drawer · Form · Select · Date Picker가 Button 다음인 이유
  • 모두 JS와 layer(z-index) 정책이 함께 걸려 있어 단일 CSS 작업이 어려움

  • Select · Date Picker는 custom UI 강제 정책으로 인해 단독 CSS 작업으로 처리하지 않음
Table을 마지막에 두는 이유
  • 전체 게시판 / 모든 화면에 영향 · 회귀 시 파급 효과가 가장 큼
  • P0-07(page-level h-scroll 금지) · P0-08(컬럼 설정 저장) 확정 후 진입 권장
  • Backend API Boundary와 함께 검토 — 컬럼 schema 책임 분리 필요
동시 구현 금지
  • Button / Drawer / Form / Dialog / Table 중 1개만 진행
  • 각 단계 staging QA 통과 후 다음 단계로 진입
  • 위 순서를 임의 변경 시 본 시트와 Decision Register v1을 함께 갱신

08 Risk Notes — 승인안 채택 시 남는 위험 위험

본 Approval Candidate를 그대로 채택했을 때 남는 위험과 후속 결정 의존성을 정리한다. 결재 시 PM이 함께 인지해야 한다.

Layer — Global Loading / Toast 우선순위
  • 전역 차단 정책과 알림 UX가 충돌할 수 있음(예: 저장 중 Toast 노출 보류 정책 필요)
  • Loading이 너무 오래 잡히면 사용자가 “멈춤”으로 인지 — UX 정책으로 보완 필요
  • 관련 항목 — P0-01 / P0-09 / Layer Mapping LY-01
Drawer — Full Detail URL routing
  • v1 내부 상태 전환만 채택 → 공유 URL · 새로고침 복원 미지원
  • v2 도입 시 권한 가드 · SSR 호환 · 라우터 전반 점검 필요
  • 관련 항목 — P0-04 / Drawer Mapping
Form — Select / Date Picker custom UI
  • custom UI 강제는 JS 공통화 의존 — 단독 CSS 작업으로 처리 금지
  • 접근성 · 키보드 인터랙션 회귀 위험 — staging QA 항목으로 명시
  • 관련 항목 — P0-05 / P0-06 / Form Mapping FM-02 · FM-03
Table — 컬럼 설정 저장
  • v1은 게시판 schema 단일 기준 → 사용자 개인화 요구는 v2까지 보류
  • 데이터 구조 · API Boundary와 연결되므로 v1은 문서 기준만 확정
  • 관련 항목 — P0-08 / Table Mapping TB-02 · TB-03 / Backend Architecture Guide
후속 결정 의존성 본 P0 Approval이 통과되어도 P1 / P2 항목은 별도 결재가 필요하다. 특히 Button BT-02 / Form FM-04 · FM-07 / Table TB-04 등은 Mapping 문서 본문과 함께 후속 결재로 처리한다.
